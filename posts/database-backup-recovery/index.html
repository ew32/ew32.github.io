데이터베이스 백업과 복구 전략 - 풀백업, 증분, PITR, 재해 복구

작성일 2026-04-21 Edited on 2026-05-19 In Backend , Database
데이터베이스 백업의 종류와 PITR(시점 복구), 재해 복구(DR) 전략을 비교하고 MySQL, PostgreSQL의 실제 백업 방법과 운영 노하우를 정리합니다.

왜 백업이 중요한가?

데이터베이스 장애는 하드웨어 고장, 소프트웨어 버그, 인적 실수, 보안 사고(랜섬웨어 등) 등 어떤 형태로든 일어납니다. 백업이 없거나 복구 절차가 검증되지 않은 상태에서 장애가 발생하면 데이터 손실은 곧 비즈니스의 치명적 손실로 이어집니다. “백업이 있는 시스템이 아니라 복구가 검증된 시스템 이 안전한 시스템”이라는 격언이 있을 정도입니다.

RPO와 RTO

백업·복구 전략의 두 가지 핵심 지표입니다.

  • RPO(Recovery Point Objective): 허용 가능한 데이터 손실량 (예: 5분)
  • RTO(Recovery Time Objective): 허용 가능한 복구 시간 (예: 1시간)

RPO가 작을수록 백업 주기가 짧아야 하고, RTO가 작을수록 복구 절차가 빠르고 자동화되어야 합니다.

백업의 종류

1. 논리적 백업(Logical Backup)

SQL 문 또는 데이터 export 형태로 저장. 이식성이 좋지만 큰 데이터에는 느립니다.

2. 물리적 백업(Physical Backup)

데이터 파일을 그대로 복사. 빠르고 대용량에 적합합니다.

3. 풀 백업(Full Backup)

전체 데이터를 백업. 가장 단순하지만 시간과 공간을 많이 사용합니다.

4. 증분 백업(Incremental Backup)

지난 백업 이후 변경된 부분만 저장. 빠르지만 복구 시 풀 + 증분들을 모두 적용해야 합니다.

5. 차등 백업(Differential Backup)

마지막 풀 백업 이후의 모든 변경을 저장. 증분보다 복구가 단순하지만 크기가 점점 커집니다.

PITR (Point-In-Time Recovery)

특정 시점으로 데이터베이스를 되돌리는 기법입니다. 풀 백업 + 트랜잭션 로그(MySQL의 binlog, PostgreSQL의 WAL)를 함께 사용합니다.

3-2-1 백업 규칙

업계에서 널리 사용되는 백업 규칙입니다.

  • 3 개의 데이터 복사본 유지
  • 2 개의 서로 다른 저장 매체에 보관
  • 1 개는 오프사이트(원격지)에 보관

클라우드 환경에서는 다음과 같이 응용할 수 있습니다.

  • 운영 DB
  • 같은 리전의 자동 백업(스냅샷)
  • 다른 리전의 백업(S3 cross-region replication)

재해 복구(DR) 전략

Cold Standby

백업만 보관. 장애 시 새 인프라를 구축하고 복원. RTO가 길지만 비용이 가장 낮음.

Warm Standby

복제본 서버를 유지하되 적극 사용하지 않음. 복제 지연 정도의 데이터 손실 발생 가능.

Hot Standby (Active-Active 포함)

실시간 동기화된 복제본. 페일오버가 거의 즉시. 비용이 가장 높음.

전략 RPO RTO 비용
Cold 시간~일 시간 낮음
Warm 분 분 중간
Hot 초~0 초 높음

복구 훈련 (Disaster Recovery Drill)

백업의 가치는 복구 가능 여부에서 나옵니다. 정기적인 복구 훈련을 통해 다음을 검증해야 합니다.

  1. 백업 파일이 손상 없이 복원 가능한가?
  2. PITR로 임의 시점 복구가 동작하는가?
  3. 복구에 걸리는 실제 시간이 RTO 이내인가?
  4. 절차서가 최신 상태인가?
  5. 담당자가 부재중일 때도 절차가 수행 가능한가?

클라우드 환경의 백업

  • AWS RDS: 자동 백업(점진적), 스냅샷, PITR(최대 35일)
  • AWS Aurora: continuous backup, 5분 단위 PITR
  • GCP Cloud SQL: 자동/수동 백업, PITR
  • Azure Database: 지역 중복 백업, long-term retention

매니지드 서비스도 자체 백업 외에 별도 외부 백업 을 유지하는 것이 안전합니다. 클라우드 계정 자체가 손상되는 경우를 대비해야 하기 때문입니다.

백업 시 흔한 실수

  1. 백업만 하고 복구를 검증하지 않음
  2. 암호화 누락으로 백업 파일이 보안 위협이 됨
  3. 백업 파일을 같은 서버에 저장
  4. 메타데이터(스키마, 사용자, 권한) 누락
  5. 재해 시 백업 접근 권한이 없는 경우 (계정 잠김 등)
  6. 백업 모니터링 없음 (실패 알림 미설정)

마무리

데이터베이스 백업은 단순히 명령어를 실행하는 일이 아니라, RPO·RTO를 정의하고 정책을 수립하며 정기적으로 복구를 검증하는 종합적인 운영 활동입니다. 3-2-1 규칙을 기억하고 자동화·암호화·DR 훈련까지 갖춘다면, 어떤 장애가 발생해도 비즈니스를 빠르게 복구할 수 있는 안정적인 시스템을 만들 수 있습니다.
