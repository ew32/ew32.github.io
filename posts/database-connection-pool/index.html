데이터베이스 커넥션 풀(HikariCP) 이해와 튜닝

작성일 2026-05-11 Edited on 2026-06-15 In Backend , Database
커넥션 풀이 필요한 이유와 HikariCP의 동작 원리, 핵심 설정 및 튜닝 포인트를 살펴봅니다.

커넥션은 왜 비싼가

데이터베이스 연결을 새로 만들 때마다 다음 작업이 일어납니다.

  1. TCP 3-way handshake
  2. (TLS 사용 시) TLS 핸드셰이크
  3. 인증, 권한 확인
  4. 세션 변수 초기화
  5. 메모리 할당

매 요청마다 이 과정을 거치면 응답 시간이 수십 ms 이상 늘어나고, DB 서버 측에도 부하가 큽니다. 커넥션 풀(Connection Pool) 은 미리 만들어둔 커넥션을 재사용해 이 비용을 제거합니다.

커넥션 풀의 동작

close()는 실제로 커넥션을 끊는 것이 아니라 풀에 반환합니다. 따라서 try-with-resources나 finally에서 반드시 반환해야 합니다.

HikariCP의 특징

HikariCP는 Spring Boot 2.x부터 기본 커넥션 풀입니다.

  • 매우 가벼운 코드 베이스
  • ConcurrentBag 자료구조로 락 경합 최소화
  • 빠른 가져오기·반환 경로
  • 정확한 메트릭(HikariPool.MBean)

핵심 설정

속성 의미 권장 기본값
maximumPoolSize 최대 커넥션 수 CPU/워크로드별 산정
minimumIdle 최소 유휴 커넥션 maximumPoolSize와 동일(권장)
connectionTimeout 풀에서 커넥션을 기다리는 최대 시간 30,000 ms
idleTimeout 유휴 커넥션 유지 시간 600,000 ms
maxLifetime 커넥션 최대 수명 1,800,000 ms
validationTimeout 커넥션 유효성 검사 시간 5,000 ms
leakDetectionThreshold 누수 감지 시간 운영에선 5,000 ms 정도

풀 크기 산정

HikariCP 위키의 공식이 가장 자주 인용됩니다.

대부분의 OLTP 워크로드에서는 10~30 정도면 충분합니다. 풀이 클수록 좋다는 것은 흔한 오해이며, DB 측 컨텍스트 스위칭과 락 경합으로 오히려 성능이 떨어집니다.

작은 풀이 더 빠른 이유

  • DB는 동시에 처리할 수 있는 작업 수가 제한적(CPU·디스크)
  • 풀이 크면 DB에 대기 큐가 쌓여 응답 시간이 오히려 증가
  • 작은 풀 + 빠른 회전이 더 높은 처리량을 만듭니다

maxLifetime과 DB 타임아웃 정렬

DB의 wait_timeout(MySQL) 또는 방화벽의 idle TCP 타임아웃보다 maxLifetime이 짧아야 합니다. 그렇지 않으면 끊어진 좀비 커넥션을 풀이 모르고 내주게 됩니다.

누수 감지

leakDetectionThreshold를 설정하면 정해진 시간 동안 반환되지 않은 커넥션을 로그로 알려줍니다.

스택 트레이스를 통해 close() 누락 지점을 찾아낼 수 있습니다.

메트릭 모니터링

HikariCP는 Micrometer와 통합되어 다음 지표를 노출합니다.

  • hikaricp.connections.active
  • hikaricp.connections.idle
  • hikaricp.connections.pending (대기 중 스레드)
  • hikaricp.connections.usage (사용 시간 분포)
  • hikaricp.connections.acquire (가져오는 데 걸린 시간)

pending이 0보다 자주 크다면 풀 크기 부족, acquire 시간이 길다면 DB 응답 지연을 의심합니다.

흔한 함정

  • 트랜잭션을 매우 길게 잡음: 외부 API 호출을 트랜잭션 안에 두지 마세요.
  • @Transactional이 붙은 메서드에서 동기 외부 호출: 풀이 빠르게 고갈됩니다.
  • N+1 쿼리: 커넥션 사용 시간을 증가시킵니다.
  • 장기 실행 배치와 OLTP가 같은 풀을 공유: 별도 데이터소스로 분리.
  • PreparedStatement 캐시 미사용: cachePrepStmts=true를 켜두세요.

다중 데이터소스

읽기 복제본을 활용할 때 풀을 분리합니다.

@Transactional(readOnly = true)와 라우팅 데이터소스를 조합하면 부하 분산이 가능합니다.

튜닝 절차 요약

  1. 처음엔 작게 시작 (10~20)
  2. 부하 테스트로 pending, acquire 관찰
  3. DB CPU/IO를 함께 모니터링
  4. DB가 한계라면 풀을 늘리지 말고 DB·쿼리부터 최적화
  5. maxLifetime을 DB·네트워크 타임아웃 미만으로 설정

마무리

커넥션 풀은 단순한 설정 같지만 시스템의 응답 시간과 안정성에 직접 영향을 줍니다. HikariCP의 기본값은 합리적이지만, 풀 크기와 수명 설정은 환경에 맞게 조정해야 합니다. “풀을 크게 잡으면 빨라진다”는 직관 대신, DB가 실제로 소화할 수 있는 만큼만 보내고 나머지는 큐로 흘러가게 두는 것이 가장 좋은 튜닝입니다.
