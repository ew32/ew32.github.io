Spring Boot 테스트 전략

작성일 2026-05-09 Edited on 2026-05-19 In Backend , Java
단위 테스트, 슬라이스 테스트, 통합 테스트를 조합한 Spring Boot 테스트 전략과 도구 활용법을 정리합니다.

좋은 테스트가 필요한 이유

테스트는 회귀(regression) 방지와 설계 피드백이라는 두 가지 가치를 동시에 제공합니다. Spring Boot는 풍부한 테스트 도구를 기본 제공하지만, 잘못 사용하면 느리고 깨지기 쉬운 테스트 더미가 됩니다. 이 글은 단위, 슬라이스, 통합 테스트를 어떻게 조합해야 하는지 그 전략을 정리합니다.

테스트 피라미드

레벨 도구 속도 비중
Unit JUnit, Mockito 매우 빠름 많음
Slice @WebMvcTest, @DataJpaTest 빠름 중간
Integration @SpringBootTest 느림 적음
E2E RestAssured, Selenium 매우 느림 매우 적음

피라미드의 아래쪽일수록 많고 빠르게, 위쪽일수록 적고 신중하게 작성합니다.

단위 테스트

도메인 객체와 서비스 로직은 Spring 컨테이너 없이 순수 JUnit으로 테스트합니다.

Mockito를 사용해 협력 객체를 가짜로 만들 수 있습니다.

슬라이스 테스트

특정 레이어만 로딩해 빠르게 검증합니다.

@WebMvcTest

컨트롤러 계층만 구성하고 Service는 @MockBean으로 대체합니다.

@DataJpaTest

JPA 레이어만 구성하며, 내장 DB(H2)와 트랜잭션 롤백을 자동 적용합니다.

@DataJpaTest는 실제 운영 DB와 다를 수 있어, Testcontainers로 실제 DB 이미지를 사용하는 것이 더 안전합니다.

Testcontainers 통합

운영 환경과 동일한 DB로 테스트해 SQL 호환성 문제를 잡을 수 있습니다.

통합 테스트

@SpringBootTest는 전체 컨텍스트를 띄우는 가장 무거운 테스트입니다. 자주 사용하기보다는 핵심 시나리오에 집중해야 합니다.

테스트 데이터 관리

  • Builder 패턴으로 테스트 픽스처 생성 간소화
  • 트랜잭션 롤백을 활용해 데이터 격리 (@Transactional)
  • 공통 데이터는 @Sql 또는 TestEntityManager로 셋업
  • Random 데이터는 Instancio나 EasyRandom 활용

모범 사례

  • 테스트 이름은 의도가 드러나게: should_return_404_when_order_not_found
  • Given-When-Then 구조로 가독성 확보
  • 테스트는 독립적: 실행 순서에 의존하지 않기
  • 외부 시스템은 모킹: WireMock, MockServer 사용
  • 느린 테스트는 분리: @Tag("slow") 후 CI 파이프라인에서 분기
  • 컨텍스트 캐시 활용: @MockBean을 남용하면 컨텍스트가 자주 재생성되어 느려짐

마무리

좋은 테스트 전략은 단순히 커버리지 숫자를 채우는 것이 아니라, 빠른 피드백과 신뢰할 수 있는 회귀 방지를 균형 있게 제공하는 것입니다. 단위 테스트로 도메인 규칙을 단단히 잡고, 슬라이스 테스트로 레이어 간 계약을 검증하며, 통합 테스트로 핵심 시나리오를 보장하는 피라미드 구조를 유지해야 합니다.
