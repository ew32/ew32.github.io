모놀리식과 MSA 아키텍처 비교

작성일 2026-05-14 Edited on 2026-05-12 In Backend , Infrastructure
모놀리식과 마이크로서비스 아키텍처의 구조, 장단점, 선택 기준을 비교합니다.

두 아키텍처의 개념

모놀리식 아키텍처(Monolithic Architecture) 는 모든 기능을 하나의 코드베이스와 배포 단위로 묶은 전통적인 구조입니다. 반면 MSA(Microservices Architecture) 는 도메인 단위로 분리된 여러 독립 서비스의 집합으로 시스템을 구성합니다.

두 방식은 단순한 코드 구성 차이가 아니라, 배포·확장·조직·데이터 관리 전반에 걸친 트레이드오프를 동반합니다.

항목별 비교

항목 모놀리식 MSA
배포 단위 단일 서비스별 독립
데이터베이스 공유 DB 서비스별 DB

운영 복잡도 낮음 높음
기술 다양성 제한적 자유로움

코드 관점의 차이

모놀리식 - 모듈 직접 호출

MSA - 네트워크 호출

네트워크 경계가 생기는 순간 타임아웃, 재시도, 회로 차단기, 분산 트레이싱 같은 새로운 고민이 추가됩니다.

모놀리식이 적합한 경우

  • 초기 스타트업·MVP 단계
  • 도메인이 아직 불확실한 시스템
  • 팀 규모가 작은 경우 (5~10명 이하)
  • 운영·DevOps 인프라가 부족한 조직

MSA가 적합한 경우

  • 수십 명 이상의 다중 팀 구성
  • 트래픽이 서비스별로 크게 차이 나는 경우
  • 잦은 부분 배포가 필요한 도메인
  • 장애 격리와 고가용성이 중요한 비즈니스

전환 전략

모놀리식에서 MSA로 전환할 때는 Strangler Fig 패턴을 자주 사용합니다.

  1. 기존 모놀리식 앞에 API Gateway를 둠
  2. 도메인 단위로 일부 기능을 신규 서비스로 분리
  3. 게이트웨이가 해당 경로를 새 서비스로 라우팅
  4. 점진적으로 모놀리식 코드 제거

마무리

MSA는 모놀리식의 상위 호환이 아니라 다른 트레이드오프를 가진 선택지입니다. 조직 규모, 도메인 성숙도, 운영 역량을 종합적으로 따져 결정해야 하며, 많은 경우 잘 설계된 모듈러 모놀리식으로 시작하는 것이 현명한 출발점입니다.
