CORS 이해하기 - 동작 원리부터 설정까지

작성일 2026-05-13 Edited on 2026-06-05 In Network , HTTP
브라우저의 동일 출처 정책과 CORS의 동작 원리, Preflight 요청과 안전한 서버 설정 방법을 정리합니다.

CORS는 왜 존재하는가

브라우저는 기본적으로 **동일 출처 정책(Same-Origin Policy, SOP)**을 따릅니다. 한 출처에서 로드된 스크립트는 다른 출처의 응답을 읽을 수 없습니다. 이 단순한 규칙이 없다면, 사용자가 악성 사이트에 방문했을 때 그 사이트의 JavaScript가 사용자의 은행 계정 API를 조용히 호출하고 응답을 가져갈 수 있게 됩니다.

하지만 현대 웹은 SPA, 마이크로서비스, API 서버 분리 등 출처를 넘나드는 통신이 일상입니다. 이를 안전하게 허용하기 위한 표준화된 우회로가 바로 **CORS(Cross-Origin Resource Sharing)**입니다.

“출처(Origin)”의 정의

출처는 scheme + host + port의 조합입니다. 세 가지 중 하나라도 다르면 다른 출처입니다.

CORS의 핵심: 서버가 허락해야 한다

CORS는 서버 측에서 응답 헤더로 허락을 표현하면, 브라우저가 그에 따라 응답을 JavaScript에 노출할지 결정하는 메커니즘입니다. 핵심 헤더는 다음과 같습니다.

Simple Request vs Preflight

브라우저는 요청을 두 종류로 구분합니다.

Simple Request

다음 조건을 모두 만족하면 Simple입니다.

  • 메서드: GET, HEAD, POST
  • 헤더: CORS-safelisted 헤더만 사용 (Accept, Content-Language, Content-Type 일부 등)
  • Content-Type: application/x-www-form-urlencoded, multipart/form-data, text/plain 중 하나

이 경우 브라우저는 즉시 요청을 보내고, 응답 헤더만 검사합니다.

Preflight Request

Simple 조건을 벗어나면(예: Content-Type: application/json, 커스텀 헤더 사용) 브라우저는 본 요청 전에 OPTIONS Preflight를 먼저 보냅니다.

서버가 허용 헤더로 응답해야 본 요청이 진행됩니다.

Max-Age로 Preflight 결과를 캐시해 성능 저하를 줄일 수 있습니다.

인증 정보 전송: withCredentials

쿠키나 HTTP 인증 정보를 함께 보내려면 클라이언트는 credentials: 'include'를, 서버는 다음을 응답해야 합니다.

주의: Allow-Credentials: true일 때는 Allow-Origin에 *를 사용할 수 없습니다. 반드시 명시적 출처여야 합니다.

자주 하는 실수

  • Access-Control-Allow-Origin: * + Allow-Credentials: true 동시 사용 (브라우저가 거부)
  • 요청 Origin을 그대로 반사하면서 화이트리스트 검증 누락 → 사실상 *와 동일
  • Preflight 응답에 Authorization 헤더가 누락되어 401 발생
  • 리버스 프록시(NGINX)와 애플리케이션이 헤더를 중복 추가해 브라우저가 거부

CORS는 보안인가?

흔한 오해 중 하나가 “CORS를 풀면 보안이 약해진다”는 표현입니다. 정확히 말하면 CORS는 SOP라는 보안 정책을 우회하기 위한 메커니즘이며, 서버 자체의 인증/인가를 대체하지 않습니다. CORS가 막아주는 것은 “다른 사이트의 JS가 내 응답을 읽는 것”이지, “다른 사이트가 내 API에 요청을 보내는 것”이 아닙니다(요청 자체는 가능).

따라서 CSRF 방어, 인증 토큰 검증 등은 CORS와 별개로 반드시 수행해야 합니다.

Best Practices

  • Allow-Origin은 명시적 화이트리스트로 관리
  • Allow-Credentials: true는 꼭 필요할 때만
  • Preflight 비용을 줄이려면 Max-Age 적절히 설정
  • 와일드카드 서브도메인은 정규식으로 엄격히 검증
  • CORS와 인증/인가는 별개의 관심사로 다룰 것

마무리

CORS는 “막는 도구”가 아니라 “열어주는 도구”입니다. SOP라는 강력한 기본 정책 위에서, 서버가 명시적으로 신뢰하는 출처에게만 응답 접근을 허락하는 합의된 통로죠. 헤더의 의미와 Preflight 동작 원리를 정확히 이해하면, 디버깅 시간이 절반으로 줄어들 것입니다.
