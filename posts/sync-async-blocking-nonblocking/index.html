동기/비동기 vs 블로킹/논블로킹 차이

작성일 2026-05-24 Edited on 2026-06-05 In System , OS
헷갈리기 쉬운 동기/비동기와 블로킹/논블로킹 개념을 명확히 구분해 설명합니다.

들어가며

동기(Synchronous), 비동기(Asynchronous), 블로킹(Blocking), 논블로킹(Non-Blocking). 이 네 가지 용어는 자주 함께 등장하지만, 자주 혼동됩니다. 두 쌍은 서로 다른 관점에서 프로세스의 동작을 설명합니다. 이 글에서는 각각의 정의와 조합 가능한 네 가지 동작 모델을 정리합니다.

두 가지 관점 구분하기

  • 동기/비동기: 작업의 완료를 누가 신경 쓰는지에 대한 관점입니다. 호출자가 결과를 직접 확인하면 동기, 호출자가 신경 쓰지 않고 다른 일을 하면 비동기입니다.
  • 블로킹/논블로킹: 호출된 함수가 제어권을 즉시 반환하는지에 대한 관점입니다. 즉시 반환하면 논블로킹, 작업이 끝날 때까지 반환하지 않으면 블로킹입니다.

즉, 동기/비동기는 작업의 순서, 블로킹/논블로킹은 제어권의 이동에 관한 개념입니다.

블로킹 vs 논블로킹

블로킹은 함수를 호출한 후 작업이 끝날 때까지 호출자가 대기합니다. 논블로킹은 함수가 즉시 반환되며, 호출자는 다른 작업을 계속할 수 있습니다.

동기 vs 비동기

동기는 호출자가 결과를 기다리거나 직접 확인합니다. 비동기는 호출자가 결과에 관심을 두지 않고, 작업이 끝나면 콜백이나 이벤트, Future 같은 방식으로 알림을 받습니다.

네 가지 조합

조합 설명 예시
동기 + 블로킹 호출자가 대기하며 결과를 직접 받음 일반 함수 호출, read()
동기 + 논블로킹 호출자가 직접 결과를 폴링(polling)함 O_NONBLOCK + while 루프
비동기 + 블로킹 결과를 신경 안 쓰지만 호출은 멈춤 (드문 경우) select() 일부 사용 패턴
비동기 + 논블로킹 호출 즉시 반환되고 콜백/이벤트로 알림 epoll, Node.js, async/await

I/O 멀티플렉싱

논블로킹과 비동기 모델은 실제로 운영체제의 I/O 멀티플렉싱 기법을 활용합니다.

  • select: 가장 오래된 방식, FD 수에 제한
  • poll: select의 제한 해결, 여전히 O(N) 순회
  • epoll(Linux), kqueue(BSD), IOCP(Windows): 이벤트 기반, 대규모 동시 연결에 적합

Node.js, Nginx, Netty 같은 고성능 서버는 이런 메커니즘 위에 동작합니다.

언제 무엇을 써야 할까?

  • 단순 스크립트, CPU 작업: 동기 + 블로킹이 코드가 가장 단순합니다.
  • 소수의 I/O 작업: 멀티스레드 + 블로킹도 충분합니다.
  • 수천~수만 개의 동시 연결: 비동기 + 논블로킹이 필요합니다.
  • GUI 애플리케이션: 메인 스레드 멈춤 방지를 위해 비동기 사용이 권장됩니다.

마무리

동기/비동기와 블로킹/논블로킹은 비슷해 보이지만 서로 다른 차원의 개념입니다. 작업의 순서와 결과 처리 방식을 본다면 동기/비동기, 제어권이 즉시 반환되는지를 본다면 블로킹/논블로킹입니다. 두 축을 분리해 생각하면 비동기 프로그래밍의 다양한 패턴(콜백, Promise, async/await, 이벤트 루프)이 자연스럽게 이해됩니다.
