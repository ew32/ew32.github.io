Spring Data JPA 심화 활용

작성일 2026-05-07 Edited on 2026-06-15 In Backend , Database
페치 조인, 영속성 컨텍스트, Querydsl 등 Spring Data JPA를 효율적으로 사용하는 심화 기법을 다룹니다.

Spring Data JPA 다시 보기

Spring Data JPA는 JPA 표준 위에 리포지토리 추상화를 더해 반복적인 DAO 코드를 줄여주는 모듈입니다. 단순히 JpaRepository를 상속하는 것에 그치면 ORM의 진짜 힘을 활용하지 못합니다. 이 글에서는 영속성 컨텍스트의 동작부터 N+1 문제, 동적 쿼리, 성능 튜닝까지 실전 관점에서 정리합니다.

영속성 컨텍스트와 1차 캐시

JPA의 핵심은 영속성 컨텍스트(Persistence Context) 입니다. 트랜잭션 단위로 살아있는 엔티티 캐시이자 변경 감지(dirty checking)의 근거지입니다.

save()를 호출하지 않아도 트랜잭션 커밋 시점에 변경 사항이 SQL로 반영됩니다. 이는 영속성 컨텍스트가 스냅샷을 비교해 변경된 필드만 UPDATE 하기 때문입니다.

N+1 문제와 페치 조인

연관 엔티티를 LAZY 로딩하면 단일 조회 후 컬렉션을 사용할 때 추가 쿼리가 발생합니다. 사용자 100명을 조회하고 각 사용자의 주문을 사용하면 1 + 100 쿼리가 실행됩니다.

해결책 중 가장 직관적인 방법은 페치 조인(Fetch Join) 입니다.

@EntityGraph도 같은 효과를 제공합니다.

페치 조인 사용 시 주의점은 다음과 같습니다.

  • 컬렉션 페치 조인은 결과 행이 중복되므로 distinct 또는 Set 사용
  • 페이지네이션과 함께 사용 시 메모리에서 페이징하므로 위험
  • 둘 이상의 컬렉션 동시 페치 조인은 MultipleBagFetchException 발생

페이징과 성능

대용량 데이터에서는 Pageable로 페이지네이션을 구현합니다.

다만 count 쿼리가 매번 실행되어 부담이 될 수 있으므로, 다음과 같이 분리할 수 있습니다.

매우 큰 테이블에서는 cursor 기반 페이징(키셋 페이지네이션)이 OFFSET 방식보다 성능이 좋습니다.

동적 쿼리: Specification과 Querydsl

조건이 가변적인 검색 화면에서는 동적 쿼리가 필요합니다. JPA Criteria 기반 Specification은 가능하지만 가독성이 떨어집니다. 실무에서는 Querydsl을 선호합니다.

null을 반환하면 해당 조건이 자동으로 무시되어 매우 깔끔합니다.

벌크 연산과 영속성 컨텍스트

JPA의 UPDATE/DELETE는 영속성 컨텍스트를 거치지 않고 DB에 직접 적용됩니다.

clearAutomatically = true로 1차 캐시를 비우지 않으면 이후 조회한 엔티티가 옛 상태일 수 있습니다.

성능 튜닝 체크리스트

항목 권장 사항
연관관계 기본 페치 전부 LAZY
컬렉션 조회 페치 조인 또는 @BatchSize
쓰기 지연 hibernate.jdbc.batch_size 설정
읽기 전용 @Transactional(readOnly = true)
DTO 직접 매핑 JPQL constructor expression 또는 Querydsl projection

DTO 프로젝션

엔티티를 그대로 응답하면 의도치 않은 lazy 로딩이나 순환 참조 문제가 발생합니다. DTO로 직접 매핑하세요.

모범 사례

  • 양방향 연관관계에서는 연관관계 편의 메서드 작성
  • @OneToMany의 cascade는 정말 필요한 경우에만
  • 식별자 전략은 가능한 한 IDENTITY 또는 SEQUENCE를 사용
  • 로깅에 p6spy를 적용해 실제 SQL과 파라미터 확인

마무리

Spring Data JPA는 편리하지만 ORM의 동작을 이해하지 못하면 성능 문제와 마주칠 가능성이 큽니다. 영속성 컨텍스트, 페치 전략, 동적 쿼리, 벌크 연산을 정확히 이해하고 실제 SQL을 항상 확인하는 습관을 들이면 강력한 도구가 됩니다.
