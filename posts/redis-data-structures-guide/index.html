Redis 자료구조 완벽 정리 - String, List, Hash, Set, ZSet, Stream

작성일 2026-04-20 Edited on 2026-06-05 In Backend , Database
Redis의 핵심 자료구조인 String, List, Hash, Set, Sorted Set, Stream의 특징과 명령어, 실무 활용 사례를 코드 예제와 함께 정리합니다.

Redis와 자료구조

Redis는 단순한 key-value 캐시가 아니라 “자료구조 서버(data structures server)” 라고 불릴 정도로 다양한 자료구조를 제공합니다. 메모리에서 동작하기 때문에 마이크로초 단위 응답이 가능하며, 자료구조마다 최적화된 명령어가 있어 복잡한 비즈니스 로직을 간결하게 표현할 수 있습니다.

1. String

Redis의 가장 기본적인 자료구조이며 모든 키의 값은 최대 512MB까지 저장 가능합니다. 문자열뿐 아니라 숫자(원자적 증감 지원)와 직렬화된 객체(JSON 등)도 다룰 수 있습니다.

활용: 캐시, 카운터, 분산 락, 세션 저장

2. List

양방향 연결 리스트로, 양 끝에 O(1)로 요소를 추가/제거할 수 있습니다. 큐와 스택으로 활용 가능합니다.

활용: 작업 큐, 최근 N개 로그, 메시지 버퍼

3. Hash

필드와 값으로 구성된 맵. 객체를 표현할 때 적합합니다. 작은 hash는 ziplist로 저장되어 메모리 효율이 매우 높습니다.

활용: 사용자 프로필, 객체 필드 단위 업데이트, 메모리 효율적인 데이터 저장

4. Set

중복 없는 비순서 집합. 합집합·교집합·차집합 연산을 지원합니다.

활용: 태그, 친구 관계, 좋아요 목록, 추천 시스템, 중복 제거

5. Sorted Set (ZSet)

score를 기준으로 정렬된 집합. 랭킹 시스템의 핵심입니다. 추가/삭제는 O(log N), 범위 조회는 O(log N + M)으로 매우 효율적입니다.

활용: 리더보드, 우선순위 큐, 시간순 정렬 피드, 지연 작업 큐

지연 작업 큐 패턴

6. Stream

Redis 5.0에 추가된 로그 자료구조. Kafka와 유사한 append-only 로그를 제공하며 consumer group을 지원합니다.

활용: 이벤트 소싱, 메시지 브로커, 감사 로그, 비동기 작업 디스패치

7. 추가 자료구조

Bitmap

String 위에 비트 단위 연산을 수행합니다. 메모리 효율적인 출석 체크, on/off 플래그에 유용합니다.

HyperLogLog

근사적 cardinality 추정. 12KB로 수십억 개의 유니크 카운트를 약 0.81% 오차로 측정할 수 있습니다.

Geo

지리적 좌표 저장 및 거리 계산.

Bloom Filter (Redis Stack)

존재 여부를 확률적으로 판단. 캐시 미스 방지에 유용합니다.

키 네이밍 컨벤션

콜론(:)으로 네임스페이스를 구분하면 키 검색과 관리가 수월합니다.

운영 시 주의사항

  • TTL 설정: 메모리 누수를 막기 위해 적절한 만료 시간 부여
  • KEYS * 금지: 운영 환경에서는 SCAN 사용
  • 큰 키 주의: 단일 명령으로 큰 데이터를 다루면 다른 명령을 블로킹할 수 있음
  • maxmemory-policy 설정: allkeys-lru 등으로 명시적 정책 지정
  • persistence 선택: RDB(스냅샷) + AOF(append-only) 조합 권장
