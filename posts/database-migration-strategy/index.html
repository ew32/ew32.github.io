데이터베이스 마이그레이션 전략

작성일 2026-05-11 Edited on 2026-05-26 In Backend , Database
스키마 변경부터 데이터 이관까지 안전하게 수행하기 위한 데이터베이스 마이그레이션 전략과 도구를 정리합니다.

데이터베이스 마이그레이션의 두 얼굴

데이터베이스 마이그레이션은 크게 두 가지를 의미합니다.

  1. 스키마 마이그레이션: 테이블 구조, 인덱스, 제약 등을 버전 관리하며 변경
  2. 데이터 마이그레이션: 데이터를 다른 DB/스토리지로 이관

두 작업 모두 “되돌리기 어렵다”는 공통점이 있어 신중한 전략이 필요합니다.

스키마 마이그레이션 도구

Flyway

SQL 파일 기반으로 단순합니다. 파일명 규칙을 따르면 자동으로 적용됩니다.

Spring Boot에서는 application.yml에 의존성만 추가하면 부팅 시 자동 적용됩니다.

Liquibase

XML/YAML/JSON으로 변경 사항을 선언합니다. DB 종류별 추상화가 좋고 롤백 지원이 강합니다.

항목 Flyway Liquibase
변경 정의 SQL XML/YAML/SQL
학습 곡선 낮음 보통
롤백 유료 또는 수동 기본 지원
DB 추상화 낮음 높음

무중단 스키마 변경

운영 중인 시스템에서 단순한 ALTER TABLE이 테이블 잠금을 유발해 장애가 될 수 있습니다. 다음 원칙을 따릅니다.

Expand-Migrate-Contract 패턴

  1. Expand: 새 컬럼/테이블을 추가 (기존 코드와 호환)
  2. Migrate: 새 구조에 데이터 채우기, 코드 점진 전환
  3. Contract: 더 이상 쓰이지 않는 옛 구조 제거

예: 컬럼 이름을 name → full_name으로 변경

온라인 DDL

  • MySQL: ALGORITHM=INPLACE, LOCK=NONE
  • PostgreSQL: CREATE INDEX CONCURRENTLY
  • 대용량 테이블엔 pt-online-schema-change, gh-ost 등 외부 도구 활용

데이터 이관 전략

Big Bang

서비스를 잠시 중단하고 한 번에 옮깁니다.

  • 장점: 단순
  • 단점: 다운타임, 롤백 어려움
  • 적합: 소규모 데이터, 야간 점검 가능 시스템

Trickle (점진적 이관)

새 시스템에 점진적으로 트래픽을 옮기며 데이터를 동기화합니다.

  • 장점: 다운타임 최소
  • 단점: 양방향 동기화 복잡, 데이터 충돌 처리 필요

Dual Write

애플리케이션이 두 DB에 동시에 씁니다.

분산 트랜잭션이 어려우므로 Outbox 패턴 이나 CDC(Change Data Capture) 를 조합하는 것이 안전합니다.

CDC 기반

Debezium 같은 도구로 원본 DB의 변경 로그(WAL/binlog)를 캡처해 새 DB로 스트리밍합니다.

  • 거의 실시간 동기화
  • 원본 부하 적음
  • 초기 스냅샷 + 증분 동기화의 정석

검증과 롤백

이관 후 반드시 검증 절차를 포함해야 합니다.

  • 행 개수 비교: SELECT COUNT(*) 양쪽 비교
  • 체크섬 비교: 페이지 단위 해시
  • 비즈니스 무결성: 핵심 보고서 결과 비교
  • 다크 런치: 새 DB로 읽기만 분기해 실제 트래픽으로 검증

롤백 계획은 작성하는 것을 넘어 연습해야 합니다.

흔한 실수

  • 마이그레이션을 코드 리뷰 없이 수행 → 운영 DB에서 사고
  • NOT NULL 컬럼을 기본값 없이 추가 → 테이블 락 또는 실패
  • 외래키를 한 번에 추가 → 검증 비용으로 장기 락
  • 인덱스를 운영 시간에 생성 → CONCURRENTLY 옵션 누락
  • 롤백 스크립트 미작성 → 사고 복구 지연

체크리스트

  • 변경 스크립트가 버전 관리되는가
  • 운영과 동일 구조의 스테이지에서 리허설했는가
  • 백업과 PITR(Point-In-Time Recovery)이 준비되었는가
  • 무중단이 필요한 변경인지 분류했는가
  • 모니터링 지표(락 대기, 슬로우 쿼리)가 준비되었는가

마무리

데이터베이스 마이그레이션은 코드 변경보다 훨씬 위험합니다. Flyway/Liquibase로 변경을 버전 관리하고, Expand-Migrate-Contract 같은 패턴으로 무중단을 보장하며, 대규모 데이터 이관에는 CDC와 검증 절차를 반드시 포함하세요. 잘 설계된 마이그레이션은 보이지 않게 지나가지만, 부실한 마이그레이션은 새벽 호출의 원인이 됩니다.
