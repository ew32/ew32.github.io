JavaScript 모듈 시스템 - CommonJS와 ESM

작성일 2026-05-18 In Frontend , JavaScript
JavaScript의 대표적인 모듈 시스템인 CommonJS와 ES Modules의 차이와 사용법을 정리합니다.

모듈 시스템이 필요한 이유

초창기 JavaScript는 모든 스크립트가 전역 스코프를 공유했습니다. 이는 변수 충돌, 의존성 관리의 어려움, 코드 재사용 곤란 등 많은 문제를 일으켰습니다. 이를 해결하기 위해 모듈 시스템이 등장했습니다.

모듈 시스템의 목적은 다음과 같습니다.

  • 코드를 파일 단위로 캡슐화
  • 명시적인 의존성 선언
  • 이름 충돌 방지
  • 코드 재사용성 향상

CommonJS

CommonJS(CJS) 는 Node.js의 기본 모듈 시스템입니다. 동기적 로딩 방식이며, 서버 환경에 최적화되어 있습니다.

특징:

  • require로 import, module.exports로 export
  • 동기적 실행
  • 파일 단위 캐싱(같은 모듈은 한 번만 로드)
  • 동적 로드 가능 (require를 어디서든 호출 가능)

ES Modules (ESM)

ES Modules는 ES6에서 표준화된 공식 모듈 시스템입니다. 브라우저와 Node.js 양쪽에서 사용 가능합니다.

특징:

  • import/export 키워드 사용
  • 정적 분석 가능 → 트리 셰이킹(tree-shaking) 지원
  • 비동기 로딩 가능
  • 엄격 모드(strict mode) 자동 적용
  • this가 undefined

동적 import

ESM에서도 동적 import는 가능하지만, Promise를 반환합니다.

이는 코드 스플리팅(code splitting) 의 핵심 기법으로, 필요한 시점에만 모듈을 로드해 초기 번들 크기를 줄일 수 있습니다.

CommonJS vs ESM 비교

항목 CommonJS ESM
키워드 require/module.exports import/export
로딩 방식 동기 비동기
정적 분석 어려움 가능 (tree-shaking)
동적 로드 자유로움 import() 함수
엄격 모드 선택 자동
파일 확장자 .js, .cjs .js(type:module), .mjs
순환 참조 부분 객체 반환 라이브 바인딩

Node.js에서 ESM 사용

Node.js에서 ESM을 사용하려면 다음 중 하나를 따릅니다.

  1. package.json에 "type": "module" 추가
  2. 파일 확장자를 .mjs로 지정

인터롭(Interop)

ESM에서 CommonJS 모듈을 가져올 수는 있지만, 반대는 직접적이지 않습니다.

번들러(Webpack, Rollup, Vite 등)는 두 시스템을 자동으로 변환해주지만, Node.js 환경에서는 명시적인 처리가 필요합니다.

순환 참조

CommonJS에서는 부분 초기화된 객체가 반환될 수 있어 undefined가 나오기 쉽지만, ESM은 라이브 바인딩(live binding) 이므로 나중에 값이 초기화되어도 참조가 갱신됩니다.

트리 셰이킹

ESM은 정적 구조 덕분에 번들러가 사용되지 않는 export를 컴파일 시점에 제거할 수 있습니다.

이는 최종 번들 크기를 크게 줄여줍니다.

마무리

CommonJS는 Node.js의 역사를 만든 시스템이고, ESM은 JavaScript의 표준 미래입니다. 현재는 두 시스템이 공존하는 과도기지만, 새 프로젝트에서는 가능한 ESM을 사용하는 것이 권장됩니다. 트리 셰이킹, 정적 분석, 비동기 로딩 등 모던 빌드 도구의 이점을 최대한 활용할 수 있기 때문입니다.
