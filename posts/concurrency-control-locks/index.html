데이터베이스 동시성 제어와 락(Lock)

작성일 2026-05-10 Edited on 2026-05-22 In Backend , Database
데이터베이스가 동시 트랜잭션을 안전하게 처리하기 위해 사용하는 락의 종류와 MVCC, 데드락 처리 방식을 정리합니다.

왜 동시성 제어가 필요한가

여러 사용자가 동시에 같은 데이터를 읽고 쓰면 다음과 같은 문제가 발생할 수 있습니다.

  • Lost Update: 두 트랜잭션이 같은 행을 수정해 한 쪽 변경이 사라짐
  • Inconsistent Read: 도중에 다른 트랜잭션이 변경해 일관성이 깨짐
  • Phantom Read: 범위 쿼리에 새 행이 끼어듦

동시성 제어(Concurrency Control)는 이를 방지하면서도 처리량을 최대화하는 것이 목표입니다.

락의 분류

잠금 단위(Granularity)

단위 설명
Row Lock 행 단위, 동시성 최대
Page Lock 페이지 단위 (SQL Server 등)
Table Lock 테이블 전체
Database Lock DB 전체 (백업 등 특수 상황)

잠금 모드

  • Shared Lock (S): 읽기 락. 여러 트랜잭션이 동시에 보유 가능
  • Exclusive Lock (X): 쓰기 락. 단독 보유
  • Intent Lock (IS/IX): 하위 레벨 락의 의도를 상위에 표시

호환성 표:

S X
S O X
X X X

SQL에서 직접 락 걸기

FOR UPDATE는 같은 행에 대한 다른 쓰기를 블록합니다.

MVCC: 락 없는 읽기

MVCC(Multi-Version Concurrency Control) 는 데이터의 여러 버전을 유지해 읽기와 쓰기를 분리합니다.

  • 트랜잭션 시작 시점의 스냅샷을 읽음
  • 쓰기는 새 버전을 만들고, 읽기는 락을 잡지 않음
  • “읽는 사람은 쓰는 사람을 막지 않고, 쓰는 사람은 읽는 사람을 막지 않는다”

PostgreSQL은 오래된 버전을 VACUUM 으로 정리하고, MySQL InnoDB는 undo log에 보관합니다.

낙관적 vs 비관적 락

비관적 락 (Pessimistic Lock)

충돌이 자주 일어난다고 가정하고 미리 락을 잡습니다.

낙관적 락 (Optimistic Lock)

충돌이 드물다고 보고 락을 잡지 않은 채 진행하다 커밋 시점에 충돌을 검사합니다. 보통 버전 컬럼을 사용합니다.

업데이트 결과가 0행이면 충돌이 발생한 것입니다.

항목 비관적 락 낙관적 락
충돌 빈도 높음 낮음
대기 있음 없음
재시도 보통 불필요 필요
적합 워크로드 쓰기 충돌 잦음 읽기 중심

갭 락과 넥스트 키 락 (MySQL)

InnoDB는 팬텀 리드 방지를 위해 갭 락(Gap Lock) 과 넥스트 키 락(Next-Key Lock) 을 사용합니다.

갭 락은 동시성에 영향을 주므로 인덱스 설계가 중요합니다.

데드락

서로가 가진 자원을 기다리며 무한히 대기하는 상태입니다.

  • 트랜잭션 A: row 1 잠금 → row 2 요청
  • 트랜잭션 B: row 2 잠금 → row 1 요청

대부분의 DBMS는 데드락을 감지해 한쪽을 강제로 롤백합니다.

데드락 줄이는 방법

  • 같은 순서로 자원을 잠그기
  • 트랜잭션을 짧게 유지
  • 인덱스를 잘 설계해 락 범위를 줄이기
  • 재시도 로직 구현

모니터링

  • PostgreSQL: pg_locks, pg_stat_activity
  • MySQL: INFORMATION_SCHEMA.INNODB_TRX, INNODB_LOCKS
  • 슬로우 쿼리 로그와 함께 락 대기 시간을 추적

마무리

동시성 제어는 RDBMS의 핵심 기능이며, 락과 MVCC는 그 두 기둥입니다. 워크로드 특성에 맞게 격리 수준과 락 전략을 선택하고, 데드락이나 갭락 같은 미묘한 동작을 이해해야 안정적인 시스템을 만들 수 있습니다. 모든 충돌을 락으로 해결하기보다는 트랜잭션을 짧게 유지하고 인덱스를 잘 설계하는 것이 가장 효과적인 처방입니다.
