Linux cron과 crontab으로 스케줄링하기

작성일 2026-07-04 Edited on 2026-05-22 In System , OS
리눅스의 전통적 작업 스케줄러 cron의 동작 원리, crontab 문법, 자주 발생하는 함정과 대안을 정리합니다.

cron이란

cron은 정해진 시간에 명령을 주기적으로 실행하는 유닉스 계열의 전통적 스케줄러입니다. crond 데몬이 분 단위로 깨어나 등록된 작업 목록을 점검하고 일치하는 항목을 실행합니다. 백업, 로그 로테이션, 헬스 체크처럼 정형적인 반복 작업에 널리 쓰입니다.

crontab 파일 종류

  • 사용자 crontab: crontab -e로 편집. 각 사용자 권한으로 실행.
  • 시스템 crontab(/etc/crontab, /etc/cron.d/*): user 필드가 추가됨. 관리자가 작성.
  • 드롭인 디렉토리: /etc/cron.hourly, /etc/cron.daily, /etc/cron.weekly, /etc/cron.monthly. anacron이 함께 동작해 시스템이 꺼져 있었어도 누락된 작업을 보정합니다.

crontab 관리

흔한 함정

1. PATH 부족

cron 환경은 매우 제한적이라 PATH가 거의 비어 있습니다. 명령은 절대 경로로 지정하거나 crontab 상단에 PATH를 선언합니다.

2. 출력이 메일로 사라진다

stdout/stderr는 기본적으로 사용자 메일함으로 전송됩니다. 메일이 설정되어 있지 않으면 사라지므로, 로그 파일로 명시적으로 리다이렉트하는 것이 안전합니다.

3. 중복 실행

긴 작업이 다음 실행 시각과 겹치면 동시에 두 개가 돌 수 있습니다. flock으로 락을 걸어 방지합니다.

4. 타임존

cron은 시스템 타임존을 따릅니다. 컨테이너 환경에서는 의도와 다른 UTC로 동작할 수 있으므로 /etc/localtime 또는 TZ 환경 변수를 명시적으로 설정합니다.

5. % 문자 처리

명령에 %가 있으면 cron이 줄바꿈으로 해석합니다. \%로 이스케이프해야 합니다.

디버깅 팁

  • journalctl -u cron(또는 crond)으로 실행 이력을 확인합니다.
  • 임시로 1~2분 주기로 등록해 동작을 검증합니다.
  • 스크립트 첫 줄에 set -euo pipefail을 두어 무음 실패를 막습니다.
  • run-parts --test /etc/cron.daily로 어떤 스크립트가 실행되는지 미리 점검합니다.

대안

  • systemd timer: 의존성, 로그 통합, Persistent 보정, RandomizedDelay 등에서 우수.
  • anacron: 데스크톱처럼 24시간 켜져 있지 않은 시스템에서 누락 보정.
  • Kubernetes CronJob: 컨테이너 워크로드의 스케줄링.
  • 클라우드 스케줄러: AWS EventBridge Scheduler, GCP Cloud Scheduler 등.

마무리

cron은 단순하지만 환경 차이와 락, 로깅 같은 운영 디테일을 놓치면 의외로 사고가 잦은 도구입니다. 절대 경로, 락, 로그 리다이렉트 세 가지를 습관화하고, 복잡한 의존성이 필요한 작업은 systemd timer나 전용 스케줄러로 옮기는 것을 검토하세요.
