HTTP/2와 HTTP/3 차이점 정리

작성일 2026-05-23 Edited on 2026-06-15 In Network , HTTP
HTTP 프로토콜의 진화 과정을 따라 HTTP/2와 HTTP/3의 핵심 차이점을 살펴봅니다.

HTTP 진화의 흐름

HTTP는 1991년 HTTP/0.9부터 시작해 30년 이상 진화해왔습니다. 각 버전은 이전의 성능 한계를 극복하는 방향으로 설계되었으며, **HTTP/2(2015)**와 **HTTP/3(2022)**는 현대 웹의 표준이 되었습니다.

  • HTTP/1.1 (1997): 텍스트 기반, Keep-Alive, 파이프라이닝
  • HTTP/2 (2015): 바이너리, 멀티플렉싱, 헤더 압축
  • HTTP/3 (2022): QUIC 기반, UDP 사용

HTTP/1.1의 한계

HTTP/1.1은 단일 TCP 연결에서 한 번에 하나의 요청만 처리합니다. 이를 우회하기 위해 브라우저는 도메인당 보통 6개의 동시 연결을 사용하지만, 다음과 같은 문제가 남습니다.

  • Head-of-Line Blocking: 앞 요청이 지연되면 뒤 요청도 대기
  • 헤더 중복: 매 요청마다 동일한 헤더 반복 전송
  • TCP 핸드셰이크 비용: 새 연결마다 RTT 소모

HTTP/2의 핵심 특징

1. 바이너리 프레이밍

HTTP/2는 텍스트가 아닌 바이너리 프레임으로 통신합니다. 메시지를 작은 프레임으로 쪼개고 인터리브하여 전송합니다.

프레임 타입 역할
HEADERS 헤더 정보
DATA 본문 데이터
SETTINGS 연결 파라미터
PRIORITY 우선순위 지정

2. 멀티플렉싱

하나의 TCP 연결에서 여러 요청과 응답이 동시에 인터리브되어 흐릅니다. HTTP/1.1의 Head-of-Line Blocking이 응용 계층에서 해결됩니다.

3. 헤더 압축 (HPACK)

이전에 전송한 헤더를 인덱스로 참조하여 중복을 제거합니다. 동일한 사이트의 후속 요청에서는 헤더가 거의 0바이트에 가깝습니다.

4. Server Push

서버가 클라이언트 요청 전에 필요한 리소스를 미리 전송합니다.

(다만 효과적 활용이 어려워 Chrome은 2022년 Server Push를 deprecated 처리)

5. 스트림 우선순위

각 스트림에 우선순위를 부여하여 중요한 리소스를 먼저 받을 수 있습니다.

HTTP/2의 남은 문제

HTTP/2는 여전히 TCP 위에서 동작합니다. 따라서 TCP 자체의 한계가 그대로 남습니다.

  • TCP 레벨의 Head-of-Line Blocking: 한 패킷이 손실되면 같은 연결의 모든 스트림이 대기
  • 연결 설정 비용: TCP 3-way + TLS 1.2 핸드셰이크로 2~3 RTT 필요
  • 연결 마이그레이션 불가: 네트워크 전환 시 연결 재수립

HTTP/3의 등장

HTTP/3은 QUIC라는 새로운 전송 프로토콜 위에서 동작합니다. QUIC는 UDP를 기반으로 신뢰성, 보안, 멀티플렉싱을 응용 계층에서 구현합니다.

QUIC의 특징

  1. UDP 기반: 운영체제와 무관하게 빠른 진화 가능
  2. TLS 1.3 통합: 핸드셰이크와 암호화가 1 RTT (0 RTT 재연결 가능)
  3. 스트림 레벨 신뢰성: 한 스트림의 패킷 손실이 다른 스트림에 영향 없음
  4. 연결 ID: IP가 바뀌어도 연결 유지 (Wi-Fi <-> LTE 전환)

HTTP/3 핸드셰이크

HTTP/2 vs HTTP/3 비교

항목 HTTP/2 HTTP/3
전송 계층 TCP QUIC (UDP)
암호화 TLS 1.2/1.3 (선택) TLS 1.3 (필수)
Head-of-Line Blocking TCP 레벨 발생 없음
연결 설정 2~3 RTT 1 RTT (0 RTT 가능)
헤더 압축 HPACK QPACK
연결 마이그레이션 불가 가능
보급률 매우 높음 빠르게 증가 중

도입 시 고려사항

  • HTTP/3는 UDP 443 포트를 사용하므로 방화벽 정책 확인 필요
  • 대부분의 환경에서 Alt-Svc 헤더로 자동 업그레이드
  • 캐싱과 CDN 호환성 확인
  • 클라이언트가 미지원이면 HTTP/2로 fallback

실제 성능 차이

시나리오 HTTP/2 HTTP/3
안정된 유선 환경 비슷 약간 우위
모바일/패킷 손실 환경 큰 저하 안정적 유지
네트워크 전환 재연결 필요 연결 유지

마무리

HTTP/2는 멀티플렉싱과 헤더 압축으로 웹 성능을 크게 끌어올렸고, HTTP/3는 QUIC라는 새로운 기반 위에서 TCP의 구조적 한계를 극복했습니다. 특히 모바일과 불안정한 네트워크 환경에서 HTTP/3의 이점은 두드러지며, 앞으로 웹의 기본 프로토콜로 빠르게 자리잡을 것입니다.
