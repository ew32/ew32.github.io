웹폰트 최적화 - font-display, preload, subset

작성일 2026-06-14 Edited on 2026-05-20 In Frontend
웹폰트로 인한 렌더링 지연과 레이아웃 시프트를 줄이기 위한 font-display, preload, 서브셋 등 핵심 최적화 기법을 정리합니다.

웹폰트가 성능에 미치는 영향

웹폰트는 사이트의 아이덴티티를 결정하지만, 잘못 사용하면 다음과 같은 문제를 일으킵니다.

  • FOIT(Flash of Invisible Text) — 폰트 로딩 동안 텍스트가 보이지 않음
  • FOUT(Flash of Unstyled Text) — 시스템 폰트로 잠깐 보였다가 바뀜
  • CLS(Cumulative Layout Shift) — 폰트 교체 시 줄 높이/너비가 변해 레이아웃이 흔들림
  • LCP(Largest Contentful Paint) 지연 — 큰 텍스트가 폰트 의존이면 LCP가 늦어짐

font-display 전략

@font-face 안에 font-display를 지정해 폰트 로딩 중의 동작을 제어합니다.

값 동작
auto 브라우저 기본 (대개 block과 유사)
block 최대 3초 invisible → 폰트 등장
swap 즉시 fallback 표시 → 폰트 로드되면 교체
fallback 100ms invisible → fallback → 3초 안에 들어오면 swap
optional 100ms invisible → 못 받으면 그냥 fallback 유지

본문 텍스트는 보통 swap을 권장합니다. 다만 FOUT가 큰 시각 변화를 일으킨다면 optional 또는 fallback 매칭(다음 절)을 고려합니다.

preload로 우선순위 올리기

가장 먼저 표시되는 폰트는 <link rel="preload">로 우선 로드합니다.

crossorigin을 빠뜨리면 두 번 다운로드되므로 반드시 포함합니다. 폰트는 일반적으로 CSS가 파싱된 후에야 발견되므로 preload가 매우 효과적입니다.

WOFF2와 형식 지정

현대 브라우저는 모두 WOFF2를 지원합니다. WOFF2는 WOFF 대비 약 30% 작아 가장 우선해야 할 형식입니다.

오래된 브라우저 지원이 정말로 필요한 경우에만 woff, ttf를 추가합니다.

서브셋(subset)

한글 폰트는 글자 수가 많아 파일이 큽니다. 사용하는 글자만 남기면 크게 줄어듭니다.

  • 유니코드 범위로 분할
  • 자주 쓰는 KS X 1001 한글 2350자만 포함
  • 영문/숫자/기호만 별도 서브셋

브라우저는 해당 범위의 글자가 페이지에 등장할 때만 파일을 받습니다.

가변 폰트(variable font)

여러 굵기·기울임을 하나의 파일로 합친 가변 폰트는 6~9개 굵기를 따로 받는 것보다 훨씬 효율적입니다.

font-weight: 100 900 범위 지정이 가변 폰트의 표식입니다.

레이아웃 시프트 줄이기

size-adjust, ascent-override, descent-override로 fallback 폰트의 메트릭을 웹폰트와 비슷하게 맞추면 교체 시 흔들림을 거의 없앨 수 있습니다.

Next.js의 next/font는 이 작업을 자동화해 CLS를 0에 가깝게 만들어 줍니다.

체크리스트

  • WOFF2 사용
  • 필요한 굵기/스타일만 등록
  • font-display: swap(또는 optional) 적용
  • 첫 화면 폰트는 <link rel="preload" ... crossorigin>
  • 한글이면 서브셋 또는 unicode-range 분할
  • fallback 메트릭 조정으로 CLS 완화
  • 가능한 경우 가변 폰트 사용

마무리

웹폰트 최적화는 LCP, CLS, FOIT 같은 핵심 지표에 직접 영향을 줍니다. preload·서브셋·font-display·fallback 메트릭 매칭만 잘 적용해도 체감 로딩 속도와 시각적 안정성이 눈에 띄게 좋아집니다. 디자인을 해치지 않으면서 빠른 사이트를 만드는 데에 가장 효과적인 투자 중 하나입니다.
