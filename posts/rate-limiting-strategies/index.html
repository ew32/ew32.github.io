Rate Limiting 알고리즘과 구현

작성일 2026-05-14 Edited on 2026-05-20 In Backend , Security
API 보호를 위한 Rate Limiting 알고리즘을 비교하고, 분산 환경에서의 구현과 운영 전략을 정리합니다.

Rate Limiting이 필요한 이유

Rate Limiting은 단위 시간당 처리할 수 있는 요청 수를 제한해, 시스템과 사용자 모두를 보호하는 메커니즘입니다. 주요 목적은 다음과 같습니다.

  • 남용 방지: 크롤러, 봇, 무차별 대입(Brute Force) 차단
  • 공정성: 특정 사용자가 시스템 자원을 독점하지 못하도록
  • 비용 통제: 과금형 API에서 폭주 비용 방지
  • 장애 격리: 백엔드 의존 서비스가 다운되더라도 폭주 차단

특히 로그인, 비밀번호 재설정, OTP 발송 같은 엔드포인트에는 보안 차원에서 반드시 적용해야 합니다.

주요 알고리즘 4가지

1. Fixed Window

고정된 시간 윈도우(예: 1분) 안에서 카운트를 셉니다. 가장 단순합니다.

단점: 윈도우 경계에서 2배의 트래픽 폭주가 가능합니다(00:59에 100회 + 01:00에 100회).

2. Sliding Window Log

요청 시각 자체를 기록하고, 현재 시각 기준 N초 이내의 요청 수를 셉니다. 가장 정확하지만 메모리 비용이 큽니다.

3. Sliding Window Counter

Fixed Window 두 개를 가중 평균하는 근사 방식입니다. 메모리 효율이 좋고 경계 폭주를 완화합니다.

4. Token Bucket

가장 널리 쓰이는 알고리즘입니다. 일정 속도로 토큰이 버킷에 채워지고, 요청은 토큰을 1개 소모합니다. 토큰이 없으면 거부됩니다.

**버스트(burst)**를 허용하면서도 평균 속도를 제한할 수 있다는 장점이 있습니다.

5. Leaky Bucket

요청이 큐에 쌓이고, 일정 속도로 새어 나가도록(처리되도록) 구성합니다. 고정된 처리율을 보장하는 데 유리합니다. 다만 버스트를 흡수하지 못하고 지연이 발생합니다.

비교 표

알고리즘 정확도 메모리 버스트 허용 구현 난이도
Fixed Window 낮음 낮음 일부 (경계 폭주) 매우 쉬움
Sliding Log 매우 높음 높음 없음 보통
Sliding Counter 높음 낮음 부드러움 보통
Token Bucket 높음 낮음 허용 보통
Leaky Bucket 높음 보통 없음 보통

분산 환경에서의 구현

서버가 여러 대인 경우 메모리에 카운트를 두면 인스턴스별로 한도가 따로 계산되는 문제가 생깁니다. 일반적인 해법은 Redis를 중앙 카운터로 사용하는 것입니다.

Lua 스크립트로 묶으면 원자성이 보장돼 race condition을 피할 수 있습니다. 더 정교한 알고리즘이 필요하면 redis-cell 모듈이나 Envoy/API Gateway 내장 기능을 활용하는 것도 좋습니다.

키 설계

무엇을 기준으로 제한할지 정하는 것이 중요합니다.

키 용도
사용자 ID 인증된 사용자별 제한
API Key 외부 파트너
IP 주소 비로그인 트래픽
엔드포인트 + 사용자 민감 엔드포인트 차등 적용
디바이스 핑거프린트 IP 우회 방지 보조

IP만 기준으로 삼으면 CGNAT, 회사망, 모바일 캐리어 NAT 환경에서 다수 사용자가 공유돼 부작용이 생깁니다. 가능하면 계정 단위 키를 우선하세요.

HTTP 응답 표준

클라이언트가 한도를 인지할 수 있도록 표준 헤더를 노출하세요. IETF draft인 RateLimit-* 헤더가 권장 방향입니다.

보안 관점의 Rate Limit

다음 엔드포인트는 일반 API보다 훨씬 엄격한 한도가 필요합니다.

  • 로그인, 비밀번호 재설정
  • OTP/SMS/이메일 발송 (비용 폭탄 위험)
  • 회원가입, 검색 자동완성

또한 계정 잠금과 결합하면 단순 Rate Limit보다 강력한 brute-force 방어가 됩니다.

Best Practices

  • 계정 + IP + 엔드포인트 다중 키 조합
  • 분산 환경은 Redis Lua 또는 API Gateway에서 일괄 처리
  • 표준 RateLimit-* 헤더와 429 + Retry-After 응답
  • 민감 엔드포인트는 별도의 엄격한 정책 적용
  • DDoS 방어와는 별개 계층 — CDN/WAF로 분리

마무리

Rate Limiting은 단순한 카운팅 기능처럼 보이지만, 분산 환경의 정확성과 보안 위협 모델을 함께 고려해야 하는 시스템 설계 주제입니다. 알고리즘 선택보다 중요한 것은 “무엇을 기준으로 제한할지”, “어떤 엔드포인트에 차등 적용할지”라는 정책 설계입니다. 작게 시작하되 일찍 도입하세요. 한 줄의 Rate Limit이 수만 건의 무차별 대입을 막아줄 수 있습니다.
