MySQL vs PostgreSQL 비교

작성일 2026-05-09 Edited on 2026-05-22 In Backend , Database
대표적인 오픈소스 RDBMS인 MySQL과 PostgreSQL의 차이점을 기능, 성능, 사용 사례 관점에서 비교합니다.

두 데이터베이스의 위치

MySQL과 PostgreSQL은 가장 널리 사용되는 오픈소스 관계형 데이터베이스(RDBMS)입니다. MySQL은 단순성과 빠른 읽기 성능으로 웹 애플리케이션 시장을 장악해왔고, PostgreSQL은 표준 준수와 확장성을 강점으로 엔터프라이즈 환경에서 채택이 빠르게 늘고 있습니다.

선택은 단순한 취향 문제가 아니라, 데이터 모델·트랜잭션 요구사항·운영 환경에 따라 달라집니다.

아키텍처 차이

항목 MySQL PostgreSQL
프로세스 모델 스레드 기반 프로세스 기반
기본 스토리지 엔진 InnoDB (다중 엔진 지원) 단일 엔진
MVCC InnoDB에서 지원 핵심 기능으로 내장
라이선스 GPL (Oracle 소유) PostgreSQL License (BSD 유사)
표준 SQL 준수 일부 비표준 허용 표준 SQL에 가깝게 엄격

PostgreSQL은 ORDBMS(객체-관계형)로 분류되며 사용자 정의 타입, 상속, 함수형 인덱스 등 고급 기능이 풍부합니다.

데이터 타입 비교

PostgreSQL은 풍부한 내장 타입을 제공합니다.

MySQL도 8.0부터 JSON을 지원하지만, JSONB처럼 인덱싱 친화적인 이진 포맷은 PostgreSQL 쪽이 더 성숙합니다.

인덱스와 쿼리 최적화

  • MySQL: B-Tree, Full-Text, R-Tree(공간), Hash(MEMORY 엔진) 인덱스를 지원합니다.
  • PostgreSQL: B-Tree 외에 GIN, GiST, BRIN, SP-GiST, Hash 등 다양한 인덱스 타입을 제공합니다.

부분 인덱스, 함수 기반 인덱스, 표현식 인덱스 등은 PostgreSQL이 훨씬 자유롭게 다룹니다.

트랜잭션과 동시성

두 데이터베이스 모두 ACID를 지원하지만 구현 방식이 다릅니다.

  • PostgreSQL: 모든 작업이 MVCC 위에서 동작하여 읽기 작업이 쓰기를 막지 않습니다.
  • MySQL InnoDB: MVCC를 지원하지만 갭락(Gap Lock) 등으로 동시성 처리 방식이 다릅니다.

PostgreSQL의 SSI(Serializable Snapshot Isolation) 는 직렬화 가능 수준에서도 락 없이 동작합니다.

복제와 확장성

기능 MySQL PostgreSQL
비동기 복제 기본 지원 기본 지원
동기 복제 반동기(semi-sync) 동기 복제 지원
논리 복제 8.0부터 강화 10부터 강화
멀티 마스터 Group Replication 외부 도구(BDR 등)

MySQL은 운영 도구(Vitess, Galera)가 풍부하고, PostgreSQL은 Citus, Patroni 같은 도구로 보완합니다.

언제 무엇을 쓸까

MySQL이 유리한 경우:

  • 읽기 위주의 웹 트래픽
  • 단순한 CRUD 위주 도메인
  • LAMP/LEMP 스택 호환성
  • 빠른 학습 곡선이 필요할 때

PostgreSQL이 유리한 경우:

  • 복잡한 쿼리, 분석 작업
  • JSON·배열·지리 데이터 활용
  • 엄격한 데이터 무결성 요구
  • 사용자 정의 함수·확장이 많은 환경

성능에 대한 흔한 오해

“MySQL이 무조건 빠르다”는 과거의 통념입니다. 현재는 워크로드에 따라 우위가 갈립니다.

  • 단순 PK 조회, 짧은 트랜잭션: MySQL이 약간 우위
  • 복잡한 조인, 집계, 분석: PostgreSQL이 우위
  • 대규모 동시 쓰기: 튜닝에 따라 달라짐

마무리

MySQL과 PostgreSQL은 모두 검증된 RDBMS이며, 어느 한쪽이 절대적으로 우월하지 않습니다. 데이터 모델의 복잡성, 트랜잭션 격리 요구사항, 팀의 운영 경험을 종합해 선택해야 합니다. 새 프로젝트라면 PostgreSQL의 기능 풍부함이 장기적으로 유리한 경우가 많고, 기존 MySQL 자산이 많다면 굳이 옮길 이유는 없습니다.
