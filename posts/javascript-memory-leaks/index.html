JavaScript 메모리 누수 - 원인과 진단

작성일 2026-05-30 Edited on 2026-05-26 In Frontend , JavaScript
JavaScript에서 발생하는 대표적인 메모리 누수 원인과 Chrome DevTools를 이용한 진단 및 해결 방법을 정리합니다.

메모리 누수란

가비지 컬렉터가 있어도 더 이상 필요 없는 메모리가 회수되지 않는 상황을 메모리 누수라고 합니다. JS 엔진은 도달 가능성(reachability)을 기준으로 메모리를 해제하므로, 어딘가에서 참조가 살아 있으면 회수되지 않습니다.

흔한 누수 원인

1. 의도치 않은 전역 변수

"use strict" 또는 모듈 모드를 활성화하면 이런 실수를 방지할 수 있습니다.

2. 해제되지 않은 타이머와 이벤트 리스너

3. 분리된 DOM 노드

DOM에서 제거된 노드가 JS 객체에 의해 참조되면 메모리에 남습니다.

4. 클로저로 인한 누수

V8은 일부 최적화로 사용하지 않는 변수를 제거하기도 하지만, 모든 엔진과 상황에서 보장되지는 않습니다.

5. Map/Set의 강한 참조

Map과 Set은 키/값을 강하게 참조합니다. 캐시 용도라면 WeakMap, WeakSet을 고려하세요.

Chrome DevTools로 진단하기

Memory 탭 - Heap Snapshot

  1. DevTools → Memory → Heap snapshot으로 스냅샷 1 촬영
  2. 의심되는 동작을 반복(예: 모달 열기/닫기 10회)
  3. GC 버튼으로 강제 수집 후 스냅샷 2 촬영
  4. “Comparison” 모드에서 스냅샷 1과 2를 비교

객체 수가 계속 증가하는 클래스가 있다면 누수 의심 대상입니다. Retainers 패널에서 어떤 참조가 객체를 붙잡고 있는지 추적할 수 있습니다.

Performance 탭

Performance 녹화 시 Memory 체크박스를 켜면 JS 힙 크기가 시간에 따라 어떻게 변하는지 그래프로 볼 수 있습니다. 톱니 모양으로 오르내리면 정상, 계속 우상향이면 누수 의심입니다.

performance.memory API

예방 패턴

  • 컴포넌트 unmount 시 removeEventListener, clearInterval, AbortController로 fetch 취소
  • SPA 라우팅 시 이전 페이지의 구독 해제 보장
  • 캐시는 LRU 또는 WeakMap 기반으로 구현
  • React라면 useEffect의 cleanup 함수, Vue라면 onUnmounted에서 자원 정리

마무리

메모리 누수는 천천히 누적되어 한참 후에야 드러나는 무서운 버그입니다. 정기적으로 Heap snapshot으로 검사하는 습관, 그리고 “내가 만든 구독은 내가 해제한다”는 원칙을 코드 리뷰 기준으로 삼아보세요.
