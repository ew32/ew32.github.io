비밀번호 해싱 - bcrypt와 Argon2 비교

작성일 2026-05-13 Edited on 2026-05-26 In Security , Cryptography
안전한 비밀번호 저장을 위한 해싱의 원칙을 살펴보고, bcrypt와 Argon2의 차이와 선택 기준을 정리합니다.

비밀번호를 평문이나 단순 해시로 저장하면 안 되는 이유

비밀번호를 평문으로 저장하면 DB가 유출되는 순간 모든 계정이 즉시 탈취됩니다. 그렇다고 MD5, SHA-256 같은 일반 해시도 해법이 될 수 없습니다. 이런 해시는 너무 빠르도록 설계되어 있어, 현대 GPU로 초당 수십억 번의 시도가 가능합니다. 레인보우 테이블과 결합하면 8자리 이하 비밀번호는 분 단위로 깨집니다.

비밀번호 해시는 느려야 하며, 각 사용자별로 달라야 합니다. 이를 위해 등장한 것이 Salt + Slow Hash 방식이고, 그 대표가 bcrypt와 Argon2입니다.

비밀번호 해싱의 3대 원칙

  1. Salt: 같은 비밀번호라도 사용자마다 다른 해시가 나오도록 무작위 값을 추가
  2. Slow: 의도적으로 계산을 느리게 해 brute-force 비용 증가
  3. Parameterized: 하드웨어 발전에 맞춰 비용 파라미터를 조절 가능

bcrypt

1999년에 발표된 Blowfish 기반 해시 알고리즘으로, 가장 오랫동안 검증된 비밀번호 해싱 함수입니다.

특징:

  • cost factor 하나로 계산량 조절
  • 자체 salt 생성 및 인코딩
  • 입력이 72바이트로 잘림(긴 비밀번호의 뒷부분이 무시됨)
  • 메모리 사용량이 작아 GPU/ASIC 공격에 비교적 약함

Argon2

2015년 Password Hashing Competition 우승작으로, 현재 OWASP가 신규 시스템에 가장 권장하는 알고리즘입니다. 세 가지 변종이 있습니다.

변종 특징 용도
Argon2d GPU 저항 강함, 사이드 채널 약함 암호화폐 채굴
Argon2i 사이드 채널 저항 강함 사이드 채널 우려 환경
Argon2id i + d의 장점을 결합 비밀번호 해싱 권장

특징:

  • 메모리 하드(Memory-Hard) — 대량의 RAM을 요구해 GPU/ASIC 공격을 무력화
  • 메모리, 반복, 병렬도를 독립적으로 튜닝 가능
  • 입력 길이 제한 없음

bcrypt vs Argon2 비교

항목 bcrypt Argon2id
발표 1999 2015
튜닝 파라미터 cost(시간) memory + time + parallelism
메모리 하드 아니오 예
GPU 저항 보통 강함
입력 길이 72바이트 제한 무제한
표준화 사실상 표준 RFC 9106
라이브러리 성숙도 매우 높음 점점 성숙

어떤 것을 선택해야 할까?

  • 신규 시스템: Argon2id 권장 (OWASP 가이드라인)
  • 레거시 시스템: bcrypt를 유지하되 cost 12 이상
  • 환경에 Argon2 라이브러리가 부족하다면 bcrypt(cost ≥ 12) 또는 scrypt도 무리는 없습니다.

파라미터 가이드(2026 기준)

OWASP Password Storage Cheat Sheet 기준의 일반 권고치입니다.

운영 환경에서는 로그인 응답 시간이 250~500ms 정도 되도록 튜닝하는 것이 일반적인 균형점입니다.

절대 하지 말아야 할 것

  • 비밀번호를 평문/MD5/SHA-256/SHA-512만으로 저장
  • 모든 사용자에게 동일한 Salt 사용
  • 비밀번호와 함께 **Pepper(서버 비밀)**를 같은 DB에 저장 (Pepper는 별도 KMS에)
  • 직접 만든 자작 해싱 알고리즘 사용

비밀번호 외 추가 방어

해시만으로는 부족합니다. 다음과 같은 정책을 함께 운영하세요.

  • 로그인 Rate Limit + 계정 잠금
  • MFA(다중 요소 인증) 도입
  • 유출된 비밀번호 차단(haveibeenpwned API)
  • 정기적 해시 알고리즘 재해싱(Rehash on Login)

마무리

비밀번호 해싱은 “DB가 유출됐다고 가정했을 때 얼마나 시간을 벌어줄 수 있느냐”의 게임입니다. bcrypt는 여전히 안전한 선택이고, Argon2id는 더 나은 기본값입니다. 중요한 것은 알고리즘 이름이 아니라, 충분한 비용 파라미터와 정기적인 재평가 그리고 다층 방어라는 점을 기억하세요.
