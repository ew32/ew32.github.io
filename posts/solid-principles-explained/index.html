SOLID 5원칙 한 번에 정리하기

작성일 2026-05-17 Edited on 2026-05-26 In Backend
객체지향 설계의 5대 원칙 SOLID(SRP, OCP, LSP, ISP, DIP)를 코드 예제와 함께 명확히 정리합니다.

SOLID란?

SOLID는 로버트 마틴(Uncle Bob)이 정리한 객체지향 설계 5대 원칙의 머리글자입니다. 좋은 설계의 기준이 되어 유연성, 유지보수성, 확장성을 높이는 가이드라인 역할을 합니다.

  • SRP - Single Responsibility Principle
  • OCP - Open/Closed Principle
  • LSP - Liskov Substitution Principle
  • ISP - Interface Segregation Principle
  • DIP - Dependency Inversion Principle

S - 단일 책임 원칙(SRP)

“클래스는 단 하나의 변경 이유만을 가져야 한다.”

O - 개방-폐쇄 원칙(OCP)

“확장에는 열려 있고, 변경에는 닫혀 있어야 한다.”

새 기능을 추가할 때 기존 코드를 수정하지 않고 새 코드만 추가해야 합니다.

Strategy, Factory, Template Method 패턴이 대표적인 OCP 구현 도구입니다.

L - 리스코프 치환 원칙(LSP)

“하위 타입은 상위 타입을 대체할 수 있어야 한다.”

상속 관계라면 부모 자리에 자식을 넣어도 프로그램이 정상 동작해야 합니다. 흔한 위반 예가 “정사각형은 직사각형”입니다.

LSP 위반은 상속을 잘못 사용한 신호입니다. 합성(composition)으로 대체하거나 인터페이스를 다시 설계해야 합니다.

I - 인터페이스 분리 원칙(ISP)

“클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 한다.”

거대한 인터페이스 하나보다 역할별로 잘게 나눈 인터페이스가 낫습니다.

D - 의존성 역전 원칙(DIP)

“고수준 모듈은 저수준 모듈에 의존하지 말아야 하며, 둘 다 추상화에 의존해야 한다.”

Spring의 DI 컨테이너는 DIP를 실현하기 위한 인프라입니다.

SOLID 적용 시 주의점

  • 원칙은 도구이지 목적이 아닙니다. 과도하게 적용하면 추상화 비용이 더 큽니다.
  • 변경 가능성이 낮은 코드에 미리 적용할 필요는 없습니다(YAGNI).
  • 책임은 “변경 이유”로 판단하세요. 유사한 이유로 함께 바뀌는 코드는 같이 두어도 됩니다.

마무리

SOLID는 디자인 패턴, 클린 아키텍처, DDD의 토대가 되는 원칙입니다. 코드를 작성하거나 리뷰할 때 “이 클래스의 책임은 무엇인가?”, “확장 가능한가?”, “추상화에 의존하는가?”를 자문하는 습관을 들이면 자연스럽게 좋은 설계를 향하게 됩니다.
