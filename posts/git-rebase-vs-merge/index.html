Git Rebase vs Merge 전략

작성일 2026-07-14 Edited on 2026-05-19 In 개발환경
Git의 두 통합 방법인 rebase와 merge의 차이, 장단점, 그리고 팀에서의 선택 가이드를 정리합니다.

두 가지 통합 방식

기능 브랜치를 메인 브랜치에 합칠 때 Git은 크게 두 가지 방법을 제공합니다.

  • merge: 두 브랜치의 끝을 묶는 새로운 머지 커밋을 만든다.
  • rebase: 기능 브랜치의 커밋들을 메인 브랜치 끝으로 옮겨 재작성한다.

결과적으로 코드 상태는 같지만 히스토리의 형태와 의미가 완전히 달라집니다.

Merge

장점은 사실을 그대로 기록한다는 점입니다. “이 시점에 이 브랜치를 합쳤다”가 머지 커밋으로 남고, 기존 커밋의 SHA가 바뀌지 않아 안전합니다. 단점은 머지 커밋이 누적되면 히스토리가 거미줄처럼 복잡해진다는 것입니다.

Rebase

기능 브랜치의 B, C가 D 위로 재배치되어 일직선 히스토리가 됩니다. git log --oneline이 깔끔해지고 변경의 흐름을 따라가기 쉽습니다. 다만 커밋 SHA가 바뀌므로 이미 푸시된 공유 브랜치에서는 위험합니다.

Golden Rule

공개된 브랜치(이미 다른 사람이 받아간 브랜치)는 rebase하지 말 것.

이를 어기면 협업자의 로컬과 원격 히스토리가 어긋나 강제 푸시와 강제 풀의 악순환이 시작됩니다.

Interactive Rebase로 커밋 정리

PR을 올리기 전에 자신의 브랜치를 정돈할 때 유용합니다.

에디터에서 다음과 같이 편집합니다.

  • pick: 그대로 사용
  • squash / fixup: 이전 커밋에 합치기
  • reword: 메시지 수정
  • drop: 제거

실무 전략

Squash and merge

GitHub의 기본 옵션 중 하나. 기능 브랜치의 모든 커밋을 하나로 합쳐 메인에 추가합니다. 메인 히스토리가 매우 깔끔해지지만 세부 커밋 단위는 사라집니다.

Rebase and merge

기능 브랜치 커밋을 그대로 메인 위에 일직선으로 얹습니다. 세부 커밋이 보존되면서도 머지 커밋이 없습니다.

Merge commit

모든 머지를 머지 커밋으로 기록. 흐름을 정확히 보존하지만 가지가 많아집니다.

팀 단위 선택 가이드

상황 권장
오픈소스, 많은 기여자, PR 단위 추적 Squash merge
일직선 히스토리 선호, 세부 커밋 보존 Rebase merge
대규모 장기 브랜치, 흐름 기록 중요 Merge commit

중요한 것은 팀이 한 가지 전략으로 합의하는 것입니다. 정책을 README에 명시하고 PR 템플릿에서 안내하면 일관성이 유지됩니다.

사고 복구

rebase 중 꼬였다면 당황하지 말고 git rebase --abort로 되돌리거나, git reflog로 이전 HEAD를 찾아 복구할 수 있습니다.

마무리

merge와 rebase는 옳고 그름이 아니라 트레이드오프입니다. 사실 보존이냐 깔끔한 히스토리냐의 선택이며, 팀 컨벤션으로 정해두는 것이 가장 중요합니다.
