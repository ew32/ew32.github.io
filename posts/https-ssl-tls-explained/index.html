HTTPS와 SSL/TLS 동작 원리

작성일 2026-05-06 Edited on 2026-06-15 In Network , Security
HTTPS가 어떻게 안전한 통신을 보장하는지 SSL/TLS 핸드셰이크 과정과 인증서 구조를 통해 설명합니다.

HTTP의 한계

HTTP는 평문(plain text)으로 데이터를 주고받습니다. 이로 인해 다음과 같은 보안 문제가 발생합니다.

  • 도청(Eavesdropping): 중간자가 통신 내용을 그대로 볼 수 있음
  • 변조(Tampering): 통신 도중 데이터를 바꿀 수 있음
  • 위장(Impersonation): 가짜 서버로 위장해 정보를 탈취

HTTPS는 이러한 문제를 해결하기 위해 HTTP에 SSL/TLS 레이어를 추가한 프로토콜입니다.

SSL과 TLS의 관계

  • SSL(Secure Sockets Layer): Netscape가 1990년대 개발, SSL 1.0~3.0
  • TLS(Transport Layer Security): SSL의 후속 표준, TLS 1.0~1.3

오늘날 우리가 “SSL”이라고 부르는 것은 사실 대부분 TLS입니다. SSL 3.0 이하는 취약점으로 인해 사용이 금지되었으며, TLS 1.2와 1.3이 주로 사용됩니다.

대칭키 vs 비대칭키 암호화

대칭키 암호화

  • 같은 키로 암호화/복호화 (예: AES)
  • 빠름
  • 키 전달 문제 존재

비대칭키 암호화 (공개키 암호화)

  • 공개키로 암호화, 개인키로 복호화 (예: RSA, ECDSA)
  • 느림
  • 키 전달 문제 해결

TLS는 두 방식의 장점을 결합합니다. 핸드셰이크 단계에서 비대칭키로 안전하게 대칭키를 공유한 뒤, 실제 데이터 통신은 대칭키로 수행합니다.

TLS 1.2 핸드셰이크 과정

핵심 흐름

  1. 클라이언트가 지원하는 암호 스위트 목록을 전송
  2. 서버가 사용할 암호 스위트를 결정하고 인증서 전송
  3. 클라이언트가 인증서를 검증하고 PreMasterSecret 생성
  4. 양쪽이 PreMasterSecret과 랜덤값을 조합하여 동일한 세션 키(대칭키) 생성
  5. 이후 모든 통신은 세션 키로 암호화

TLS 1.3의 개선

TLS 1.3은 핸드셰이크를 1-RTT로 단축하고(이전은 2-RTT), 이미 연결한 적 있는 서버와는 0-RTT 재연결도 가능합니다.

  • 취약한 암호 스위트 제거 (RSA 키 교환, RC4, MD5 등)
  • Perfect Forward Secrecy 필수화
  • 핸드셰이크의 대부분을 암호화
  • 빠른 연결 속도

인증서(Certificate) 구조

서버 인증서는 다음과 같은 정보를 담은 공증된 문서입니다.

  • 도메인 이름 (Common Name, SAN)
  • 발급 대상(Subject) 정보
  • 서버의 공개키
  • 발급자(Issuer) 정보
  • 유효 기간
  • CA의 디지털 서명

인증서 체인

브라우저는 OS에 내장된 Root CA 목록을 신뢰합니다. 서버가 보낸 인증서가 신뢰된 Root까지 체인이 연결되면 유효한 인증서로 인정됩니다.

인증서 검증 과정

  1. 유효 기간 확인
  2. 도메인 일치 확인 (URL의 호스트와 인증서의 CN/SAN)
  3. 서명 검증: 상위 CA의 공개키로 서명 확인
  4. 체인 검증: Root CA까지 신뢰 체인 확인
  5. 폐기 여부 확인 (CRL, OCSP)

검증 실패 시 브라우저는 경고 페이지를 표시합니다.

Let’s Encrypt와 무료 인증서

Let’s Encrypt는 무료로 자동화된 인증서를 발급해주는 비영리 CA입니다. certbot 같은 도구로 손쉽게 발급/갱신할 수 있습니다.

유효 기간은 90일이므로 자동 갱신 설정이 필수입니다.

HSTS (HTTP Strict Transport Security)

HTTPS 적용 후에도 사용자가 처음에 http://로 접속하면 중간자 공격에 노출될 수 있습니다. HSTS는 브라우저에 이 사이트는 항상 HTTPS로만 접속하라고 알려주는 헤더입니다.

흔한 오해와 진실

오해 진실
HTTPS는 느리다 TLS 1.3과 HTTP/2 덕분에 거의 차이 없음
HTTPS는 SEO에 영향 없다 Google은 HTTPS를 랭킹 신호로 사용
자체 서명 인증서로 충분하다 브라우저 경고 발생, 신뢰 불가
HTTPS면 모든 보안 문제가 해결된다 전송 보안일 뿐, 애플리케이션 보안은 별도 필요

마무리

HTTPS는 더 이상 선택이 아니라 필수입니다. 핸드셰이크 과정과 인증서 구조를 이해하면 HTTPS 관련 오류를 만났을 때 원인을 빠르게 파악할 수 있고, 보안 설정을 더 견고하게 다듬을 수 있습니다.
