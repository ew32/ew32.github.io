Saga 패턴으로 분산 트랜잭션 처리하기

작성일 2026-05-16 Edited on 2026-05-19 In Backend , Infrastructure
마이크로서비스 환경에서 분산 트랜잭션을 다루는 Saga 패턴의 두 가지 구현 방식과 보상 트랜잭션을 정리합니다.

왜 분산 트랜잭션이 어려운가?

모놀리식에서는 하나의 DB 트랜잭션으로 일관성을 보장할 수 있지만, 마이크로서비스에서는 각 서비스가 자체 데이터베이스를 가집니다. 이 환경에서 여러 서비스에 걸친 작업의 원자성을 어떻게 보장할 수 있을까요?

2PC(2-Phase Commit)는 이론적으로 가능하지만 블로킹·가용성 저하·확장성 문제 때문에 실무에서는 거의 쓰이지 않습니다. 대신 Saga 패턴이 일반적인 해법으로 자리 잡았습니다.

Saga 패턴이란?

Saga는 여러 로컬 트랜잭션의 시퀀스로 전역 트랜잭션을 구성하는 패턴입니다. 각 단계는 자체 트랜잭션으로 커밋되며, 어느 단계에서든 실패하면 이전 단계들에 대해 보상 트랜잭션(Compensating Transaction) 을 실행하여 비즈니스 상태를 되돌립니다.

핵심 아이디어: ACID 대신 BASE(Basically Available, Soft state, Eventual consistency).

예시 시나리오

주문 처리 흐름을 가정해 봅니다.

  1. 주문 생성 (Order Service)
  2. 결제 처리 (Payment Service)
  3. 재고 차감 (Inventory Service)
  4. 배송 요청 (Shipping Service)

3단계에서 재고가 부족하다면 결제를 환불(보상)하고 주문을 취소(보상)해야 합니다.

단계 정방향 보상
1 주문 생성 주문 취소
2 결제 승인 결제 환불
3 재고 차감 재고 복구
4 배송 요청 배송 취소

구현 방식 1: Choreography

각 서비스가 이벤트를 발행/구독하여 분산된 방식으로 협력합니다. 중앙 조정자가 없습니다.

장점

  • 단순한 구조, 서비스 간 결합 낮음
  • 추가 컴포넌트 불필요

단점

  • 흐름 추적이 어려움 (관측성 도구 필수)
  • 참여 서비스가 늘수록 이벤트 의존 그래프 복잡화
  • 순환 의존 발생 위험

구현 방식 2: Orchestration

오케스트레이터가 중앙에서 각 서비스를 명시적으로 호출하고, 실패 시 보상을 지시합니다.

실무에서는 상태 머신 형태로 구현하며 Axon, Camunda, Temporal, AWS Step Functions 같은 도구가 자주 활용됩니다.

장점

  • 흐름이 코드 한곳에 명시 - 가독성/디버깅 우수
  • 보상 시나리오 정의 용이

단점

  • 오케스트레이터에 로직 집중 - 단일 책임 비대화 위험
  • 추가 서비스 운영 부담

보상 트랜잭션 설계 원칙

  1. 의미적 되돌리기: 물리적 롤백이 아니라 반대 행위로 표현 (환불, 취소)
  2. 멱등성 보장: 같은 보상이 여러 번 호출돼도 안전
  3. 항상 성공해야 함: 보상 실패 시를 위한 재시도/알림 체계 필요
  4. 데이터 흔적 보존: 단순 삭제보다 상태 변경이 안전

멱등성 키 활용

Choreography vs Orchestration 비교

항목 Choreography Orchestration
결합도 낮음 중간
가시성 낮음 높음
복잡도 (소규모) 단순 복잡
복잡도 (대규모) 매우 복잡 관리 가능
적합 사례 2~3개 서비스 4개 이상 서비스

마무리

Saga 패턴은 분산 환경에서 결과적 일관성을 받아들이고, 비즈니스 의미의 보상으로 실패를 흡수하는 현실적인 접근입니다. 단순한 흐름은 Choreography로, 복잡한 워크플로는 Orchestration으로 시작하되, 멱등성·관측성·재시도 정책을 처음부터 함께 설계하는 것이 안정적인 운영의 비결입니다.
