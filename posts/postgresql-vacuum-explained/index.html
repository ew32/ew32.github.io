PostgreSQL VACUUM 완전 정복 - MVCC와 autovacuum의 모든 것

작성일 2026-04-13 Edited on 2026-05-20 In Backend , Database
PostgreSQL의 MVCC 구조와 VACUUM의 동작 원리, autovacuum 튜닝, 그리고 dead tuple과 bloat 문제 해결 방법을 실무 관점에서 설명합니다.

PostgreSQL과 MVCC

PostgreSQL은 MVCC(Multi-Version Concurrency Control) 를 통해 동시성을 제어합니다. MVCC에서는 데이터 행(row)을 수정하거나 삭제해도 즉시 제거하지 않고, 새로운 버전의 튜플을 생성하여 트랜잭션 간 격리를 보장합니다. 이로 인해 읽기와 쓰기가 서로를 차단하지 않는 장점이 있지만, dead tuple 이라는 부산물이 발생합니다.

이 dead tuple을 정리하지 않으면 테이블이 비대해지고(bloat), 성능이 저하되며, 결국 트랜잭션 ID 고갈이라는 치명적인 문제까지 발생할 수 있습니다.

VACUUM이란?

VACUUM은 dead tuple을 회수하고 공간을 재사용 가능하게 만드는 PostgreSQL의 핵심 유지보수 작업입니다. VACUUM의 주요 역할은 다음과 같습니다.

  1. Dead tuple 회수: 더 이상 어떤 트랜잭션도 볼 수 없는 튜플을 제거
  2. Visibility Map 갱신: 인덱스 전용 스캔(Index Only Scan)을 가능하게 함
  3. Free Space Map 갱신: 새 데이터 삽입 위치 결정에 활용
  4. Transaction ID Wraparound 방지: XID 고갈 방지를 위한 freezing 수행
  5. 통계 정보 수집(VACUUM ANALYZE): 옵티마이저가 사용할 통계 정보 갱신

VACUUM의 종류

1. VACUUM (lazy vacuum)

기본 VACUUM은 dead tuple을 회수하지만 디스크 공간을 OS로 반환하지는 않습니다. 즉, 테이블 크기는 줄어들지 않고 내부 공간만 재사용됩니다.

2. VACUUM FULL

테이블 전체를 다시 쓰면서 dead tuple을 완전히 제거하고 디스크 공간을 회수합니다. 하지만 ACCESS EXCLUSIVE 락을 잡기 때문에 운영 중 사용에 주의가 필요합니다.

운영 환경에서는 pg_repack 같은 확장을 사용하면 락 없이 테이블 재구성이 가능합니다.

3. VACUUM ANALYZE

VACUUM과 함께 통계 정보까지 수집합니다.

Autovacuum의 동작

PostgreSQL은 기본적으로 autovacuum을 활성화하여 자동으로 VACUUM과 ANALYZE를 수행합니다. autovacuum launcher가 워커 프로세스를 띄워 테이블별로 임계값에 도달했는지 검사합니다.

Dead Tuple과 Bloat 모니터링

dead_ratio가 0.1(10%)을 넘는 큰 테이블은 튜닝 대상이 됩니다.

Transaction ID Wraparound 문제

PostgreSQL의 트랜잭션 ID는 32비트(약 40억)로 제한되어 있으며, 순환(wraparound) 구조입니다. 이를 방지하기 위해 VACUUM은 오래된 튜플의 XID를 frozen 상태로 표시합니다.

autovacuum_freeze_max_age(기본 2억)에 도달하면 강제로 anti-wraparound VACUUM이 실행되며, 이때는 사용자가 VACUUM 명령으로도 막을 수 없습니다.

흔한 문제와 해결

  • Long-running 트랜잭션: 오래 열린 트랜잭션은 VACUUM이 dead tuple을 제거하지 못하게 함. pg_stat_activity에서 추적
  • Replication slot 미사용: 잊혀진 슬롯이 WAL을 잡아두면 비슷한 문제 발생
  • bloat 누적: pgstattuple 확장으로 정확한 bloat 측정

VACUUM이 안 되는 상황 디버깅

다음 항목들을 순서대로 점검합니다.

  1. 오래된 트랜잭션이 있는지 (pg_stat_activity)
  2. 사용되지 않는 replication slot (pg_replication_slots)
  3. prepared transaction (pg_prepared_xacts)
  4. hot_standby_feedback 옵션이 켜진 standby

마무리

VACUUM은 PostgreSQL의 MVCC를 지탱하는 필수 도구입니다. autovacuum의 동작을 이해하고 모니터링 지표를 꾸준히 살피며, 테이블 특성에 맞는 튜닝을 적용하면 bloat과 XID wraparound 같은 문제를 사전에 막을 수 있습니다. “PostgreSQL은 VACUUM을 자주 해야 행복하다”는 격언을 기억합시다.
