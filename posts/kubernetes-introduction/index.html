Kubernetes 입문 - 핵심 개념 정리

작성일 2026-05-25 Edited on 2026-05-18 In Cloud , Infrastructure
컨테이너 오케스트레이션의 표준인 Kubernetes의 핵심 개념과 동작 원리를 정리합니다.

들어가며

Docker를 이용한 컨테이너화는 이제 표준이 되었지만, 수십~수백 개의 컨테이너를 운영 환경에서 관리하려면 별도의 도구가 필요합니다. Kubernetes(K8s) 는 컨테이너 오케스트레이션의 사실상 표준으로, 컨테이너의 배포, 확장, 복구, 네트워킹을 자동화해 줍니다. 이 글에서는 Kubernetes의 핵심 구성 요소와 개념을 정리합니다.

Kubernetes란?

구글이 2014년 오픈소스로 공개한 컨테이너 관리 플랫폼입니다. 핵심 기능은 다음과 같습니다.

  • 자동 배포 및 롤백: 선언적 설정으로 원하는 상태 유지
  • 셀프 힐링: 실패한 컨테이너 자동 재시작
  • 수평 확장: 트래픽 증가 시 컨테이너 자동 추가
  • 서비스 디스커버리 및 로드 밸런싱
  • 시크릿 및 구성 관리

클러스터 아키텍처

Kubernetes는 컨트롤 플레인(Control Plane) 과 워커 노드(Worker Node) 로 구성됩니다.

컨트롤 플레인 구성 요소

  • kube-apiserver: 모든 요청의 진입점, REST API 제공
  • etcd: 클러스터 상태를 저장하는 분산 키-값 저장소
  • kube-scheduler: 새 파드를 어떤 노드에 배치할지 결정
  • kube-controller-manager: 다양한 컨트롤러 실행 (Deployment, ReplicaSet 등)

워커 노드 구성 요소

  • kubelet: 노드 위에서 파드를 실행하고 상태 보고
  • kube-proxy: 노드의 네트워크 라우팅 관리
  • 컨테이너 런타임: containerd, CRI-O 등

핵심 오브젝트

Pod

Kubernetes의 최소 배포 단위로, 하나 이상의 컨테이너가 함께 묶여 실행됩니다. 같은 파드 내 컨테이너는 네트워크와 스토리지를 공유합니다.

Deployment

파드의 선언적 관리, 롤링 업데이트, 롤백을 제공합니다. 실무에서는 파드를 직접 생성하기보다 Deployment를 사용합니다.

Service

파드는 동적으로 생성/삭제되며 IP가 바뀝니다. Service는 파드 그룹에 안정적인 진입점을 제공합니다.

서비스 타입은 다음과 같습니다.

타입 설명
ClusterIP 클러스터 내부 IP (기본값)
NodePort 노드의 특정 포트로 외부 노출
LoadBalancer 클라우드 로드밸런서 연결
ExternalName 외부 DNS 이름 매핑

ConfigMap & Secret

구성과 비밀 정보를 컨테이너 이미지와 분리하는 방법입니다.

Ingress

HTTP/HTTPS 트래픽을 클러스터 내부 서비스로 라우팅합니다.

kubectl 자주 쓰는 명령어

선언적 vs 명령형

Kubernetes의 핵심 철학은 선언적(Declarative) 관리입니다. 사용자는 “원하는 상태”를 YAML로 기술하고, Kubernetes는 실제 상태를 그 상태로 맞춰갑니다.

  • 명령형: kubectl run nginx --image=nginx
  • 선언적: kubectl apply -f nginx.yaml

실무에서는 선언적 방식을 사용해 Git에서 버전 관리(GitOps)하는 것이 표준입니다.

운영 시 알아두면 좋은 개념

  • Namespace: 클러스터 내 가상 분리 단위
  • HPA(Horizontal Pod Autoscaler): CPU/메모리 기반 자동 스케일
  • Liveness/Readiness Probe: 헬스체크
  • Resource Requests/Limits: 리소스 보장과 제한
  • PV/PVC: 영구 스토리지 관리

마무리

Kubernetes는 학습 곡선이 가파르지만, 한 번 익히면 컨테이너 운영의 강력한 무기가 됩니다. Pod, Deployment, Service, Ingress 같은 기본 오브젝트와 선언적 관리 철학을 먼저 익히고, 점차 헬스체크, 오토스케일링, 스토리지, 보안으로 확장해 나가는 것을 추천합니다. AWS EKS, GCP GKE, Azure AKS 같은 매니지드 서비스를 사용하면 컨트롤 플레인 관리 부담도 줄일 수 있습니다.
