Java Virtual Threads(Project Loom) 실전 가이드

작성일 2026-04-04 Edited on 2026-07-20 In Backend , Java
Java 21에서 정식화된 Virtual Threads의 동작 원리, 플랫폼 스레드와의 차이, 캐리어 스레드와 핀(pinning) 이슈, 실제 사용 시 주의점을 정리합니다.

왜 Virtual Threads인가

전통적인 자바 스레드(플랫폼 스레드)는 OS 스레드와 1:1로 매핑됩니다. JVM 스레드 하나를 만들 때마다 OS 자원 하나가 소비되고, 스택 메모리(보통 ~1MB)가 할당됩니다. 결과적으로 한 프로세스에서 수천 개를 띄우는 것이 한계입니다.

웹 서버는 보통 “요청당 스레드” 모델로 동작합니다. 한 요청을 처리하는 동안 DB·외부 API I/O 대기로 스레드가 블록되면 그 자원은 낭비되고, 동시에 처리할 수 있는 요청 수가 제한됩니다. 이 문제를 회피하려고 리액티브 프로그래밍, CompletableFuture, async/await 같은 비동기 모델이 등장했지만 코드가 복잡해지고 디버깅이 어려워지는 단점이 있습니다.

Project Loom이 제안한 Virtual Thread는 이 두 세계의 장점을 합치려 합니다. 코드는 동기적으로 짜되, 스레드 자체를 매우 가볍게 만들어 수십~수백만 개를 띄울 수 있게 하는 것입니다. Java 21에서 정식 기능이 되었습니다.

기본 사용법

가상 스레드를 만드는 방법은 여러 가지가 있습니다.

핵심은 Executors.newVirtualThreadPerTaskExecutor()입니다. 요청당 가상 스레드를 만들어 동시 작업을 손쉽게 처리할 수 있습니다.

플랫폼 스레드 vs 가상 스레드

항목 플랫폼 스레드 가상 스레드
OS 자원 OS 스레드 1:1 매핑 캐리어 스레드 풀에서 다중화
메모리 수백 KB ~ 수 MB 스택 수백 바이트 ~ 수 KB
생성 비용 비쌈 (풀링 필요) 매우 저렴 (풀링 불필요)
적합한 작업 CPU 집약적 I/O 집약적

가상 스레드는 내부적으로 ForkJoinPool 기반의 캐리어 스레드 위에서 실행됩니다. 블로킹 I/O를 만나면 JVM이 가상 스레드를 캐리어에서 떼어내(unmount) 다른 가상 스레드를 실행하고, I/O가 끝나면 다시 캐리어에 붙여(mount) 재개합니다. 이를 continuations 기반 스케줄링이라고 합니다.

스레드 풀을 버려라

가상 스레드에서는 스레드 풀이 안티 패턴입니다. 가상 스레드는 충분히 가볍기 때문에 작업마다 새로 만드는 것이 정답입니다.

자원 사용량을 제한하려면 스레드 풀 대신 세마포어를 사용하세요.

핀(Pinning) 이슈

가상 스레드는 두 가지 경우에 캐리어 스레드에 고정(pinned) 되어 다른 가상 스레드에 양보하지 못합니다.

  1. synchronized 블록 내부에서 블로킹 작업을 수행할 때
  2. 네이티브 메서드(JNI) 안에 있을 때

핀이 잦으면 가상 스레드의 이점을 잃습니다. 다행히 Java 24부터는 synchronized 블록 안에서도 unmount가 가능해지는 방향으로 개선이 이뤄지고 있습니다. 그 전 버전에서는 가능하면 **synchronized 대신 ReentrantLock**을 사용하세요.

핀 발생 여부는 -Djdk.tracePinnedThreads=full 옵션으로 추적할 수 있습니다.

ThreadLocal 주의

가상 스레드는 매우 많이 생성되므로 무거운 ThreadLocal을 그대로 쓰면 메모리 문제가 생깁니다. JEP 446에서 제안된 ScopedValue(미리보기)가 더 적합한 대안이 될 수 있습니다.

어떤 워크로드에 적합한가

  • 잘 맞음: 외부 API 다수 호출, DB I/O 많은 웹 서비스, 마이크로서비스 게이트웨이
  • 별 효과 없음: 순수 CPU 연산, 짧은 메모리 작업
  • 부적합: synchronized와 JNI에 깊이 얽힌 레거시 코드

Spring Boot 3.2부터는 spring.threads.virtual.enabled=true 한 줄로 Tomcat 워커, @Async, 스케줄러 등이 가상 스레드 위에서 동작하게 할 수 있습니다.

마무리

Virtual Threads는 자바 동시성 모델의 패러다임을 바꾸는 기능입니다. 비동기·리액티브의 복잡함 없이 단순한 동기 코드로 높은 동시성을 달성할 수 있습니다. 다만 핀 이슈와 ThreadLocal 관리, 자원 제한(세마포어) 같은 새로운 규칙을 익혀야 합니다. I/O 바운드 워크로드를 가진 서비스라면 가장 먼저 검토할 가치가 있는 기술입니다.
