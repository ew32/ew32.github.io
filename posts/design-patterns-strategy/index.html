Strategy 패턴으로 알고리즘 깔끔하게 교체하기

작성일 2026-05-15 Edited on 2026-07-20 In Backend
알고리즘 군을 캡슐화하고 런타임에 교체 가능하게 만드는 Strategy 패턴의 구조와 실전 예제를 살펴봅니다.

Strategy 패턴이란?

Strategy 패턴은 동일한 목적을 가진 알고리즘들을 인터페이스로 추상화하고, 런타임에 구체 알고리즘을 선택해 사용할 수 있게 해주는 행동 패턴입니다. if-else 또는 switch 분기로 흩어져 있는 알고리즘을 클래스로 분리해, 변경에 닫혀 있고 확장에 열려 있는(OCP) 설계를 만듭니다.

GoF의 정의: “알고리즘 군을 정의하고 각각을 캡슐화하여 교환 가능하게 만든다.”

언제 사용하는가

  • 동일한 작업을 여러 방식(정렬, 결제, 압축, 인증 등)으로 수행해야 할 때
  • 조건문이 비대해지면서 새 케이스 추가가 부담스러울 때
  • 알고리즘을 런타임에 동적으로 교체해야 할 때

안티 패턴: 거대한 if-else

새 결제 수단이 추가될 때마다 클래스를 수정해야 하고, 단위 테스트도 어렵습니다.

Strategy 패턴 적용

새 결제 수단은 PaymentStrategy 구현체만 추가하면 됩니다.

Spring에서의 Strategy

Spring DI와 결합하면 더 우아해집니다.

Strategy vs State

두 패턴 구조는 거의 같지만 의도가 다릅니다.

  • Strategy: 클라이언트가 알고리즘을 선택
  • State: 객체 내부 상태에 따라 동작이 자동으로 바뀜

함수형 언어에서의 Strategy

Java 8 람다 등장 이후 인터페이스 구현체를 만들지 않고도 함수로 전달할 수 있습니다.

마무리

Strategy 패턴은 가장 자주 쓰이고 효과적인 패턴 중 하나입니다. if-else가 비대해지고 있다면 가장 먼저 떠올려야 할 도구이며, Spring DI나 람다와 결합하면 더 간결한 코드를 얻을 수 있습니다. 단, 알고리즘이 정말 1~2개뿐이고 변경 가능성이 낮다면 굳이 도입하지 않아도 됩니다.
