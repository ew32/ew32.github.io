Android 16KB Page Size 대응이란? (Google Play Console 경고 해결)

최근 Google Play Console에서 다음과 같은 경고를 볼 수 있다.

Your app must support 16 KB memory page sizes

처음 보면 생소하지만, 이는 Android 운영체제의 메모리 관리 방식과 관련된 내용이다.

16KB Page Size란?

운영체제는 메모리를 한 번에 관리하지 않고 일정한 크기의 페이지(Page) 단위로 나누어 관리한다.

기존 Android 기기 대부분은 다음과 같이 4KB 페이지를 사용했다.

하지만 최신 ARM64 기반 Android 기기에서는 다음과 같이 16KB 페이지를 사용하는 경우가 등장하고 있다.

즉, 이번 이슈는 운영체제의 메모리 페이징(Page Size) 자체와 관련된 이야기이다.

왜 Google이 16KB 지원을 요구하는가?

16KB 페이지를 사용하면 다음과 같은 장점이 있다.

  • Page Table 크기 감소
  • TLB(Translation Lookaside Buffer) 효율 향상
  • 메모리 관리 비용 감소
  • 성능 및 전력 효율 향상

특히 대용량 메모리를 사용하는 최신 ARM64 기기에서 효과가 크다.

어떤 앱이 영향을 받을까?

주로 네이티브 코드(.so) 를 사용하는 앱이 영향을 받는다.

영향이 큰 경우

  • NDK 사용
  • C/C++ 라이브러리
  • OpenCV
  • FFmpeg
  • SQLCipher
  • TensorFlow Lite
  • Unity
  • Flutter 플러그인
  • React Native Native Module

대부분 영향이 없는 경우

  • 순수 Java 앱
  • 순수 Kotlin 앱
  • Retrofit
  • Room
  • Firebase

Java/Kotlin 코드는 ART(Android Runtime) 위에서 실행되므로 직접적인 영향을 거의 받지 않는다.

.so 파일이 왜 문제가 되는가?

네이티브 라이브러리(.so)는 실행 시 메모리에 직접 매핑(mmap)된다.

예전에는 대부분 4KB 기준으로 만들어졌다.

하지만 16KB 페이지를 사용하는 환경에서는 다음과 같이 16KB 정렬이 필요하다.

Google Play는 APK나 AAB 내부의 .so 파일을 검사하여 이를 확인한다.

결론

16KB Page Size 이슈는 운영체제의 메모리 페이지 크기가 4KB에서 16KB로 확대되는 흐름에 대응하기 위한 것이다.

“네이티브 라이브러리(.so)가 16KB 페이지를 사용하는 Android 환경에서도 정상적으로 메모리에 로딩될 수 있도록 빌드되어야 한다.”
