Tree Shaking과 번들 최적화 실전 가이드

작성일 2026-06-21 Edited on 2026-05-19 In Frontend
Tree Shaking의 원리, sideEffects 플래그, 번들 분석과 청크 분할까지 프론트엔드 번들을 가볍게 만드는 방법을 정리합니다.

Tree Shaking이란

Tree Shaking은 “사용되지 않는 코드를 번들에서 잘라내는” 정적 분석 기법입니다. 나무를 흔들면 죽은 잎이 떨어지는 모습에서 이름이 왔습니다. webpack 2가 도입했고 지금은 Vite, esbuild, Rollup, Parcel 등 거의 모든 번들러가 지원합니다.

핵심 전제는 ES Modules의 정적 import입니다. CommonJS의 require()는 런타임에 무엇이 임포트되는지가 결정되므로 정적 분석이 불가능합니다.

1. ESM 빌드를 우선 사용한다

라이브러리를 고를 때 package.json의 "module" 또는 "exports" 필드가 있는지를 확인하세요. 대표적으로 lodash 대신 lodash-es, date-fns, nanoid, zod처럼 ESM 친화적인 패키지가 좋습니다.

2. sideEffects 플래그를 정확히 설정한다

package.json에서 부수 효과가 없는 모듈을 번들러에 알려줄 수 있습니다.

CSS import처럼 부수 효과가 있는 파일이 있다면 배열로 지정합니다.

이 플래그가 없으면 번들러는 “혹시 모르니” 모든 모듈을 포함시킵니다.

3. Named Import만 사용한다

또한 라이브러리에서 default export 하나로 모든 기능을 묶어두는 패턴은 Tree Shaking을 무력화합니다. 가능하면 named export 위주로 작성하세요.

4. Babel/TypeScript 설정 점검

module을 CommonJS로 두면 TypeScript가 ESM을 CJS로 변환해 Tree Shaking이 사실상 사라집니다.

5. 번들 분석하기

무엇이 들어 있는지 모르면 줄일 수도 없습니다.

가장 자주 마주치는 문제는 다음과 같습니다.

  • moment.js + locale 전체 (~250KB)
  • lodash 전체 (~70KB)
  • 큰 SVG/PNG가 base64로 인라인됨
  • 같은 라이브러리의 ESM/CJS 두 버전이 동시에 들어옴

6. 코드 스플리팅(Code Splitting)

지연 로딩과 짝을 이루는 기법입니다. dynamic import를 만나면 번들러는 자동으로 청크를 분리합니다.

Vite에서는 build.rollupOptions.output.manualChunks로 벤더 청크 분리를 세밀하게 제어할 수 있습니다.

7. 압축과 폴리필

  • 서버에서 Brotli 압축을 켜면 gzip 대비 15~25% 추가 절감.
  • core-js/regenerator-runtime을 무차별 적용하지 말고 browserslist에 실제 타깃 브라우저만 명시.
  • 모던 ESM과 레거시 청크를 함께 빌드하는 모듈/노모듈 패턴을 활용하면 모던 브라우저에는 가벼운 코드만 전달됩니다.

측정 지표

지표 목표
초기 JS 전송 크기 (gzip) < 150KB
전체 JS 파싱·실행 시간 < 300ms (중급 모바일 기준)
라우트 청크 평균 < 50KB
LCP < 2.5s

실전 체크리스트

  • package.json에 "sideEffects" 명시
  • CommonJS-only 라이브러리는 대체재 검토
  • import * as, default-only export 패턴 제거
  • dynamic import로 라우트·무거운 모듈 분리
  • 번들 분석기로 정기 점검
  • Brotli/HTTP2 적용
