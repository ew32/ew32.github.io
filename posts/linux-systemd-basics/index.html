Linux systemd 기초 - 유닛 작성과 서비스 관리

작성일 2026-07-03 Edited on 2026-05-18 In System , OS
현대 리눅스의 init 시스템인 systemd의 구조와 유닛 파일 작성법, 자주 쓰는 systemctl/journalctl 명령을 정리합니다.

systemd란

systemd는 대부분의 현대 리눅스 배포판이 사용하는 init 시스템이자 서비스 매니저입니다. PID 1로 부팅 직후 실행되어 모든 사용자 공간 프로세스를 관리하고, 의존성 기반 병렬 부팅, 소켓·타이머 활성화, 로깅(journal), cgroup 자원 제어를 통합 제공합니다.

유닛(Unit) 종류

  • .service: 데몬·일반 프로세스.
  • .socket: 소켓 활성화.
  • .timer: 시간 기반 트리거(크론 대체).
  • .mount / .automount: 파일시스템 마운트.
  • .target: 다른 유닛의 묶음(런레벨 역할).

유닛 파일은 다음 경로에서 우선순위 순으로 검색됩니다.

기본 service 유닛

간단한 Node.js 애플리케이션을 데몬으로 등록하는 예시입니다.

Type 옵션

  • simple: ExecStart 프로세스가 곧 메인 프로세스.
  • forking: 전통적 데몬, 자식 프로세스로 분기.
  • oneshot: 한 번 실행 후 종료(스크립트성 작업).
  • notify: 프로세스가 준비 완료를 sd_notify로 알림.

명령어

로그 - journalctl

systemd는 표준 출력/에러를 자동 수집해 바이너리 저널에 저장합니다.

Timer 유닛 - cron 대체

systemd timer는 cron과 달리 로깅, 의존성, 누락된 실행 보정(Persistent) 등 강력한 기능을 제공합니다.

자원 제한과 보안

systemd는 cgroup으로 CPU·메모리·IO를 제한할 수 있습니다.

샌드박싱 옵션(ProtectSystem, ProtectHome, PrivateTmp, CapabilityBoundingSet 등)을 활용해 서비스 권한을 최소화하면 보안이 크게 강화됩니다.

마무리

systemd는 init 이상의 통합 운영 기반입니다. 서비스 자동 재시작, 타이머, 로그 통합, cgroup 제어를 적극 활용하면 별도 프로세스 매니저 없이도 견고한 운영 환경을 구성할 수 있습니다.
