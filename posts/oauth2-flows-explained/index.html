OAuth 2.0 인증 흐름 완벽 정리

작성일 2026-05-12 Edited on 2026-05-20 In Backend , Authentication
OAuth 2.0의 4가지 표준 인증 흐름과 OIDC, PKCE 확장을 정리하고 각 상황에 맞는 선택 기준을 알아봅니다.

OAuth 2.0이란?

OAuth 2.0은 사용자가 자신의 비밀번호를 제3자 애플리케이션에 직접 노출하지 않고도, 특정 리소스(이메일, 프로필, 파일 등)에 대한 위임된 접근 권한을 부여할 수 있게 해주는 표준 프로토콜입니다. RFC 6749로 명세화되어 있으며, 오늘날 거의 모든 SaaS 플랫폼이 이를 채택하고 있습니다.

OAuth 2.0의 핵심 아이디어는 단순합니다. “비밀번호를 공유하지 말고, 권한이 제한된 Access Token을 발급하라.”

주요 역할(Role)

역할 설명
Resource Owner 보호된 리소스의 소유자(보통 최종 사용자)
Client 리소스에 접근하려는 애플리케이션
Authorization Server 인증 후 토큰을 발급하는 서버
Resource Server 보호된 자원을 호스팅하는 API 서버

이 4개의 주체가 메시지를 주고받는 순서를 정의한 것이 곧 OAuth 2.0 Flow입니다.

Authorization Code Flow

가장 표준적이고 안전한 흐름으로, 서버 사이드 웹 애플리케이션에서 사용합니다.

code를 한 번 더 토큰으로 교환하는 단계가 있기 때문에, 토큰이 브라우저 URL에 노출되지 않습니다.

PKCE 확장

**PKCE(Proof Key for Code Exchange)**는 모바일/SPA처럼 client_secret을 안전하게 보관할 수 없는 환경을 위해 RFC 7636에서 정의된 확장 사양입니다.

Authorize 단계에 code_challenge를 보내두고, Token 교환 시 code_verifier를 검증함으로써 **인가 코드 탈취 공격(Authorization Code Interception)**을 막습니다. 현재 OAuth 2.1 초안에서는 모든 Public Client에 PKCE를 필수로 권고합니다.

Client Credentials Flow

사용자가 개입하지 않는 서버 간(M2M) 통신에서 사용합니다.

Cron 작업, 백엔드 → 백엔드 호출, CI/CD 토큰 발급 등에 적합합니다.

Device Authorization Flow

TV, IoT 디바이스처럼 키보드 입력이 어려운 환경을 위한 흐름입니다. 디바이스가 user_code를 화면에 표시하면, 사용자는 다른 기기(스마트폰)에서 해당 코드를 입력해 인증을 완료합니다. Google TV, AWS CLI 로그인에서 흔히 볼 수 있습니다.

Implicit Flow는 더 이상 권장되지 않는다

과거 SPA에서 사용되던 Implicit Flow는 Access Token이 URL Fragment로 직접 노출되어 보안에 취약합니다. 현재 IETF는 Authorization Code + PKCE로 대체할 것을 권고합니다.

OpenID Connect(OIDC)

OAuth 2.0이 인가(Authorization) 프로토콜이라면, OIDC는 그 위에 인증(Authentication) 계층을 얹은 것입니다. 핵심 차이는 id_token이라는 JWT가 추가되어, 사용자의 신원 정보를 표준 클레임(sub, iss, aud, exp)으로 전달한다는 점입니다.

Best Practices

  • Authorization Code + PKCE를 기본으로 선택하세요. SPA, Mobile, Server 모두 적용 가능합니다.
  • state 파라미터로 CSRF 방어를 반드시 수행하세요.
  • redirect_uri는 화이트리스트로 엄격히 검증하고 와일드카드 사용을 피하세요.
  • Access Token은 짧게(5~30분), Refresh Token은 회전(Rotation) 정책으로 관리하세요.
  • 토큰을 localStorage에 저장하면 XSS에 노출됩니다. SPA에서는 HttpOnly 쿠키 + BFF 패턴을 고려하세요.
  • scope는 최소 권한 원칙으로 좁게 요청하세요.

마무리

OAuth 2.0은 단순한 로그인 도구가 아니라, 권한 위임을 위한 프로토콜의 표준 언어입니다. 흐름의 명칭만 외우기보다 각 흐름이 등장한 위협 모델을 이해해야 올바른 선택을 할 수 있습니다. 오늘날 새로운 서비스를 설계한다면 거의 모든 경우에 Authorization Code + PKCE + OIDC의 조합이 출발점이 될 것입니다.
