WebSocket vs SSE vs Polling - 실시간 전송 기법 비교

작성일 2026-06-24 Edited on 2026-06-15 In Network , HTTP
폴링, 롱폴링, SSE, WebSocket의 동작 원리와 선택 기준을 실전 관점에서 비교합니다.

실시간 통신, 무엇으로 구현할까

웹에서 “실시간”을 만들 수 있는 길은 사실 네 가지뿐입니다. 폴링(Polling), 롱폴링(Long Polling), Server-Sent Events(SSE), WebSocket. 각각 등장한 시기와 동작 원리가 다르며, 어떤 것을 고르느냐에 따라 인프라 비용과 사용자 경험이 모두 달라집니다.

네 가지 방식의 동작

1. Polling

클라이언트가 일정 주기로 요청을 보냅니다.

가장 단순하지만 사건이 없어도 요청이 계속 나갑니다. 평균 지연은 폴링 주기의 절반입니다.

2. Long Polling

서버가 변화가 생길 때까지 응답을 보류합니다.

지연은 거의 0에 가깝지만, 서버가 다수의 열린 연결을 메모리에 들고 있어야 합니다.

3. SSE

HTTP 위에서 서버가 일방적으로 메시지를 흘려보냅니다. 클라이언트는 EventSource로 듣기만 합니다.

자동 재연결, Last-Event-ID로 누락 보완 같은 기능이 표준에 포함되어 있습니다.

4. WebSocket

HTTP로 핸드셰이크 후 같은 TCP 연결을 풀-듀플렉스 채널로 업그레이드합니다.

양방향, 저지연, 바이너리 지원 등 가장 강력하지만 직접 챙겨야 하는 운영 항목도 많습니다.

한눈에 비교

방향 C → S C ↔ S (반복) S → C C ↔ S

바이너리 가능 가능 불가 가능
자동 재연결 자연스러움 자연스러움 내장 직접 구현

서버 부하 낮음(비효율) 중간 낮음 낮음

구현 난이도 가장 쉬움 쉬움 쉬움 중간

사례별 추천

  • 알림 배지, 진행률 표시, 주식 시세, 로그 라이브 뷰 → SSE.
  • 채팅, 협업 도구, 게임, 화이트보드 → WebSocket.
  • 상태가 자주 바뀌지 않고 외부 시스템 통합이 단순해야 함 → Polling.
  • AI 토큰 스트리밍 → SSE가 사실상 표준.
  • 모바일 푸시 → 위 어느 것도 아닌 OS 푸시 채널(FCM/APNs).

운영 고려사항

수평 확장과 메시지 라우팅

WebSocket/SSE 모두 연결이 특정 인스턴스에 묶입니다. 메시지를 전 인스턴스에 뿌리려면 Redis Pub/Sub, NATS, Kafka 같은 메시지 버스가 필요합니다.

하트비트와 idle 종료

대부분의 로드 밸런서는 60초 정도의 idle timeout을 가집니다. WebSocket은 ping/pong, SSE는 주기적 주석 라인(: ping\n\n)으로 연결을 유지하세요.

인증

WebSocket은 핸드셰이크 시점에만 헤더를 검사합니다. 토큰을 쿼리에 담아 보내거나(?token=...) 첫 메시지로 인증을 마치는 패턴이 일반적입니다. SSE는 쿠키 기반 인증이 가장 단순합니다.

CORS와 프록시

SSE/Polling은 일반 HTTP CORS 규칙을 따릅니다. WebSocket은 CORS가 아닌 Origin 헤더 검증으로 처리하며, Nginx 프록시 뒤에 둘 때는 Upgrade, Connection 헤더 통과 설정을 반드시 추가해야 합니다.

의사 결정 트리

  1. 양방향 통신이 꼭 필요한가? → 예: WebSocket, 아니오: 다음 단계.
  2. 서버에서 클라이언트로 푸시가 필요한가? → 예: SSE, 아니오: 다음 단계.
  3. 사건이 자주 발생하고 지연이 중요한가? → 예: Long Polling, 아니오: Polling.

마무리

“실시간이라면 무조건 WebSocket”이라는 신화는 이제 통하지 않습니다. 단방향 알림에는 SSE, 단순 조회에는 Polling이 더 잘 어울리는 경우가 많습니다. 요구 사항을 먼저 정리하고 가장 단순한 방식부터 검토하세요.
