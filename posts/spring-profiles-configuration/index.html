Spring Profiles와 외부 설정 - 환경별 application.yml 분리 전략

작성일 2026-04-11 Edited on 2026-05-20 In Backend , Java
Spring Boot의 Profiles, application.yml 환경별 분리, @Profile, 환경변수와 프로퍼티 우선순위, 비밀값 관리까지 실전 가이드로 정리합니다.

왜 환경별 설정 분리가 필요한가

같은 애플리케이션 코드를 로컬, 개발 서버, 스테이징, 운영에서 돌릴 때 DB URL·외부 API 키·로그 레벨·기능 토글이 모두 달라집니다. 코드에서 환경을 분기하는 방식은 곧 한계에 부딪힙니다. Spring Boot는 Profiles와 외부 설정(Externalized Configuration) 두 가지 메커니즘으로 이를 깔끔하게 해결합니다.

Profiles 기본

Profile은 단순히 “이름이 붙은 설정 묶음”입니다. 활성 Profile은 다음 중 하나로 지정합니다.

여러 개를 콤마로 결합할 수도 있습니다: --spring.profiles.active=prod,monitoring.

application.yml 파일 분리

전통적인 방법은 파일을 나누는 것입니다.

application.yml에 공통값을 두고, 활성 Profile에 해당하는 파일이 덮어쓰기 방식으로 합쳐집니다.

단일 파일에서 멀티 도큐먼트로 분리

YAML의 --- 구분자로 한 파일에 여러 Profile 설정을 작성할 수도 있습니다.

작은 프로젝트나 라이브러리 모듈에서 유용합니다.

@Profile로 빈 활성화

특정 Profile에서만 빈을 등록하고 싶다면 @Profile을 씁니다.

!prod, prod & monitoring, prod | staging 같은 표현식도 가능합니다.

설정 값 주입: @Value와 @ConfigurationProperties

@Value

간단한 값 한두 개를 주입할 때 사용합니다.

@ConfigurationProperties (권장)

여러 속성을 하나의 객체로 묶어서 받습니다. 타입 안전성, 자동완성, 검증을 제공합니다.

JSR-303 검증도 적용할 수 있습니다.

프로퍼티 우선순위

같은 키가 여러 곳에 정의되면 Spring Boot가 명시된 우선순위로 해석합니다(상위가 우선).

  1. 명령행 인자 (--key=value)
  2. SPRING_APPLICATION_JSON
  3. JNDI / 서블릿 컨텍스트
  4. OS 환경 변수
  5. Java 시스템 프로퍼티 (-Dkey=value)
  6. 외부 application-{profile}.yml
  7. 외부 application.yml
  8. 패키지 내부 application-{profile}.yml
  9. 패키지 내부 application.yml
  10. @PropertySource
  11. 기본값

운영 환경에서는 환경 변수 또는 명령행 인자로 비밀값을 주입하고, 코드 저장소에는 비밀값을 절대 두지 않는 것이 원칙입니다.

환경 변수 매핑 규칙

mail.smtp.host 같은 프로퍼티는 환경 변수에서 MAIL_SMTP_HOST로 자동 매핑됩니다.

Spring Boot는 _를 .로, 대문자를 소문자로 매핑해 줍니다. 도커 컨테이너와 쿠버네티스 운영에서 가장 흔히 쓰는 방식입니다.

${} 변수 치환과 기본값

YAML 안에서 다른 프로퍼티를 참조하거나 환경 변수의 기본값을 지정할 수 있습니다.

비밀값(Secret) 관리

평문으로 비밀값을 저장하면 안 됩니다. 다음 방법을 권장합니다.

  • 환경 변수 주입 (가장 단순)
  • Kubernetes Secret + 환경 변수
  • HashiCorp Vault + Spring Cloud Vault
  • AWS Secrets Manager / Parameter Store + Spring Cloud AWS

테스트에서의 Profile

@ActiveProfiles로 테스트마다 활성 Profile을 지정할 수 있습니다.

application-test.yml을 두면 테스트 전용 설정이 자동으로 로드됩니다.

흔히 하는 실수와 팁

  • 프로파일 충돌: 여러 profile에 같은 키가 있으면 마지막에 활성화된 profile이 이긴다는 사실을 기억하세요.
  • 운영 설정을 패키지 내부에 두지 말 것: 비밀값이 빌드 아티팩트에 박혀 유출 위험.
  • default profile: 활성 Profile이 없으면 default profile이 동작합니다. 의도치 않게 default profile에 운영 설정이 들어가지 않도록 주의.
  • Spring Boot 2.4+의 변화: spring.profiles(deprecated) 대신 spring.config.activate.on-profile을 사용해야 합니다.
  • @ConfigurationProperties를 적극 사용해 타입 안전한 설정을 유지하세요.

마무리

Profiles와 외부 설정은 Spring Boot의 가장 큰 장점 중 하나입니다. 코드 한 줄 바꾸지 않고 동일 바이너리를 모든 환경에서 돌릴 수 있는 것은 운영의 큰 무기입니다. 핵심 원칙은 세 가지입니다 — 공통은 application.yml에, 환경 차이는 profile 파일이나 환경 변수로, 비밀은 절대 저장소에 두지 말 것.
