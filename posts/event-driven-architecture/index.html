이벤트 기반 아키텍처(EDA) 이해

작성일 2026-05-15 Edited on 2026-07-20 In Backend , Infrastructure
이벤트 기반 아키텍처의 개념, 구성 요소, Kafka 기반 구현 예시와 장단점을 살펴봅니다.

이벤트 기반 아키텍처란?

이벤트 기반 아키텍처(Event-Driven Architecture, EDA) 는 시스템 구성 요소들이 직접 호출이 아닌 이벤트(Event) 라는 비동기 메시지로 상호작용하는 아키텍처입니다. 이벤트는 *”이미 발생한 사실”*을 표현하며, 발행자(Producer)는 누가 소비할지 알 필요가 없습니다.

EDA는 마이크로서비스, 실시간 데이터 파이프라인, IoT, 알림 시스템 등 다양한 영역에서 활용됩니다.

핵심 구성 요소

구성 요소 설명
Event Producer 이벤트를 생성·발행하는 주체
Event Broker 이벤트 라우팅·저장 (Kafka, RabbitMQ, AWS SNS)
Event Consumer 이벤트를 구독·처리하는 주체
Event Schema 이벤트 페이로드의 구조 (Avro, JSON Schema)

이벤트 모델링

이벤트는 보통 다음 정보를 포함합니다.

이벤트 이름은 과거 시제 동사로 작성하는 것이 관례입니다 (OrderCreated, PaymentCompleted).

주요 패턴

1. Publish/Subscribe

여러 구독자가 동일한 이벤트를 받아 각자 다른 작업을 수행하는 기본 패턴입니다.

2. Event Sourcing

상태가 아닌 이벤트의 시퀀스를 저장소로 삼는 방식입니다. 현재 상태는 이벤트를 재생(replay)하여 도출합니다.

3. CQRS와의 결합

명령(Command)은 이벤트로 기록하고, 조회(Query)는 별도 Read Model에서 처리하는 구조와 결합되곤 합니다.

4. Outbox 패턴

DB 트랜잭션과 메시지 발행의 원자성을 보장하기 위해, 이벤트를 outbox 테이블에 함께 저장한 뒤 별도 프로세스가 발행합니다.

장점

  • 느슨한 결합: Producer와 Consumer는 서로 알지 못함
  • 확장성: 컨슈머를 독립적으로 추가/스케일아웃
  • 회복력: 일시적 장애가 발생해도 메시지 큐가 흡수
  • 감사 추적: 이벤트 로그가 곧 시스템 히스토리

단점과 주의사항

  • 순서 보장이 까다로움 (파티션 키 설계 필요)
  • 중복 메시지 처리 - 멱등성(idempotency) 설계 필수
  • 분산 트랜잭션 - 강한 일관성 대신 결과적 일관성
  • 디버깅 난이도 증가 - 분산 트레이싱 필수

EDA는 대규모 분산 시스템에서 결합도를 낮추고 확장성을 확보하는 데 강력한 패러다임입니다. 다만 동기 호출에 익숙한 팀이라면 이벤트 스키마 관리, 멱등성, 순서 보장, 관측 가능성 같은 새로운 고민에 대비해야 합니다.
