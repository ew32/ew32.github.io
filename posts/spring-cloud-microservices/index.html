Spring Cloud로 시작하는 마이크로서비스 - Eureka, Config Server, Gateway

작성일 2026-04-09 Edited on 2026-05-26 In Backend , Java
Spring Cloud의 핵심 구성요소인 Eureka 서비스 디스커버리, Config Server 중앙 설정, Spring Cloud Gateway 라우팅을 실전 구성과 함께 살펴봅니다.

왜 Spring Cloud인가

마이크로서비스 아키텍처로 가면 모놀리식에서는 자연스럽던 것들이 모두 인프라 문제가 됩니다.

  • 서비스가 다른 서비스의 주소를 어떻게 알까? → 서비스 디스커버리
  • 설정값(DB 비밀번호, 외부 API 키)을 어떻게 일관되게 관리할까? → 중앙 설정 관리
  • 외부 클라이언트는 수십 개의 서비스를 어떻게 호출할까? → API Gateway
  • 장애가 전파되지 않게 하려면? → 서킷 브레이커

Spring Cloud는 이런 분산 시스템 패턴들을 Spring Boot 위에 표준화해 제공합니다. 핵심 모듈만 살펴봅시다.

서비스 디스커버리: Eureka

Netflix Eureka는 가장 널리 쓰이는 디스커버리 서버 중 하나입니다. 각 서비스 인스턴스가 시작 시 자신을 등록하고, 다른 서비스는 서비스 이름으로 조회합니다.

Eureka 서버

Eureka 클라이언트

이렇게만 하면 order-service라는 이름으로 등록되고, 다른 서비스가 http://order-service/...로 호출할 수 있습니다. RestTemplate이나 WebClient에 @LoadBalanced를 붙이면 자동으로 인스턴스 중 하나를 선택해 호출합니다.

중앙 설정 관리: Config Server

수십 개의 서비스가 환경(개발/스테이징/운영)마다 서로 다른 설정을 갖는데 이를 각자 파일로 관리하면 운영이 지옥이 됩니다. Spring Cloud Config Server는 Git 저장소를 백엔드로 두고 설정을 중앙에서 제공합니다.

Config Server

Git 저장소에는 order-service.yml, order-service-prod.yml처럼 파일을 둡니다.

클라이언트

서비스가 시작될 때 order-service-prod.yml을 자동으로 가져옵니다. @RefreshScope + /actuator/refresh 조합으로 재시작 없이 설정을 갱신하거나, Spring Cloud Bus로 모든 인스턴스에 한 번에 반영할 수도 있습니다.

API Gateway: Spring Cloud Gateway

외부 트래픽의 단일 진입점입니다. 라우팅, 인증, 속도 제한, 로드 밸런싱을 처리합니다.

lb://order-service는 Eureka에서 인스턴스를 찾아 로드밸런싱합니다. 자바 DSL로 라우트를 정의할 수도 있습니다.

Gateway는 WebFlux 기반이라 논블로킹이며 높은 동시성을 처리합니다.

서킷 브레이커: Resilience4j

장애 전파를 막기 위한 패턴입니다. Spring Cloud Circuit Breaker는 여러 구현(Resilience4j, Sentinel 등)을 추상화합니다.

설정으로 실패율 임계치, 슬라이딩 윈도우 크기, half-open 전환 시간 등을 정의합니다.

분산 추적: Micrometer Tracing

Spring Boot 3 이후에는 Sleuth가 Micrometer Tracing으로 대체되었습니다. 요청 ID(traceId, spanId)가 자동으로 로그에 포함되고, Zipkin·Tempo·Jaeger 등으로 내보낼 수 있습니다. 마이크로서비스 운영에는 사실상 필수입니다.

구성 예시 아키텍처

도입할 때 고려사항

  • 운영 복잡도 증가: 디스커버리·Config·Gateway 등 인프라 서비스 자체도 HA로 운영해야 합니다.
  • 쿠버네티스 환경에서는 디스커버리를 K8s Service로, Config를 ConfigMap/Secret으로 대체하는 경우가 많습니다. Spring Cloud Kubernetes 모듈이 그 역할을 합니다.
  • 로컬 개발: Spring Boot devtools, profile 분리, Testcontainers 활용을 적극 권장.
  • 버전 매트릭스: Spring Cloud의 릴리스 트레인(예: 2024.0.x)이 Spring Boot 버전과 호환되는지 반드시 확인.

마무리

Spring Cloud는 마이크로서비스 인프라를 직접 만들지 않아도 되도록 표준 컴포넌트를 제공합니다. 처음부터 모든 것을 도입하려 하기보다는 서비스 디스커버리 → Config Server → Gateway → 서킷 브레이커 → 추적 순으로 점진적으로 도입하는 것이 안전합니다. 쿠버네티스 친화적인 구성 또한 함께 고려하면 운영 부담을 줄일 수 있습니다.
