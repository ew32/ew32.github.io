GraphQL 쿼리 최적화 - N+1 문제와 DataLoader

작성일 2026-06-25 Edited on 2026-05-26 In Backend
GraphQL의 고질적인 N+1 문제를 DataLoader로 해결하고, 쿼리 복잡도와 캐싱까지 최적화하는 방법을 다룹니다.

GraphQL의 매력과 그늘

GraphQL은 클라이언트가 필요한 필드만 골라 가져올 수 있다는 점에서 매우 매력적인 API 스타일입니다. 다만 그 유연함이 곧 서버 성능의 위험이 되기도 합니다. 가장 유명한 것이 N+1 쿼리 문제입니다.

N+1 문제가 발생하는 순간

다음과 같은 스키마를 가정해 봅시다.

리졸버를 순진하게 구현하면 이렇게 됩니다.

게시글이 100개라면 사용자 조회만 100번 발생합니다. 합계 1+N=101번. 이것이 N+1 문제입니다.

DataLoader: 배치와 캐싱

Facebook이 만든 DataLoader는 한 이벤트 루프 틱 동안 들어온 요청을 모았다가 한 번에 처리해 줍니다.

핵심 규칙은 두 가지입니다.

  1. 반환 배열의 순서와 길이가 입력 키와 정확히 일치해야 한다.
  2. DataLoader는 요청 단위로 새로 만들어야 한다. 그래야 사용자 간 데이터가 섞이지 않고 캐시가 안전합니다.

일대다 관계 배칭

게시글에 달린 댓글처럼 1:N 관계도 비슷한 패턴으로 배칭합니다.

쿼리 복잡도 제한

DataLoader가 N+1을 해결해 주더라도, 클라이언트가 깊이 50의 중첩 쿼리를 보내면 서버는 무너집니다. 복잡도 분석을 함께 사용하세요.

스키마 레벨에서 비용을 명시할 수도 있습니다.

Persisted Query와 응답 캐싱

운영 환경에서는 임의 쿼리를 받지 말고 persisted query(클라이언트가 미리 등록한 쿼리 해시만 허용)로 좁히는 것이 안전합니다. Apollo Server의 responseCachePlugin이나 Cloudflare/Stellate 같은 GraphQL CDN을 결합하면 HTTP 캐시까지 적극 활용할 수 있습니다.

리졸버나 스키마에 @cacheControl(maxAge: 60) 디렉티브를 붙이면 자동으로 Cache-Control 헤더에 반영됩니다.

데이터베이스 차원의 최적화

DataLoader는 “여러 번의 단일 조회”를 “한 번의 IN 조회”로 바꿔주지만, 그 IN 조회가 비효율적이면 의미가 없습니다.

  • 외래키 컬럼(author_id, post_id)에는 반드시 인덱스를 둡니다.
  • SELECT * 대신 GraphQL이 실제로 요청한 필드만 SELECT 하도록 info 객체에서 selection set을 분석하는 라이브러리(예: graphql-fields)를 활용할 수 있습니다.
  • ORM의 lazy loading을 끄고 명시적으로 join/IN을 쓰는 편이 예측 가능합니다.

관측과 추적

문제를 측정할 수 없으면 고칠 수 없습니다.

  • Apollo Studio / GraphQL Inspector로 필드별 호출 횟수와 지연을 봅니다.
  • OpenTelemetry의 GraphQL/SQL 인스트루멘테이션으로 한 요청에서 발생한 쿼리 수를 추적합니다.
  • 개발 모드에서 pg의 log 옵션이나 ORM의 query 로깅을 켜 N+1이 다시 슬그머니 들어오지 않았는지 확인합니다.

체크리스트

  • 관계 필드 리졸버는 모두 DataLoader 경유
  • DataLoader는 요청 단위 생성
  • 쿼리 깊이/복잡도 제한
  • 운영용 persisted query 운영
  • 자주 호출되는 쿼리에 응답 캐시 적용
  • FK 인덱스와 IN 쿼리 실행계획 점검

마무리

GraphQL의 성능 문제는 대부분 “유연한 쿼리 + 순진한 리졸버”의 결합에서 시작됩니다. DataLoader, 복잡도 제한, 캐싱이라는 세 축을 함께 갖추면 GraphQL은 REST 못지않게 안정적이고 빠른 API가 될 수 있습니다.
