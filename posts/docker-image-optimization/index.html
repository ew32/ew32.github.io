Docker 이미지 사이즈 최적화 전략

작성일 2026-06-27 Edited on 2026-05-26 In Cloud , Infrastructure
멀티스테이지 빌드, 베이스 이미지 선택, 레이어 캐시 활용 등 Docker 이미지를 가볍고 빠르게 만드는 실전 기법을 소개합니다.

왜 이미지 크기가 중요한가

Docker 이미지가 클수록 빌드, 푸시, 풀, 스케일 아웃 모든 단계가 느려집니다. 또한 불필요한 패키지가 많을수록 공격 표면이 넓어져 보안 취약점도 증가합니다. 수백 MB짜리 Node.js 이미지가 멀티스테이지와 슬림 베이스 적용 후 수십 MB로 줄어드는 일은 흔합니다.

베이스 이미지 선택

가장 큰 영향을 주는 요소는 베이스 이미지입니다.

  • ubuntu, debian: 호환성은 좋지만 100MB 이상.
  • *-slim: 데비안 기반의 슬림 변종, 약 40~80MB.
  • alpine: musl libc 기반, 5~10MB. 일부 네이티브 바이너리 호환성 이슈 주의.
  • distroless: Google이 제공, OS 도구가 없고 런타임만 포함. 보안에 유리.

Node, Python 등은 공식 slim 또는 alpine 태그를 적극 활용합니다.

멀티스테이지 빌드

빌드 도구와 런타임을 분리하면 최종 이미지에서 컴파일러, 패키지 매니저 등을 제거할 수 있습니다.

Go, Rust 같은 정적 바이너리는 한층 더 극단적으로 최적화할 수 있습니다.

레이어 캐시 활용

COPY와 RUN은 각각 레이어를 만들고, 앞 단계가 바뀌면 뒤 레이어 캐시도 무효화됩니다. 자주 바뀌지 않는 파일을 먼저 복사하는 패턴이 중요합니다.

RUN을 합치고 캐시·임시 파일을 같은 레이어에서 제거해야 실제로 용량이 줄어듭니다.

.dockerignore 활용

node_modules, .git, 빌드 산출물, 테스트 결과물 등을 .dockerignore로 제외하면 빌드 컨텍스트가 줄어들고 캐시 효율이 올라갑니다.

분석 도구

  • docker image ls로 사이즈 확인.
  • docker history <image>로 레이어별 용량 확인.
  • dive 도구로 레이어 내부를 시각적으로 탐색.
  • docker scout 또는 trivy로 취약점 스캔.

추가 기법

  • BuildKit 활성화(DOCKER_BUILDKIT=1)로 병렬 빌드와 --mount=type=cache 사용.
  • 최종 이미지를 --platform 옵션으로 멀티 아키텍처 빌드.
  • 비루트 사용자(USER) 지정으로 보안 강화.

마무리

이미지 최적화는 한 번에 끝나는 작업이 아니라, 베이스 선택·멀티스테이지·캐시 전략을 꾸준히 점검하는 과정입니다. 매 빌드마다 사이즈와 빌드 시간을 메트릭으로 남기면 회귀를 빠르게 잡아낼 수 있습니다.
