프로세스와 스레드의 차이

작성일 2026-05-24 Edited on 2026-05-19 In System , OS
운영체제의 핵심 개념인 프로세스와 스레드의 차이점과 동시성 처리 방식을 정리합니다.

들어가며

운영체제를 공부할 때 가장 먼저 마주치는 개념이 바로 프로세스(Process) 와 스레드(Thread) 입니다. 두 개념은 비슷해 보이지만, 메모리 구조와 자원 공유 방식에서 큰 차이를 가집니다. 이 글에서는 프로세스와 스레드의 정의, 차이점, 그리고 실제 동시성 처리에서 어떤 선택을 해야 하는지를 살펴봅니다.

프로세스란?

프로세스는 실행 중인 프로그램의 인스턴스입니다. 디스크에 저장된 실행 파일(예: notepad.exe)을 메모리에 로드해 실행하면 운영체제는 하나의 프로세스를 생성합니다.

각 프로세스는 다음과 같은 독립된 자원을 가집니다.

  • Code 영역: 실행될 명령어
  • Data 영역: 전역 변수, 정적 변수
  • Heap 영역: 동적으로 할당된 메모리
  • Stack 영역: 함수 호출 정보, 지역 변수
  • PCB(Process Control Block): PID, 상태, 레지스터 값 등 운영체제가 관리하는 정보

프로세스 간 메모리는 격리되어 있어, 한 프로세스가 다른 프로세스의 메모리에 직접 접근할 수 없습니다. 통신이 필요하다면 IPC(Inter-Process Communication) 메커니즘을 사용해야 합니다.

스레드란?

스레드는 프로세스 내부에서 실행되는 흐름의 단위입니다. 하나의 프로세스는 최소 하나(메인 스레드)에서 여러 개의 스레드를 가질 수 있습니다.

스레드는 같은 프로세스 안에서 다음 자원을 공유합니다.

  • Code, Data, Heap 영역
  • 파일 디스크립터, 열린 소켓 등

반면 다음 자원은 스레드마다 독립적으로 가집니다.

  • Stack 영역
  • 레지스터, 프로그램 카운터(PC)

핵심 차이 비교

항목 프로세스 스레드
메모리 독립적 Code/Data/Heap 공유
생성 비용 높음 낮음
컨텍스트 스위칭 무거움 가벼움
통신 방식 IPC(파이프, 소켓 등) 공유 메모리
안정성 한 프로세스 죽어도 다른 프로세스 영향 없음 한 스레드 죽으면 프로세스 전체 영향

멀티프로세스 vs 멀티스레드 선택 기준

멀티프로세스를 선택해야 할 때

  • 작업 간 강한 격리가 필요할 때(보안, 안정성)
  • CPU 바운드 작업이며, Python처럼 GIL이 있는 언어에서 진정한 병렬 처리가 필요할 때
  • 한 작업의 실패가 다른 작업에 영향을 주면 안 될 때

멀티스레드를 선택해야 할 때

  • I/O 바운드 작업이 많을 때(파일, 네트워크)
  • 작업 간 빈번한 데이터 공유가 필요할 때
  • 자원 사용을 최소화해야 할 때

컨텍스트 스위칭

CPU가 하나의 작업에서 다른 작업으로 전환하는 과정을 컨텍스트 스위칭이라고 합니다.

  • 프로세스 간 스위칭: PCB 저장/복원, 메모리 맵 교체, TLB(Translation Lookaside Buffer) 플러시 → 비용이 큼
  • 스레드 간 스위칭(같은 프로세스 내): 레지스터와 스택 포인터만 교체 → 비용이 작음

동시성 이슈

스레드는 메모리를 공유하기 때문에 경쟁 조건(Race Condition) 이 발생할 수 있습니다. 이를 해결하기 위해 다음과 같은 동기화 기법을 사용합니다.

  • Mutex / Lock: 한 번에 하나의 스레드만 임계 영역에 진입
  • Semaphore: N개의 스레드만 진입 허용
  • Monitor: 자바의 synchronized 키워드처럼 객체 단위 락

마무리

프로세스와 스레드는 운영체제가 동시성을 구현하는 두 가지 방식입니다. 프로세스는 격리와 안정성을, 스레드는 효율적인 자원 공유와 가벼운 스위칭을 제공합니다. 어떤 모델을 선택할지는 작업의 특성(CPU 바운드 vs I/O 바운드), 격리 요구사항, 성능 요구사항에 따라 달라집니다. 현대의 많은 시스템은 두 방식을 조합해 사용한다는 점도 기억해 두면 좋습니다.
