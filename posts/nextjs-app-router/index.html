Next.js App Router 사용 가이드

작성일 2026-05-19 Edited on 2026-05-18 In Frontend , JavaScript
Next.js 13 이후 도입된 App Router의 파일 기반 라우팅, 서버 컴포넌트, 레이아웃, 데이터 페칭을 정리합니다.

App Router란?

App Router는 Next.js 13에서 정식 도입되어 14, 15를 거치며 안정화된 새로운 라우팅 시스템입니다. 기존의 pages/ 디렉토리 기반 Pages Router를 대체하며, React Server Components(RSC) 를 기반으로 한 강력한 서버 중심 모델을 제공합니다.

주요 차이점은 다음과 같습니다.

  • 디렉토리: app/ 기반 (기존은 pages/)
  • 기본 컴포넌트: 서버 컴포넌트 ('use client' 선언으로 클라이언트 전환)
  • 데이터 페칭: fetch 확장과 React Cache 통합
  • 레이아웃: 중첩 레이아웃 기본 지원

파일 기반 라우팅

app/ 디렉토리의 폴더 구조가 URL이 됩니다. 각 라우트는 특수 파일을 가질 수 있습니다.

파일 역할
page.tsx 라우트의 실제 화면
layout.tsx 자식 라우트를 감싸는 공통 UI
loading.tsx Suspense 로딩 상태
error.tsx 에러 경계
not-found.tsx 404 화면

서버 컴포넌트와 클라이언트 컴포넌트

App Router의 컴포넌트는 기본적으로 서버에서 렌더링됩니다. 서버 컴포넌트는 번들 크기에 포함되지 않고, DB나 파일 시스템에 직접 접근할 수 있습니다.

상호작용이 필요한 컴포넌트는 'use client' 지시어를 파일 상단에 둡니다.

데이터 페칭과 캐싱

App Router는 fetch를 확장해 캐싱 동작을 제어합니다.

서버 액션 또는 API 라우트에서 revalidateTag('posts')를 호출하면 해당 태그의 캐시가 무효화됩니다.

레이아웃과 템플릿

루트 레이아웃은 반드시 존재해야 하며 <html>, <body>를 포함합니다.

레이아웃은 라우트 전환 시 상태가 유지됩니다. 매번 새로 마운트되어야 한다면 template.tsx를 사용합니다.

동적 라우트와 메타데이터

대괄호 표기로 동적 세그먼트를 만듭니다.

Pages Router와의 비교

기능 Pages Router App Router
디렉토리 pages/ app/
기본 렌더링 클라이언트 서버
데이터 페칭 getServerSideProps 등 async 컴포넌트 + fetch
레이아웃 _app.js 단일 중첩 레이아웃
메타데이터 next/head Metadata API

모범 사례

  • 가능한 한 서버 컴포넌트로 유지하고 'use client'는 잎(leaf) 컴포넌트에만 적용합니다.
  • 데이터 페칭은 사용하는 컴포넌트에서 직접 수행하여 props drilling을 피합니다.
  • 캐시 전략(revalidate, tags)을 의식적으로 설계합니다.
  • 폼 처리는 서버 액션을 우선 검토합니다.

마무리

App Router는 단순한 폴더 구조 변경이 아니라 렌더링 모델의 전환입니다. 서버 컴포넌트, 향상된 데이터 페칭, 중첩 레이아웃은 더 빠르고 단순한 애플리케이션을 만들 수 있게 해줍니다. 작은 프로젝트로 충분히 익힌 뒤 실제 서비스에 도입해보세요.
