MySQL 복제 전략 완벽 가이드 - Master-Slave부터 Group Replication까지

작성일 2026-04-12 Edited on 2026-05-19 In Backend , Database
MySQL 복제의 핵심 전략인 Master-Slave, Group Replication, GTID를 비교하고 실무에서의 구성, 장애 대응, 모니터링 방법을 상세히 설명합니다.

MySQL 복제란?

MySQL 복제(Replication)는 한 MySQL 서버의 데이터를 다른 MySQL 서버로 비동기 또는 준동기 방식으로 전달하는 기술입니다. 이를 통해 읽기 부하 분산, 고가용성(HA), 백업, 분석 워크로드 분리 등 다양한 운영상 이점을 얻을 수 있습니다. 복제는 단순히 데이터를 복사하는 것에 그치지 않고, 운영 환경의 안정성을 책임지는 핵심 기반 인프라입니다.

복제의 기본 동작 원리

MySQL 복제는 바이너리 로그(binlog) 를 기반으로 동작합니다. 마스터(소스) 서버는 모든 데이터 변경 사항(INSERT, UPDATE, DELETE, DDL)을 binlog에 기록하고, 슬레이브(레플리카) 서버는 이를 읽어와서 자기 자신에게 다시 실행합니다.

복제 처리 흐름은 다음과 같습니다.

  1. 마스터: 트랜잭션 커밋 시 binlog에 이벤트 기록
  2. 슬레이브의 IO Thread: 마스터에서 binlog 이벤트를 읽어와 로컬의 relay log에 저장
  3. 슬레이브의 SQL Thread: relay log를 읽어 데이터에 적용

Master-Slave (전통적 비동기 복제)

가장 기본적인 구성으로, 단일 마스터가 쓰기를 받고 N개의 슬레이브가 비동기로 데이터를 복제받습니다.

장점

  • 구성이 단순하고 운영 부담이 적다
  • 읽기 부하를 슬레이브로 분산할 수 있다
  • 마스터 장애 시 슬레이브를 승격(promote)하여 빠르게 복구 가능

단점

  • 비동기 방식이므로 복제 지연(lag) 발생 가능
  • 슬레이브로 읽기 시 stale read 위험
  • 마스터 장애 시 아직 복제되지 않은 데이터 손실 가능성

구성 예시

준동기 복제(Semi-Synchronous Replication)

비동기 복제의 데이터 손실 위험을 줄이기 위해 등장한 방식입니다. 마스터는 적어도 하나의 슬레이브가 binlog 이벤트를 relay log에 기록했다고 ACK할 때까지 커밋 응답을 지연시킵니다.

성능과 안정성의 균형이 좋아 금융권 등에서 자주 사용됩니다.

GTID 기반 복제

GTID(Global Transaction Identifier) 는 각 트랜잭션에 전역 고유 ID를 부여하는 방식입니다. 기존의 binlog 파일명·포지션 기반 복제는 페일오버 시 정확한 위치를 찾기 어려웠지만, GTID는 이러한 문제를 해결합니다.

GTID 형식은 server_uuid:transaction_id 입니다. 예: 3E11FA47-71CA-11E1-9E33-C80AA9429562:23.

GTID의 장점

  • 페일오버가 단순해진다 (SOURCE_AUTO_POSITION=1)
  • 어떤 트랜잭션이 어디까지 적용되었는지 추적 용이
  • 복제 토폴로지 변경이 쉬움

MySQL Group Replication

MySQL 5.7부터 등장한 다중 마스터 동기식 복제 솔루션입니다. Paxos 기반의 합의 알고리즘(XCom)을 사용하며, 일관성과 가용성을 모두 제공합니다.

주요 특징

  • 그룹 내 모든 노드가 동등 (single-primary 또는 multi-primary 모드)
  • 트랜잭션은 그룹의 과반수 합의 후 커밋
  • 노드 장애 시 자동 감지 및 그룹 재구성
  • 충돌 감지 및 인증(certification) 단계 포함

구성 예시

Group Replication을 쓰면 좋은 경우

  • 자동 페일오버가 필요한 미션 크리티컬 서비스
  • 여러 데이터센터 간 데이터 일관성이 중요한 환경
  • InnoDB Cluster를 통한 통합 운영을 원할 때

복제 지연 모니터링

추가로 pt-heartbeat나 PMM 같은 도구를 활용하면 더 정확한 지연 측정이 가능합니다.

실무 운영 팁

  • binlog_format은 ROW로 통일: 비결정적 SQL 문제를 피할 수 있음
  • 슬레이브는 read_only=ON 으로 설정해 실수 방지
  • 복제 채널별 모니터링: 멀티 소스 복제 사용 시 채널별로 지연 추적
  • VIP 또는 ProxySQL 등을 활용한 페일오버 자동화 고려
  • 백업 전략과 결합: 슬레이브에서 mysqldump 또는 Xtrabackup 수행

마무리

MySQL 복제는 단순한 데이터 복사가 아니라 가용성, 일관성, 성능의 균형을 잡는 도구입니다. 서비스 특성에 따라 비동기, 준동기, Group Replication 중 적절한 방식을 선택하고, GTID 활성화와 자동 페일오버 도구를 함께 사용하면 더 안정적인 운영이 가능합니다.
