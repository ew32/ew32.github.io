데이터베이스 쿼리 최적화 기법 - EXPLAIN부터 N+1 문제까지

작성일 2026-04-18 Edited on 2026-07-20 In Backend , Database
실행 계획 분석, 인덱스 활용, JOIN 최적화, N+1 문제 해결 등 실무에서 즉시 적용할 수 있는 SQL 쿼리 최적화 기법을 종합적으로 정리합니다.

쿼리 최적화가 중요한 이유

데이터베이스는 대부분의 백엔드 서비스에서 성능의 병목 지점입니다. 같은 결과를 만드는 SQL이라도 작성 방식에 따라 응답 시간이 10배, 100배 차이 날 수 있습니다. 쿼리 최적화는 단순히 SQL 문법을 개선하는 일이 아니라, 옵티마이저의 동작 원리·인덱스 구조·실행 계획을 종합적으로 이해하는 작업입니다.

실행 계획 분석: EXPLAIN

쿼리 최적화의 첫 단계는 항상 실행 계획을 확인하는 것입니다.

MySQL

주요 칼럼:

  • type: const, eq_ref, ref, range, index, ALL (오른쪽으로 갈수록 느림)
  • key: 사용된 인덱스
  • rows: 예상 검사 행 수
  • Extra: Using filesort, Using temporary 등 추가 정보

PostgreSQL

Seq Scan, Index Scan, Bitmap Heap Scan, Hash Join, Nested Loop 같은 노드 정보가 표시됩니다.

인덱스 활용 원칙

1. 좌측 컬럼 우선(Leftmost Prefix)

복합 인덱스 (a, b, c)는 다음 조건에서만 활용됩니다.

  • WHERE a = ?
  • WHERE a = ? AND b = ?
  • WHERE a = ? AND b = ? AND c = ?

WHERE b = ?만 있는 경우는 인덱스를 못 씁니다.

2. 함수·연산을 피한다

3. 카디널리티가 높은 컬럼을 선택성으로 활용

성별처럼 카디널리티가 낮은 컬럼은 단독 인덱스로는 효과가 적습니다. (status, created_at)처럼 결합해 선택성을 높입니다.

4. 커버링 인덱스(Covering Index)

쿼리에 필요한 모든 컬럼이 인덱스에 포함되면 테이블 조회 없이 인덱스만으로 결과를 반환합니다.

JOIN 최적화

JOIN 종류와 동작

  • Nested Loop Join: 외부 테이블의 각 행에 대해 내부 테이블을 인덱스로 검색. 행 수가 적을 때 유리
  • Hash Join: 작은 테이블을 해시 테이블로 만들고 큰 테이블을 스캔하며 매칭. 동등 조인에 강함
  • Merge Join: 정렬된 두 테이블을 병합. 양쪽에 인덱스가 있을 때 효율적

작성 팁

  1. 조인 키에는 반드시 인덱스를 만든다
  2. 가능하면 작은 결과 집합부터 줄여 나간다
  3. 불필요한 컬럼은 SELECT에서 제외
  4. LEFT JOIN이 의미상 필요 없으면 INNER JOIN으로 바꿔 옵티마이저에 자유도를 준다
  5. 서브쿼리보다 JOIN이 일반적으로 빠르지만, DBMS 버전에 따라 다르므로 EXPLAIN으로 비교

N+1 문제

ORM 사용 시 가장 흔히 발생하는 성능 문제입니다.

발생 예시

해결 방법

  1. Fetch Join (JPQL)

  2. EntityGraph

  3. Batch Size 설정

  4. 단일 쿼리로 직접 작성 (네이티브 SQL 또는 QueryDSL)

페이징 최적화

자주 만나는 안티 패턴

1. SELECT *

필요 컬럼만 선택해 네트워크와 메모리 사용을 줄입니다. 커버링 인덱스 활용도 가능해집니다.

2. 묵시적 형 변환

3. OR 남용

4. 불필요한 DISTINCT, ORDER BY

집계 결과가 이미 유일하면 DISTINCT 불필요. 정렬이 필요 없다면 ORDER BY 제거.

5. NOT IN과 NULL

서브쿼리에 NULL이 포함되면 결과가 비어 나옵니다. NOT EXISTS나 LEFT JOIN ... IS NULL 사용을 고려합니다.

통계와 옵티마이저

옵티마이저는 통계 정보를 바탕으로 실행 계획을 결정합니다. 데이터 분포가 크게 바뀐 후에는 통계를 갱신해야 합니다.

캐싱과 결과 활용

  • 결과 캐싱: Redis 등으로 자주 조회되는 결과 저장
  • 머터리얼라이즈드 뷰: 집계 결과를 미리 계산해 두기
  • 읽기 전용 복제본: 분석성 쿼리는 슬레이브에서

최적화 워크플로

  1. 느린 쿼리 식별: slow query log, pg_stat_statements
  2. EXPLAIN 분석: 풀 스캔, 불필요 정렬 확인
  3. 인덱스 또는 쿼리 수정
  4. before/after 비교: 응답 시간, 검사 행 수
  5. 운영 모니터링: 데이터 증가에 따른 성능 변화 관찰

마무리

쿼리 최적화는 한 번의 작업으로 끝나지 않습니다. 데이터가 변하고 트래픽 패턴이 달라지면 같은 쿼리도 다르게 동작합니다. EXPLAIN을 습관화하고 인덱스 설계 원칙과 안티 패턴을 머릿속에 두면, 가벼우면서 확장 가능한 데이터베이스 운영이 가능해집니다.
