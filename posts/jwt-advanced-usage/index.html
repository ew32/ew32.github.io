JWT 심화 활용과 보안 주의사항

작성일 2026-05-12 Edited on 2026-05-26 In Backend , Authentication
JWT의 구조와 서명 알고리즘을 깊이 살펴보고, 실무에서 자주 발생하는 보안 함정과 회피 방법을 정리합니다.

JWT란?

**JWT(JSON Web Token)**는 RFC 7519에서 정의한 자가 검증 가능한 토큰 포맷입니다. 서버가 세션 저장소를 유지하지 않아도, 토큰 자체에 클레임과 서명을 담아 상태 비저장(Stateless) 인증을 가능하게 합니다. 그러나 그 단순함 뒤에 자주 간과되는 보안 함정이 숨어 있습니다.

JWT의 구조

JWT는 점(.)으로 구분된 3개의 BASE64URL 인코딩 부분으로 이루어집니다.

서명 알고리즘 선택

알고리즘 키 형태 용도
HS256 공유 비밀키(HMAC) 단일 서비스 내부 토큰
RS256 RSA 공개/개인키 다수 검증자, 공개 JWK 제공
ES256 ECDSA(P-256) 모바일/IoT, 짧은 키 길이 선호
EdDSA Ed25519 최신, 빠르고 안전

팁: 여러 서비스가 토큰을 검증해야 한다면 비대칭 키(RS256/ES256/EdDSA)를 사용해 개인키 노출 범위를 최소화하세요.

악명 높은 alg: none 공격

초기 JWT 라이브러리들이 헤더의 alg 값을 그대로 신뢰해 발생한 취약점입니다. 공격자가 alg를 none으로 바꾸면 서명이 비어 있어도 검증을 통과합니다.

방어: 서버에서 허용된 알고리즘 목록을 명시적으로 지정해 검증하세요.

RS256 → HS256 혼동 공격

서버가 알고리즘을 자동으로 선택하면, 공격자가 alg를 HS256으로 바꾸고 공개키를 비밀키처럼 사용해 서명할 수 있습니다. 역시 알고리즘 화이트리스트로 막아야 합니다.

클레임 검증 체크리스트

JWT의 서명을 검증했다고 끝이 아닙니다. 다음 클레임도 반드시 확인해야 합니다.

  • iss (Issuer): 신뢰하는 발급자인가?
  • aud (Audience): 내가 받아야 할 토큰인가?
  • exp (Expiration): 만료되지 않았는가?
  • nbf (Not Before): 사용 가능한 시점인가?
  • iat (Issued At): 너무 오래된 토큰인가?

JWT의 한계 — 즉시 폐기가 어렵다

JWT는 발급 후 만료 시각까지 유효하므로, 로그아웃이나 권한 변경 즉시 무효화가 어렵습니다. 대표적 해결책은 다음과 같습니다.

  1. 짧은 Access Token + Refresh Token Rotation
  2. 서버 측 블랙리스트(jti 기반)
  3. Token Versioning: 사용자별 토큰 버전을 DB에 두고, 클레임의 버전과 비교

토큰 저장 위치

저장소 XSS 위험 CSRF 위험 비고
localStorage 높음 낮음 XSS 발생 시 즉시 탈취
sessionStorage 높음 낮음 탭 닫으면 사라짐
HttpOnly Cookie 낮음 높음 SameSite로 보완
메모리(JS 변수) 새로고침 시 손실 낮음 SPA에서 권장

권장: HttpOnly + Secure + SameSite=Lax 쿠키 또는 BFF(Backend-For-Frontend) 패턴으로 토큰을 브라우저에 노출하지 않는 방식을 검토하세요.

민감정보를 페이로드에 담지 말 것

JWT 페이로드는 암호화가 아니라 인코딩입니다. 누구나 디코딩할 수 있으므로 비밀번호, 주민번호, 카드번호 등을 절대 넣지 마세요. 진짜 암호화가 필요하다면 **JWE(JSON Web Encryption)**를 사용해야 합니다.

Best Practices

  • 알고리즘은 화이트리스트로 강제하세요.
  • iss, aud, exp를 항상 검증하세요.
  • 키는 kid 헤더로 식별하고 주기적으로 회전하세요.
  • Refresh Token은 1회용 + 회전 + 재사용 탐지로 운영하세요.
  • 페이로드 크기를 작게 유지하세요(헤더 크기 폭증을 막기 위함).

마무리

JWT는 강력하지만 잘못 쓰면 위험한 도구입니다. 구조만 외우지 말고, 각 클레임이 막아주는 위협과 서명 알고리즘의 신뢰 모델을 함께 이해해야 합니다. “서명만 맞으면 신뢰한다”는 사고를 버리고, 클레임 단위로 방어를 쌓는 습관이 안전한 인증 시스템의 시작입니다.
