다중 인증(MFA) 완전 정리 - TOTP, HOTP, WebAuthn

작성일 2026-04-25 Edited on 2026-05-20 In Backend , Authentication
2FA와 MFA의 차이부터 TOTP/HOTP 알고리즘 동작 원리, WebAuthn(Passkey) 기반 피싱 저항 인증까지 한 번에 정리합니다.

MFA와 2FA의 차이

다중 인증(MFA, Multi-Factor Authentication)은 두 가지 이상의 서로 다른 인증 요소를 결합하는 방식입니다. 2FA는 MFA의 한 형태로, 정확히 두 가지 요소를 사용합니다. 요소는 다음 세 가지 범주로 나뉩니다.

  • 지식(something you know): 비밀번호, PIN
  • 소유(something you have): 휴대폰, 보안 키, OTP 생성기
  • 존재(something you are): 지문, 얼굴 등 생체 정보

같은 범주의 두 요소(예: 비밀번호 + 보안 질문)는 진정한 MFA가 아닙니다. 반드시 서로 다른 범주를 조합해야 합니다.

HOTP - 카운터 기반 OTP

HOTP(HMAC-based OTP, RFC 4226)는 카운터 값을 HMAC-SHA1로 해시한 뒤 6자리 숫자로 변환합니다.

서버와 클라이언트가 같은 비밀키 K와 카운터 C를 공유해야 합니다. 사용자가 OTP를 사용할 때마다 C가 증가합니다. 카운터 동기화가 깨지면 인증이 실패하므로, 일반적으로 look-ahead window를 둡니다.

TOTP - 시간 기반 OTP

TOTP(Time-based OTP, RFC 6238)는 HOTP의 카운터를 시간으로 대체한 것입니다.

기본값은 T0 = 0, X = 30초입니다. 30초마다 새 코드가 생성되며, 시계 오차를 위해 보통 ±1 윈도우를 허용합니다.

서버 구현 예시는 다음과 같습니다.

Google Authenticator, Authy 등은 모두 같은 표준을 따르므로 위 알고리즘으로 검증 가능합니다.

TOTP의 한계

TOTP는 편리하지만 다음 약점이 있습니다.

  • 피싱 취약: 사용자가 가짜 사이트에 6자리 코드를 입력하면 공격자가 30초 안에 재사용 가능
  • 공유 비밀키: 서버 DB 유출 시 모든 사용자의 OTP가 위조 가능
  • 시계 동기화 의존: 클라이언트 시계가 크게 어긋나면 실패

이 한계를 극복하기 위해 등장한 것이 WebAuthn입니다.

WebAuthn과 Passkey

WebAuthn(W3C 표준)은 공개키 암호 기반의 인증 표준입니다. 클라이언트가 키 쌍을 생성하고, 공개키만 서버에 등록합니다. 인증 시 서버가 보낸 챌린지를 개인키로 서명해 응답합니다.

WebAuthn의 핵심 장점은 피싱 저항입니다. 브라우저가 키와 도메인(rp.id)을 바인딩하므로, 가짜 사이트에서는 인증이 작동하지 않습니다.

Passkey는 WebAuthn 자격증명을 동일 사용자의 디바이스 간에 동기화하는 사용자 경험을 표준화한 것으로, Apple iCloud Keychain, Google Password Manager 등이 지원합니다.

어떤 방식을 선택할까

  • SMS OTP: 최소한의 보안. SIM 스와핑 공격으로 우회 가능하므로 가급적 회피.
  • TOTP: 무료, 간단, 광범위. 일반 사용자에게 적합한 기본값.
  • 푸시 알림: UX가 좋지만 MFA fatigue 공격 주의.
  • WebAuthn/Passkey: 가장 강력하고 피싱에 강함. 신규 서비스의 기본 선택지로 권장.

마무리

비밀번호만으로는 더 이상 안전하지 않습니다. 최소 TOTP, 가능하다면 WebAuthn을 도입해 피싱과 자격증명 도용에 대한 방어선을 확보해야 합니다. MFA는 비용 대비 보안 효과가 가장 큰 투자입니다.
