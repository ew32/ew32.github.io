Java 가비지 컬렉션(GC) 이해하기

작성일 2026-05-08 Edited on 2026-07-20 In Backend , Java
JVM 힙 구조와 GC 알고리즘, 튜닝 포인트를 정리하여 메모리 관리의 원리를 설명합니다.

가비지 컬렉션이 필요한 이유

C/C++과 달리 Java는 개발자가 직접 메모리를 해제하지 않습니다. JVM의 가비지 컬렉터(Garbage Collector) 가 더 이상 참조되지 않는 객체를 자동으로 회수해 메모리 누수와 댕글링 포인터 문제를 줄여줍니다. 하지만 GC는 공짜가 아니며, 동작 방식을 이해하지 못하면 응답 시간 스파이크나 OOM(OutOfMemoryError)으로 이어집니다.

JVM 힙 구조

전통적인 HotSpot JVM의 힙은 세대(generation) 가설에 기반합니다. 대부분의 객체는 짧은 시간 안에 죽고, 살아남은 객체는 오래 산다는 관찰에 기초합니다.

  • Minor GC: Young 영역에서 발생, 빠르고 빈번
  • Major / Full GC: Old 또는 전체 힙 대상, 비싸고 느림
  • Stop-the-World(STW): GC 동안 애플리케이션 스레드가 멈추는 구간

객체의 생애 흐름

  1. 객체는 Eden에 할당됩니다.
  2. Eden이 차면 Minor GC가 발생, 살아남은 객체는 Survivor로 이동합니다.
  3. Survivor 영역을 여러 번 거치며 age가 임계치에 도달하면 Old로 승격됩니다.
  4. Old가 차면 Major GC가 일어납니다.

주요 GC 알고리즘

GC 특징 대표 사용처
Serial 단일 스레드, 단순 작은 데스크톱 앱
Parallel 다중 스레드, 처리량 최적화 배치 작업
CMS (deprecated) 동시 마킹, 낮은 지연 레거시 서비스
G1 영역(region) 기반, 균형 Java 9+ 기본
ZGC 매우 큰 힙, sub-ms 지연 대형 서비스
Shenandoah 동시 압축 Red Hat 진영

Java 17 LTS 이후로는 G1이 기본이며, 일정 규모 이상에서는 ZGC 또는 Shenandoah가 선택지에 들어옵니다.

G1 GC의 핵심 개념

G1은 힙을 동일 크기의 region으로 나누고, 각 region을 Eden/Survivor/Old/Humongous 중 하나로 동적으로 분류합니다. 살아있는 객체가 적은 region부터 우선 회수해 목표 일시 정지 시간을 맞춥니다.

ZGC: 초저지연 GC

ZGC는 컬러 포인터와 로드 배리어를 사용해 거의 모든 작업을 애플리케이션과 동시에 수행합니다. 수십 GB 힙에서도 STW가 1ms 미만으로 유지됩니다.

참조 종류

가비지 컬렉터가 객체를 회수할지 결정하는 데에는 참조의 종류가 영향을 줍니다.

  • Strong Reference: 일반 참조, GC 대상이 되지 않음
  • Soft Reference: 메모리가 부족할 때 회수, 캐시에 적합
  • Weak Reference: 다음 GC에서 회수, WeakHashMap에서 사용
  • Phantom Reference: 사후 정리용

GC 로그 분석

운영 환경에서는 반드시 GC 로그를 남깁니다.

분석 도구로는 GCViewer, GCEasy가 유용합니다. 다음 지표를 관찰합니다.

  • Pause Time: STW 시간의 분포
  • Throughput: 애플리케이션 처리 시간 비율
  • Promotion Rate: Old로 승격되는 객체량
  • Allocation Rate: 초당 할당량

흔한 문제와 진단

증상 의심 원인
Full GC가 자주 발생 메모리 누수, Old 영역 부족
Young GC가 너무 잦음 Eden이 작거나 할당률 과다
응답 시간 스파이크 큰 객체 승격, 메타스페이스 GC
OutOfMemoryError: Heap 힙 부족 또는 누수
OutOfMemoryError: Metaspace 클래스 로딩 과다, 핫 리로드 누수

힙 덤프(-XX:+HeapDumpOnOutOfMemoryError)와 Eclipse MAT을 활용하면 누수의 주체를 파악할 수 있습니다.

튜닝 원칙

  1. 측정 먼저: 추측보다 GC 로그와 프로파일 데이터에 의존
  2. 목표 설정: 처리량 vs 지연 시간 우선순위 결정
  3. 힙 크기: Xms와 Xmx를 동일하게 설정해 리사이즈 비용 제거
  4. 변경은 한 번에 하나씩: 어떤 옵션이 효과를 냈는지 명확히
  5. 할당률을 줄이는 코드 변경이 GC 옵션보다 효과적인 경우가 많음

마무리

GC는 블랙박스가 아니라 명확한 규칙으로 동작하는 시스템입니다. 세대별 가설, 알고리즘 차이, 참조 종류, 로그 분석 도구를 이해하면 메모리 문제를 추측이 아닌 데이터로 해결할 수 있습니다. 무엇보다 가장 좋은 튜닝은 불필요한 객체 할당을 줄이는 코드 변경에서 시작합니다.
