Java Optional 올바르게 사용하기

작성일 2026-05-09 Edited on 2026-05-20 In Backend , Java
Optional의 등장 배경, 안티패턴, 권장 사용 패턴과 함수형 조합 방법을 정리합니다.

Optional이 도입된 배경

NullPointerException은 Java 개발자가 가장 자주 마주치는 예외입니다. 토니 호어(Tony Hoare)는 null 참조를 “10억 달러의 실수”라고 평했습니다. Java 8은 이 문제를 완화하기 위해 Optional<T> 을 도입했습니다. Optional은 “값이 있을 수도, 없을 수도 있다”는 사실을 타입 시스템에 명시적으로 표현합니다.

Optional의 진짜 목적은 반환 타입에서 부재(absence)를 표현하는 것입니다. 모든 null 가능성을 Optional로 감싸는 것이 아니라, API 설계 의도를 명확히 하기 위한 도구입니다.

orElse는 항상 인자가 평가되는 반면, orElseGet은 비어있을 때만 평가됩니다. 비용이 큰 기본값에는 orElseGet을 사용하세요.

함수형 조합

flatMap은 결과가 다시 Optional인 경우에 사용합니다.

ifPresent와 ifPresentOrElse는 부수 효과에 사용합니다.

비교: null vs Optional

측면 null Optional
의도 표현 암묵적 명시적
NPE 위험 높음 낮음
메모리 없음 객체 1개 추가
API 가독성 떨어짐 좋음
컬렉션 사용 적합 부적합

성능이 매우 민감한 경로에서는 Optional 객체 할당 비용이 무시 못 할 수도 있지만, 대부분의 비즈니스 로직에서는 가독성 이점이 훨씬 큽니다.

기본형 Optional

박싱을 피하기 위해 OptionalInt, OptionalLong, OptionalDouble도 제공합니다.

권장 사용 패턴

  • 반환 타입에 사용, 필드/파라미터/컬렉션 요소에는 지양
  • get()보다 orElse, orElseGet, orElseThrow 사용
  • isPresent/get 조합 대신 map/flatMap 체이닝
  • orElse(null)는 가능한 한 피하기

마무리

Optional은 단순히 null을 감싸는 wrapper가 아니라, “값이 없을 수 있음”을 API에 명시하는 타입입니다. 함수형 조합과 결합해 사용하면 분기로 가득 찬 null 체크 코드를 선언적으로 다시 쓸 수 있습니다. 적절한 위치에서만 사용한다면 코드 가독성과 안전성을 동시에 끌어올릴 수 있는 강력한 도구입니다.
