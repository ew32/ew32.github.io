API Gateway 패턴과 역할

작성일 2026-05-16 Edited on 2026-05-18 In Backend , Infrastructure
마이크로서비스 아키텍처에서 API Gateway의 역할, 주요 기능, 구현 예시와 모범 사례를 정리합니다.

API Gateway란?

API Gateway 는 클라이언트와 다수의 백엔드 서비스 사이에 위치하는 단일 진입점(Single Entry Point) 입니다. 모든 외부 요청은 게이트웨이를 거쳐 적절한 서비스로 라우팅되며, 인증·로깅·요청 변환 같은 횡단 관심사(cross-cutting concerns) 를 한곳에서 처리합니다.

마이크로서비스를 운영할 때 클라이언트가 수십 개의 서비스 엔드포인트를 직접 알도록 강제하는 것은 비효율적이며 보안상 위험합니다. API Gateway는 이 문제를 해결합니다.

주요 역할

역할 설명
라우팅 경로/호스트/헤더 기반으로 적절한 서비스 라우팅
인증/인가 JWT 검증, OAuth, API Key 검사
속도 제한 Rate Limiting, Throttling
로드 밸런싱 같은 서비스의 인스턴스 분산
요청/응답 변환 프로토콜 변환, 헤더 조작
응답 캐싱 자주 호출되는 응답 캐시
회로 차단 장애 서비스로의 요청 차단
관측 통합 로깅·메트릭·트레이싱 수집

BFF 패턴 (Backend for Frontend)

클라이언트 종류(모바일, 웹, 외부 파트너)별로 별도의 API Gateway를 두는 패턴입니다. 각 클라이언트의 화면 요구에 맞춰 응답을 집계(aggregation) 하거나 변환합니다.

각 BFF는 자신의 클라이언트만 책임지므로 변경이 다른 클라이언트에 영향을 주지 않습니다.

주의해야 할 안티 패턴

1. 비즈니스 로직 침투

게이트웨이는 횡단 관심사만 다뤄야 합니다. 도메인 로직을 게이트웨이에 두면 분산 모놀리식이 됩니다.

2. 단일 장애 지점

게이트웨이가 다운되면 전체 시스템이 마비됩니다. 다중 인스턴스 + Auto Scaling + 헬스 체크가 필수입니다.

3. 과도한 응답 집계

게이트웨이에서 너무 많은 서비스를 호출해 응답을 합치면 지연·결합도·실패 면적 모두 증가합니다.

대표 솔루션

솔루션 특징
Spring Cloud Gateway JVM 기반, Reactive
Kong OpenResty(NGINX+Lua) 기반, 풍부한 플러그인
Envoy 고성능 C++, 서비스 메시(Istio)에서도 활용
AWS API Gateway 매니지드 서비스, Lambda 연동
Nginx 가벼운 라우팅·LB

모범 사례

  1. 타임아웃 항상 설정 - downstream이 응답하지 않을 때 게이트웨이가 자원 고갈되지 않도록
  2. 회로 차단기 적용 - 장애 전파 방지
  3. 표준화된 에러 응답 - 게이트웨이에서 통일된 형식으로 변환
  4. 헬스 체크 분리 - liveness와 readiness 구분
  5. 관측성 우선 - 트레이스 ID를 모든 요청에 부여하여 분산 트레이싱 가능하게

서비스 메시와의 관계

서비스 메시(Istio, Linkerd)는 서비스 간(내부) 통신을 다루는 반면, API Gateway는 외부→내부(north-south) 트래픽을 다룹니다. 두 가지는 보완 관계이며 동시에 사용되는 경우가 많습니다.

마무리

API Gateway는 마이크로서비스의 현관문입니다. 보안, 성능, 관측, 회복력 같은 공통 관심사를 한곳에서 처리함으로써 각 서비스를 비즈니스 로직에 집중시켜 줍니다. 다만 비즈니스 로직 침투, 단일 장애점, 과도한 책임 같은 함정을 경계해야 안정적인 시스템을 운영할 수 있습니다.
