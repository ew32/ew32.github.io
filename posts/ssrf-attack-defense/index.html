SSRF 공격과 방어 - 내부망을 노리는 요청 변조

작성일 2026-04-27 Edited on 2026-05-26 In Security , Cryptography
SSRF의 작동 원리, 클라우드 메타데이터 탈취 같은 실제 피해 사례, 그리고 DNS Rebinding까지 방어하는 강력한 대응 전략을 정리합니다.

SSRF란 무엇인가

SSRF(Server-Side Request Forgery)는 공격자가 서버 측 애플리케이션을 속여 임의의 URL로 HTTP 요청을 보내게 만드는 취약점입니다. 일반적인 패턴은 사용자가 입력한 URL을 서버가 그대로 fetch할 때 발생합니다.

공격자는 ?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/처럼 내부 메타데이터 엔드포인트를 지정해 AWS 자격증명을 탈취할 수 있습니다.

실제 피해 사례

2019년 Capital One 사고는 SSRF로 EC2 IMDSv1에서 IAM 토큰을 훔쳐 S3 버킷 1억 건 이상의 데이터를 유출한 대표적 사례입니다. 단일 SSRF가 클라우드 인프라 전체를 노출시킬 수 있다는 점이 충격적이었습니다.

SSRF 공격이 노리는 표적은 다양합니다.

  • 클라우드 메타데이터(169.254.169.254, GCP/Azure 동등 엔드포인트)
  • 내부 관리 API(http://localhost:8500/v1/kv/... 같은 Consul, etcd)
  • 사내망 서비스(http://10.0.0.5/admin)
  • 파일 프로토콜(file:///etc/passwd)

기본 방어 - 충분하지 않은 접근

흔히 시도하는 방어는 URL 파싱 후 호스트가 사설 IP인지 검사하는 것입니다.

문제는 두 가지입니다.

  1. TOCTOU: 검사 시점의 DNS 결과와 실제 요청 시점의 결과가 다를 수 있음(DNS Rebinding).
  2. 우회 기법: 10진수 IP(http://2852039166/), IPv6 매핑, 리다이렉트 체이닝 등.

강력한 방어 전략

다층 방어가 필요합니다.

1. 허용 목록(Allowlist)

가능하다면 호출 가능한 도메인을 명시적으로 한정합니다. 차단 목록(blocklist)보다 항상 안전합니다.

2. DNS 결과 고정

DNS Rebinding을 막기 위해 직접 IP를 해석한 뒤, 그 IP로 connection을 만들고 Host 헤더만 원래 호스트로 설정합니다.

Python의 requests는 urllib3의 connection pool을 후킹하면 됩니다.

3. 리다이렉트 처리

서버가 응답으로 30x 리다이렉트를 보내면 클라이언트는 새 URL로 다시 요청합니다. 따라서 리다이렉트마다 같은 검증을 반복해야 합니다.

4. 메타데이터 보호

AWS는 IMDSv2(세션 토큰 기반)를 사용해야 합니다. IMDSv2는 PUT 요청으로 토큰을 받고, GET 요청에 토큰 헤더를 포함시켜야 하므로 단순 SSRF로는 자격증명을 탈취하기 어렵습니다.

5. 네트워크 격리

가장 강력한 방어는 네트워크 수준입니다. 외부 호출 전용 프록시를 두고, 애플리케이션 서브넷에서 내부 메타데이터/관리 IP로의 egress를 차단합니다. 보안 그룹과 VPC 라우팅으로 강제하면 코드 버그가 있어도 피해를 막을 수 있습니다.

점검 체크리스트

  • 사용자 입력 URL을 직접 fetch하지 않는가
  • 사설/링크로컬/예약 IP를 차단하는가
  • DNS Rebinding 대응이 있는가
  • 리다이렉트마다 재검증하는가
  • IMDSv2를 강제하고 있는가
  • 외부 호출용 egress 정책이 있는가

마무리

SSRF는 단순한 URL 파라미터에서 시작해 클라우드 전체를 위협할 수 있는 강력한 취약점입니다. 코드 레벨의 검증, 네트워크 분리, 클라우드 설정(IMDSv2)을 모두 적용해야 비로소 안전합니다.
