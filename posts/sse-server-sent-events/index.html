Server-Sent Events(SSE) 완벽 이해

작성일 2026-06-19 Edited on 2026-05-18 In Network , HTTP
Server-Sent Events의 동작 원리, 구현 방법, WebSocket과의 차이를 실전 예제로 정리합니다.

SSE란 무엇인가

Server-Sent Events(SSE)는 서버에서 클라이언트로 단방향 스트림을 흘려보내기 위한 HTML5 표준입니다. WebSocket과 달리 별도의 프로토콜 업그레이드 없이 일반 HTTP 위에서 동작하며, 브라우저는 EventSource API로 손쉽게 소비할 수 있습니다.

핵심 아이디어는 간단합니다. 서버가 응답을 닫지 않고 text/event-stream 형식의 메시지를 계속 흘려보내면, 브라우저가 그것을 한 줄씩 파싱해 이벤트로 변환합니다.

메시지 포맷

규칙은 다음과 같습니다.

  • 각 이벤트는 빈 줄로 구분됩니다.
  • event:는 이벤트 타입, 생략하면 message가 기본값.
  • data:는 본문(여러 줄로 나눠 보내면 줄바꿈으로 합쳐짐).
  • id:는 이벤트 식별자. 연결이 끊겼다 재접속하면 브라우저가 Last-Event-ID 헤더로 자동으로 보냅니다.
  • retry:는 재연결 대기 시간(ms).

클라이언트 구현

EventSource는 연결이 끊어지면 자동으로 재접속을 시도하고, 마지막으로 받은 id를 Last-Event-ID 헤더로 보내줍니다. 서버는 이 값을 보고 빠진 이벤트를 재전송할 수 있습니다.

서버 구현 (Node.js / Express)

핵심은 (1) 헤더를 먼저 flush, (2) 이벤트가 발생할 때마다 res.write, (3) 클라이언트가 끊기면 리스너를 정리하는 것입니다.

프록시와 버퍼링 주의

Nginx 같은 리버스 프록시 뒤에 SSE를 두면 응답이 버퍼링되어 메시지가 즉시 전달되지 않을 수 있습니다.

CDN을 거치는 경우도 마찬가지로 SSE 경로를 캐시·버퍼링 대상에서 제외해야 합니다.

SSE vs WebSocket vs Long Polling

항목 SSE WebSocket Long Polling
방향 서버 → 클라이언트 단방향 양방향 양방향(반복)
프로토콜 HTTP TCP 위 별도 프로토콜 HTTP
자동 재연결 내장 직접 구현 자연스러움
바이너리 불가 가능 텍스트 위주
프록시 호환 매우 좋음 업그레이드 필요 좋음
구현 난이도 가장 낮음 중간 낮음

적합한 사례

  • 단방향 실시간 피드: 주식 시세, 스포츠 점수, 알림, 진행률 표시줄.
  • AI 응답 스트리밍: 토큰 단위 출력에 SSE가 사실상 표준입니다.
  • 서버 로그 라이브 뷰.
  • 장시간 작업 진행 상황: 빌드 로그, 데이터 가공 진행률.

반대로 채팅·게임·협업 도구처럼 양방향 통신이 필요하다면 WebSocket이 적합합니다.

운영 팁

  • 연결당 메모리 사용량이 낮긴 하지만 한 인스턴스의 동시 연결 수에 한계가 있으므로 수평 확장과 sticky session을 고려하세요.
  • 30~60초마다 빈 주석 라인(: ping\n\n)을 보내 idle 연결이 끊기는 것을 방지합니다.
  • 인증은 일반 HTTP와 동일하게 쿠키 또는 헤더로 처리합니다.

마무리

SSE는 “HTTP 위에서 가장 단순하게 실시간을 구현하는 방법”입니다. 양방향이 필요 없다면, 복잡한 WebSocket보다 먼저 SSE를 떠올려 보세요.
