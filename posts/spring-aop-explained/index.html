Spring AOP 동작 원리와 활용

작성일 2026-05-07 Edited on 2026-06-15 In Backend , Java
Spring AOP의 프록시 기반 동작 원리, 포인트컷, 어드바이스 종류와 실전 활용 사례를 설명합니다.

AOP가 필요한 이유

관점 지향 프로그래밍(AOP, Aspect-Oriented Programming) 은 횡단 관심사(cross-cutting concern)를 비즈니스 로직과 분리하기 위한 패러다임입니다. 로깅, 트랜잭션, 권한 검사, 캐시, 성능 측정과 같은 코드는 여러 메서드에 흩어져 중복되기 쉽습니다. AOP는 이 공통 코드를 한 곳에 모아 선언적으로 적용할 수 있게 해줍니다.

Spring은 자체적으로 프록시 기반의 가벼운 AOP를 제공합니다. AspectJ와 같이 바이트코드를 조작하지는 않지만, 대부분의 실무 요구사항을 충족할 수 있습니다.

핵심 용어 정리

용어 설명
Aspect 횡단 관심사를 모듈화한 단위
Join Point 어드바이스가 적용될 수 있는 지점(메서드 실행)
Pointcut Join Point를 선택하는 표현식
Advice 실제 수행되는 부가 동작
Target 어드바이스가 적용되는 대상 객체
Proxy 타깃을 감싸 어드바이스를 적용하는 객체

프록시 기반 동작

Spring AOP는 빈을 등록할 때 대상 빈을 프록시 객체로 감쌉니다. 외부에서 빈을 호출하면 실제 객체가 아니라 프록시가 호출되고, 프록시가 어드바이스를 실행한 후 타깃 메서드로 위임합니다.

  • 인터페이스가 있는 빈은 JDK 동적 프록시로 생성
  • 인터페이스가 없으면 CGLIB 프록시(상속 기반) 생성
  • Spring Boot 2.x부터는 기본이 CGLIB

이 구조에서 주의할 점은 자기 호출(self-invocation) 입니다. 같은 클래스 내부에서 this.method()로 호출하면 프록시를 거치지 않으므로 어드바이스가 동작하지 않습니다.

어드바이스 종류

@Around가 가장 강력하지만, 단순 로깅이나 후처리에는 @Before, @AfterReturning이 더 안전합니다.

포인트컷 표현식

자주 쓰는 패턴을 알아두면 유용합니다.

여러 표현식을 결합할 수도 있습니다.

실전 예제: 커스텀 어노테이션 기반 캐시

특정 메서드의 결과를 짧게 캐시하는 어노테이션을 만들어 봅시다.

어드바이스 실행 순서

여러 Aspect가 같은 join point에 적용될 때는 @Order로 순서를 지정합니다.

낮은 숫자가 바깥쪽에서 실행됩니다. 트랜잭션처럼 핵심에 가까울수록 안쪽에 두는 것이 일반적입니다.

모범 사례

  • @Transactional은 Spring AOP의 대표 활용 사례이므로 동작 원리를 반드시 이해
  • 자기 호출 문제는 빈을 별도 클래스로 분리하거나 AopContext.currentProxy()로 우회
  • 포인트컷은 가능한 한 좁게 잡아 성능과 예측 가능성 확보
  • Aspect의 의존성은 최소화하고, 도메인 로직을 침범하지 않도록 유지

마무리

Spring AOP는 강력하지만 프록시 기반이라는 본질을 이해해야 의도대로 동작합니다. 횡단 관심사를 식별하고, 포인트컷을 명확히 정의하면 코드 중복을 크게 줄이고 비즈니스 로직을 깔끔하게 유지할 수 있습니다.
