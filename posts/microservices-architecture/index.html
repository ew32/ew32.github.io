마이크로서비스 아키텍처 입문

작성일 2026-05-14 Edited on 2026-05-22 In Backend , Infrastructure
마이크로서비스 아키텍처(MSA)의 개념, 구성 요소, 장단점과 도입 시 고려사항을 정리합니다.

마이크로서비스 아키텍처란?

마이크로서비스 아키텍처(Microservices Architecture, MSA) 는 하나의 큰 애플리케이션을 작고 독립적인 서비스들의 집합으로 구성하는 아키텍처 스타일입니다. 각 서비스는 자체적인 비즈니스 도메인을 가지며, 독립적으로 배포·확장·운영할 수 있습니다.

Netflix, Amazon, Uber와 같은 대형 서비스들이 트래픽과 조직 규모가 커지면서 모놀리식의 한계를 극복하기 위해 도입한 패턴입니다.

핵심 특성

마이크로서비스를 정의하는 일반적인 특성은 다음과 같습니다.

  • 단일 책임: 한 서비스는 하나의 비즈니스 기능에 집중
  • 독립 배포: 다른 서비스에 영향 없이 배포 가능
  • 분산 데이터 관리: 각 서비스가 자체 데이터베이스를 보유
  • 언어/기술 다양성: 서비스마다 적합한 기술 스택 선택
  • 자동화된 인프라: CI/CD, IaC, 컨테이너 오케스트레이션 필수

서비스 간 통신

서비스 간 통신은 크게 동기와 비동기 방식으로 나뉩니다.

동기 통신 (REST/gRPC)

비동기 통신 (메시지 브로커)

비동기 방식은 서비스 간 결합도를 낮추고 장애 격리에 유리합니다.

주요 구성 요소

MSA를 구성하기 위해 일반적으로 다음과 같은 인프라가 필요합니다.

구성 요소 역할 대표 도구
API Gateway 라우팅, 인증, 트래픽 제어 Spring Cloud Gateway, Kong
Service Discovery 서비스 위치 등록/조회 Eureka, Consul
Config Server 중앙 설정 관리 Spring Cloud Config
Message Broker 비동기 메시징 Kafka, RabbitMQ
Container Orchestrator 배포/스케일링 Kubernetes
Observability 로그/메트릭/트레이싱 Prometheus, Grafana, Jaeger

장점과 단점

장점

  • 서비스별 독립 배포로 릴리스 주기 단축
  • 부분 수평 확장 가능 (병목 서비스만 스케일아웃)
  • 작은 팀이 자율적으로 서비스를 소유
  • 장애 격리로 시스템 전체 가용성 향상

단점

  • 분산 시스템 복잡도 증가 (네트워크, 일관성, 트랜잭션)
  • 운영·모니터링 비용 상승
  • 서비스 간 API 호환성 관리 필요
  • 데이터 정합성 확보가 까다로움 (Saga 패턴 등 필요)

도입 시 고려사항

마이크로서비스가 항상 정답은 아닙니다. 다음 항목을 점검하세요.

  1. 조직 규모: 콘웨이 법칙에 따라 팀 구조가 아키텍처를 결정합니다.
  2. DevOps 성숙도: 자동화된 빌드/배포/관측 없이 MSA는 운영 부담만 커집니다.
  3. 도메인 명확성: 잘못된 서비스 경계는 더 큰 결합을 만듭니다.
  4. 트래픽/규모: 작은 서비스라면 모놀리식이 더 효율적입니다.

처음에는 모듈러 모놀리식으로 시작해 도메인이 명확해지면 점진적으로 분리하는 Strangler Fig 전략이 권장됩니다.

마무리

마이크로서비스는 큰 시스템의 복잡성을 분할 정복으로 다루기 위한 강력한 도구지만, 그 자체로 분산 시스템의 어려움을 해결해 주지는 않습니다. 도메인 경계 설계, 운영 자동화, 관측 가능성 확보가 성공의 핵심입니다.
