Java Sealed 클래스와 인터페이스로 도메인 모델링하기

작성일 2026-04-02 Edited on 2026-05-22 In Backend , Java
Java 17에서 정식화된 sealed 클래스와 인터페이스의 문법, permits 절, exhaustive switch와의 결합, 도메인 모델링에서의 실전 활용을 정리합니다.

Sealed 타입이 등장한 배경

Java의 클래스 계층은 전통적으로 두 극단을 오갔습니다. 완전히 열린 상속(public class)이거나, 완전히 닫힌 final 클래스거나. 그 사이에 “내가 정한 몇 개의 자식만 허용하고 싶다”는 요구가 항상 있어 왔지만, 이는 패키지-프라이빗 트릭이나 enum으로 우회해야 했습니다.

Java 15에서 프리뷰로 도입되어 Java 17에서 정식 기능이 된 sealed 타입은 이 빈틈을 메웁니다. sealed 키워드와 permits 절을 사용해 어떤 클래스·인터페이스가 상속·구현할 수 있는지 명시적으로 선언할 수 있습니다.

기본 문법

Payment 인터페이스는 정확히 3개의 구현체만 허용합니다. 외부에서 임의로 implements Payment를 시도하면 컴파일 에러가 발생합니다.

하위 타입의 3가지 선택지

sealed 타입을 상속·구현하는 자식 타입은 반드시 다음 중 하나로 선언되어야 합니다.

  • final — 더 이상 상속 불가 (가장 일반적)
  • sealed — 또 다른 permits 목록 정의 (계층을 깊게 가져갈 때)
  • non-sealed — 봉인을 풀고 자유로운 상속 허용

permits 절 생략 규칙

permits 절은 모든 자식 타입이 같은 모듈, 또는 같은 패키지에 있을 때 생략할 수 있습니다. 같은 파일 안에서 모두 정의한다면 가장 간결합니다.

Exhaustive Switch와의 결합

sealed의 진짜 매력은 패턴 매칭과 결합될 때 드러납니다. 컴파일러는 sealed 타입의 모든 하위 타입을 알고 있기 때문에, switch에서 모든 경우를 다뤘는지 검증해 줍니다. default 절 없이도 안전한 분기가 가능합니다.

만약 나중에 KakaoPay라는 새로운 결제 수단을 permits에 추가했는데 switch는 갱신하지 않았다면, 컴파일 시점에 “exhaustiveness 부족” 에러가 발생합니다. 타입 시스템이 도메인 변경을 강제로 추적해 주는 효과가 있습니다.

도메인 모델링: Result, State, Event

sealed는 함수형 언어의 ADT(대수적 데이터 타입)와 비슷한 표현을 자바에 가져옵니다. 결과 타입, 상태 머신, 이벤트 정의 등에 잘 어울립니다.

Result 타입

주문 상태 머신

각 상태가 고유한 데이터를 가질 수 있고, switch로 분기하면 각 상태에 맞는 필드에 안전하게 접근할 수 있습니다.

enum vs sealed

비슷해 보이지만 차이가 있습니다.

  • enum: 상수의 집합. 각 값이 서로 다른 데이터를 갖기 어렵고, 인스턴스가 고정됨.
  • sealed: 타입의 집합. 각 하위 타입이 서로 다른 구조의 데이터를 가질 수 있고, 인스턴스를 다양하게 만들 수 있음.

“상태 자체는 같은데 라벨만 다르다”면 enum, “상태마다 들고 있는 데이터가 다르다”면 sealed가 맞습니다.

리플렉션과 도구 지원

sealed 정보는 런타임에도 보존됩니다.

마무리

Sealed 타입은 자바 타입 시스템에 “닫힌 계층”을 도입하는 작지만 강력한 변화입니다. 패턴 매칭과 결합하면 분기 누락을 컴파일러가 잡아 주어, 도메인이 변할 때 영향을 받는 코드를 즉시 알 수 있습니다. Record와 함께 활용하면 표현력 높은 자바 코드를 작성할 수 있습니다.
