Observer 패턴과 이벤트 시스템 설계

작성일 2026-05-14 Edited on 2026-05-15 In Backend
객체 간 일대다 의존 관계를 정의하는 Observer 패턴의 구조와 실전 이벤트 시스템 구현 방법을 알아봅니다.

Observer 패턴이란?

Observer 패턴은 한 객체(Subject)의 상태가 변할 때, 그것에 의존하는 다른 객체들(Observer)에게 자동으로 알림이 가도록 설계하는 행동(Behavioral) 디자인 패턴입니다. 발행-구독(Publish-Subscribe) 모델의 가장 기본적인 형태로, 객체 간 결합도를 낮추면서 상태 변화를 전파할 수 있게 해줍니다.

GUI 이벤트 처리, 메시지 브로커, 반응형 프로그래밍 등 다양한 영역에서 사용됩니다.

구조

  • Subject(Publisher): 관찰 대상. Observer 목록 관리, 등록/해제, 알림 발송 책임
  • Observer(Subscriber): Subject로부터 알림을 받아 처리하는 객체
  • ConcreteSubject / ConcreteObserver: 구체 구현체

Java 예제

Push vs Pull

  • Push 방식: Subject가 변경된 데이터를 Observer에 직접 전달. 간단하지만 Observer가 필요 없는 정보까지 받음
  • Pull 방식: Subject는 변경 사실만 알리고, Observer가 필요 시 Subject를 조회. 유연하지만 Observer가 Subject에 의존

자주 발생하는 함정

  1. 메모리 누수: Observer를 unsubscribe 하지 않으면 GC 대상이 되지 않음. WeakReference나 명시적 해제 필요
  2. 순환 알림: Observer가 다시 Subject 상태를 바꾸면 무한 루프 위험
  3. 동기 처리: 한 Observer가 느리면 전체 흐름이 막힘. 비동기/이벤트 큐 도입 고려
  4. 예외 전파: 한 Observer의 예외가 다른 Observer 알림을 막지 않도록 격리

실전 응용

Spring 외에도 RxJava/Reactor의 Flux, JavaScript의 EventEmitter, DOM 이벤트 리스너 모두 Observer 패턴의 변형입니다.

이벤트 시스템 설계 팁

  • 이벤트는 불변(Immutable) 객체로 정의
  • 이벤트 이름은 과거형(OrderPlaced, UserRegistered)
  • 도메인 이벤트와 통합 이벤트(다른 시스템에 전달)를 구분
  • 트랜잭션 경계와 이벤트 발행 시점을 명확히 (Outbox 패턴 고려)

마무리

Observer 패턴은 결합도를 낮추고 확장성을 높이는 강력한 도구이지만, 알림 순서, 메모리 관리, 동기/비동기 처리 같은 운영 이슈를 함께 고려해야 합니다. 대규모 시스템이라면 메시지 큐나 이벤트 버스 인프라로 한 단계 추상화하는 것을 권장합니다.
