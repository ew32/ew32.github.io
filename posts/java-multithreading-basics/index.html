Java 멀티스레딩 기초

작성일 2026-05-08 Edited on 2026-05-15 In Backend , Java
Java에서 스레드를 생성하고 동기화하는 기본 메커니즘과 동시성 도구를 소개합니다.

왜 멀티스레딩인가

현대의 CPU는 멀티 코어이며, 단일 스레드 프로그램은 코어의 일부만을 사용합니다. 멀티스레딩은 여러 작업을 동시에 처리해 응답성과 처리량을 높이는 핵심 기술입니다. 다만, 동시 실행이 가져오는 공유 자원 경쟁과 예측 불가능성을 다루지 못하면 데이터가 손상되거나 데드락이 발생할 수 있습니다.

스레드 생성

가장 기본적인 방법은 Thread 클래스 또는 Runnable 인터페이스를 사용하는 것입니다.

Thread를 상속하기보다는 Runnable을 구현하는 것이 일반적입니다. Java 21부터는 가벼운 Virtual Thread도 사용할 수 있습니다.

ExecutorService

실무에서는 스레드를 직접 생성하기보다 ExecutorService 로 관리합니다.

Executor의 종류는 다음과 같습니다.

생성 메서드 특징
newFixedThreadPool(n) 고정 크기 풀
newCachedThreadPool() 필요 시 생성, 60초 후 회수
newSingleThreadExecutor() 단일 스레드
newScheduledThreadPool(n) 지연/주기 실행
newVirtualThreadPerTaskExecutor() Java 21+ 가상 스레드

공유 자원과 동기화

여러 스레드가 같은 변수에 접근하면 race condition이 발생합니다.

count++는 읽기-증가-쓰기 세 단계라 동시에 호출되면 손실이 발생합니다. 해결 방법은 여러 가지입니다.

synchronized

암시적인 객체 락을 사용하며, 메서드 또는 블록 단위로 적용 가능합니다. 단순하지만 락 범위가 커지면 성능이 떨어집니다.

java.util.concurrent.atomic

원자적 연산을 제공하는 클래스로 락 없이 동시성을 처리합니다.

ReentrantLock

tryLock, 공정성, 인터럽트 가능 락 등 synchronized보다 유연합니다.

volatile과 가시성

Java 메모리 모델은 스레드별 캐시를 허용하기 때문에 한 스레드의 쓰기가 다른 스레드에 즉시 보이지 않을 수 있습니다. volatile 키워드는 읽기/쓰기를 메인 메모리에 직접 반영하도록 보장합니다.

다만 volatile은 원자성을 보장하지 않으므로 count++ 같은 합성 연산에는 사용할 수 없습니다.

동시성 컬렉션

java.util.concurrent 패키지의 컬렉션은 멀티스레드에서 안전합니다.

  • ConcurrentHashMap: 세그먼트 락 기반의 해시맵
  • CopyOnWriteArrayList: 쓰기 시 배열 복사, 읽기 다중 시 유리
  • BlockingQueue 구현체: 생산자-소비자 패턴

CompletableFuture

비동기 결과를 조합하는 모던 API입니다.

콜백 지옥 없이 의존성 그래프를 표현할 수 있고, 예외 처리도 일관적입니다.

데드락 회피

두 스레드가 서로 다른 락을 점유한 채 상대의 락을 기다리면 데드락이 발생합니다.

  • 락 획득 순서를 전역적으로 통일
  • tryLock(timeout)으로 회피 가능성을 열어두기
  • 가능한 한 락 범위를 좁히고 한 번에 하나의 락만 보유

모범 사례

  • 스레드를 직접 만들기보다 ExecutorService 또는 가상 스레드 사용
  • 공유 가변 상태를 줄이고 불변 객체와 메시지 패싱을 선호
  • 동시성 도구는 본인이 의미를 정확히 아는 것만 사용
  • 부하 테스트로 race condition을 적극 노출

마무리

멀티스레딩은 강력하지만 함정이 많은 영역입니다. synchronized, volatile, atomic, 동시성 컬렉션의 의미 차이를 정확히 이해하고, 추상 수준이 높은 도구(ExecutorService, CompletableFuture, Virtual Thread)를 우선 활용하는 것이 안전하고 생산적인 길입니다.
