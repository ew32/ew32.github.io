Git 브랜치 전략 비교 (Git Flow, GitHub Flow, Trunk Based)

작성일 2026-05-02 Edited on 2026-05-19 In 개발환경
대표적인 Git 브랜치 전략의 특징과 장단점을 비교하여 프로젝트에 맞는 전략을 선택하는 기준을 제시합니다.

브랜치 전략이 필요한 이유

팀 단위로 코드를 관리할 때 명확한 브랜치 전략 없이 개발하면 충돌이 잦아지고 릴리즈 관리가 어려워집니다. 브랜치 전략은 팀의 협업 규칙이며, 프로젝트 규모와 배포 주기에 맞춰 선택해야 합니다.

Git Flow

Vincent Driessen이 2010년에 제안한 전통적인 브랜치 모델입니다.

주요 브랜치

  • master (또는 main): 제품 출시 버전 관리
  • develop: 다음 릴리즈를 위한 통합 브랜치
  • feature/*: 새로운 기능 개발
  • release/*: 릴리즈 준비 브랜치
  • hotfix/*: 출시된 버전의 긴급 수정

워크플로

  1. develop에서 feature/login 브랜치 생성
  2. 기능 개발 완료 후 develop에 머지
  3. 릴리즈 시점에 release/1.2.0 생성, 테스트 및 버그 수정
  4. master와 develop에 머지 후 태그 부여
  5. 운영 환경 버그는 hotfix/*로 처리

장단점

  • 장점: 명확한 역할 구분, 다중 버전 관리에 유리
  • 단점: 브랜치가 복잡, CD(지속적 배포)에 부적합

GitHub Flow

GitHub에서 제안한 단순화된 모델입니다.

주요 규칙

  • main 브랜치는 항상 배포 가능한 상태
  • 모든 작업은 main에서 분기한 feature 브랜치에서 진행
  • Pull Request로 코드 리뷰 후 머지
  • 머지 즉시 배포

장단점

  • 장점: 단순, 빠른 배포 주기에 적합, 학습 곡선 낮음
  • 단점: 여러 버전을 동시 유지하기 어려움

Trunk Based Development

대형 IT 기업(Google, Facebook 등)이 선호하는 전략입니다.

주요 규칙

  • 모든 개발자가 main(trunk)에 직접 또는 매우 짧은 브랜치를 통해 커밋
  • 브랜치 수명은 보통 하루 이내
  • 미완성 기능은 Feature Flag로 숨김
  • 강력한 CI 파이프라인과 자동화된 테스트가 필수

장단점

  • 장점: 머지 충돌 최소화, 진정한 CI/CD 가능
  • 단점: 테스트 자동화 수준이 높아야 함, 작은 단위 커밋 문화 필요

비교 정리

전략 적합한 환경 배포 주기 복잡도
Git Flow 패키지 소프트웨어, 다중 버전 지원 주/월 단위 높음
GitHub Flow 웹 서비스, SaaS 일 단위 낮음
Trunk Based 대규모 조직, 강력한 CI 시간/분 단위 중간 (자동화 의존)

어떤 전략을 선택할까?

  • 소규모 팀 + 빠른 배포: GitHub Flow
  • 모바일 앱처럼 다중 버전 운영: Git Flow
  • CI/CD가 잘 갖춰진 대규모 팀: Trunk Based

실무 팁

  1. 브랜치 명명 규칙: feature/JIRA-123-login-page처럼 이슈 번호 포함
  2. 머지 방식: PR 머지 시 squash merge로 깔끔한 히스토리 유지
  3. 보호 규칙: main 브랜치는 직접 푸시 금지, PR 리뷰 필수
  4. 자동 삭제: 머지된 브랜치는 자동 삭제로 정리

마무리

완벽한 브랜치 전략은 없습니다. 팀 규모, 배포 주기, 테스트 자동화 수준에 따라 적절히 선택하고, 필요에 따라 변형해서 사용하는 것이 좋습니다.
