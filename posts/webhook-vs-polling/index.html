Webhook vs Polling - 무엇을 언제 써야 할까

작성일 2026-06-17 Edited on 2026-06-05 In Network , HTTP
폴링과 웹훅의 동작 원리, 장단점, 보안 검증 방법까지 실전 관점에서 정리합니다.

두 방식이 풀려는 문제

서버 A에서 어떤 사건이 일어났을 때 서버 B에 그 사실을 알려야 한다고 합시다. 가장 단순한 방법은 B가 주기적으로 “변한 게 있어?”라고 묻는 것(폴링)이고, 더 영리한 방법은 A가 사건이 발생한 시점에 B에게 직접 알려주는 것(웹훅)입니다. 둘은 같은 문제를 정반대 방향에서 해결합니다.

Polling: 클라이언트가 주기적으로 묻는다

폴링은 클라이언트가 정해진 주기로 서버에 요청을 보내 변경 여부를 확인하는 방식입니다.

장점은 단순함과 방화벽 친화성입니다. B는 외부에서 들어오는 연결을 열어둘 필요가 없습니다. 단점은 명확합니다.

  • 사건이 없어도 계속 요청 → 대부분의 응답이 비어 있음
  • 폴링 주기에 따라 지연이 결정됨 (5초 주기 = 평균 2.5초 지연)
  • 트래픽 비용과 서버 부하 증가

Long Polling: 폴링의 개량형

서버가 즉시 빈 응답을 주지 않고, 사건이 발생할 때까지 응답을 보류하는 방식입니다. 변화가 생기면 그 즉시 응답을 내려보내고 클라이언트는 곧바로 새로운 요청을 보냅니다. 거의 실시간에 가까운 지연을 얻지만, 서버 측에서 다수의 열린 연결을 관리해야 합니다.

Webhook: 서버가 사건이 일어나면 알려준다

웹훅은 사건 발생 시 서버가 미리 등록된 URL로 HTTP 요청(보통 POST)을 보내는 방식입니다.

장점은 실시간성과 효율성입니다. 변화가 없을 때는 트래픽이 0이고, 변화가 있을 때만 즉시 통지됩니다. 단점은 수신 측이 공개 엔드포인트와 일정 수준의 가용성을 갖추어야 한다는 점, 그리고 보안과 재시도 처리가 까다롭다는 점입니다.

웹훅 보안: 서명 검증

웹훅 URL은 공개되어 있으므로 누구든 가짜 페이로드를 보낼 수 있습니다. 따라서 발신자는 HMAC 서명을 함께 보내고, 수신자는 이를 검증해야 합니다.

멱등성과 재시도

네트워크가 불안정하면 같은 웹훅이 여러 번 도착할 수 있습니다. 수신 측은 이벤트 ID를 키로 하는 멱등 처리 테이블을 두는 것이 안전합니다.

요청이 들어오면 먼저 INSERT ... ON CONFLICT DO NOTHING을 시도하고, 충돌이 나면 이미 처리한 이벤트로 간주합니다.

선택 기준

상황 권장
외부 시스템에 엔드포인트 노출 어려움 Polling
단순한 상태 조회, 트래픽 적음 Polling
결제, 알림 등 실시간성이 중요 Webhook
대규모 이벤트 (수천 TPS) Webhook + 큐
양방향 실시간 통신 WebSocket (별도 글 참조)

하이브리드 패턴

실무에서는 웹훅이 누락될 수 있다는 가정 하에 폴링을 보조 수단으로 사용하기도 합니다. 예를 들어 결제 시스템은 웹훅으로 즉시 통지하되, 클라이언트는 결제 완료 페이지에서 5초마다 상태를 한 번 더 확인합니다. 두 방식을 함께 쓰면 신뢰성과 실시간성을 동시에 챙길 수 있습니다.

마무리

폴링은 단순하고 안전하지만 낭비가 많고, 웹훅은 효율적이지만 운영 비용이 듭니다. “이벤트 빈도”와 “실시간성 요구”라는 두 축으로 선택하면 거의 실패하지 않습니다.
