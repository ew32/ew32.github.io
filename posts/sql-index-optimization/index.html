SQL 인덱스 최적화 가이드

작성일 2026-05-05 Edited on 2026-06-05 In Backend , Database
인덱스의 동작 원리부터 복합 인덱스 설계, 실행 계획 분석까지 실무에서 바로 쓰는 최적화 노하우를 정리합니다.

인덱스란?

인덱스(Index)는 책의 색인과 같습니다. 책에서 특정 단어를 찾을 때 전체를 읽지 않고 색인을 보듯이, DB도 인덱스를 통해 원하는 데이터를 빠르게 찾을 수 있습니다.

인덱스의 본질은 추가 저장 공간을 사용해 검색 속도를 끌어올리는 트레이드오프입니다. 무조건 많이 만든다고 좋은 것이 아닙니다.

인덱스의 동작 원리

대부분의 RDBMS는 B-Tree (정확히는 B+Tree) 구조를 사용합니다.

B-Tree의 특징

  • 정렬된 상태로 데이터 유지
  • 균형 트리 → 어떤 데이터를 찾아도 거의 동일한 시간 (O(log N))
  • 범위 검색에 유리
  • 리프 노드끼리 연결 리스트로 이어져 있음

인덱스가 빠른 이유

1만 건 테이블에서 특정 값을 찾는다고 가정하면:

  • 풀 스캔: 평균 5,000번 비교
  • B-Tree 인덱스(차수 100 가정): 약 3번의 노드 탐색

인덱스가 효과적인 경우

인덱스가 무력화되는 경우

복합 인덱스(Composite Index)

여러 컬럼을 묶은 인덱스로, 컬럼 순서가 매우 중요합니다.

이 인덱스는 다음 조건에서만 효과적입니다:

  • WHERE user_id = ?
  • WHERE user_id = ? AND status = ?

다음에는 사용되지 않습니다:

  • WHERE status = ? (선두 컬럼 누락)

컬럼 순서 결정 원칙

  1. 선택도(Selectivity)가 높은 컬럼을 앞에 (고유값이 많을수록 좋음)
  2. 등호 조건을 범위 조건보다 앞에
  3. 자주 사용되는 조건을 앞에

커버링 인덱스(Covering Index)

쿼리에 필요한 모든 컬럼이 인덱스에 포함되어 있어, 테이블 접근 없이 인덱스만으로 결과를 만들 수 있는 인덱스입니다.

실행 계획 분석

MySQL/PostgreSQL에서는 EXPLAIN으로 쿼리의 실행 계획을 확인할 수 있습니다.

주요 확인 사항

  • type: ALL(풀 스캔)은 위험, ref/range/const 등이 좋음
  • key: 실제 사용된 인덱스
  • rows: 예상 검색 행 수 (적을수록 좋음)
  • Extra: Using index(커버링)는 좋음, Using filesort/Using temporary는 주의

인덱스의 비용

1. 쓰기 성능 저하

INSERT/UPDATE/DELETE 시 인덱스도 함께 갱신되어야 합니다. 인덱스가 많을수록 쓰기 성능이 떨어집니다.

2. 저장 공간

인덱스 자체가 디스크 공간을 차지합니다. 큰 테이블에 인덱스를 여러 개 만들면 데이터보다 인덱스가 더 클 수도 있습니다.

3. 옵티마이저 혼란

유사한 인덱스가 많으면 옵티마이저가 잘못된 인덱스를 선택할 수 있습니다.

인덱스 설계 체크리스트

  • 자주 사용되는 WHERE/JOIN/ORDER BY 컬럼 식별
  • 선택도가 충분히 높은가? (10% 미만 권장)
  • 복합 인덱스의 컬럼 순서가 적절한가?
  • 중복되거나 사용되지 않는 인덱스는 없는가?
  • 쓰기 빈도가 높은 테이블이라면 인덱스 수를 최소화했는가?
  • 실행 계획으로 실제 사용 여부를 확인했는가?

통계 정보 갱신

옵티마이저는 통계 정보를 기반으로 실행 계획을 세웁니다. 데이터가 많이 변경되었다면 통계를 갱신해줘야 합니다.

마무리

인덱스는 강력하지만 만능이 아닙니다. 측정 없는 최적화는 추측일 뿐이므로, 반드시 실행 계획과 실제 쿼리 응답 시간을 기준으로 인덱스를 추가하거나 제거하는 습관을 들여야 합니다.
