gRPC와 Protocol Buffers 입문

작성일 2026-05-22 Edited on 2026-06-05 In Backend
고성능 RPC 프레임워크 gRPC의 동작 원리, Protocol Buffers 스키마 작성법, 그리고 REST 대비 장단점을 정리합니다.

gRPC란?

gRPC는 Google이 만들어 CNCF에 기증한 오픈소스 RPC(Remote Procedure Call) 프레임워크입니다. 클라이언트가 마치 로컬 함수처럼 원격 메서드를 호출할 수 있게 해주며, 다음 특징을 가집니다.

  • HTTP/2 기반: 다중화, 헤더 압축, 양방향 스트리밍
  • Protocol Buffers: 바이너리 직렬화 포맷으로 작고 빠름
  • 다국어 지원: Java, Go, Python, Node.js, C++, Kotlin 등 다양한 언어용 코드 자동 생성
  • 계약 기반: .proto 스키마가 곧 API 명세

Protocol Buffers 기본

.proto 파일에 메시지와 서비스를 정의합니다.

protoc 컴파일러로 각 언어의 클라이언트/서버 스텁을 생성합니다.

4가지 RPC 종류

  1. Unary: 일반 요청/응답
  2. Server Streaming: 클라이언트 요청 1 → 서버 응답 N
  3. Client Streaming: 클라이언트 요청 N → 서버 응답 1
  4. Bidirectional Streaming: 양방향 N:N

성능 특성

  • Protobuf 바이너리 + HTTP/2 다중화로 JSON/REST 대비 응답 크기 30~50%↓, 처리량 2배 이상이 일반적
  • 단일 TCP 연결로 다수 요청 동시 처리(헤드 오브 라인 블로킹 완화)
  • 정적 타입 + 코드 자동 생성으로 클라이언트 작성 비용 감소

REST 대비 장단점

장점

  • 고성능, 저지연
  • 강한 계약(스키마)
  • 양방향/스트리밍 지원
  • 다국어 클라이언트 자동 생성

단점

  • 사람이 읽기 어려운 바이너리 포맷 (디버깅 도구 필요)
  • 브라우저에서 직접 호출 불가 (gRPC-Web 또는 게이트웨이 필요)
  • HTTP/1.1 인프라(일부 프록시)와 호환 이슈
  • 학습 곡선과 툴체인 의존(protoc)

gRPC-Web과 게이트웨이

브라우저는 일반 gRPC를 호출할 수 없습니다. 두 가지 우회책이 있습니다.

  1. gRPC-Web: 프록시(Envoy)가 gRPC-Web을 일반 gRPC로 변환
  2. grpc-gateway: .proto에 HTTP 매핑을 추가하면 REST/JSON 게이트웨이를 자동 생성

사용처

  • 마이크로서비스 간 내부 통신
  • 모바일-백엔드 통신(저대역 환경에서 효율적)
  • 실시간 양방향 통신(채팅, IoT 텔레메트리, 영상 신호 처리)
  • 다국어 폴리글랏 환경

스키마 진화 팁

  • 필드 번호는 영원히 재사용 금지
  • 필드 삭제 대신 reserved로 예약
  • 필수(required) 사용 지양 (proto3는 기본적으로 optional)
  • 클라이언트는 모르는 필드를 무시하도록 동작

마무리

gRPC는 내부 마이크로서비스 통신에 거의 표준에 가까운 선택지가 되고 있습니다. 성능과 강력한 계약이 필요하다면 적극 고려할 만하며, 외부 공개 API는 REST/GraphQL, 내부 통신은 gRPC로 역할을 나누는 하이브리드 구성이 가장 실용적입니다.
