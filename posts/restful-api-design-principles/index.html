RESTful API 설계 원칙

작성일 2026-05-03 Edited on 2026-05-20 In Backend
REST의 핵심 제약 조건과 실무에서 적용 가능한 API 설계 모범 사례를 정리합니다.

REST란?

REST(Representational State Transfer)는 2000년 Roy Fielding의 박사 학위 논문에서 처음 제시된 분산 시스템을 위한 아키텍처 스타일입니다. HTTP 프로토콜의 특성을 잘 활용하도록 설계되었으며, 오늘날 가장 널리 사용되는 API 설계 방식입니다.

REST의 6가지 제약 조건

  1. 클라이언트-서버: 관심사의 분리
  2. 무상태(Stateless): 각 요청은 독립적이며 서버는 클라이언트 상태를 저장하지 않음
  3. 캐시 가능(Cacheable): 응답은 캐시 가능 여부가 명시되어야 함
  4. 계층형 시스템: 클라이언트는 직접 연결된 서버 외 다른 계층을 인지하지 못함
  5. 균일한 인터페이스: 자원 식별, 표현을 통한 자원 조작, 자기 서술적 메시지, HATEOAS
  6. Code on demand (선택): 서버가 클라이언트에 실행 가능한 코드 전송 가능

HTTP 메서드와 의미

메서드 용도 멱등성 안전성
GET 자원 조회 O O
POST 자원 생성 X X

PATCH 자원 부분 수정 X X
DELETE 자원 삭제 O X

멱등성(Idempotent): 같은 요청을 여러 번 보내도 동일한 결과
안전성(Safe): 자원의 상태를 변경하지 않음

HTTP 상태 코드 활용

성공 (2xx)

  • 200 OK: 일반적인 성공
  • 201 Created: 생성 성공 (POST)
  • 204 No Content: 성공했으나 응답 본문 없음 (DELETE)

클라이언트 오류 (4xx)

  • 400 Bad Request: 잘못된 요청
  • 401 Unauthorized: 인증 필요
  • 403 Forbidden: 권한 없음
  • 404 Not Found: 자원 없음
  • 409 Conflict: 상태 충돌
  • 422 Unprocessable Entity: 검증 실패

서버 오류 (5xx)

  • 500 Internal Server Error: 일반적인 서버 오류
  • 503 Service Unavailable: 서비스 일시 중단

페이지네이션

  • Offset 기반: 구현 간단, 데이터 변경 시 중복/누락 가능
  • Cursor 기반: 대용량/실시간 데이터에 적합

필터링, 정렬, 검색

버저닝 전략

  1. URI 버저닝: /v1/users (가장 명확)
  2. 헤더 버저닝: Accept: application/vnd.api.v1+json
  3. 쿼리 파라미터: /users?version=1

마무리
