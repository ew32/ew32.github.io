Service Worker와 PWA 기초

작성일 2026-06-22 Edited on 2026-05-18 In Frontend
Service Worker 생명주기, 캐시 전략, PWA 매니페스트와 오프라인 동작까지 기본을 정리합니다.

PWA와 Service Worker

PWA(Progressive Web App)는 웹의 보편성과 네이티브 앱의 경험을 결합한 접근 방식입니다. 그 핵심 엔진이 바로 Service Worker입니다. Service Worker는 브라우저와 네트워크 사이에 위치한 별도의 워커 스레드로, 페이지의 모든 요청을 가로채 캐시 응답을 돌려주거나 백그라운드에서 동기화를 수행할 수 있습니다.

Service Worker가 만족시켜야 하는 전제 조건이 있습니다.

  • HTTPS 필수 (localhost 예외).
  • DOM에 직접 접근 불가, window 객체 없음.
  • 비동기 API만 사용 (LocalStorage 불가).
  • 별도 스코프(보통 사이트 루트)에 등록.

등록과 생명주기

Service Worker는 다음 상태를 거칩니다.

  1. install: 처음 등록되거나 파일이 바뀌면 실행. 보통 캐시를 준비합니다.
  2. activate: 이전 버전의 SW를 대체하며 활성화. 오래된 캐시를 정리합니다.
  3. fetch / message / sync / push: 활성 상태에서 이벤트를 처리합니다.

기본 SW 코드

skipWaiting과 clients.claim은 새 SW가 즉시 활성화·제어를 가져오게 합니다.

캐시 전략

자원의 성격에 따라 다음 네 가지 전략을 조합합니다.

Cache First (정적 자산용)

Network First (HTML, API용)

Stale-While-Revalidate (자주 변하지만 최신성 덜 중요한 자원)

Network Only / Cache Only

결제 같은 민감한 API는 Network Only로, 빌드 시 동봉한 폰트는 Cache Only로 둘 수 있습니다.

Web App Manifest

PWA로 인식되려면 manifest.webmanifest가 필요합니다.

HTML에서 연결합니다.

푸시 알림과 백그라운드 동기화

디버깅과 함정

  • Chrome DevTools → Application → Service Workers에서 강제 업데이트, 오프라인 시뮬레이션, 캐시 검사가 가능합니다.
  • SW 파일은 반드시 변경 가능한 URL(보통 /sw.js)로 서비스하고, 캐시 정책은 no-cache로 두세요. 그렇지 않으면 새 버전이 영원히 배포되지 않을 수 있습니다.
  • 캐시 키에 버전 문자열을 포함시켜 청소 시점을 명확히 만드세요.
  • 워크박스(Workbox)를 사용하면 위 전략들이 간결한 API로 추상화됩니다.

마무리

Service Worker는 강력하지만 그만큼 디버깅이 까다롭습니다. 작게 시작하세요. “정적 자산은 Cache First, HTML은 Network First, API는 그대로 두기”라는 최소 구성만으로도 오프라인 동작과 체감 속도 향상을 모두 누릴 수 있습니다.
