브라우저 렌더링 과정 - Critical Rendering Path

작성일 2026-05-22 Edited on 2026-06-15 In Frontend , HTML
HTML이 화면에 그려지기까지 브라우저가 거치는 단계와 최적화 포인트를 정리합니다.

Critical Rendering Path란?

**Critical Rendering Path(CRP)**는 브라우저가 HTML, CSS, JavaScript를 받아 화면의 픽셀로 변환하는 과정입니다. 이 경로의 각 단계를 이해하면 성능 병목을 정확히 짚어내고, **첫 페인트(First Paint)**와 LCP를 효과적으로 단축할 수 있습니다.

CRP는 크게 다음 단계로 구성됩니다.

  1. DOM 구성
  2. CSSOM 구성
  3. Render Tree 결합
  4. Layout (Reflow)
  5. Paint
  6. Composite

1단계: DOM 구성

브라우저는 HTML을 받으면 토큰화 → 노드 생성 → 트리 구축 순서로 DOM 트리를 만듭니다.

위 HTML은 다음과 같은 트리 구조가 됩니다.

HTML 파싱은 점진적(streaming)으로 수행되어, 일부분만 받아도 처리가 시작됩니다.

2단계: CSSOM 구성

CSS 파일은 HTML 파싱 중 발견되는 즉시 요청되며, 별도의 CSSOM(CSS Object Model) 트리로 구성됩니다.

CSS는 **렌더링 차단 리소스(render-blocking resource)**입니다. CSSOM이 완성되기 전까지 브라우저는 Render Tree를 만들 수 없으므로 화면을 그리지 못합니다.

3단계: Render Tree 결합

DOM과 CSSOM이 결합되어 Render Tree가 생성됩니다. 이때 display: none 요소나 <head>, <script> 등 시각적으로 표현되지 않는 노드는 제외됩니다.

속성 Render Tree 포함 여부
display: none 제외
visibility: hidden 포함 (공간 차지)
opacity: 0 포함

4단계: Layout (Reflow)

각 노드의 정확한 위치와 크기를 계산하는 단계입니다. 뷰포트 크기, 박스 모델, 텍스트 줄바꿈 등이 모두 여기서 결정됩니다.

다음 작업은 Layout을 재발생시킵니다.

  • 윈도우 리사이즈
  • width, height, margin, padding 변경
  • 폰트 변경
  • DOM 노드 추가/제거
  • offsetWidth, scrollTop 등 강제 동기 레이아웃 트리거 속성 읽기

5단계: Paint

Layout이 완료되면 각 노드를 실제 픽셀로 변환합니다. 텍스트, 색상, 이미지, 보더, 그림자 등이 비트맵으로 그려집니다.

다음 속성 변경은 Layout 없이 Paint만 다시 발생시킵니다.

  • color, background-color
  • box-shadow
  • border-radius

6단계: Composite

Paint된 결과는 여러 레이어로 나뉘어 합성됩니다. GPU에서 처리되어 매우 효율적입니다.

Composite만 발생시키는 속성: transform, opacity

JavaScript의 영향

<script> 태그는 파서 차단(parser-blocking) 리소스로, HTML 파싱을 중단시킵니다.

속성 다운로드 실행 시점 순서 보장
없음 HTML 파싱 차단 다운로드 즉시 보장
async 비동기 다운로드 직후 미보장
defer 비동기 DOMContentLoaded 직전 보장

레이어 생성 트리거

다음 속성은 별도 합성 레이어(compositing layer)를 만들어 성능을 향상시킵니다.

단, 과도한 레이어는 메모리를 소모하므로 필요한 곳에만 사용해야 합니다.

최적화 체크리스트

  • CSS는 <head>에서 빠르게 로드, 비핵심은 비동기
  • JS는 defer 또는 async 활용
  • 애니메이션은 transform과 opacity 위주
  • 강제 동기 레이아웃 방지 (읽기/쓰기 분리)
  • 큰 DOM 변경은 DocumentFragment 사용
  • 자주 변경되는 요소는 will-change 또는 contain 속성

마무리

브라우저 렌더링 과정을 이해하면 어디서 비용이 발생하는지 정확히 파악할 수 있습니다. CRP의 단계마다 차단을 최소화하고, 애니메이션은 합성 단계에서 처리되도록 설계하면 60fps의 부드러운 사용자 경험을 안정적으로 제공할 수 있습니다.
