Helm Chart 입문 - 작성부터 배포까지

작성일 2026-06-30 Edited on 2026-05-15 In Cloud , Infrastructure
Kubernetes 패키지 매니저 Helm의 핵심 개념과 Chart 구조, 템플릿 작성, 배포 방법을 실습 중심으로 정리합니다.

Helm을 사용하는 이유

Kubernetes 매니페스트는 단순한 애플리케이션도 Deployment, Service, ConfigMap, Secret, Ingress 등 여러 YAML이 필요합니다. 환경별로 값만 다른 경우에도 파일을 복제·수정해야 하므로 변경 관리가 어렵습니다. Helm은 매니페스트를 **차트(Chart)**라는 패키지로 묶고, **값(values)**으로 환경 차이를 표현해 이 문제를 해결합니다.

핵심 개념

  • Chart: 템플릿화된 매니페스트 묶음.
  • Release: 차트가 특정 클러스터·네임스페이스에 설치된 인스턴스.
  • Repository: 차트를 배포하는 HTTP 저장소 또는 OCI 레지스트리.
  • Values: 템플릿에 주입되는 값(values.yaml 또는 --set).

차트 구조

생성된 디렉토리는 다음과 같습니다.

Chart.yaml에는 이름, 버전, 앱 버전을 명시합니다.

템플릿 문법

Helm은 Go 템플릿과 Sprig 함수를 사용합니다. 들여쓰기·이름 충돌을 다루기 위한 헬퍼가 풍부합니다.

values.yaml에서 기본값을 정의합니다.

배포와 관리

의존성과 서브차트

Chart.yaml의 dependencies 필드로 다른 차트를 포함할 수 있습니다.

helm dependency update로 의존성을 다운로드한 뒤 함께 배포됩니다.

패키징과 배포

helm package mychart로 .tgz를 만들고 HTTP 저장소나 OCI 레지스트리에 푸시합니다.

모범 사례

  • values.schema.json을 작성해 잘못된 값을 사전 검증합니다.
  • helm lint와 helm template로 CI에서 정적 검증합니다.
  • 시크릿은 차트에 평문으로 두지 말고 SealedSecrets, External Secrets, SOPS 등과 결합합니다.
  • 차트 버전과 앱 버전을 구분 관리해 변경 이력을 명확히 합니다.

마무리

Helm은 Kubernetes 워크로드를 재사용 가능한 패키지로 관리할 수 있게 해 줍니다. 운영 규모가 커질수록 차트의 가치는 크게 늘어나며, ArgoCD·Flux 같은 GitOps 도구와 결합하면 선언적 배포의 핵심 구성 요소가 됩니다.
