ORM vs Query Builder vs Raw SQL 비교

작성일 2026-05-11 Edited on 2026-05-19 In Backend , Database
데이터 접근 계층을 구성하는 세 가지 방식인 ORM, Query Builder, Raw SQL의 장단점과 선택 기준을 정리합니다.

데이터 접근 계층의 세 갈래

애플리케이션이 DB와 대화하는 방식은 크게 세 가지입니다.

  • Raw SQL: 손으로 작성한 SQL을 그대로 실행
  • Query Builder: 프로그래밍 언어로 SQL을 조립
  • ORM(Object-Relational Mapping): 객체 그래프로 DB를 다룸

각각은 추상화 수준이 다르며, 그에 따라 생산성·성능·제어력의 균형이 달라집니다.

Raw SQL

가장 낮은 추상화입니다. JDBC, MyBatis, jdbcTemplate 같은 것이 여기 속합니다.

MyBatis는 XML/어노테이션에 SQL을 두고 결과 매핑을 자동화합니다.

장점

  • SQL을 그대로 보고 튜닝 가능
  • 가장 빠른 실행 경로
  • DB 고유 기능 활용 자유

단점

  • 보일러플레이트 많음
  • 타입 안정성 부족
  • 리팩토링 어려움 (컬럼 이름 변경 등)

Query Builder

코드로 SQL을 조립하지만 객체 그래프로 매핑하지는 않습니다. jOOQ(Java), QueryDSL(Java), Knex(Node.js), SQLAlchemy Core(Python) 등이 대표적입니다.

장점

  • 컴파일 타임에 컬럼/타입 검증
  • SQL과 1:1 가까운 구조라 예측 가능
  • 동적 쿼리(조건부 where) 작성 편리
  • 리팩토링 친화적

단점

  • 학습 곡선이 ORM보다 가파를 수 있음
  • 매핑 로직은 여전히 일부 필요

ORM (객체-관계 매핑)

객체 모델과 테이블을 매핑하고, 객체 그래프를 통해 자동으로 SQL을 생성합니다. Hibernate/JPA, TypeORM, Django ORM, ActiveRecord 등이 있습니다.

장점

  • 생산성 최고: 단순 CRUD는 거의 무료
  • 객체 그래프 탐색이 자연스러움
  • 1차 캐시, 변경 감지, 캐스케이드 등 풍부한 기능
  • DB 종류에 비교적 독립적

단점

  • 자동 생성 SQL이 비효율적일 수 있음
  • N+1 문제 등 함정이 많음
  • 내부 동작 이해 없이는 디버깅 어려움
  • 복잡한 집계나 윈도우 함수에는 부족

N+1 문제와 해결

해결 방법:

ORM의 강점을 살리려면 이런 패턴을 반드시 익혀야 합니다.

세 방식 비교

항목 Raw SQL Query Builder ORM
추상화 수준 낮음 중간 높음
생산성 낮음 중간~높음 높음
성능 제어 매우 좋음 좋음 보통
타입 안전 약함 강함 강함
학습 곡선 낮음 중간 높음
복잡 쿼리 적합도 매우 좋음 좋음 보통

언제 무엇을 쓸까

  • 단순 CRUD 위주 도메인: ORM
  • 복잡한 검색/리포팅, 동적 쿼리: Query Builder
  • 대량 배치, 복잡 분석, 고성능 요구: Raw SQL
  • 혼합: 한 프로젝트에서 ORM + Query Builder + Native Query를 함께 쓰는 것이 일반적

베스트 프랙티스

  • ORM을 쓸 때도 생성된 SQL을 항상 확인하세요(show-sql, p6spy).
  • 페치 전략(LAZY/EAGER)을 의식적으로 선택하세요.
  • 1차 캐시와 2차 캐시의 동작을 이해하세요.
  • 대량 처리에는 batch insert/update 또는 raw SQL을 사용하세요.
  • 비즈니스 로직을 SQL에 깊이 묻지 말고, 코드와 SQL 사이의 책임을 명확히 나누세요.

마무리

ORM, Query Builder, Raw SQL은 경쟁자가 아니라 계층별 도구입니다. ORM의 생산성을 누리되 한계를 알고, 필요한 곳에서는 Query Builder나 Native SQL로 내려가 직접 제어해야 합니다. 좋은 백엔드 개발자는 “어떤 도구가 옳은가”보다 “지금 이 쿼리에 어떤 도구가 맞는가”를 판단할 수 있는 사람입니다.
