CAP 정리와 PACELC - 분산 데이터베이스의 트레이드오프

작성일 2026-04-17 Edited on 2026-06-15 In Backend , Database
분산 시스템 설계의 기본 원칙인 CAP 정리와 이를 확장한 PACELC 모델을 살펴보고, 실제 분산 데이터베이스들이 어떤 선택을 했는지 정리합니다.

CAP 정리란?

CAP 정리(CAP Theorem) 는 2000년 Eric Brewer가 제안하고, 2002년 Seth Gilbert와 Nancy Lynch가 정리하여 발표한 분산 시스템의 근본적 한계에 관한 정리입니다. 분산 데이터 시스템은 다음 세 가지 속성 중 두 개까지만 동시에 만족할 수 있다 고 말합니다.

  • C (Consistency, 일관성): 모든 노드가 같은 시점에 같은 데이터를 보여줌
  • A (Availability, 가용성): 장애가 발생해도 모든 요청에 응답함
  • P (Partition tolerance, 분할 내성): 네트워크 분할이 발생해도 시스템이 동작함

왜 동시에 셋을 만족할 수 없는가?

분산 시스템에서 네트워크 장애는 언제든 일어나므로 P는 사실상 필수 입니다. 따라서 실제 선택은 P를 전제로 한 CP 와 AP 사이의 결정입니다.

다음과 같은 시나리오를 생각해 봅시다.

이때 클라이언트가 Node A에 쓰기 요청을 보내면 시스템은 두 가지 중 하나를 선택해야 합니다.

  1. A(가용성) 선택: Node A에서 쓰기를 받아들이고 응답. 단, Node B는 최신 데이터를 모름 → 일관성 위반
  2. C(일관성) 선택: Node A에서 쓰기를 거부하거나 대기. 응답하지 않음 → 가용성 위반

CP 시스템

일관성과 분할 내성을 우선하는 시스템입니다. 네트워크 분할 시 일부 노드가 응답을 거부할 수 있습니다.

  • HBase, MongoDB(기본 설정), etcd, ZooKeeper, Spanner

예: ZooKeeper는 과반수 합의가 안 되면 쓰기를 거부합니다. 이로써 모든 노드가 동일한 상태를 유지합니다.

AP 시스템

가용성과 분할 내성을 우선하는 시스템입니다. 어떤 상황에서도 응답하지만, 결과적 일관성(eventual consistency)을 가질 뿐입니다.

  • Cassandra, DynamoDB, Couchbase, Riak

예: Cassandra는 노드 일부가 끊겨도 쓰기를 받아들이며, 나중에 read repair나 hinted handoff로 데이터를 동기화합니다.

CA 시스템?

이론적으로는 가능하지만, 네트워크 분할이 일어나지 않는다고 가정해야 합니다. 단일 노드 RDBMS 가 여기에 해당한다고 볼 수 있습니다. 분산 환경에서는 CA만 만족하는 시스템은 사실상 존재하지 않습니다.

CAP의 한계

CAP는 매우 단순화된 모델입니다. 실제 시스템은 다음과 같은 부분에서 더 복잡합니다.

  1. CAP는 정상 상황과 장애 상황을 구분하지 않음
  2. “일관성”이 정확히 무엇인지(선형 가능성, 순차적 일관성 등) 명시하지 않음
  3. 지연(latency) 트레이드오프를 다루지 않음

이를 보완하기 위해 등장한 것이 PACELC 입니다.

PACELC 모델

Daniel Abadi가 2010년에 제시한 모델로, CAP를 확장합니다.

If Partition, then A vs C; Else, then Latency vs Consistency.

즉,

  • 분할 시: A와 C 중 선택 (CAP와 동일)
  • 분할이 없을 때: 지연(L)과 일관성(C) 중 선택

이로써 평상시에도 시스템이 어떤 트레이드오프를 하는지 명확히 표현할 수 있습니다.

PACELC 분류 예시

시스템 분할 시 평상시 분류
Cassandra AP EL PA/EL
DynamoDB(기본) AP EL PA/EL
MongoDB CP EC PC/EC
Spanner CP EC PC/EC
MySQL Group Replication CP EC PC/EC
RDBMS 단일 노드 - EC -/EC

PA/EL은 가용성·낮은 지연을 우선, PC/EC는 일관성을 우선하는 시스템입니다.

일관성 모델의 다양성

“일관성”이라는 단어는 여러 의미를 가집니다.

  • Strong Consistency (Linearizability): 모든 연산이 실시간 순서대로 관찰됨
  • Sequential Consistency: 어떤 순서로 일관되게 보임
  • Causal Consistency: 인과관계가 있는 작업의 순서만 보장
  • Read Your Writes: 자기 자신의 쓰기는 즉시 볼 수 있음
  • Eventual Consistency: 결국엔 같아짐

분산 시스템 설계는 이 중 어떤 수준이 필요한지 명확히 정의하는 데서 시작합니다.

실무 적용 가이드

CP를 선택해야 할 경우

  • 금융 거래, 재고 관리 등 강한 일관성이 필수
  • 분산 락, 리더 선출, 설정 저장소
  • 잘못된 데이터보다 응답 거부가 나은 도메인

AP를 선택해야 할 경우

  • 소셜 미디어 피드, 추천 시스템
  • 로그 수집, 메트릭 저장
  • 일부 stale data를 허용해도 비즈니스 영향이 작은 경우

튜닝 가능한 일관성

Cassandra, DynamoDB 같은 시스템은 요청별로 일관성 수준을 조정할 수 있습니다.

ONE, QUORUM, ALL 등 선택지에 따라 응답 시간과 일관성 보장 수준이 달라집니다.

CAP를 둘러싼 오해

  1. “세 개 중 두 개만”이라는 슬로건은 단순화된 것이며, 실제로는 분할 시에만 선택의 문제가 발생함
  2. CP 시스템이 100% 일관적인 것은 아님 (구현에 따라 다름)
  3. AP 시스템이 일관성을 전혀 못 보장하는 것은 아님 (eventual consistency 제공)

마무리

CAP와 PACELC는 분산 시스템 설계의 본질적 트레이드오프를 보여주는 프레임워크입니다. 모든 비즈니스 요구를 만족하는 마법의 시스템은 없으며, 우리가 만들고자 하는 서비스의 특성에 맞춰 일관성·가용성·지연 사이의 균형을 의식적으로 결정해야 합니다.
