WebSocket 동작 원리와 활용

작성일 2026-05-23 Edited on 2026-07-20 In Network , HTTP
실시간 양방향 통신의 표준인 WebSocket의 핸드셰이크부터 실전 활용까지 다룹니다.

WebSocket이란?

WebSocket은 단일 TCP 연결 위에서 양방향 풀 듀플렉스 통신을 제공하는 프로토콜입니다. 2011년 RFC 6455로 표준화되었으며, 채팅, 알림, 실시간 협업, 게임, 주식 시세 등 지연 시간이 중요한 애플리케이션의 핵심 기술로 자리잡았습니다.

HTTP의 요청-응답 패턴은 클라이언트가 먼저 묻지 않으면 서버가 데이터를 보낼 수 없는 단방향 구조입니다. WebSocket은 이 한계를 극복하기 위해 설계되었습니다.

기존 방식의 한계

WebSocket 이전에는 실시간성을 흉내내기 위해 여러 우회 기법이 사용되었습니다.

방식 동작 방식 단점
Polling 주기적으로 요청 지연, 불필요한 요청
Long Polling 응답을 보류하다 데이터 발생 시 반환 연결 재수립 비용
SSE 서버 -> 클라이언트 단방향 양방향 불가
WebSocket 풀 듀플렉스 별도 프로토콜

WebSocket 핸드셰이크

WebSocket은 HTTP Upgrade 요청으로 시작합니다.

클라이언트 요청

서버 응답

Sec-WebSocket-Accept는 클라이언트가 보낸 키에 매직 GUID 258EAFA5-E914-47DA-95CA-C5AB0DC85B11를 더해 SHA-1 해시 후 Base64 인코딩한 값입니다. 이로써 임의의 서버가 아닌 WebSocket을 이해하는 서버임이 검증됩니다.

핸드셰이크 이후에는 HTTP 메시지가 아닌 WebSocket 프레임이 흐릅니다.

프레임 구조

WebSocket 프레임은 다음 요소로 구성됩니다.

필드 크기 설명
FIN 1 bit 마지막 프레임 여부
Opcode 4 bit 프레임 종류 (text, binary, ping 등)
Mask 1 bit 마스킹 여부 (클라이언트는 필수)
Payload Length 7~64 bit 본문 길이
Masking Key 0/32 bit 마스킹 키
Payload 가변 실제 데이터

주요 Opcode

  • 0x1: 텍스트
  • 0x2: 바이너리
  • 0x8: 연결 종료
  • 0x9: Ping
  • 0xA: Pong

클라이언트 사용 예제

브라우저는 표준 WebSocket API를 제공합니다.

  • ws://: 평문 (포트 80)
  • wss://: TLS 암호화 (포트 443) - 프로덕션은 wss 필수

Heartbeat과 재연결

장시간 유휴 상태인 연결은 NAT나 로드밸런서에 의해 끊길 수 있습니다. 이를 방지하기 위해 주기적 ping/pong과 자동 재연결 로직이 필요합니다.

보안 고려사항

WebSocket은 HTTP와 다른 보안 모델을 가집니다.

  1. Origin 검증: 서버는 Origin 헤더로 출처를 확인해야 함 (CORS 미적용)
  2. 인증: 핸드셰이크 시 토큰 검증 또는 쿠키 활용
  3. wss 필수: 평문은 중간자 공격에 취약
  4. 메시지 검증: 클라이언트 입력은 항상 신뢰하지 않음
  5. 레이트 리미팅: 메시지 폭주 방지

활용 시나리오

분야 활용 사례
메신저 실시간 채팅, 타이핑 표시
협업 공유 문서, 화이트보드
금융 실시간 시세, 호가창
게임 멀티플레이어 동기화
알림 푸시 알림, 상태 업데이트
IoT 디바이스 모니터링

WebSocket vs SSE vs HTTP/3

특성 WebSocket SSE HTTP/3 + Streaming
방향 양방향 서버->클라이언트 양방향 가능
프로토콜 ws/wss HTTP HTTP/3
재연결 수동 구현 자동 자동
바이너리 지원 미지원 지원
복잡도 중간 낮음 중간

마무리

WebSocket은 단순한 HTTP의 한계를 넘어 진정한 양방향 통신을 가능하게 한 핵심 기술입니다. 핸드셰이크 이후 가벼운 프레임 구조로 낮은 지연 시간을 제공하며, 적절한 heartbeat와 재연결, 보안 처리만 갖추면 안정적인 실시간 서비스를 구축할 수 있습니다.
