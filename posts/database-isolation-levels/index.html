트랜잭션 격리 수준 완벽 정리 - Read Uncommitted부터 Serializable까지

작성일 2026-04-14 Edited on 2026-05-19 In Backend , Database
SQL 표준 4가지 트랜잭션 격리 수준과 Dirty Read, Non-repeatable Read, Phantom Read 같은 동시성 문제를 실제 예시와 함께 설명합니다.

트랜잭션과 ACID

데이터베이스 트랜잭션은 ACID 속성을 만족해야 합니다.

  • Atomicity: 모두 성공하거나 모두 실패
  • Consistency: 트랜잭션 전후 데이터 무결성 유지
  • Isolation: 동시 실행 트랜잭션끼리의 간섭 차단
  • Durability: 커밋된 데이터는 영속적으로 저장

이 중 격리성(Isolation)은 동시성과 성능의 트레이드오프 때문에 가장 복잡하며, 여러 단계의 격리 수준이 정의되어 있습니다.

동시성 문제 4가지

격리 수준을 이해하려면 먼저 어떤 문제들이 발생할 수 있는지 알아야 합니다.

1. Dirty Read

다른 트랜잭션이 커밋하지 않은 데이터를 읽는 현상.

2. Non-repeatable Read (반복 불가능한 읽기)

같은 트랜잭션 내에서 같은 행을 두 번 읽었을 때 값이 달라지는 현상.

3. Phantom Read (유령 읽기)

같은 조건의 범위 쿼리를 두 번 실행했을 때 결과 집합의 행 수가 달라지는 현상.

4. Lost Update

두 트랜잭션이 동시에 같은 데이터를 갱신해 한쪽 변경이 사라지는 문제.

SQL 표준 격리 수준 4단계

격리 수준 Dirty Read Non-repeatable Phantom
READ UNCOMMITTED 허용 허용 허용
READ COMMITTED 방지 허용 허용
REPEATABLE READ 방지 방지 허용(표준)
SERIALIZABLE 방지 방지 방지

READ UNCOMMITTED

가장 낮은 격리 수준. 다른 트랜잭션의 미커밋 데이터까지 볼 수 있어 거의 쓰지 않습니다.

READ COMMITTED

커밋된 데이터만 볼 수 있는 수준. PostgreSQL, Oracle의 기본값입니다. 같은 쿼리를 두 번 실행하면 다른 결과가 나올 수 있습니다.

REPEATABLE READ

트랜잭션 시작 시점의 스냅샷을 유지합니다. MySQL InnoDB의 기본값입니다. SQL 표준에서는 phantom read를 허용하지만, InnoDB는 갭 락(gap lock)을 사용해 사실상 phantom까지 방지합니다.

SERIALIZABLE

가장 엄격한 수준. 모든 동시성 문제를 방지하지만 성능 비용이 큽니다. PostgreSQL은 SSI(Serializable Snapshot Isolation)를 통해 락 없이 직렬화를 구현합니다.

DBMS별 차이

MySQL (InnoDB)

  • 기본: REPEATABLE READ
  • 일관된 스냅샷 + 넥스트 키 락으로 phantom 방지
  • SELECT ... FOR UPDATE는 현재 읽기(current read)로 동작

PostgreSQL

  • 기본: READ COMMITTED
  • REPEATABLE READ 사용 시 직렬화 충돌 발생하면 트랜잭션 abort
  • SERIALIZABLE에서 could not serialize access 에러 발생 가능, 애플리케이션 재시도 필요

Oracle

  • 기본: READ COMMITTED
  • REPEATABLE READ는 지원하지 않음 (SERIALIZABLE만 제공)

격리 수준 확인 및 변경

실무에서의 선택 가이드

  1. 일반 OLTP 서비스: READ COMMITTED 또는 REPEATABLE READ가 적절
  2. 금융·재고처럼 강한 일관성 필요: SERIALIZABLE 또는 명시적 락(SELECT ... FOR UPDATE)
  3. 분석 쿼리: READ COMMITTED로 충분하며, 가능한 경우 read replica 활용
  4. 재시도 로직: SERIALIZABLE에서는 직렬화 실패가 발생할 수 있으니 애플리케이션 단에서 재시도 구현

락(Lock)과 격리 수준의 관계

격리 수준은 락의 종류와 범위에도 영향을 줍니다.

  • 공유 락(S Lock): 읽기 시 사용, 다른 공유 락과는 호환
  • 배타 락(X Lock): 쓰기 시 사용, 다른 락과 호환 안 됨
  • 갭 락(Gap Lock): 인덱스 갭에 대한 락 (InnoDB REPEATABLE READ에서 phantom 방지)
  • 넥스트 키 락: 인덱스 레코드 + 갭 락의 조합

낙관적 동시성 제어

DB의 비관적 락 대신 버전 번호로 충돌을 감지하는 방식도 자주 사용됩니다.

JPA의 @Version도 이와 같은 원리입니다.

마무리

트랜잭션 격리 수준은 “데이터 정합성과 성능의 균형”을 결정하는 중요한 설계 결정입니다. SQL 표준의 4단계 외에도 DBMS별 구현 차이를 이해하고, 비즈니스 요구사항에 맞춰 적절한 수준과 락 전략을 조합해야 안정적인 시스템을 만들 수 있습니다.
