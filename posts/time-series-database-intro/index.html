시계열 데이터베이스 입문 - InfluxDB, TimescaleDB 비교와 사용 시나리오

작성일 2026-04-22 Edited on 2026-06-15 In Backend , Database
시계열 데이터의 특징, 일반 RDBMS로는 부족한 이유, InfluxDB와 TimescaleDB의 구조·쿼리 차이, 도입 시 고려사항을 실무 관점에서 정리합니다.

시계열 데이터란 무엇인가

시계열(time-series) 데이터는 시간 순서로 기록되는 측정값을 의미한다. 서버 CPU 사용률, 주식 가격, IoT 센서값, 사용자 이벤트 로그 등 우리가 매일 다루는 데이터 상당수가 시계열에 속한다. 시계열 데이터의 공통적인 특징은 다음과 같다.

  • 시간 + 측정값(metric) 구조: 모든 행이 timestamp를 가짐
  • append 위주: 새 데이터는 거의 항상 가장 최근 시각으로 들어옴 (random write 거의 없음)
  • 읽기는 범위 스캔: “지난 1시간 평균”, “지난 7일 95퍼센타일” 같이 시간 범위 + 집계 쿼리가 절대 다수
  • 데이터 양이 매우 큼: 초당 수천~수십만 포인트가 들어오는 경우가 흔함
  • 오래된 데이터의 가치 감소: 1년 전 1초 단위 데이터는 분 단위로 다운샘플링해도 충분

이러한 패턴은 일반 RDBMS의 트랜잭션·정규화 위주 설계와 다르므로, 전용 시계열 DB를 사용하면 저장 효율과 쿼리 성능을 크게 끌어올릴 수 있다.

왜 일반 RDBMS만으로는 부족한가

PostgreSQL이나 MySQL로 시계열 데이터를 저장하는 것 자체는 가능하다. 다만 다음 문제가 생긴다.

  1. B-Tree 인덱스 비대화: 측정값 1억 행이 쌓이면 timestamp 인덱스만으로도 수 GB가 된다.
  2. 압축 부재: 같은 메트릭 이름·태그가 매 행 반복 저장된다.
  3. 데이터 보존 정책(retention) 부재: “30일 이상 된 데이터 자동 삭제”를 직접 구현해야 한다.
  4. 다운샘플링 부재: 1초 → 1분 평균으로 자동 압축 같은 기능이 없다.
  5. 시간 함수 부족: time_bucket, last, first 같은 시계열 함수가 표준 SQL에 없다.

전용 시계열 DB는 이런 문제를 시간축에 특화된 저장 엔진·쿼리 함수로 해결한다.

InfluxDB

InfluxData가 만든 대표 시계열 DB로, Go 기반 단일 바이너리로 동작한다. v2 이후 자체 쿼리 언어 Flux(InfluxQL도 지원)를 사용한다.

데이터 모델

  • measurement: 테이블에 대응
  • tag: 인덱싱되는 메타데이터 (호스트명, 리전 등)
  • field: 인덱싱되지 않는 실제 측정값
  • timestamp: 나노초 정밀도

TimescaleDB

PostgreSQL의 확장(extension)으로 동작하므로, 기존 SQL 지식과 도구를 그대로 쓸 수 있다는 것이 가장 큰 강점이다.

핵심 개념: Hypertable

일반 테이블을 시간 구간(chunk)별로 자동 파티셔닝한 가상 테이블이다.

이후 INSERT/SELECT는 일반 테이블처럼 쓰지만, 내부적으로 일자별 청크로 분리되어 저장된다.

InfluxDB vs TimescaleDB 비교

항목 InfluxDB TimescaleDB
기반 자체 엔진 (Go) PostgreSQL 확장
쿼리 언어 Flux / InfluxQL 표준 SQL + 시계열 함수
조인 제한적 PostgreSQL 전체 기능
JSON·관계형 데이터 부족 강력 (JSONB, FK 등)
운영 도구 Telegraf·Chronograf 풍부한 PG 생태계
학습 곡선 Flux 학습 필요 SQL에 익숙하면 즉시
압축비 매우 높음 높음 (10배 이상)

도입 시 체크리스트

  • 쓰기 처리량: 초당 몇 포인트인가? 단일 노드로 충분한지 확인
  • 보존 기간과 다운샘플링 정책: 1초 → 1분 → 1시간 단계가 필요한가?
  • 분석 쿼리 복잡도: 조인·서브쿼리가 많다면 TimescaleDB가 유리
  • 기존 운영 스택: Grafana만 있으면 InfluxDB가 쉽고, PG가 이미 있다면 TimescaleDB가 자연스러움
  • 고가용성·클러스터링: 두 제품 모두 상용 버전에서 더 강한 클러스터링 제공

시계열 DB의 일반적인 안티패턴

  • 태그(라벨)에 고-카디널리티 값 넣기: user_id, request_id 같은 값을 태그로 쓰면 인덱스가 폭발한다. fields에 넣자.
  • 모든 데이터를 영구 보관: retention 정책 없이 운영하면 디스크가 금세 차고 쿼리 성능이 떨어진다.
  • 단일 측정값 테이블에 모든 메트릭 몰아넣기: 의미별로 measurement / hypertable을 분리해야 압축·쿼리 효율이 좋다.

마무리

시계열 DB는 “시간 + 메트릭” 패턴의 워크로드를 위한 도구다. 이미 PostgreSQL을 운영 중이고 SQL 친화적인 환경이라면 TimescaleDB가, 메트릭 수집·시각화 파이프라인을 빠르게 세우고 싶다면 InfluxDB가 적합하다. 두 제품 모두 retention과 다운샘플링을 적극 활용해야 장기적으로 비용·성능 모두 만족시킬 수 있다.
