React Suspense와 Error Boundary 패턴

작성일 2026-06-05 Edited on 2026-06-15 In Frontend , JavaScript
비동기 UI를 안전하게 다루기 위한 Suspense와 Error Boundary의 조합 패턴을 정리합니다.

왜 Suspense와 Error Boundary가 함께 다뤄지나

React 컴포넌트에서 데이터를 패칭하면 로딩 상태와 에러 상태를 반드시 다뤄야 합니다. 과거에는 isLoading, error 같은 상태를 직접 관리했지만, Suspense와 Error Boundary를 함께 쓰면 선언적으로 분리할 수 있습니다.

  • Suspense: 자식 트리가 “준비되지 않았을 때(pending)” 보여줄 fallback을 선언
  • Error Boundary: 자식 트리에서 던져진 에러를 잡아 fallback UI 렌더

Suspense 기본 사용

React 18부터는 서버 컴포넌트나 use() 훅, React Query, Relay 등이 Promise를 throw해 Suspense를 트리거합니다.

Error Boundary 구현

Error Boundary는 클래스 컴포넌트로만 만들 수 있습니다. 함수형으로 쓰고 싶다면 react-error-boundary 라이브러리가 표준처럼 사용됩니다.

Suspense + Error Boundary 조합

이 구조에서는 다음과 같이 흐름이 분리됩니다.

  1. 데이터가 로딩 중이면 Spinner 표시
  2. 데이터 로드 성공 시 UserProfile 표시
  3. 에러 발생 시 ErrorBoundary fallback 표시

세분화된 로딩 UI

페이지 전체가 아닌 일부만 Suspense로 감싸 점진적 로딩 UI를 만들 수 있습니다.

Header는 빨리 표시되고, PostList는 데이터가 준비되는 대로 표시됩니다. 사용자 입장에서는 “전체 빈 화면 → 한 번에 채워짐”이 아니라 점진적으로 채워지는 UX를 얻을 수 있습니다.

주의점

  • Error Boundary는 렌더 단계의 에러만 잡습니다. 이벤트 핸들러나 비동기 콜백 안의 에러는 잡지 못하므로 직접 try/catch 또는 상태로 보내야 합니다.
  • Suspense는 fallback이 자주 깜빡일 수 있으므로 startTransition이나 useDeferredValue로 우선순위를 조절합니다.
  • Next.js App Router에서는 loading.tsx, error.tsx가 Suspense·Error Boundary 역할을 자동으로 합니다.

마무리

Suspense와 Error Boundary는 React의 비동기 UI를 선언적으로 표현하는 두 축입니다. 둘을 의식적으로 조합해 사용하면 “로딩 상태 변수”와 “에러 상태 변수”가 컴포넌트 곳곳에 흩어지는 일을 막을 수 있고, 코드 가독성과 사용자 경험이 동시에 개선됩니다.
