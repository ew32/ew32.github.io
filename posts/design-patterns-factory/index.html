Factory와 Abstract Factory 패턴 완벽 정리

작성일 2026-05-13 Edited on 2026-05-15 In Backend
객체 생성 책임을 분리하는 Factory Method와 Abstract Factory 패턴의 차이와 실전 활용법을 코드 예시와 함께 정리합니다.

Factory 패턴이란?

Factory 패턴은 객체 생성 로직을 별도의 클래스나 메서드로 분리하여, 클라이언트 코드가 구체 클래스에 의존하지 않도록 만드는 생성(Creational) 디자인 패턴입니다. new 키워드를 직접 사용하는 코드는 구체 타입에 강하게 결합되어 유연성이 떨어지지만, Factory를 도입하면 인터페이스에만 의존하므로 확장에 열려 있고 변경에 닫힌(OCP) 설계를 만들 수 있습니다.

GoF 디자인 패턴에서는 크게 두 가지로 구분합니다.

  • Factory Method: 객체 생성을 서브클래스에 위임
  • Abstract Factory: 관련된 객체 군(family)을 생성하는 인터페이스 제공

Factory Method 패턴

객체 생성 메서드를 추상화하고, 어떤 구체 클래스를 만들지는 서브클래스가 결정합니다.

클라이언트는 Logistics 추상 타입만 알면 되고, 실제 어떤 Transport가 생성되는지는 신경 쓰지 않습니다.

Abstract Factory 패턴

여러 종류의 관련 객체를 일관된 방식으로 생성해야 할 때 사용합니다. 예를 들어 OS 별 UI 컴포넌트(버튼, 체크박스)를 동시에 교체하는 경우입니다.

런타임에 GUIFactory 구현체만 교체하면 전체 UI 군이 일관되게 바뀝니다.

Factory Method vs Abstract Factory

구분 Factory Method Abstract Factory
목적 단일 객체 생성 위임 관련 객체 군 생성
구조 상속 기반 합성(컴포지션) 기반
확장 단위 새 Product 추가 새 Product 군 추가

실전 활용 사례

  • Spring의 BeanFactory, ApplicationContext
  • JDBC DriverManager.getConnection
  • java.util.Calendar.getInstance
  • 로깅 라이브러리의 LoggerFactory

마무리

Factory 계열 패턴은 객체 생성과 사용을 분리해 결합도를 낮추고 확장성을 높이는 가장 기본적인 도구입니다. 단일 제품이면 Factory Method, 제품 군이면 Abstract Factory를 떠올리세요. 다만 지나치게 적용하면 클래스 수가 폭증하므로, 실제 변경 가능성이 있는 곳에 한정해 도입하는 것이 좋습니다.
