Spring Batch 입문 - Job/Step/ItemReader/Processor/Writer

작성일 2026-04-08 Edited on 2026-06-15 In Backend , Java
대용량 배치 처리를 위한 Spring Batch의 핵심 개념과 청크 지향 처리, 재시작·스킵·재시도, 실무에서 자주 쓰는 패턴까지 정리합니다.

Spring Batch가 해결하는 문제

대량의 데이터를 안정적으로 일괄 처리해야 하는 시나리오 — 매일 자정 결제 정산, 회원 등급 재계산, 외부 시스템과의 동기화, 보고서 생성 — 에서는 재시작 가능성, 트랜잭션 관리, 실패 처리, 모니터링이 핵심 과제입니다. 단순한 스케줄러 + 비즈니스 로직 조합으로 시작했다가, 데이터가 커지고 장애 처리가 복잡해지면서 누구나 비슷한 인프라를 다시 만들게 됩니다.

Spring Batch는 이 공통 요구사항을 표준화한 프레임워크입니다. Job 메타데이터를 DB에 저장해 중단된 곳부터 재시작할 수 있고, 청크 단위 트랜잭션·재시도·스킵을 선언적으로 설정할 수 있습니다.

핵심 도메인 모델

  • Job — 하나의 배치 작업 단위. 여러 Step으로 구성됨
  • Step — Job의 한 단계. 청크 또는 태스크릿 방식으로 동작
  • ItemReader — 데이터 소스에서 한 건씩 읽어옴
  • ItemProcessor — 한 건을 가공·변환·필터링
  • ItemWriter — 가공된 데이터를 일괄로 출력
  • JobRepository — 실행 메타데이터(시작 시각, 종료 상태, 파라미터) 저장
  • JobLauncher — Job 실행기

청크 지향 처리

Spring Batch의 대표적인 처리 모델입니다.

청크 크기를 100으로 설정하면 100건을 읽고 가공한 뒤 한 번에 쓰고, 트랜잭션을 커밋합니다. 중간에 실패하면 해당 청크만 롤백되고, 재시작 시 마지막 성공 청크 다음부터 진행됩니다.

간단한 Job 정의

Spring Boot + Spring Batch 5 기준 예시입니다.

Reader / Processor / Writer 구현

Reader 예: JPA 페이징

다른 옵션: JdbcCursorItemReader, FlatFileItemReader, MultiResourceItemReader, RepositoryItemReader.

Processor 예: 변환 및 필터링

Writer 예: 배치 insert

실패 처리: Skip, Retry, Restart

  • Skip: 특정 예외가 발생한 아이템을 건너뛰고 계속 진행
  • Retry: 일시적 장애에 대해 재시도
  • Restart: 실패한 청크 이후부터 다음 실행에서 재개

JobParameters를 잘 활용해야 재시작이 가능합니다. 동일한 파라미터로 실행하면 이미 완료된 Job은 다시 실행되지 않고, 실패한 Job은 같은 파라미터로 재실행 시 이어서 진행됩니다. 따라서 매번 새로 실행해야 하는 작업이라면 --job.runDate=2026-04-08 같은 파라미터로 구분해야 합니다.

Job 실행 방법

  • CommandLineJobRunner / Spring Boot 자동 실행: 부팅 시 자동 실행
  • JobLauncher 직접 호출: 스케줄러나 API에서 트리거
  • Quartz, Airflow 등 외부 스케줄러와 연동

성능 최적화 팁

  • 청크 크기는 너무 작으면 트랜잭션 오버헤드, 너무 크면 메모리·롤백 부담. 보통 100~1000 사이에서 튜닝.
  • 페이징 Reader는 페이지 크기 = 청크 크기로 맞추면 효율적.
  • Multi-threaded Step: 단일 Step을 여러 스레드로 분할 처리.
  • 파티셔닝(Partitioning): 데이터를 분할해 여러 Step을 병렬 실행. 대용량에서 가장 효과적.
  • Bulk Insert: JdbcBatchItemWriter는 PreparedStatement 배치를 사용해 빠른 쓰기를 제공.

모니터링과 운영

  • Spring Batch 메타데이터 테이블(BATCH_JOB_INSTANCE, BATCH_JOB_EXECUTION 등)을 직접 조회
  • Spring Boot Actuator + Micrometer로 메트릭 노출
  • Spring Batch Admin(별도)나 직접 대시보드 구축
  • 실패 알림은 JobExecutionListener에서 Slack/이메일로 발송

마무리

Spring Batch는 화려한 신기능보다는 신뢰성과 운영성을 제공하는 프레임워크입니다. 단순 스케줄러로 만든 배치가 데이터가 늘면서 흔들리기 시작했다면, 그때가 Spring Batch를 도입할 시점입니다. 청크 지향 처리, 메타데이터 기반 재시작, 표준화된 실패 처리는 직접 구현하기 어려운 자산입니다.
