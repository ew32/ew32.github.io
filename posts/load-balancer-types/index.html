로드 밸런서 종류와 알고리즘

작성일 2026-05-24 Edited on 2026-05-26 In Network , Theory
L4와 L7 로드 밸런서의 차이점과 트래픽 분산 알고리즘을 정리합니다.

로드 밸런서란?

**로드 밸런서(Load Balancer)**는 들어오는 네트워크 트래픽을 여러 백엔드 서버에 분산시켜 단일 서버의 부하를 줄이고 가용성을 높이는 장치 또는 소프트웨어입니다. 현대 웹 서비스의 사실상 모든 구간에 배치되어, 확장성과 장애 격리의 핵심 역할을 합니다.

로드 밸런서가 제공하는 주요 가치는 다음과 같습니다.

  • 가용성: 일부 서버 장애 시 정상 서버로만 라우팅
  • 확장성: 수평 확장(scale-out) 용이
  • 성능: 트래픽 분산으로 응답 시간 단축
  • 보안: SSL 종료, WAF, DDoS 방어 통합 가능

OSI 계층별 로드 밸런서

로드 밸런서는 동작 계층에 따라 크게 두 가지로 분류됩니다.

L4 로드 밸런서

전송 계층(TCP/UDP)에서 동작. IP 주소와 포트를 기반으로 분산합니다.

  • 페이로드를 들여다보지 않음
  • 매우 빠르고 처리량이 큼
  • TLS 종료 불가능 (TCP passthrough)
  • 예: AWS NLB, HAProxy(TCP 모드), LVS

L7 로드 밸런서

응용 계층(HTTP/HTTPS)에서 동작. URL, 헤더, 쿠키 등 메시지 내용을 기반으로 분산합니다.

  • 콘텐츠 기반 라우팅 가능 (/api -> API 서버, /static -> 정적 서버)
  • TLS 종료, 압축, 캐싱 등 부가 기능
  • 약간 느림 (페이로드 파싱 필요)
  • 예: AWS ALB, Nginx, HAProxy(HTTP 모드), Envoy

L4 vs L7 비교

항목 L4 L7
동작 계층 TCP/UDP HTTP/HTTPS
라우팅 기준 IP, 포트 URL, 헤더, 쿠키
처리량 매우 높음 높음
부가 기능 제한적 풍부
TLS 처리 패스스루 종료 가능
사용 예 게임, DB, gRPC 웹, REST API

분산 알고리즘

1. 라운드 로빈 (Round Robin)

요청을 순서대로 서버에 분배합니다. 가장 단순하지만 서버 성능 차이를 반영하지 못합니다.

2. 가중 라운드 로빈 (Weighted Round Robin)

각 서버에 가중치를 부여하여 분배 비율을 조정합니다.

3. 최소 연결 (Least Connections)

현재 활성 연결 수가 가장 적은 서버로 라우팅합니다. 요청 처리 시간이 가변적인 환경에 적합합니다.

4. 최소 응답 시간 (Least Response Time)

응답 시간이 가장 빠른 서버를 선택합니다. 헬스체크와 결합해 동적으로 적응합니다.

5. IP 해시 (IP Hash)

클라이언트 IP를 해시하여 동일 IP는 항상 같은 서버로 라우팅합니다. **세션 어피니티(Sticky Session)**가 필요할 때 유용합니다.

6. Consistent Hashing

서버 추가/제거 시 영향을 최소화하는 해시 기법. 캐시 서버 클러스터에 자주 사용됩니다.

알고리즘 균등성 세션 유지 적합한 환경
Round Robin 좋음 불가 동일 사양 서버
Weighted RR 좋음 불가 서로 다른 사양
Least Conn 매우 좋음 불가 가변 처리 시간
IP Hash 보통 가능 세션 필요 시
Consistent Hash 좋음 가능 캐시 분산

헬스 체크

로드 밸런서는 주기적으로 백엔드 상태를 확인하여 문제가 있는 서버를 풀에서 제외합니다.

  • Active Health Check: 별도 요청으로 명시적 확인
  • Passive Health Check: 실제 트래픽 응답으로 판단

세션 유지 (Sticky Session)

상태 저장이 필요한 애플리케이션은 동일 사용자를 같은 서버로 라우팅해야 합니다.

  • IP Hash: IP 기반 (NAT 환경에서 한계)
  • 쿠키 기반: 로드 밸런서가 쿠키 발급
  • JWT/공유 세션 스토어: 무상태로 설계 (권장)

글로벌 로드 밸런싱

여러 리전에 분산된 사용자를 적절한 데이터 센터로 라우팅합니다.

  • GeoDNS: 지리적 위치 기반 DNS 응답
  • Anycast: 같은 IP를 여러 위치에서 광고 (Cloudflare, AWS Global Accelerator)
  • CDN 통합: Edge 서버에서 분기

베스트 프랙티스

  • 헬스 체크 엔드포인트는 가볍게 유지
  • 무상태 설계로 세션 어피니티 의존 최소화
  • TLS는 로드 밸런서에서 종료 (백엔드는 평문 또는 mTLS)
  • 충분한 타임아웃 설정 (느린 백엔드 격리)
  • 메트릭과 로그 수집 (P99 지연 시간 모니터링)

마무리

로드 밸런서는 단순한 트래픽 분배 장치가 아니라 가용성, 확장성, 보안의 교차점입니다. L4와 L7 중 어떤 계층이 필요한지, 어떤 분산 알고리즘이 워크로드에 맞는지, 세션 유지를 어떻게 처리할지를 종합적으로 설계하면 안정적이고 확장 가능한 서비스를 구축할 수 있습니다.
