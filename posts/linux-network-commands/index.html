Linux 네트워크 진단 명령어 정리 - ip, ss, netstat, tcpdump

작성일 2026-07-05 Edited on 2026-05-18 In System , OS
리눅스 서버에서 네트워크 상태를 점검할 때 자주 쓰는 명령어들의 사용법과 실전 활용 패턴을 정리합니다.

기본 흐름

서버 네트워크 문제를 진단할 때는 보통 다음 순서로 접근합니다.

  1. 인터페이스와 라우팅이 정상인지(ip).
  2. 어떤 포트가 열려 있고 누가 듣고 있는지(ss).
  3. 외부 호스트와 통신이 되는지(ping, traceroute, curl).
  4. 실제 패킷이 오가는지(tcpdump).

각 단계의 핵심 명령을 정리합니다.

ip - 주소·라우팅·링크

ip 명령은 ifconfig, route, arp를 대체하는 최신 도구입니다.

ss - 소켓 상태

ss는 netstat보다 훨씬 빠르고 정보가 풍부합니다. 컨테이너·고부하 서버에서 권장됩니다.

netstat은 여전히 동작하지만 deprecated된 배포판이 많습니다. 신규 스크립트는 ss 사용을 권장합니다.

ping / traceroute / mtr

방화벽이 ICMP를 차단하면 ping이 실패해도 HTTP는 동작할 수 있으므로, 항상 curl과 병행 확인합니다.

curl - HTTP 진단

dig / host - DNS 조회

tcpdump - 패킷 캡처

가장 강력한 디버깅 도구입니다. 운영 환경에서는 부하·보안에 주의해 짧게 캡처하고 파일로 저장합니다.

자주 쓰는 필터 표현식

  • host 1.2.3.4, src host, dst host
  • port 53, portrange 5000-6000
  • tcp, udp, icmp
  • tcp[tcpflags] & tcp-syn != 0 (SYN 패킷만)

pcap 파일은 Wireshark로 열어 GUI 분석이 가능합니다.

방화벽과 NAT

연결이 안 될 때는 보안 그룹·클라우드 NACL·호스트 방화벽을 순서대로 점검합니다.

시나리오 예시

“애플리케이션이 외부 API 호출에서 가끔 타임아웃이 난다”는 보고가 들어왔을 때.

  1. ss -s로 TIME_WAIT, ESTABLISHED 수 폭증 여부 확인.
  2. ip route get <api-ip>로 의도한 경로/인터페이스로 나가는지 확인.
  3. dig <api-domain>으로 DNS 응답 시간 점검(여러 번 측정).
  4. mtr <api-host>로 중간 홉의 패킷 손실 확인.
  5. tcpdump로 SYN/SYN-ACK가 오가는지 캡처하고, 핸드셰이크 단계인지 응답 지연인지 분리.

마무리

네트워크 디버깅의 핵심은 “어느 계층에서 멈춰 있는지”를 빠르게 좁히는 것입니다. ip, ss, curl, dig, tcpdump 다섯 가지만 손에 익혀도 대부분의 운영 이슈를 진단할 수 있습니다.
