BEM 네이밍 방법론 정리

작성일 2026-06-13 Edited on 2026-05-20 In Frontend , CSS
CSS 클래스 네이밍 규칙인 BEM(Block-Element-Modifier)의 원리와 실전 적용법을 정리합니다.

BEM이란?

BEM은 Yandex에서 시작된 CSS 클래스 네이밍 방법론으로, Block, Element, Modifier의 약자입니다. 클래스 이름만 보고도 구조와 역할을 알 수 있게 만드는 것이 목적입니다.

  • Block — 독립적으로 의미 있는 컴포넌트. 예: card, menu
  • Element — 블록을 구성하는 부분. 블록 없이는 의미가 없음. 예: card__title
  • Modifier — 변형/상태. 예: card--featured, menu__item--active

왜 BEM을 쓰는가

  • 충돌 방지: 클래스 이름이 길지만 의미상 유일해 다른 컴포넌트와 충돌이 없음
  • 선택자 단순화: .menu li > a:hover 같은 후손 선택자 대신 .menu__link 한 줄
  • specificity 평탄화: 모두 클래스 한 개, 0,1,0 우선순위로 통일
  • 읽기 쉬움: HTML만 봐도 컴포넌트 구조가 드러남

기본 예시

Element는 한 단계로

Element의 중첩(block__element__sub)은 BEM 가이드에서 명시적으로 금지합니다. 구조는 HTML이 표현하고, 클래스는 평탄하게 유지합니다. 깊이가 더 필요하면 새로운 Block을 분리해야 한다는 신호입니다.

Modifier 사용 패턴

수정자는 두 가지 표기 방식이 있습니다.

BEM 표준은 1번이지만, 빌드 도구가 2번 형태([data-*])도 함께 자주 씁니다. 두 방식을 혼용하지 않고 프로젝트에서 하나를 정해 일관되게 적용하는 것이 중요합니다.

블록 안의 블록

블록 안에 다른 블록을 자유롭게 넣을 수 있습니다. 이때 안쪽 블록은 바깥의 Element가 아니라 별도의 Block으로 작성합니다.

button은 어디서나 재사용되는 독립 컴포넌트이므로 card__button이 아닙니다.

CSS 모듈/Sass와의 조합

Sass에서는 & 연산자로 BEM을 간결하게 표현할 수 있습니다.

CSS 모듈을 사용하면 클래스가 자동으로 스코프되므로 BEM의 필요성이 약해지지만, 팀에서 일관된 네이밍을 유지하기 위해 BEM 규칙을 그대로 적용하는 경우도 많습니다.

BEM의 한계와 대안

  • 클래스 이름이 길어집니다.
  • HTML 마크업이 무거워 보일 수 있습니다.
  • 단순한 마크업에는 과한 느낌이 있습니다.

이를 보완하는 흐름으로 Tailwind CSS의 유틸리티 퍼스트, CSS-in-JS, CSS 모듈이 있습니다. 다만 디자인 시스템처럼 의미 있는 컴포넌트 라이브러리를 만들 때 BEM은 여전히 강력한 선택지입니다.

마무리

BEM은 단순한 규칙이지만 CSS의 가장 골치 아픈 문제인 “어디서 스타일이 깨졌는가”에 명확한 답을 줍니다. Block 중심으로 사고하고 Element는 한 단계로, Modifier로 상태를 표현한다는 세 가지 원칙만 지키면 큰 프로젝트에서도 흔들리지 않는 CSS 구조를 만들 수 있습니다.
