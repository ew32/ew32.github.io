Content Security Policy(CSP) 헤더 작성 가이드

작성일 2026-05-01 Edited on 2026-06-15 In Security , Cryptography
XSS 방어의 마지막 보루인 CSP의 주요 지시문, nonce/hash 기반 strict-dynamic 패턴, report-only 모드를 활용한 점진적 도입 전략을 정리합니다.

CSP가 필요한 이유

XSS는 여전히 가장 빈번한 웹 취약점 중 하나입니다. 입력 검증과 출력 인코딩이 1차 방어선이라면, CSP(Content Security Policy)는 마지막 안전망입니다. 코드에 XSS 버그가 있더라도 CSP가 강력하면 공격자가 임의 스크립트를 실행할 수 없습니다.

CSP는 HTTP 응답 헤더로 전달되며, 브라우저가 페이지에서 로드/실행할 수 있는 리소스의 출처를 제한합니다.

주요 지시문

지시문 의미
default-src 다른 지시문의 폴백
script-src JavaScript 출처
style-src CSS 출처
img-src 이미지 출처
connect-src XHR, fetch, WebSocket
font-src 폰트
frame-src iframe 임베드
frame-ancestors 이 페이지를 누가 임베드할 수 있는가
object-src <object>, <embed>
base-uri <base> 태그
form-action form action 출처

흔한 약한 설정

다음 설정은 XSS를 막지 못합니다.

unsafe-inline을 허용하는 순간 인라인 스크립트가 실행 가능하므로 XSS 방어 효과가 사라집니다.

nonce 기반 strict CSP

현대적 권장 패턴은 nonce와 strict-dynamic을 결합한 strict CSP입니다.

  • 매 요청마다 새로운 nonce를 생성
  • 신뢰하는 스크립트 태그에만 nonce 속성을 부여
  • strict-dynamic은 nonce가 부여된 스크립트가 동적으로 로드하는 스크립트도 허용

hash 기반 허용

정적 인라인 스크립트가 있다면 SHA 해시로 허용할 수 있습니다.

해시 값은 브라우저 콘솔의 위반 메시지에 표시되거나, 직접 계산할 수 있습니다.

Trusted Types

DOM XSS를 원천 차단하려면 Trusted Types를 도입합니다.

이 설정 하에서 element.innerHTML = userInput은 예외가 발생합니다. 개발자는 정책 함수로 검증된 값만 할당할 수 있습니다.

Report-Only로 점진 도입

기존 사이트에 갑자기 strict CSP를 적용하면 페이지가 깨집니다. 먼저 Report-Only로 운영해 위반을 수집합니다.

이 모드에서는 차단 없이 위반만 보고되므로, 실제 운영 영향 없이 정책을 다듬을 수 있습니다.

일반적 함정

  • third-party 스크립트: Google Analytics, 광고 SDK 등이 인라인 스크립트나 eval을 사용하면 strict CSP와 충돌
  • CDN 와일드카드: script-src https://cdn.example.com처럼 보일 수 있지만, 그 CDN에 누구나 업로드 가능하다면 위험
  • JSONP: 사실상 CSP를 우회하는 패턴, 대안 검토 필요

점검 체크리스트

  • unsafe-inline, unsafe-eval이 제거되었는가
  • nonce 또는 hash 기반으로 스크립트를 제어하는가
  • object-src 'none', base-uri 'none'이 설정되었는가
  • frame-ancestors로 클릭재킹도 방어되는가
  • Report-To 또는 report-uri로 위반을 수집하는가

마무리

CSP는 한 번에 완벽한 정책을 만들기 어렵지만, Report-Only 모드와 점진적 강화 전략을 활용하면 안전하게 도입할 수 있습니다. nonce + strict-dynamic + Trusted Types 조합이 현재 가장 권장되는 형태입니다.
