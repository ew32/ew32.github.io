데이터베이스 데드락 발생 원인과 해결 전략

작성일 2026-04-16 Edited on 2026-07-20 In Backend , Database
데이터베이스 데드락이 발생하는 원리와 잠금 순서, 타임아웃, 재시도 등 실무에서 사용하는 데드락 예방 및 해결 방법을 자세히 설명합니다.

데드락이란?

데드락(Deadlock)은 두 개 이상의 트랜잭션이 서로가 보유한 자원의 해제를 기다리면서 영원히 진행되지 않는 상태를 말합니다. 데이터베이스에서는 행 락, 갭 락, 테이블 락 등 다양한 락이 얽혀 발생합니다.

데드락 발생 조건 (Coffman 조건)

데드락은 다음 네 가지 조건이 동시에 만족될 때 발생합니다.

  1. 상호 배제(Mutual Exclusion): 자원은 한 번에 한 트랜잭션만 점유
  2. 점유와 대기(Hold and Wait): 자원을 점유한 상태에서 다른 자원을 기다림
  3. 비선점(No Preemption): 자원을 강제로 빼앗을 수 없음
  4. 순환 대기(Circular Wait): 자원 점유와 대기 관계가 순환을 이룸

이 중 하나라도 깨면 데드락은 발생하지 않습니다. 실무에서는 주로 순환 대기 를 막는 전략을 사용합니다.

전형적인 데드락 시나리오

잠금 순서 차이로 인한 데드락

T1은 id=1, id=2 순서로 락을 잡으려 했고, T2는 id=2, id=1 순서로 잡으려 했기 때문에 순환 대기가 만들어졌습니다.

인덱스 갭 락에 의한 데드락 (MySQL)

InnoDB의 REPEATABLE READ에서는 갭 락이 추가로 잡혀 의외의 데드락이 발생합니다.

데드락 감지와 처리

대부분의 DBMS는 데드락을 자동으로 감지하고, 비용이 적은 트랜잭션을 victim으로 선택해 롤백시킵니다.

MySQL 데드락 로그 확인

PostgreSQL은 deadlock_timeout(기본 1초) 이상 락 대기 시 데드락 검사를 수행하고, 발견되면 에러를 반환합니다.

데드락 예방 전략

1. 잠금 순서 통일

가장 효과적인 방법입니다. 여러 행을 갱신할 때 항상 동일한 순서(예: id 오름차순)로 락을 잡도록 규약을 만듭니다.

2. 트랜잭션을 짧게 유지

트랜잭션 길이가 길수록 락 보유 시간이 길어져 충돌 확률이 올라갑니다. 외부 API 호출이나 무거운 계산을 트랜잭션 안에 두지 않습니다.

3. 적절한 인덱스 설계

인덱스가 없으면 풀 스캔이 발생해 더 많은 행에 락이 걸립니다. 핵심 갱신 쿼리에는 반드시 인덱스를 만듭니다.

4. 격리 수준 조정

REPEATABLE READ의 갭 락이 자주 문제를 일으킨다면 READ COMMITTED로 낮추는 것을 고려할 수 있습니다. 단, 비즈니스 요구사항을 충분히 검토해야 합니다.

5. 락 타임아웃 설정

영원히 대기하지 않도록 타임아웃을 두는 것이 중요합니다.

재시도 로직 구현

데드락은 완전히 없앨 수 없으므로, 애플리케이션에서 자동 재시도 를 구현하는 것이 표준 패턴입니다.

Spring의 @Retryable이나 Resilience4j 같은 도구를 사용해도 됩니다.

마무리

데드락은 다중 트랜잭션 환경에서 피할 수 없는 현상이지만, 일관된 잠금 순서·짧은 트랜잭션·적절한 인덱스·재시도 로직을 통해 충분히 관리할 수 있습니다. 발생 시에는 로그를 분석해 원인을 파악하고 구조적 개선을 적용하는 습관이 안정적인 서비스를 만듭니다.
