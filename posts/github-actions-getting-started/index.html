GitHub Actions로 CI/CD 시작하기

작성일 2026-05-26 Edited on 2026-05-19 In Cloud , Infrastructure
GitHub Actions의 기본 개념과 워크플로 작성법, 실전 CI/CD 예제를 정리합니다.

들어가며

소프트웨어 개발에서 CI/CD(Continuous Integration / Continuous Delivery) 는 이제 선택이 아닌 필수입니다. GitHub Actions는 GitHub에 내장된 자동화 플랫폼으로, 별도의 서버 구축 없이 PR 검증, 테스트, 빌드, 배포까지 한 곳에서 처리할 수 있습니다. 이 글에서는 GitHub Actions의 핵심 개념과 자주 쓰는 워크플로를 정리합니다.

핵심 개념

  • Workflow: 자동화 프로세스. .github/workflows/ 안의 YAML 파일로 정의
  • Event: 워크플로를 트리거하는 사건 (push, pull_request, schedule 등)
  • Job: 워크플로 안의 작업 단위. 같은 러너 위에서 실행
  • Step: Job 안의 개별 명령어 또는 액션
  • Action: 재사용 가능한 단위 (actions/checkout, actions/setup-node 등)
  • Runner: 워크플로를 실행하는 서버 (GitHub-hosted 또는 self-hosted)

첫 워크플로

리포지토리에 .github/workflows/hello.yml 파일을 만듭니다.

이제 push할 때마다 자동으로 실행됩니다.

시크릿 관리

API 키, AWS 자격 증명 같은 민감 정보는 Repository Secrets 또는 Environment Secrets에 저장합니다.

GitHub UI: Settings → Secrets and variables → Actions

더 안전한 방법으로 OIDC를 통한 임시 자격 증명을 사용할 수 있습니다.

환경(Environment)과 승인

배포 전 수동 승인을 받으려면 Environment를 사용합니다.

환경 설정에서 reviewer를 지정하면 해당 사용자의 승인 후에만 배포가 진행됩니다.

작업 의존성과 병렬 실행

캐시 활용

빌드 시간 단축을 위해 의존성 캐시를 사용합니다.

actions/setup-node, setup-python은 내장 캐시 옵션을 제공해 더 편리합니다.

자주 사용하는 컨텍스트

컨텍스트 설명
github.sha 커밋 SHA
github.ref 브랜치/태그 ref
github.actor 워크플로를 트리거한 사용자
github.event_name 트리거 이벤트 종류
secrets.* 시크릿 값
env.* 환경 변수

모범 사례

  • 워크플로 분리: CI(테스트), CD(배포)를 별도 파일로
  • 시크릿 노출 금지: 시크릿을 echo로 출력하지 않기
  • 버전 고정: actions/checkout@v4처럼 메이저 또는 SHA로 고정
  • 불필요한 트리거 최소화: paths, branches 필터 활용
  • 타임아웃 설정: timeout-minutes로 무한 실행 방지
  • Self-hosted Runner: 대용량 빌드는 자체 러너 활용

마무리

GitHub Actions는 설정이 간단하면서도 매우 유연한 CI/CD 도구입니다. 단순한 테스트 자동화부터 멀티 환경 배포, OIDC 기반의 안전한 클라우드 인증까지 폭넓은 시나리오를 지원합니다. 작은 워크플로 하나로 시작하고, 매트릭스 테스트, 환경 승인, 캐시 등을 점차 추가해 가면서 효율적인 자동화 파이프라인을 구축해 보세요.
