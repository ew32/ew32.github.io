도메인 주도 설계(DDD) 기초

작성일 2026-05-15 In Backend , Infrastructure
도메인 주도 설계의 전략·전술적 패턴과 핵심 개념을 예시와 함께 살펴봅니다.

DDD란?

도메인 주도 설계(Domain-Driven Design, DDD) 는 Eric Evans가 2003년 정립한 소프트웨어 설계 방법론으로, 복잡한 비즈니스 문제를 도메인 모델 중심으로 풀어내는 접근입니다.

기술 스택이 아닌 도메인 전문가의 언어를 코드와 일치시키는 것이 핵심이며, 이를 통해 비즈니스와 코드 사이의 간극을 줄이고 변경에 강한 설계를 추구합니다.

전략적 설계 vs 전술적 설계

DDD는 두 가지 큰 범주로 나뉩니다.

범주 관심사 주요 개념
전략적 설계 시스템 전체 경계 Bounded Context, Context Map, Ubiquitous Language
전술적 설계 코드 수준 모델링 Entity, Value Object, Aggregate, Repository, Domain Service

핵심 개념

Ubiquitous Language (보편 언어)

도메인 전문가와 개발자가 같은 단어로 같은 의미를 표현합니다. “고객은 주문을 결제한다” 라는 문장이 곧 코드의 메서드명이 되어야 합니다.

Bounded Context

같은 단어라도 컨텍스트마다 의미가 다릅니다. 예를 들어 “상품(Product)” 은 카탈로그 컨텍스트와 재고 컨텍스트에서 다른 속성을 가집니다. 각 컨텍스트는 독립된 모델과 언어를 가집니다.

Entity vs Value Object

  • Entity: 식별자(ID)로 구분되는 객체 (예: User, Order)
  • Value Object: 속성 값으로 동등성이 결정되는 불변 객체 (예: Money, Address)

Aggregate와 Aggregate Root

Aggregate 는 일관성 경계 안에 있는 객체들의 묶음입니다. 외부에서는 Aggregate Root 를 통해서만 접근할 수 있습니다.

Aggregate 외부에서 OrderLine을 직접 조작할 수 없게 함으로써 불변식(invariant) 을 보호합니다.

Repository

Aggregate의 영속성을 추상화하는 인터페이스입니다. 도메인 계층은 구현이 아닌 인터페이스에 의존합니다.

Domain Service

여러 Aggregate에 걸친 도메인 로직이 필요한 경우 사용합니다. 상태가 없고, 도메인 용어로 명명해야 합니다.

Context Map - 컨텍스트 간 관계

서로 다른 컨텍스트가 어떻게 협력하는지를 표현합니다.

  • Partnership: 두 팀이 함께 진화
  • Customer/Supplier: 한 쪽이 다른 쪽의 요구를 수용
  • Conformist: 변경 권한 없이 그대로 따름
  • Anti-Corruption Layer: 외부 모델을 자체 모델로 변환하는 보호막

모범 사례

  1. 풍부한 도메인 모델 - Anemic Model(빈약한 모델)을 피하고 로직을 객체 안으로
  2. 작은 Aggregate - 한 트랜잭션에서 하나의 Aggregate만 변경
  3. 이벤트 활용 - Aggregate 간 결합은 도메인 이벤트로
  4. 이름이 곧 설계 - 도메인 용어를 정확히 반영

도입 시 주의점

DDD는 복잡한 도메인에서 빛을 발하지만, 단순 CRUD에는 오히려 오버엔지니어링이 될 수 있습니다. 모든 시스템에 일률적으로 적용할 패턴이 아니며, 핵심 도메인(Core Domain) 에 우선 집중하는 것이 현실적입니다.

마무리

DDD는 코드 패턴이 아닌 사고방식입니다. 도메인 전문가와의 협업, 보편 언어 정착, 명확한 경계 설정이 어떤 전술적 패턴보다 중요합니다. 작은 컨텍스트에서부터 점진적으로 도입해 보면서 팀의 도메인 이해도를 함께 끌어올리는 것이 효과적입니다.
