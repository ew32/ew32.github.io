Spring WebFlux로 시작하는 리액티브 프로그래밍

작성일 2026-04-07 Edited on 2026-05-15 In Backend , Java
Spring WebFlux와 Project Reactor의 Mono/Flux 개념, MVC와의 차이, WebClient·R2DBC 활용, 그리고 리액티브가 적합한 시나리오를 정리합니다.

WebFlux란

Spring WebFlux는 Spring 5에서 도입된 논블로킹 리액티브 웹 스택입니다. 기존 Spring MVC가 서블릿 기반의 동기·블로킹 모델이라면, WebFlux는 Reactor Netty를 기본으로 하는 비동기 이벤트 루프 위에서 동작합니다.

리액티브 프로그래밍의 핵심은 데이터를 스트림으로 다루며, 비동기적으로 발생하는 이벤트를 선언적으로 합성하는 것입니다. 적은 수의 스레드로 많은 요청을 처리해야 하는 I/O 집약적 서비스에 적합합니다.

Mono와 Flux

Project Reactor가 제공하는 두 가지 핵심 타입입니다.

  • Mono<T> — 0 또는 1개의 결과를 비동기적으로 방출
  • Flux<T> — 0개부터 N개의 결과를 비동기적으로 방출

구독(subscribe) 전까지는 아무 일도 일어나지 않는다는 점이 핵심입니다. Reactor의 파이프라인은 선언일 뿐이고, 실제 실행은 subscribe 시점에 시작됩니다.

기본 컨트롤러 작성

WebFlux는 두 가지 프로그래밍 모델을 지원합니다.

1) 어노테이션 기반 (MVC와 유사)

2) 함수형 라우터

연산자(Operator) 활용

리액티브의 강점은 연산자 합성에 있습니다.

  • flatMap — 비동기 연쇄
  • zip — 여러 Mono를 병렬 실행 후 합치기
  • switchIfEmpty — 비어 있을 때 대체
  • timeout — 지연 시 예외
  • onErrorResume — 에러 발생 시 폴백

WebClient — 논블로킹 HTTP 클라이언트

RestTemplate의 리액티브 대체재입니다.

여러 외부 API를 호출해 결과를 합치는 시나리오에서 빛을 발합니다.

세 호출이 병렬로 실행되어 전체 응답 시간이 가장 느린 호출에 수렴합니다.

데이터 접근 — R2DBC

JDBC는 블로킹 API라서 WebFlux의 이점을 살리지 못합니다. 리액티브 환경에서는 R2DBC(Reactive Relational Database Connectivity) 또는 MongoDB/Redis 리액티브 드라이버를 사용하세요.

JPA는 리액티브 환경에 부적합합니다. 굳이 써야 한다면 별도 스레드 풀로 격리하세요.

백프레셔(Backpressure)

리액티브 스트림의 또 다른 핵심은 소비자가 자신이 감당할 수 있는 양만큼 데이터를 요청할 수 있다는 점입니다. Reactor는 request(n) 신호로 이를 구현합니다. 대용량 데이터를 다룰 때 자동으로 흐름을 조절해 OOM을 막아 줍니다.

WebFlux vs MVC, 언제 무엇을?

상황 권장
외부 API 다수 호출, 게이트웨이 WebFlux
스트리밍·SSE·웹소켓 위주 WebFlux
일반 CRUD, JPA 사용 MVC
CPU 집약적 작업 MVC
팀의 비동기 경험 부족 MVC

Java 21의 Virtual Threads가 등장하면서, “동시성을 위해 리액티브가 꼭 필요한가”라는 질문이 다시 나오고 있습니다. 동기 코드를 유지하면서 수십만 동시 요청을 처리하려면 MVC + 가상 스레드도 강력한 대안입니다.

디버깅 팁

리액티브 스택은 스택 트레이스가 끊기기 쉽습니다.

  • 개발 환경에서 Hooks.onOperatorDebug() 또는 Reactor Tools BlockHound 활용
  • .log() 연산자로 신호 추적
  • checkpoint("name")으로 파이프라인 위치 표시

마무리

WebFlux는 적합한 워크로드에서는 효율적이고 우아한 도구입니다. 하지만 학습 곡선이 가파르고, 한 곳에서라도 블로킹 호출이 섞이면 전체 성능이 무너집니다. “꼭 필요한가”를 먼저 자문하고, 도입 시에는 끝에서 끝까지 논블로킹을 유지하는 일관성을 지키는 것이 핵심입니다.
