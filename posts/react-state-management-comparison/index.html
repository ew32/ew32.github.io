React 상태 관리 라이브러리 비교 - Redux, Zustand, Recoil

작성일 2026-05-20 Edited on 2026-05-19 In Frontend , JavaScript
React 생태계의 대표적인 상태 관리 라이브러리 Redux, Zustand, Recoil의 특징과 사용법을 비교합니다.

왜 상태 관리 라이브러리가 필요한가?

React는 컴포넌트 단위로 useState, useReducer를 통해 상태를 관리할 수 있지만, 여러 컴포넌트가 동일한 상태를 공유하거나 깊은 트리 구조 전반에 상태를 전달해야 할 때는 한계가 있습니다. 이때 등장하는 것이 전역 상태 관리 라이브러리입니다.

Context API로도 일부 해결되지만, 값이 바뀔 때 구독한 모든 컴포넌트가 리렌더되는 성능 문제가 있어 대규모 앱에서는 전용 라이브러리가 더 적합합니다.

Redux와 Redux Toolkit

Redux는 가장 오래된 React 상태 관리 라이브러리로, 단방향 데이터 흐름과 불변성 원칙을 따릅니다. 최근에는 보일러플레이트를 줄인 Redux Toolkit(RTK) 사용이 표준입니다.

컴포넌트에서는 다음과 같이 사용합니다.

장점은 예측 가능한 상태 변화와 강력한 DevTools, 그리고 미들웨어 생태계입니다.

Zustand

Zustand는 훅 기반의 매우 가벼운(<1KB) 라이브러리로, Provider 없이 간단히 전역 스토어를 만들 수 있습니다.

컴포넌트는 필요한 부분만 선택하여 구독합니다.

선택자(selector)를 사용하면 세밀한 리렌더 최적화가 자동으로 이뤄집니다. 미들웨어로 persist, devtools, immer 등을 손쉽게 추가할 수 있습니다.

Recoil

Recoil은 Facebook에서 시작한 라이브러리로, Atom과 Selector 개념을 통해 React의 컴포넌트 트리와 자연스럽게 통합됩니다.

사용은 React의 useState와 매우 유사합니다.

비동기 selector를 통해 Suspense와 자연스럽게 통합되는 것이 특징입니다.

비교표

항목 Redux Toolkit Zustand Recoil
번들 크기 중간 매우 작음 중간
보일러플레이트 중간 매우 적음 적음
Provider 필요 예 아니오 예
학습 곡선 중간 낮음 낮음
DevTools 강력함 지원 기본
비동기 처리 RTK Query, Thunk 자유롭게 selector
유지보수 활발 활발 둔화

어떻게 선택할까?

  • 대규모 팀, 표준화 필요, 복잡한 비동기 흐름 -> Redux Toolkit + RTK Query
  • 개인/소규모 프로젝트, 빠른 시작, 단순함 -> Zustand
  • 세밀한 의존성 그래프, Suspense 통합 -> Recoil 또는 Jotai

서버 상태(API 데이터)는 TanStack Query나 SWR처럼 전용 도구를 별도로 사용하는 것이 좋습니다. 클라이언트 상태와 서버 상태를 같은 도구로 다루지 않는 것이 권장 패턴입니다.

모범 사례

  • 전역 상태로 두기 전에 정말 전역인지 다시 고민합니다.
  • 상태를 작게 쪼개어 리렌더 범위를 줄입니다.
  • 서버 데이터와 UI 상태를 분리합니다.
  • DevTools가 디버깅에 결정적이므로 초기 설정 시 함께 도입합니다.

마무리

Redux는 안정성과 생태계, Zustand는 단순함, Recoil은 React 친화적 모델이라는 각자의 강점이 있습니다. 프로젝트의 규모와 팀 상황에 맞게 선택하되, 클라이언트 상태와 서버 상태를 분리하는 원칙은 어느 라이브러리를 쓰더라도 동일하게 적용하세요.
