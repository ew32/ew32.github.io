시크릿 관리와 HashiCorp Vault 개요

작성일 2026-05-03 Edited on 2026-06-15 In Backend , Authentication
시크릿을 코드와 환경변수에서 분리해야 하는 이유와 HashiCorp Vault의 핵심 개념(secret engine, auth method, lease, dynamic secret)을 정리합니다.

시크릿 관리가 필요한 이유

DB 비밀번호, API 키, 인증서 같은 시크릿을 .env 파일이나 환경변수에 그대로 두는 것은 다음 문제를 야기합니다.

  • Git 저장소에 실수로 커밋
  • 컨테이너 이미지에 베이크되어 영구 보존
  • 운영자가 평문을 알게 되어 책임 추적 불가
  • 로테이션 시 모든 서비스 재배포 필요

시크릿 관리 시스템은 시크릿을 중앙화하고, 접근을 감사하며, 자동 로테이션과 동적 발급을 가능하게 합니다. HashiCorp Vault는 이 분야의 사실상 표준입니다.

Vault의 핵심 개념

Secret Engine

시크릿을 저장하거나 동적으로 생성하는 백엔드입니다.

  • kv: 키-값 저장(정적 시크릿)
  • database: DB 자격증명을 요청 시 생성/회수
  • pki: 사설 인증서 발급
  • transit: 데이터를 Vault로 보내 암호화/복호화(키는 노출되지 않음)
  • aws: STS 임시 자격증명 발급

Auth Method

클라이언트가 Vault에 인증하는 방법입니다.

  • token: 가장 기본적인 형태
  • approle: 머신 간 인증(role_id + secret_id)
  • kubernetes: 파드의 ServiceAccount 토큰으로 인증
  • aws, gcp, azure: 클라우드 인스턴스 메타데이터 기반
  • oidc/jwt: 외부 IdP 연동

Policy

ACL 정책으로 어떤 경로에 어떤 동작을 허용할지 정의합니다.

Lease와 Revoke

Vault가 발급한 시크릿은 lease(임대 기간)를 가집니다. 만료되면 자동 폐기되며, 사고 시 명령 한 번으로 일괄 폐기할 수 있습니다.

정적 시크릿 사용 흐름

애플리케이션 입장에서는 환경변수 대신 Vault API를 호출합니다.

동적 시크릿의 강력함

Vault의 진짜 장점은 동적 시크릿입니다. 예를 들어 DB 자격증명을 요청 시점에 생성하고, 사용 후 자동 폐기합니다.

이제 애플리케이션이 database/creds/readonly를 읽으면 한 시간만 유효한 새 DB 사용자가 생성됩니다.

장점은 다음과 같습니다.

  • 자격증명이 단명하므로 유출 시 피해가 작음
  • 누구나 같은 패스워드를 쓰지 않으므로 감사 추적이 명확
  • 폐기가 즉시 가능

Kubernetes 환경 통합

Kubernetes에서 Vault를 사용하는 표준 방식은 Vault Agent Injector입니다.

사이드카가 시크릿을 파일로 마운트해 주고, lease 만료 전에 자동 갱신합니다. 애플리케이션 코드는 파일을 읽기만 하면 됩니다.

Seal/Unseal

Vault는 시작 시 봉인(seal) 상태로 시작합니다. 마스터 키는 Shamir’s Secret Sharing으로 여러 조각으로 나뉘어 있고, 일정 수 이상이 모여야 unseal됩니다. AWS KMS, GCP KMS 등을 사용한 자동 unseal도 가능합니다.

봉인 메커니즘 덕분에 서버 한 대를 탈취당해도 디스크의 시크릿이 즉시 노출되지 않습니다.

대안과 선택 기준

  • AWS Secrets Manager / Parameter Store: AWS 전용, 통합 간단
  • GCP Secret Manager: GCP 전용
  • Azure Key Vault: Azure 전용
  • HashiCorp Vault: 멀티 클라우드, 동적 시크릿, transit 등 기능 풍부
  • Doppler, 1Password Secrets Automation: SaaS형 간편 솔루션

단일 클라우드에 종속되어 있다면 해당 매니지드 서비스가 더 단순합니다. 멀티 클라우드, 온프레미스 혼합, 고급 기능이 필요하면 Vault가 강력합니다.

점검 체크리스트

  • 코드와 이미지에 평문 시크릿이 없는가
  • 모든 시크릿 접근이 감사 로그에 남는가
  • 가능한 곳은 동적 시크릿으로 전환했는가
  • 정책이 최소 권한 원칙을 따르는가
  • Vault 자체의 unseal 키와 root 토큰이 안전하게 보관되는가

마무리

시크릿은 코드와 함께 다닐 수 없습니다. Vault 같은 중앙 집중식 관리 시스템으로 분리하고, 가능한 곳은 동적 시크릿으로 전환하면 보안 수준과 운영 효율이 동시에 올라갑니다.
