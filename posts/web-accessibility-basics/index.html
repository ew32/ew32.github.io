웹 접근성(a11y) 기본 가이드

작성일 2026-05-21 Edited on 2026-05-20 In Frontend , HTML
웹 접근성의 핵심 개념인 시맨틱 HTML, ARIA, 키보드 탐색, 색상 대비, 스크린 리더 대응을 정리합니다.

웹 접근성이란?

웹 접근성(Web Accessibility, 줄여서 a11y) 은 장애 유무, 기기, 환경에 관계없이 누구나 웹 콘텐츠에 접근할 수 있도록 만드는 것을 의미합니다. 시각, 청각, 운동, 인지 장애를 가진 사용자뿐 아니라 느린 네트워크, 작은 화면, 일시적 부상 같은 상황까지 폭넓게 포함합니다.

국제 표준은 W3C의 WCAG(Web Content Accessibility Guidelines) 이며, 한국에서는 KWCAG가 이를 기반으로 합니다. 접근성은 단순한 윤리적 문제를 넘어 법적 의무이자 SEO와 사용성 향상에도 도움이 됩니다.

시맨틱 HTML이 첫걸음

가장 효과적인 접근성 개선은 올바른 HTML 요소를 사용하는 것입니다.

의도 권장 태그
페이지 영역 <header>, <nav>, <main>, <footer>, <aside>
내비게이션 <nav> + <ul><li><a>
버튼 <button> (절대 <div> 클릭으로 대체 X)
폼 <form>, <label>, <input>, <fieldset>
문서 구조 <h1> ~ <h6> 순서대로

올바른 태그는 스크린 리더가 콘텐츠를 정확히 전달하고, 키보드 사용자가 자연스럽게 탐색하게 해줍니다.

폼과 레이블

모든 입력 요소에는 명시적 레이블이 있어야 합니다.

placeholder만으로는 레이블을 대체할 수 없습니다. 입력하는 순간 사라져 사용자가 맥락을 잃기 때문입니다. 오류 메시지는 aria-describedby로 연결합니다.

이미지와 대체 텍스트

의미 있는 이미지에는 alt 속성을, 장식용 이미지에는 빈 alt=""를 부여합니다.

SVG 아이콘이 버튼의 유일한 콘텐츠라면 aria-label을 추가해 의미를 전달합니다.

키보드 탐색

마우스를 쓰지 못하는 사용자는 키보드만으로 모든 기능에 접근할 수 있어야 합니다.

  • Tab / Shift+Tab으로 포커스 이동
  • Enter, Space로 활성화
  • Esc로 닫기
  • 화살표로 메뉴/탭/슬라이더 이동

체크할 항목들입니다.

  • 모든 인터랙티브 요소는 키보드 포커스를 받을 수 있는가?
  • :focus-visible 스타일이 명확한가?
  • 포커스 순서가 시각적 순서와 일치하는가?
  • 모달이 열리면 포커스가 모달 안으로 이동하고, 닫히면 트리거로 돌아오는가?

outline: none을 무작정 제거하지 말고 대체 스타일을 반드시 제공해야 합니다.

색상 대비와 시각적 단서

WCAG는 본문 텍스트에 대해 최소 4.5:1의 명도 대비를 권장합니다(큰 텍스트는 3:1).

또한 색상만으로 정보를 전달하지 않아야 합니다. 예를 들어 빨간 글씨로만 오류를 표시하는 대신 아이콘과 텍스트를 함께 사용합니다.

ARIA - 보완 도구

ARIA(Accessible Rich Internet Applications) 는 HTML만으로 표현할 수 없는 의미를 보조 기술에 전달합니다. 다만 첫 번째 원칙은 “가능하면 ARIA를 쓰지 말고 올바른 HTML을 써라” 입니다.

자주 쓰는 ARIA 속성입니다.

속성 용도
aria-label 보이지 않는 레이블
aria-labelledby 다른 요소 id를 참조해 레이블 지정
aria-describedby 보조 설명 연결
aria-hidden="true" 스크린 리더에서 숨김
aria-live 동적으로 바뀌는 영역 알림
role 요소의 역할 명시

예시: 토스트 알림을 스크린 리더에 안내합니다.

검사 도구

  • axe DevTools - 브라우저 확장으로 페이지 검사
  • Lighthouse - 크롬 내장, 접근성 점수 산출
  • WAVE - 시각적으로 문제 영역 표시
  • 스크린 리더 - 실제 사용 검증 (NVDA, VoiceOver)

자동화 도구는 약 30% 정도의 문제만 잡아내므로 수동 키보드 탐색과 실제 스크린 리더 테스트를 병행해야 합니다.

모범 사례 체크리스트

  • 의미 있는 HTML 태그를 사용했는가?
  • 모든 폼 입력에 레이블이 있는가?
  • 이미지에 적절한 alt가 있는가?
  • 키보드만으로 전체 기능을 사용할 수 있는가?
  • 포커스 스타일이 명확한가?
  • 색상 대비가 4.5:1 이상인가?
  • 헤딩(h1~h6)이 논리적 순서로 사용되었는가?
  • 동적 변화가 보조 기술에 전달되는가?

마무리

웹 접근성은 특별한 사용자만을 위한 기능이 아닙니다. 명확한 레이블, 일관된 포커스, 충분한 대비는 모든 사용자에게 더 나은 경험을 제공합니다. 처음부터 시맨틱 HTML과 키보드 탐색을 기본기로 익혀두면, 이후 ARIA나 고급 기법은 자연스럽게 따라옵니다. 작은 개선 하나하나가 누군가에게는 큰 차이가 됩니다.
