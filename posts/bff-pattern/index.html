BFF(Backend For Frontend) 패턴 이해하기

작성일 2026-05-19 Edited on 2026-05-20 In Backend
프론트엔드별 맞춤 백엔드 계층을 두는 BFF 패턴의 등장 배경, 구조, 장단점과 실전 도입 가이드를 정리합니다.

BFF란?

BFF(Backend For Frontend)는 각 프론트엔드(웹, 모바일 iOS, Android, 스마트 TV 등)에 특화된 백엔드 계층을 두어, 클라이언트별로 다른 요구사항을 효율적으로 처리하는 아키텍처 패턴입니다. 2015년 SoundCloud의 Phil Calçado가 처음 정리했습니다.

기존에는 모든 클라이언트가 동일한 API Gateway나 단일 백엔드를 사용했지만, 모바일은 데이터 절약이 중요하고 웹은 풍부한 정보를 원하는 등 요구사항이 달라지면서 클라이언트별 백엔드의 필요성이 커졌습니다.

등장 배경

  • 모바일과 웹의 화면 구성/데이터 양 차이
  • 클라이언트마다 다른 인증/세션 방식(JWT vs Cookie)
  • 한 화면을 만들기 위해 여러 마이크로서비스를 호출하는 부담
  • 클라이언트가 직접 여러 API를 호출하면 N+1 요청과 네트워크 비용 증가

일반적인 구조

각 BFF는 자신을 사용하는 클라이언트의 요구사항에만 최적화된 API를 제공합니다.

BFF의 역할

  1. 응답 형태 가공: 클라이언트가 바로 렌더링할 수 있는 형태로 데이터 변환
  2. API 통합(Aggregation): 여러 마이크로서비스 호출을 합쳐 한 번의 응답으로 제공
  3. 인증/세션 처리: 클라이언트별 인증 방식 처리
  4. 버전 관리: 클라이언트 앱 버전별 호환성 유지
  5. 에러 처리/재시도: 다운스트림 장애를 클라이언트에 친화적으로 변환

코드 예시 (Node.js + Express)

웹 BFF는 같은 도메인 데이터를 더 풍부하게 제공하면 됩니다.

장점

  • 클라이언트별 최적화된 응답으로 UX 개선
  • 프론트엔드 팀이 BFF를 소유하면 빠른 반복 가능
  • 마이크로서비스 변화로부터 클라이언트 보호
  • 클라이언트가 단순해짐(여러 서비스 호출 불필요)

단점

  • 백엔드 수 증가로 운영 복잡도 상승
  • 중복 코드 발생 가능(공통 로직 분리 필요)
  • BFF 자체가 또 하나의 의존성과 장애점이 됨
  • 도메인 로직이 BFF로 새는 안티 패턴 주의

API Gateway와의 차이

항목 API Gateway BFF
대상 모든 클라이언트 공통 특정 클라이언트 전용
책임 라우팅, 인증, Rate Limit 응답 가공, 데이터 통합
도메인 로직 가지지 않음 가공 수준의 얇은 로직

둘은 배타적이지 않으며, Gateway 뒤에 BFF를 두는 구성도 흔합니다.

도입 가이드

  • 클라이언트가 1종이면 도입 효과가 작음
  • 마이크로서비스 수가 많고, 클라이언트 종류가 다양할 때 효과적
  • GraphQL을 BFF 구현 도구로 활용하면 클라이언트가 필요한 필드만 가져갈 수 있음
  • BFF에 비즈니스 규칙을 절대 두지 말 것 (도메인 서비스로 위임)

마무리

BFF는 마이크로서비스 환경에서 클라이언트 경험을 개선하는 검증된 패턴입니다. 모바일과 웹의 요구사항이 명확히 갈라지고 마이크로서비스가 다수일 때 가장 큰 효과를 봅니다. 단, 운영 복잡도와 책임 경계를 잘 관리해야 BFF가 또 다른 monolith로 변질되지 않습니다.
