RBAC vs ABAC - 역할 기반과 속성 기반 접근 제어 비교

작성일 2026-04-26 Edited on 2026-06-05 In Backend , Authentication
RBAC와 ABAC의 모델 차이, 장단점, 실무 적용 패턴을 비교하고 두 모델을 결합한 하이브리드 설계 사례를 살펴봅니다.

접근 제어 모델의 진화

접근 제어(Access Control)는 “누가, 무엇에, 어떤 조건에서 접근할 수 있는가”를 결정하는 메커니즘입니다. 초기 시스템은 ACL(Access Control List)을 사용했지만, 사용자 수가 늘면서 관리가 어려워졌습니다. 이를 해결한 것이 RBAC이고, 더 세밀한 제어가 필요해지면서 등장한 것이 ABAC입니다.

RBAC - 역할 기반 접근 제어

RBAC(Role-Based Access Control)는 사용자에게 직접 권한을 부여하지 않고, **역할(Role)**을 통해 간접적으로 부여합니다.

단점은 **역할 폭발(role explosion)**입니다. 부서, 지역, 프로젝트별로 권한이 달라지면 Sales-APAC-ReadOnly, Sales-EMEA-Editor 같은 역할이 끝없이 늘어납니다.

ABAC - 속성 기반 접근 제어

ABAC(Attribute-Based Access Control)는 주체(Subject), 자원(Resource), 행위(Action), 환경(Environment)의 속성을 조합한 정책으로 접근을 결정합니다.

예시 정책: “마케팅 부서 직원은 자신의 부서가 소유한 캠페인을 업무시간(09:00~18:00)에만 수정할 수 있다.”

ABAC의 강점은 표현력입니다. 역할을 미리 정의하지 않아도 동적 조건을 정책으로 표현할 수 있습니다. 단점은 복잡성과 디버깅 난이도입니다. 어떤 정책이 어떤 결정을 내렸는지 추적이 어렵습니다.

비교 정리

항목 RBAC ABAC
권한 단위 역할 속성 조합
정책 표현 정적, 명시적 동적, 조건적
관리 도구 GUI 친화적 정책 언어 필요
감사(audit) 쉬움 어려움
확장성 제한적 매우 높음
도입 비용 낮음 높음

하이브리드 접근법

실무에서는 두 모델을 결합한 RBAC + ABAC 하이브리드가 일반적입니다.

기본 권한은 RBAC로 단순하게 부여하고, 데이터 단위의 세밀한 조건(소유자, 기밀 등급, 시간대)은 ABAC로 처리합니다. AWS IAM이 대표적인 하이브리드 모델로, 역할 + 조건(Condition) 정책을 함께 사용합니다.

ReBAC라는 또 다른 방향

최근에는 ReBAC(Relationship-Based Access Control)이 주목받고 있습니다. Google의 Zanzibar 논문이 시발점이며, “사용자 A가 문서 X의 작성자 그룹에 속하는가?” 같은 관계 그래프를 평가합니다. SpiceDB, OpenFGA 등 오픈소스 구현체가 있습니다.

실무 권장 사항

  • 단순한 사내 도구라면 RBAC로 시작하세요.
  • 멀티 테넌트 SaaS라면 RBAC + 테넌트 격리 ABAC 조건을 함께 설계하세요.
  • 정책이 코드 곳곳에 분산되면 OPA 같은 정책 엔진으로 외부화하세요.
  • 모든 권한 결정은 로그를 남겨 사후 감사가 가능하게 하세요.

마무리

RBAC는 단순하고 빠르게 도입할 수 있지만, 비즈니스가 복잡해지면 ABAC의 표현력이 필요해집니다. 두 모델을 적절히 결합하고, 정책을 코드 외부로 분리하는 설계가 장기적으로 유지보수 비용을 낮춥니다.
