Java 예외 처리 모범 사례 - Checked/Unchecked부터 사용자 정의 예외까지

작성일 2026-04-06 Edited on 2026-05-19 In Backend , Java
Checked와 Unchecked 예외의 차이, try-with-resources, 사용자 정의 예외 설계, 예외 변환과 로깅 전략 등 실무에서 통하는 자바 예외 처리 패턴을 정리합니다.

예외 계층 이해하기

자바의 모든 예외는 Throwable을 정점으로 합니다.

  • Error: OutOfMemoryError, StackOverflowError 등. 애플리케이션이 처리할 수 없는 시스템적 문제.
  • Checked Exception: 컴파일러가 처리(try/catch 또는 throws)를 강제. IOException, SQLException 등.
  • Unchecked Exception: RuntimeException의 하위. 처리를 강제하지 않음. NullPointerException, IllegalArgumentException 등.

Checked vs Unchecked, 무엇을 쓸 것인가

이 주제는 자바 커뮤니티의 오랜 논쟁이지만, 현대 자바에서는 다음과 같은 경향이 자리 잡았습니다.

  • 호출자가 합리적으로 회복할 수 있는 예외 → Checked (단, 신중히)
  • 프로그래밍 오류, 잘못된 사용 → Unchecked (IllegalArgumentException, IllegalStateException)
  • 서비스/도메인 예외 → 보통 Unchecked로 정의

Spring 같은 현대 프레임워크는 대부분 예외를 unchecked로 다룹니다. Checked 예외는 호출 스택을 따라 throws가 전파되며 코드를 오염시키기 쉽기 때문입니다. 단, 명확한 회복 경로가 있고, 클라이언트가 반드시 다뤄야 하는 경우에만 Checked로 두는 것이 권장됩니다.

try-with-resources

AutoCloseable을 구현한 자원은 try-with-resources로 다루는 것이 정석입니다.

여러 자원을 동시에 선언할 수도 있고, 선언된 역순으로 자동 close됩니다. 본문에서 예외가 발생하고 close에서도 예외가 발생하면 본문 예외가 우선이며 close 예외는 suppressed로 첨부됩니다.

사용자 정의 예외 설계

도메인의 의미를 명확히 드러내는 예외 타입을 직접 정의하는 것이 좋습니다.

원칙:

  • 메시지에 컨텍스트 정보(어떤 id가 없었는지 등)를 포함
  • 원인이 있다면 cause 인자로 체이닝 — throw new MyException("...", e);
  • 도메인 예외는 보통 RuntimeException 상속
  • 너무 잘게 쪼개지 말 것. 도메인 경계 단위로 묶어 유지보수성 확보

예외 변환(translation)

낮은 레벨 예외를 그대로 위로 던지면 호출자가 인프라 디테일에 의존하게 됩니다. 추상화 경계에서 도메인 예외로 변환하세요.

원래 예외를 cause로 첨부해 스택 트레이스를 잃지 않는 것이 핵심입니다.

로깅 가이드

  • 예외는 메시지와 함께 객체 자체를 로깅해야 스택 트레이스가 남습니다. log.error("foo failed", e);

마무리

좋은 예외 처리는 에러를 숨기는 것이 아니라 에러의 의미를 드러내는 것입니다. 도메인 의미를 가진 사용자 정의 예외, 추상화 경계에서의 변환, try-with-resources, 전역 핸들러 — 이 네 가지를 일관되게 적용하면 예외가 코드를 깨끗하게 유지하는 도구가 됩니다.
