SQL Injection 원리와 방어

작성일 2026-05-13 Edited on 2026-05-19 In Security
SQL Injection의 동작 원리와 다양한 변종을 살펴보고, 매개변수 바인딩 등 근본적인 방어 방법을 정리합니다.

SQL Injection이란?

**SQL Injection(SQLi)**은 공격자가 입력값에 SQL 문법을 삽입해, 애플리케이션이 의도하지 않은 쿼리를 데이터베이스에 실행시키는 공격입니다. 20년이 넘는 시간 동안 OWASP Top 10에서 사라지지 않은, 가장 오래되고 가장 파괴적인 웹 취약점 중 하나입니다.

성공적인 SQLi는 인증 우회, 데이터 유출, 데이터 변조, 일부 환경에서는 OS 명령 실행까지 가능하게 합니다.

가장 단순한 예시

다음과 같이 문자열을 그대로 합쳐 쿼리를 만드는 코드는 즉시 취약합니다.

userId에 admin' -- 를 넣으면 쿼리는 이렇게 변합니다.

-- 이후가 주석 처리되어 비밀번호 검증이 사라지고, admin으로 로그인할 수 있습니다.

변종(Variants)

Union-Based

UNION SELECT로 다른 테이블의 데이터를 응답에 끼워 넣습니다.

Error-Based

DB 에러 메시지가 그대로 노출되는 환경에서, 에러 안에 데이터를 담아 끄집어냅니다.

Blind SQLi

응답에 데이터가 보이지 않아도, 참/거짓에 따른 응답 차이나 시간 지연을 통해 한 비트씩 추출합니다.

Second-Order SQLi

악성 입력이 저장될 때는 무해하지만, 이후 다른 쿼리에서 그대로 합쳐질 때 발동되는 형태입니다. 단순 입력 검증만으로는 잡히지 않습니다.

근본 방어: 매개변수 바인딩(Prepared Statement)

SQL Injection을 완전히 막는 유일한 본질적 방어는 쿼리 구조와 데이터를 분리하는 것입니다. 매개변수 바인딩을 사용하면 입력값은 SQL 파서가 아니라 **데이터 자리(Placeholder)**로 처리됩니다.

ORM은 자동으로 안전한가?

대부분의 ORM은 내부적으로 매개변수 바인딩을 사용해 안전합니다. 그러나 다음과 같은 Raw Query나 동적 컬럼/정렬 처리는 여전히 위험합니다.

컬럼명, 정렬 방향, 테이블명은 매개변수로 바인딩되지 않습니다. 이 경우 반드시 **허용 목록(Whitelist)**으로 검증하세요.

최소 권한 원칙

DB 계정에 필요 이상 권한을 주지 마세요. 애플리케이션 계정에는 DROP, GRANT, FILE 같은 권한이 없어야 합니다. 읽기 전용 기능은 별도 read-only 계정으로 운영하는 것이 이상적입니다.

에러 메시지 노출 방지

상세 SQL 에러를 클라이언트에 노출하면 Error-Based 공격의 단서가 됩니다. 운영 환경에서는 사용자에게는 일반화된 에러를, 내부 로그에는 상세 정보를 분리해 남기세요.

WAF는 보조 방어

Web Application Firewall이 SQLi 패턴을 어느 정도 차단할 수 있지만, 우회 기법이 무수히 많아 근본 방어를 대체할 수 없습니다. 깊이 있는 방어(Defense in Depth)의 한 층으로만 사용하세요.

Best Practices

  • 모든 쿼리에 매개변수 바인딩 사용
  • 컬럼/정렬/테이블명은 화이트리스트 검증
  • DB 계정 최소 권한 부여
  • 에러 메시지 일반화 + 내부 로깅 분리
  • 정기적인 취약점 스캐닝(sqlmap 등) 및 코드 리뷰
  • ORM의 raw query 사용 시 주의

마무리

SQL Injection은 오래된 만큼 방어법도 분명한 취약점입니다. 쿼리 구조와 데이터를 절대 섞지 않는다는 단 하나의 원칙만 일관되게 지키면 거의 모든 변종이 차단됩니다. “내 코드에는 SQLi가 없다”가 아니라, “내 코드에는 문자열 SQL 결합이 없다”가 더 정확한 자신감입니다.
