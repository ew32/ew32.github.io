XXE 공격과 방어 - 안전한 XML 파서 설정

작성일 2026-04-28 Edited on 2026-05-20 In Security , Cryptography
XXE(XML External Entity) 공격의 원리와 파일 탈취, SSRF, DoS로 이어지는 피해 양상을 살펴보고 언어별 안전한 파서 설정을 정리합니다.

XXE 공격의 원리

XXE(XML External Entity)는 XML 파서가 외부 엔티티 참조를 처리하도록 허용된 경우 발생하는 공격입니다. XML 명세는 다음과 같은 엔티티 선언을 허용합니다.

파서가 &xxe;를 만나면 /etc/passwd의 내용을 읽어와 치환합니다. 결과적으로 응답에 파일 내용이 노출될 수 있습니다.

공격 시나리오

XXE는 단순 파일 읽기를 넘어 다양한 공격 벡터를 제공합니다.

1. 파일 탈취

서버가 XML 응답을 그대로 반환하는 경우, 임의 파일을 읽을 수 있습니다.

2. SSRF 결합

file:// 대신 http://를 사용하면 SSRF로 확장됩니다.

3. Blind XXE

응답이 보이지 않을 때는 외부 DTD를 이용한 OOB(Out-of-Band) 채널로 데이터를 유출합니다.

evil.dtd는 %file; 값을 공격자 서버로 전송하는 엔티티를 정의합니다.

4. Billion Laughs DoS

지수적으로 팽창하는 엔티티로 메모리를 고갈시킵니다.

언어별 안전한 설정

Java - DocumentBuilderFactory

핵심은 disallow-doctype-decl=true입니다. DOCTYPE 선언 자체를 거부하면 모든 엔티티 공격을 차단할 수 있습니다.

Python - defusedxml

표준 xml.etree, lxml은 기본 설정이 위험할 수 있습니다. defusedxml로 교체하세요.

defusedxml은 외부 엔티티, DTD 로드, 엔티티 확장을 기본 차단합니다.

.NET - XmlReader

Node.js

libxmljs, xml2js 등 라이브러리마다 옵션이 다릅니다. xml2js는 기본적으로 외부 엔티티를 처리하지 않지만, libxmljs는 명시적으로 noent: false, noblanks: true, nonet: true를 설정해야 합니다.

SVG, DOCX, RSS도 위험

XXE는 순수 XML API뿐 아니라 XML을 내부 포맷으로 사용하는 모든 형식에 영향을 줍니다.

  • SVG 업로드 기능
  • DOCX/XLSX 파싱(내부는 XML 압축본)
  • RSS/Atom 피드 크롤러
  • SAML Response 처리

업로드 받은 파일을 내부적으로 XML 파서로 처리한다면 동일하게 안전한 설정이 필요합니다.

점검 체크리스트

  • XML을 받는 모든 엔드포인트에서 DTD가 차단되어 있는가
  • 외부 엔티티, 파라미터 엔티티, 외부 DTD 로딩이 비활성화되었는가
  • 의존 라이브러리(SVG 처리, 문서 변환 등)도 같은 설정인가
  • WAF에서 <!DOCTYPE, <!ENTITY 패턴 모니터링을 하는가
  • 가능하다면 JSON으로 API를 전환했는가

마무리

XXE는 XML 파서의 기본 동작에서 비롯되는 고전적 취약점이지만, 여전히 많은 프레임워크가 위험한 기본값을 가집니다. DOCTYPE 자체를 거부하는 설정이 가장 단순하고 안전한 방어선입니다. XML이 꼭 필요하지 않다면 JSON 사용을 우선 검토하세요.
