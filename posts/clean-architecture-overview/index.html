클린 아키텍처 개요와 레이어 구조

작성일 2026-05-16 Edited on 2026-07-20 In Backend , Infrastructure
로버트 마틴의 클린 아키텍처가 제안하는 레이어 구조와 의존성 규칙을 정리합니다.

클린 아키텍처란?

클린 아키텍처(Clean Architecture) 는 로버트 마틴(Uncle Bob)이 헥사고날, 어니언, DCI 등 여러 아키텍처를 통합·정리한 모델입니다. 핵심 가치는 프레임워크·UI·DB·외부 시스템으로부터 비즈니스 로직을 독립 시키는 것입니다.

이를 위해 시스템을 동심원 레이어로 나누고, 모든 의존성이 안쪽(상위 정책) 방향으로 흐르도록 강제합니다.

레이어 구조

레이어 책임 예시
Entities 핵심 비즈니스 규칙 Order, Money
Use Cases 애플리케이션 시나리오 PlaceOrderUseCase
Interface Adapters 데이터 형식 변환 Controller, Repository 구현
Frameworks & Drivers 외부 기술 Spring, JPA, Web

의존성 규칙

소스 코드 의존성은 반드시 바깥쪽에서 안쪽으로만 향해야 한다.

즉 Entities는 Use Cases를 모르고, Use Cases는 Adapters를 모릅니다. 안쪽 레이어는 바깥 레이어를 알아서는 안 됩니다.

이를 가능하게 만드는 핵심 도구가 의존성 역전 원칙(DIP) 입니다.

의존성 역전 예시

Use Case가 DB를 사용해야 하지만 직접 의존하면 안 되는 상황입니다.

화살표가 Adapter → Use Case 방향으로 흐르고 있음을 주목하세요. DIP를 통해 의존성이 역전된 것입니다.

Entity 예시

Order는 Spring, JPA, HTTP 어떤 것도 알지 못합니다. 순수 자바로 작성됩니다.

클린 아키텍처의 효과

  • 테스트 용이성: 도메인/유즈케이스를 외부 의존 없이 단위 테스트
  • 프레임워크 독립: Spring → Quarkus 같은 이동도 어댑터만 교체
  • DB 독립: RDB → NoSQL 이전이 도메인에 영향 없음
  • UI 독립: REST → GraphQL 전환 용이

흔한 오해

오해 1: 폴더 4개로 나누면 클린 아키텍처다?

폴더 구조가 아니라 의존성 방향이 핵심입니다. 안쪽이 바깥쪽을 import 하는 순간 클린 아키텍처가 아닙니다.

오해 2: 모든 프로젝트에 적용해야 한다?

CRUD 위주의 단순 시스템에는 과합니다. 비즈니스 로직이 풍부하고 오래 유지보수해야 하는 시스템에 유효합니다.

오해 3: 헥사고날과 다른 것이다?

표현 방식이 다를 뿐 본질은 같습니다. 헥사고날의 “포트와 어댑터”가 클린 아키텍처의 “Interface Adapters”에 대응됩니다.

모범 사례

  1. 도메인에 @Entity, @Component 같은 프레임워크 어노테이션을 두지 않기
  2. Use Case는 입력 DTO(Command)와 출력 DTO를 명시적으로 정의
  3. 매핑 코드는 어댑터 레이어에 위치
  4. 패키지 의존성을 ArchUnit 같은 도구로 강제 검증

마무리

클린 아키텍처는 “비즈니스 규칙은 가장 안쪽에, 변동이 잦은 기술은 가장 바깥쪽에” 라는 단순한 원칙을 시각화한 모델입니다. 처음에는 코드가 늘어 보이지만, 장기적으로는 변경 비용을 크게 절감하는 투자입니다.
