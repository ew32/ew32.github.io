OIDC와 OAuth2의 차이 - ID Token과 Access Token

작성일 2026-04-23 Edited on 2026-06-15 In Backend , Authentication
OAuth2는 인가 프로토콜이고 OIDC는 인증 계층입니다. ID Token과 Access Token의 차이, 사용 시점, 검증 방법을 비교 정리합니다.

OAuth2와 OIDC의 본질적 차이

OAuth2는 인가(Authorization) 프로토콜입니다. “이 클라이언트가 사용자의 리소스에 접근할 수 있는가?”라는 질문에 답합니다. 반면 OpenID Connect(OIDC)는 OAuth2 위에 구축된 인증(Authentication) 계층으로, “이 사용자가 누구인가?”를 표준화된 방식으로 알려줍니다.

OAuth2만 사용하면 액세스 토큰을 받지만 그 토큰의 소유자가 누구인지는 명세에 정의되어 있지 않습니다. OIDC는 이 공백을 메우기 위해 ID Token이라는 표준 토큰을 도입했습니다.

ID Token vs Access Token

두 토큰은 형식이 비슷해 보이지만 목적과 검증 주체가 다릅니다.

항목 ID Token Access Token
목적 사용자 신원 증명 리소스 접근 권한
형식 항상 JWT JWT 또는 불투명 문자열
대상(audience) 클라이언트 애플리케이션 리소스 서버
검증 주체 클라이언트가 직접 검증 리소스 서버가 검증
수명 짧음(수 분) 짧음~중간

ID Token은 클라이언트가 사용자 정보를 얻기 위한 것이므로, 클라이언트가 직접 서명을 검증하고 클레임을 읽습니다. Access Token은 API 호출 시 Bearer 헤더로 전송되며, 리소스 서버가 검증합니다.

ID Token 검증 절차

클라이언트는 다음 단계를 모두 통과한 ID Token만 신뢰해야 합니다.

핵심 검증 항목은 서명, iss, aud, exp, nonce입니다. nonce는 재생 공격(replay attack)을 막기 위해 인증 요청 시 생성한 값과 응답 ID Token의 값이 같아야 합니다.

Access Token을 ID Token처럼 쓰면 안 되는 이유

흔한 안티패턴은 클라이언트가 Access Token을 디코딩해 사용자 정보를 추출하는 것입니다. 이는 두 가지 문제를 만듭니다.

  • Access Token이 JWT가 아닐 수 있고, 형식이 변경될 수 있습니다.
  • audience가 리소스 서버이므로 클라이언트가 직접 검증하면 안 됩니다.

사용자 정보는 ID Token 또는 /userinfo 엔드포인트에서 얻어야 합니다.

언제 OIDC가 필요한가

  • 소셜 로그인(구글, 페이스북 등)을 통한 사용자 인증
  • 기업 내 SSO에서 사용자 신원 전달
  • B2C 서비스의 표준화된 로그인 기반

단순히 외부 API의 리소스 접근만 필요하다면 OAuth2만으로 충분합니다. 그러나 “누가 로그인했는지”를 알아야 한다면 OIDC를 선택해야 합니다.

마무리

OAuth2는 권한 위임, OIDC는 신원 확인을 담당합니다. ID Token으로 사용자가 누구인지 확인하고, Access Token으로 API를 호출한다는 역할 분리를 지키면 인증/인가 설계가 단순해집니다. JWT 검증 라이브러리에 의존하기보다, 검증 항목을 명시적으로 이해하고 적용하는 것이 안전합니다.
