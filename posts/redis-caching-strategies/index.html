Redis 캐싱 전략

작성일 2026-05-04 Edited on 2026-05-22 In Backend , Database
Cache Aside, Write Through 등 대표적인 캐싱 패턴과 Redis 활용 시 주의점을 정리합니다.

Redis란?

Redis(Remote Dictionary Server)는 메모리 기반의 Key-Value 데이터 저장소입니다. 디스크 기반 DB보다 훨씬 빠른 읽기/쓰기 성능을 제공하며, 캐시, 세션 저장소, 메시지 브로커 등 다양한 용도로 활용됩니다.

Redis의 강점

  • 인메모리 기반: 마이크로초 단위 응답
  • 다양한 자료구조 지원: String, List, Set, Hash, Sorted Set, Stream
  • 영속성 옵션 (RDB, AOF)
  • Pub/Sub, 트랜잭션, Lua 스크립트 지원
  • 클러스터링 및 복제

왜 캐시를 사용할까?

데이터베이스 부하 감소, 응답 속도 향상, 외부 API 호출 비용 절감 등이 주된 목적입니다. 하지만 캐시 도입은 일관성과 복잡도라는 비용을 수반합니다.

대표적인 캐싱 패턴

1. Cache Aside (Lazy Loading)

가장 널리 쓰이는 패턴으로, 애플리케이션이 캐시와 DB를 직접 관리합니다.

  • 장점: 구현 간단, 필요한 데이터만 캐시
  • 단점: 첫 요청은 느림(캐시 미스), 캐시-DB 불일치 가능

2. Write Through

쓰기 시 캐시와 DB를 동시에 업데이트합니다.

  • 장점: 캐시가 항상 최신 상태
  • 단점: 쓰기 지연 증가, 사용하지 않는 데이터도 캐시

3. Write Behind (Write Back)

쓰기는 캐시에만 하고, 일정 주기로 DB에 비동기 반영합니다.

  • 장점: 쓰기 성능 매우 우수
  • 단점: 캐시 장애 시 데이터 유실 위험

4. Read Through

캐시 미스 시 캐시 라이브러리가 직접 DB에서 로드합니다. 애플리케이션 코드는 캐시만 알면 됩니다.

캐시 무효화 전략

캐시된 데이터를 어떻게 제거할지 결정하는 것은 매우 중요합니다.

1. TTL(Time To Live) 기반

일정 시간이 지나면 자동 만료됩니다.

2. 명시적 삭제

데이터 변경 시 관련 캐시를 직접 삭제합니다.

3. 이벤트 기반

DB 변경 이벤트(CDC, 트리거 등)를 구독하여 캐시를 무효화합니다.

자주 발생하는 문제와 해결

1. 캐시 스탬피드(Cache Stampede)

캐시 만료 시점에 동시 요청이 몰려 DB에 한꺼번에 부하가 가는 현상입니다.

해결책:

  • 분산 락: 첫 번째 요청만 DB 조회, 나머지는 대기
  • PER(Probabilistic Early Recomputation): 만료 전에 일부 요청이 미리 갱신
  • 백그라운드 갱신: 별도 워커가 주기적으로 갱신

2. 캐시 펜트레이션(Cache Penetration)

존재하지 않는 데이터를 반복 조회하여 DB에 부담을 주는 공격입니다.

해결책:

  • 빈 결과도 짧은 TTL로 캐시
  • Bloom Filter로 사전 필터링

3. 핫 키(Hot Key)

특정 키에 트래픽이 집중되어 클러스터의 한 노드에 부하가 쏠리는 현상입니다.

해결책:

  • 키를 여러 개로 분산 (hot_key:1, hot_key:2, …)
  • 로컬 캐시 추가

자료구조 활용 예시

영속성 옵션

  • RDB: 특정 시점의 스냅샷, 빠른 재시작
  • AOF: 모든 쓰기 명령 로그, 데이터 손실 최소화
  • 두 가지 병행: 일반적으로 권장

마무리

캐시는 강력한 도구이지만 “캐시 무효화는 컴퓨터 과학의 두 가지 어려운 문제 중 하나”라는 말처럼 신중하게 설계해야 합니다. 데이터의 특성과 일관성 요구 수준에 따라 적절한 패턴을 선택하세요.
