샤딩과 파티셔닝의 차이

작성일 2026-05-11 Edited on 2026-05-12 In Backend , Database
대용량 데이터베이스 확장 기법인 파티셔닝과 샤딩의 개념적 차이, 전략, 운영상 고려사항을 비교합니다.

두 용어가 헷갈리는 이유

파티셔닝(Partitioning) 과 샤딩(Sharding) 은 모두 “큰 데이터를 작은 조각으로 나눈다”는 점에서 비슷합니다. 그러나 나누는 위치가 다릅니다.

  • 파티셔닝: 하나의 DB 인스턴스 안에서 테이블을 여러 물리적 단위로 분할
  • 샤딩: 데이터를 여러 서버(노드) 에 분산 저장하는 수평 분할의 한 형태

엄밀히 말하면 샤딩은 “노드 간 수평 파티셔닝”이라고 볼 수 있습니다.

파티셔닝의 종류

수평 파티셔닝(Horizontal)

행을 기준으로 분할합니다.

수직 파티셔닝(Vertical)

컬럼을 기준으로 분할합니다. 자주 쓰는 컬럼과 큰 BLOB 컬럼을 분리해 I/O를 줄입니다.

파티셔닝 전략

전략 설명 예
Range 값의 범위로 분할 날짜, 가격대
List 명시적 목록으로 분할 국가 코드
Hash 해시 함수 결과로 분할 사용자 ID
Composite 위 전략의 조합 Range + Hash

파티셔닝의 장점

  • 쿼리 시 파티션 프루닝으로 스캔 범위 축소
  • 오래된 파티션 삭제가 DROP TABLE로 빠름
  • 인덱스 크기 감소
  • 통계와 락 범위 축소

샤딩이란

샤딩은 서로 다른 서버에 데이터를 분산합니다. 각 서버는 독립된 DB로 동작하며, 애플리케이션이나 라우터가 어느 샤드로 갈지 결정합니다.

샤드 키 선택

샤드 키는 시스템의 운명을 좌우합니다.

좋은 샤드 키의 조건:

  • 카디널리티가 높을 것
  • 쿼리 패턴과 일치할 것 (대부분 쿼리가 단일 샤드에서 처리)
  • 분포가 균등해 핫스팟이 없을 것
  • 가능하면 변경되지 않을 것

나쁜 예: created_at(단일 샤드에 최신 트래픽 집중), country_code(국가별 분포 불균등).

샤딩 전략

해시 기반

  • 분포가 균등
  • 범위 쿼리가 어려움
  • 샤드 추가 시 리해싱 비용 큼 → Consistent Hashing으로 완화

범위 기반

  • 범위 쿼리 효율적
  • 핫스팟 위험

디렉터리 기반

조회 테이블(룩업)로 매핑을 관리합니다. 유연하지만 룩업 서비스가 SPOF가 될 수 있습니다.

샤딩의 어려움

  • 조인: 여러 샤드를 가로지르는 조인은 비싸거나 불가능
  • 분산 트랜잭션: 2PC는 성능과 가용성 비용이 큼
  • 유니크 제약: 전역 유니크 보장 어려움 → UUID, 스노우플레이크 ID
  • 리밸런싱: 샤드 추가/삭제 시 데이터 이동 필요
  • 운영 복잡도: 백업, 모니터링, 마이그레이션 모두 N배

단일 노드에서 시도해볼 수 있는 것

샤딩 전에 다음을 고려하세요.

  1. 인덱스 최적화와 쿼리 튜닝
  2. 읽기 복제본 추가
  3. 캐시 계층 도입 (Redis 등)
  4. 파티셔닝 적용
  5. 수직 분할 (서비스 단위로 DB 분리)

이 모든 것을 해도 한계라면 그때 샤딩을 도입하는 것이 일반적입니다.

파티셔닝 vs 샤딩 비교

항목 파티셔닝 샤딩
범위 단일 노드 내 여러 노드
목적 관리·성능 수평 확장
운영 복잡도 낮음 높음
트랜잭션 일반 트랜잭션 분산 트랜잭션 필요
도구 DBMS 기본 기능 Vitess, Citus, 자체 라우터

실무 도구

  • PostgreSQL: 선언적 파티셔닝, Citus 확장으로 분산화
  • MySQL: 파티션 테이블, Vitess로 샤딩
  • MongoDB: 샤드 키 기반 자동 샤딩
  • CockroachDB / TiDB: 분산 SQL이 기본

마무리

파티셔닝은 “한 서버 안에서의 정리”, 샤딩은 “여러 서버로의 분산”입니다. 파티셔닝은 비교적 안전하게 도입할 수 있지만, 샤딩은 시스템 전체 설계를 바꿔놓습니다. 충분히 작은 규모에서는 단일 노드 + 복제본 + 캐시 조합이 가장 합리적입니다. 샤딩은 정말 필요할 때, 신중한 샤드 키 선택과 함께 도입하세요.
