클릭재킹 방어 - X-Frame-Options와 frame-ancestors

작성일 2026-04-29 Edited on 2026-07-20 In Security , Cryptography
클릭재킹 공격의 원리와 UI 리드레싱 변형, X-Frame-Options와 CSP frame-ancestors를 활용한 최신 방어 기법을 정리합니다.

클릭재킹이란

클릭재킹(Clickjacking)은 공격자가 자신의 페이지에 피해자 사이트를 투명한 iframe으로 띄우고, 사용자의 클릭을 가로채 의도하지 않은 행위를 유도하는 UI 리드레싱 공격입니다. 사용자는 “무료 쿠폰 받기” 버튼을 누른 것처럼 보이지만, 실제로는 그 위에 겹쳐진 은행 송금 버튼을 클릭하게 됩니다.

전통적 공격 예시는 다음과 같습니다.

피해자는 미끼 버튼을 누른다고 생각하지만, 실제로는 은행 송금 버튼을 클릭합니다.

변형 공격

기본 클릭재킹 외에도 다양한 변종이 있습니다.

  • Likejacking: 페이스북 좋아요 버튼을 가로채는 형태
  • Cursorjacking: 마우스 커서 위치를 가상으로 이동시켜 다른 영역 클릭 유도
  • Drag-and-drop jacking: 드래그 동작으로 데이터 유출
  • Filejacking: 파일 업로드 양식을 숨겨 파일 자동 전송

X-Frame-Options

가장 오래된 방어 헤더로, 페이지가 iframe 안에서 렌더링될 수 있는지를 제어합니다.

세 가지 값이 있었지만 ALLOW-FROM은 사실상 폐기되었고, 현재는 DENY 또는 SAMEORIGIN만 의미가 있습니다.

X-Frame-Options의 한계는 다음과 같습니다.

  • 여러 도메인을 허용할 수 없음(단일 출처만)
  • 일부 브라우저(특히 구버전)에서 동작이 다름
  • 멀티 오리진 환경에서 부족함

CSP frame-ancestors

CSP(Content Security Policy)의 frame-ancestors 지시문이 현대적 대안입니다. X-Frame-Options보다 표현력이 풍부하고, 최신 브라우저는 둘이 모두 있을 때 frame-ancestors를 우선합니다.

특정 도메인 여러 개를 허용하거나 와일드카드를 사용할 수 있습니다.

JS 기반 방어 - 권장하지 않음

오래된 글에서는 “프레임 버스터” JavaScript 코드를 추천하기도 합니다.

이 방법은 다음 이유로 안전하지 않습니다.

  • JS가 꺼지면 무력화됨
  • sandbox 속성을 가진 iframe은 top 접근을 차단
  • 여러 우회 기법이 알려져 있음

반드시 HTTP 헤더 기반 방어를 사용하고, JS 방어는 보조 수단으로만 활용해야 합니다.

점검 체크리스트

  • 모든 인증된 페이지에 frame-ancestors가 적용되었는가
  • 결제, 송금, 권한 변경 같은 민감 액션에 추가 확인(비밀번호 재입력)이 있는가
  • iframe 임베드가 필요한 페이지의 허용 도메인이 최소화되었는가
  • CSP report-uri로 위반 시도를 모니터링하는가

마무리

클릭재킹은 단순하지만 실제 피해로 이어지기 쉬운 공격입니다. X-Frame-Options: DENY와 frame-ancestors 'none'을 기본값으로 두고, 필요한 페이지만 명시적으로 임베드를 허용하는 것이 가장 안전한 정책입니다.
