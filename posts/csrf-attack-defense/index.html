CSRF 공격 원리와 방어 방법

작성일 2026-05-12 Edited on 2026-06-05 In Security
CSRF 공격이 성립하는 조건을 이해하고, SameSite 쿠키와 CSRF 토큰 등 현대적인 방어 기법을 정리합니다.

CSRF란?

**CSRF(Cross-Site Request Forgery)**는 인증된 사용자의 브라우저를 속여, 사용자가 의도하지 않은 요청을 신뢰받는 사이트로 보내게 만드는 공격입니다. 사용자가 이미 로그인한 상태이고, 브라우저가 자동으로 쿠키를 첨부한다는 사실을 악용합니다.

핵심은 “공격자가 사용자의 세션 쿠키를 읽지는 못하지만, 사용자가 그 쿠키와 함께 요청을 보내도록 유도할 수 있다”는 점입니다.

공격이 성립하는 조건

CSRF가 성공하려면 다음 세 조건이 동시에 충족되어야 합니다.

  1. 피해자가 대상 사이트에 이미 로그인되어 있다(쿠키 보유).
  2. 인증 정보가 자동으로 전송된다(쿠키, Basic Auth 등).
  3. 서버가 요청의 출처(Origin)를 검증하지 않는다.

로그인한 사용자가 이 페이지를 방문하기만 해도 송금이 실행됩니다. 사용자의 브라우저가 자동으로 쿠키를 첨부했기 때문입니다.

방어 1: SameSite 쿠키

현대 브라우저는 쿠키의 SameSite 속성으로 크로스 사이트 요청 시 쿠키 자동 전송을 제한할 수 있습니다.

값 동작
Strict 크로스 사이트 요청에 쿠키 미전송. 외부 링크 클릭도 차단
Lax 안전한 Top-Level GET만 허용 (기본값)
None 모든 요청에 전송, 단 Secure 필수

Lax가 현재 대부분의 브라우저 기본값이며, 일반적인 웹 앱에서는 이것만으로도 상당수의 CSRF를 차단합니다.

방어 2: CSRF 토큰(Synchronizer Token)

서버가 세션마다 예측 불가능한 랜덤 토큰을 발급하고, 폼/요청에 포함시키도록 강제합니다. 쿠키만으로는 요청을 위조할 수 없게 됩니다.

방어 3: Double Submit Cookie

세션 저장소가 부담스러운 경우, 쿠키와 헤더에 동일한 값을 넣고 비교하는 방식이 있습니다.

공격자는 다른 도메인에서 쿠키를 읽거나 헤더를 직접 설정할 수 없으므로, 두 값이 일치한다는 사실로 합법 요청을 확인할 수 있습니다. 다만 서브도메인 인젝션 공격을 막기 위해 토큰을 HMAC 서명하는 변형이 권장됩니다.

방어 4: Origin/Referer 헤더 검증

상태 변경 요청에 대해 Origin 또는 Referer 헤더가 자기 도메인과 일치하는지 확인합니다.

브라우저가 자동으로 설정하는 헤더이므로 JS로 위조할 수 없습니다. 단, 일부 프록시나 구형 브라우저에서 헤더가 누락될 수 있어 보조적 방어로 활용합니다.

API 서버는 어떨까?

JWT를 Authorization 헤더로 전송하는 SPA의 경우, 자동 전송되는 인증 수단이 아니므로 CSRF에 본질적으로 강합니다. 하지만 인증을 쿠키 기반으로 운영한다면 SPA여도 CSRF 방어가 필요합니다.

REST 메서드 규약 지키기

GET 요청은 부수 효과(Side Effect)가 없어야 한다는 HTTP 시맨틱을 지키는 것 자체가 CSRF 방어의 출발점입니다. 상태를 변경하는 작업은 반드시 POST/PUT/DELETE를 사용하세요.

Best Practices

  • SameSite=Lax 이상을 기본값으로 적용
  • 상태 변경 엔드포인트에 CSRF 토큰 도입
  • Origin/Referer 헤더 검증을 보조 방어로 사용
  • 인증 쿠키에 HttpOnly, Secure 플래그 필수
  • HTTP 메서드 시맨틱(GET은 안전) 준수
  • 민감 작업에는 재인증(Step-up Auth) 요구

마무리

CSRF는 사용자의 브라우저가 가진 자동 인증 권한을 악용하는 공격입니다. 그래서 방어 역시 “쿠키의 자동 전송을 어떻게 제어할 것인가”라는 관점에서 출발해야 합니다. SameSite, CSRF 토큰, Origin 검증을 다층적으로 결합하면 견고한 방어선을 만들 수 있습니다.
