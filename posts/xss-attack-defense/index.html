XSS 공격 유형과 방어 패턴

작성일 2026-05-12 Edited on 2026-06-15 In Security
Stored, Reflected, DOM-based XSS의 차이와 출력 인코딩, CSP를 활용한 다층 방어 전략을 정리합니다.

XSS란?

**XSS(Cross-Site Scripting)**는 공격자가 웹 페이지에 악성 스크립트를 주입해 다른 사용자의 브라우저에서 실행시키는 취약점입니다. 한 번 성공하면 세션 쿠키 탈취, 키로깅, UI 변조, 피싱, 내부 API 호출 등 사실상 사용자의 브라우저 권한 전체가 공격자에게 넘어갑니다.

OWASP가 매년 발표하는 Top 10에서 XSS는 늘 상위권을 유지하고 있으며, “Injection” 범주의 대표 사례 중 하나입니다.

XSS의 3가지 유형

1. Stored XSS

악성 스크립트가 서버 DB에 저장되어, 페이지를 보는 모든 사용자에게 실행되는 가장 위험한 형태입니다. 댓글, 게시판, 프로필 같은 사용자 생성 콘텐츠가 표적이 됩니다.

2. Reflected XSS

URL 파라미터 등 요청 데이터가 응답에 그대로 반사되어 발생합니다. 보통 피싱 링크와 결합됩니다.

3. DOM-based XSS

서버는 무관하고, 클라이언트 JavaScript가 location.hash, document.referrer 같은 값을 DOM에 안전하지 않게 삽입할 때 발생합니다.

URL이 #<img src=x onerror=alert(1)> 라면 스크립트가 실행됩니다.

핵심 방어 원칙: 컨텍스트별 출력 인코딩

XSS 방어의 본질은 입력 차단이 아닌 출력 인코딩입니다. 같은 문자열이라도 HTML 본문, 속성, JavaScript, URL 중 어디에 들어가느냐에 따라 이스케이프 규칙이 다릅니다.

컨텍스트 위험 문자 인코딩 방식
HTML Body < > & " ' HTML Entity
HTML 속성 위 + 공백 따옴표 + Entity
JavaScript ' " \ </ \xHH 이스케이프
URL 파라미터 다수 encodeURIComponent
CSS 다수 \HH 이스케이프

대부분의 모던 템플릿 엔진은 기본적으로 자동 이스케이프를 제공합니다.

문제는 개발자가 명시적으로 raw 출력을 켤 때 발생합니다.

innerHTML 대신 textContent

HTML로 렌더링할 필요가 없는 데이터는 절대 innerHTML을 쓰지 마세요. React의 dangerouslySetInnerHTML, Vue의 v-html도 같은 위험을 가집니다.

Content Security Policy (CSP)

CSP는 브라우저에게 어떤 출처의 스크립트만 실행할 수 있는지 알려주는 응답 헤더입니다. XSS가 발생해도 외부 스크립트 로딩과 eval을 차단해 피해를 최소화합니다.

특히 'unsafe-inline'을 피하고 nonce/hash 기반으로 운영하는 것이 권장됩니다.

Trusted Types

크롬 계열 브라우저가 지원하는 최신 표준으로, innerHTML 같은 **싱크(Sink)**에 일반 문자열을 직접 대입하지 못하도록 강제합니다. DOM XSS를 구조적으로 막을 수 있습니다.

HttpOnly 쿠키

XSS가 발생하더라도 세션 쿠키를 JavaScript에서 읽을 수 없도록 HttpOnly 플래그를 설정합니다. 이는 XSS를 막는 것이 아니라 피해 범위를 줄이는 방어입니다.

입력 검증은 보조 수단

입력 단계에서 <, >를 막는 식의 방어는 우회가 너무 쉽습니다. 입력 검증은 비즈니스 검증(길이, 형식)에 집중하고, 출력 단계의 인코딩과 CSP로 본질적인 방어를 구성하세요.

Best Practices

  • 템플릿 엔진의 자동 이스케이프를 항상 켜둘 것
  • innerHTML, v-html, dangerouslySetInnerHTML은 정말 필요한 경우만 사용하고 사전에 sanitizer(DOMPurify) 적용
  • CSP nonce 기반 정책 적용
  • 쿠키에 HttpOnly + Secure + SameSite
  • 사용자 입력을 URL/JS/CSS 컨텍스트에 직접 합치지 말 것

마무리

XSS는 “사용자의 브라우저에서 코드를 실행할 수 있다”는 단순한 사실이 가진 파괴력을 보여줍니다. 방어의 핵심은 입력이 아니라 출력이며, 단일 방어선이 아닌 인코딩 + CSP + 쿠키 정책 + Trusted Types의 다층 방어로 접근해야 합니다.
