Getting Started with Reactive Programming via Spring WebFlux

Posted on 2026-04-07 Edited on 2026-07-20 In Backend , Java
The Mono/Flux concepts of Spring WebFlux and Project Reactor, differences from MVC, using WebClient and R2DBC, and scenarios where reactive fits.

What Is WebFlux?

Spring WebFlux is a non-blocking reactive web stack introduced
in Spring 5. Where the existing Spring MVC is a servlet-based,
synchronous-blocking model, WebFlux runs on an asynchronous event
loop based on Reactor Netty by default.

The core of reactive programming is treating data as a stream and
declaratively composing asynchronously occurring events. It suits
I/O-intensive services that must handle many requests with few
threads.

Mono and Flux

The two core types Project Reactor provides.

  • Mono<T> — asynchronously emits 0 or 1 result
  • Flux<T> — asynchronously emits 0 to N results

The key is that nothing happens until you subscribe. A Reactor
pipeline is just a declaration; actual execution begins at
subscribe time.

Writing a Basic Controller

WebFlux supports two programming models.

1) Annotation-Based (Similar to MVC)

2) Functional Router

Using Operators

Reactive’s strength lies in operator composition.

  • flatMap — asynchronous chaining
  • zip — run multiple Monos in parallel and combine
  • switchIfEmpty — substitute when empty
  • timeout — error on delay
  • onErrorResume — fall back on error

WebClient — A Non-Blocking HTTP Client

The reactive replacement for RestTemplate.

It shines in scenarios that call several external APIs and combine
the results.

The three calls run in parallel, so the total response time
converges to the slowest call.

Data Access — R2DBC

JDBC is a blocking API, so it doesn’t leverage WebFlux’s benefits.
In a reactive environment, use R2DBC (Reactive Relational Database
Connectivity) or reactive MongoDB/Redis drivers.

JPA is unsuitable for a reactive environment. If you must use it,
isolate it on a separate thread pool.

Backpressure

Another core of reactive streams is that the consumer can request
only as much data as it can handle. Reactor implements this with
the request(n) signal. When handling large data, it automatically
regulates the flow to prevent OOM.

WebFlux vs MVC, When to Use What

Situation Recommended
Many external API calls, a gateway WebFlux
Streaming/SSE/WebSocket-centric WebFlux
General CRUD, using JPA MVC
CPU-intensive work MVC
Team lacks async experience MVC

With Java 21’s virtual threads, the question “is reactive really
necessary for concurrency?” is being raised again. To handle
hundreds of thousands of concurrent requests while keeping
synchronous code, MVC + virtual threads is also a powerful
alternative.

Debugging Tips

The reactive stack easily loses its stack trace.

  • In development, use Hooks.onOperatorDebug() or Reactor Tools BlockHound
  • Trace signals with the .log() operator
  • Mark a pipeline position with checkpoint("name")

Wrapping Up

WebFlux is an efficient and elegant tool for the right
workloads. But the learning curve is steep, and if even one
blocking call sneaks in, overall performance collapses. Ask “is it
really necessary?” first, and when you adopt it, the key is to keep
the consistency of non-blocking from end to end.
