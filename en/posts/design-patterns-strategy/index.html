Cleanly Swapping Algorithms with the Strategy Pattern

Posted on 2026-05-15 Edited on 2026-06-15 In Backend
The structure of the Strategy pattern - which encapsulates a family of algorithms and lets you swap them at runtime - and practical examples.

What Is the Strategy Pattern?

The Strategy pattern is a behavioral pattern that abstracts a set of
algorithms with the same purpose behind an interface, letting you
choose a concrete algorithm at runtime. By moving algorithms scattered
across if-else or switch branches into separate classes, you achieve
a design that is closed to modification and open to extension (OCP).

The GoF definition: “Define a family of algorithms, encapsulate each
one, and make them interchangeable.”

When to Use It

  • When the same task needs several approaches (sorting, payment, compression, authentication)
  • When conditionals grow large and adding a new case becomes painful
  • When you need to swap algorithms dynamically at runtime

Anti-Pattern: The Giant if-else

Every new payment method forces this class to change, and unit testing
is hard.

Applying Strategy

To add a new payment method, just implement PaymentStrategy.

Strategy in Spring

Combined with Spring DI, it becomes even cleaner.

Strategy vs State

The two patterns look almost identical structurally but differ in
intent.

  • Strategy: the client picks the algorithm
  • State: behavior changes automatically based on the object’s internal state

Strategy in Functional Style

Since Java 8 lambdas, you can pass a function without implementing the
interface.

Wrapping Up

Strategy is one of the most frequently used and effective patterns.
When if-else is growing, it’s the first tool to reach for, and
combining it with Spring DI or lambdas yields very concise code. That
said, if you truly have only 1–2 algorithms and they’re unlikely to
change, you don’t need to introduce it.
