Rate Limiting Algorithms and Implementation

Posted on 2026-05-14 Edited on 2026-06-15 In Backend , Security
A comparison of Rate Limiting algorithms for protecting APIs, plus implementation and operations strategies in distributed environments.

Why Rate Limiting Is Needed

Rate Limiting is a mechanism that limits how many requests can
be processed per unit of time, protecting both the system and its
users. Its main purposes:

  • Abuse prevention: block crawlers, bots, and brute-force attempts
  • Fairness: prevent a single user from monopolizing system resources
  • Cost control: prevent runaway costs on metered APIs
  • Failure isolation: block floods even when a backend-dependent service is down

For endpoints like login, password reset, and OTP sending in
particular, it must be applied as a security measure.

Four Key Algorithms

1. Fixed Window

Counts within a fixed time window (e.g., 1 minute). The simplest.

Downside: at the window boundary, a 2x traffic burst is possible
(100 at 00:59 + 100 at 01:00).

2. Sliding Window Log

Records request timestamps themselves and counts requests within N
seconds of now. The most accurate but with high memory cost.

3. Sliding Window Counter

An approximation that weight-averages two fixed windows.
Memory-efficient and smooths out boundary bursts.

4. Token Bucket

The most widely used algorithm. Tokens fill the bucket at a
fixed rate, and a request consumes one token. If there are no
tokens, it’s rejected.

It has the advantage of allowing bursts while still limiting the
average rate.

5. Leaky Bucket

Requests pile up in a queue and leak out (get processed) at a
fixed rate. Favorable for guaranteeing a fixed processing rate.
But it can’t absorb bursts and introduces delay.

Comparison Table

Algorithm Accuracy Memory Allows Bursts Implementation Difficulty
Fixed Window Low Low Partly (boundary burst) Very easy
Sliding Log Very high High None Moderate
Sliding Counter High Low Smoothed Moderate
Token Bucket High Low Yes Moderate
Leaky Bucket High Moderate None Moderate

Implementation in a Distributed Environment

With multiple servers, keeping the count in memory causes the
problem that the limit is computed separately per instance. The
common solution is to use Redis as a central counter.

Wrapping in a Lua script guarantees atomicity and avoids race
conditions. If you need a more sophisticated algorithm, using the
redis-cell module or the built-in features of Envoy/an API gateway
is also a good option.

Key Design

Deciding what to limit on matters.

Key Use
User ID Per-authenticated-user limit
API Key External partners
IP address Unauthenticated traffic
Endpoint + user Differentiated limits for sensitive endpoints
Device fingerprint Auxiliary against IP evasion

Limiting by IP alone has side effects in CGNAT, corporate
networks, and mobile-carrier NAT environments where many users
share an IP. Where possible, prioritize per-account keys.

HTTP Response Standards

Expose standard headers so clients can be aware of the limit. The
IETF draft RateLimit-* headers are the recommended direction.

Rate Limiting from a Security Perspective

The following endpoints need much stricter limits than general
APIs.

  • Login, password reset
  • OTP/SMS/email sending (risk of a cost bomb)
  • Signup, search autocomplete

Also, combining it with account lockout makes a stronger
brute-force defense than rate limiting alone.

Best Practices

  • Combine multiple keys — account + IP + endpoint
  • In distributed environments, handle it centrally with Redis Lua or an API gateway
  • Use the standard RateLimit-* headers and a 429 + Retry-After response
  • Apply a separate, strict policy to sensitive endpoints
  • It’s a separate layer from DDoS defense — handle that with a CDN/WAF

Wrapping Up

Rate limiting looks like a simple counting feature, but it’s a
system-design topic that must consider both accuracy in
distributed environments and the security threat model. More
important than the algorithm choice is the policy design: “what to
limit on” and “which endpoints get differentiated limits.” Start
small but adopt it early. A single line of rate limiting can block
tens of thousands of brute-force attempts.
