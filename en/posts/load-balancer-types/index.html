Load Balancer Types and Algorithms

Posted on 2026-05-24 Edited on 2026-06-15 In Network , Theory
The differences between L4 and L7 load balancers and the traffic-distribution algorithms.

What Is a Load Balancer?

A load balancer is a device or software that distributes
incoming network traffic across multiple backend servers, reducing
the load on any single server and improving availability. It’s
placed at virtually every segment of modern web services, playing a
key role in scalability and failure isolation.

The main value a load balancer provides:

  • Availability: route only to healthy servers when some fail
  • Scalability: easy horizontal scale-out
  • Performance: lower response time by spreading traffic
  • Security: can integrate SSL termination, WAF, DDoS protection

Load Balancers by OSI Layer

Load balancers are broadly classified by the layer they operate at.

L4 Load Balancer

Operates at the transport layer (TCP/UDP). Distributes based on IP
address and port.

  • Doesn’t inspect the payload
  • Very fast with high throughput
  • Cannot terminate TLS (TCP passthrough)
  • Examples: AWS NLB, HAProxy (TCP mode), LVS

L7 Load Balancer

Operates at the application layer (HTTP/HTTPS). Distributes based on
message content like URL, headers, cookies.

  • Content-based routing (/api → API servers, /static → static servers)
  • Extra features like TLS termination, compression, caching
  • Slightly slower (must parse the payload)
  • Examples: AWS ALB, Nginx, HAProxy (HTTP mode), Envoy

L4 vs L7

Item L4 L7
Layer TCP/UDP HTTP/HTTPS
Routing basis IP, port URL, headers, cookies
Throughput Very high High
Extra features Limited Rich
TLS handling Passthrough Can terminate
Use cases Games, DB, gRPC Web, REST APIs

Distribution Algorithms

1. Round Robin

Distributes requests to servers in turn. Simplest, but doesn’t
reflect performance differences between servers.

2. Weighted Round Robin

Assigns weights to servers to adjust the distribution ratio.

3. Least Connections

Routes to the server with the fewest active connections. Suited to
environments with variable request processing time.

4. Least Response Time

Picks the server with the fastest response time. Combined with
health checks, it adapts dynamically.

5. IP Hash

Hashes the client IP so the same IP always routes to the same
server. Useful when you need sticky sessions.

6. Consistent Hashing

A hashing technique that minimizes impact when servers are
added/removed. Often used for cache-server clusters.

Algorithm Evenness Session Stickiness Suitable Environment
Round Robin Good No Identical-spec servers
Weighted RR Good No Differing specs
Least Conn Very good No Variable processing time
IP Hash Average Yes When sessions are needed
Consistent Hash Good Yes Cache distribution

Health Checks

A load balancer periodically checks backend health and removes
problem servers from the pool.

  • Active health check: explicit verification with a separate request
  • Passive health check: judges from real traffic responses

Sticky Sessions

A stateful application must route the same user to the same server.

  • IP Hash: IP-based (limited under NAT)
  • Cookie-based: the load balancer issues a cookie
  • JWT / shared session store: design statelessly (recommended)

Global Load Balancing

Routes users spread across regions to the appropriate data center.

  • GeoDNS: DNS responses based on geographic location
  • Anycast: advertise the same IP from many locations (Cloudflare, AWS Global Accelerator)
  • CDN integration: branch at edge servers

Best Practices

  • Keep the health-check endpoint lightweight
  • Minimize reliance on session affinity through stateless design
  • Terminate TLS at the load balancer (backends use plaintext or mTLS)
  • Set generous timeouts (to isolate slow backends)
  • Collect metrics and logs (monitor P99 latency)

Wrapping Up

A load balancer isn’t merely a traffic distributor — it’s the
intersection of availability, scalability, and security. By
holistically designing whether you need L4 or L7, which algorithm
fits your workload, and how to handle session stickiness, you can
build a stable and scalable service.
