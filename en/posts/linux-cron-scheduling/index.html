Scheduling with Linux cron and crontab

Posted on 2026-07-04 Edited on 2026-05-26 In System , OS
How Linux's traditional job scheduler cron works, crontab syntax, common pitfalls, and alternatives.

What Is cron?

cron is the traditional Unix-family scheduler that runs commands
periodically at set times. The crond daemon wakes up every minute,
checks the registered job list, and runs matching entries. It’s
widely used for routine repetitive tasks like backups, log rotation,
and health checks.

Types of crontab Files

  • User crontab: edited with crontab -e. Runs with each user’s privileges.
  • System crontab (/etc/crontab, /etc/cron.d/*): adds a user field. Written by administrators.
  • Drop-in directories: /etc/cron.hourly, /etc/cron.daily, /etc/cron.weekly, /etc/cron.monthly. anacron works alongside to make up missed jobs even if the system was off.

Time Syntax

Values can combine * (all), , (list), - (range), and /

Common Pitfalls

1. Insufficient PATH

The cron environment is very limited and PATH is nearly empty.
Specify commands with absolute paths or declare PATH at the top of
the crontab.

2. Output Vanishes into Mail

stdout/stderr are sent to the user’s mailbox by default. If mail
isn’t set up, it’s lost — redirecting explicitly to a log file is
safer.

3. Overlapping Runs

If a long job overlaps with the next run time, two can run at once.
Prevent it with flock.

4. Time Zone

cron follows the system time zone. In containers it may run in UTC
unexpectedly, so set /etc/localtime or the TZ environment
variable explicitly.

5. Handling the % Character

If a command contains %, cron interprets it as a newline. Escape
it with \%.

Debugging Tips

  • Check execution history with journalctl -u cron (or crond).
  • Temporarily register a 1–2 minute interval to verify behavior.
  • Put set -euo pipefail at the top of scripts to avoid silent failures.
  • Use run-parts --test /etc/cron.daily to preview which scripts run.

Alternatives

  • systemd timer: better for dependencies, log integration, Persistent catch-up, RandomizedDelay, etc.
  • anacron: catch-up for systems not running 24/7, like desktops.
  • Kubernetes CronJob: scheduling for container workloads.
  • Cloud schedulers: AWS EventBridge Scheduler, GCP Cloud Scheduler, etc.

Wrapping Up

cron is simple, but it causes surprisingly frequent incidents if you
overlook operational details like environment differences, locking,
and logging. Make absolute paths, locking, and log redirection a
habit, and consider moving jobs with complex dependencies to a
systemd timer or a dedicated scheduler.
