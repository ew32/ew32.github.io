Webhook vs Polling: What to Use and When

Posted on 2026-06-17 Edited on 2026-05-26 In Network , HTTP
The operating principles, pros and cons, and security verification of polling and webhooks, from a practical perspective.

The Problem Both Approaches Solve

Suppose that when an event happens on server A, it must inform
server B of that fact. The simplest method is for B to periodically
ask “is there anything new?” (polling), and the smarter method is
for A to directly notify B at the moment the event occurs
(webhook). The two solve the same problem from opposite directions.

Polling: The Client Asks Periodically

Polling is where the client sends a request to the server at a
fixed interval to check whether anything changed.

Its advantages are simplicity and firewall-friendliness. B doesn’t
need to keep an inbound connection open. Its downsides are clear.

  • Requests continue even with no events → most responses are empty
  • Latency is determined by the polling interval (a 5s interval = 2.5s average delay)
  • Increased traffic cost and server load

Long Polling: An Improved Form of Polling

Here the server doesn’t immediately return an empty response but
holds the response until an event occurs. When a change happens, it
sends the response down right away and the client immediately sends
a new request. You get near-real-time latency, but the server side
must manage many open connections.

Webhook: The Server Notifies When an Event Happens

A webhook is where, when an event occurs, the server sends an HTTP
request (usually POST) to a pre-registered URL.

Its advantages are real-time behavior and efficiency. When there’s
no change, traffic is zero, and only when there’s a change is the
notification immediate. The downsides are that the receiving side
must have a public endpoint and a certain level of availability,
and that security and retry handling are tricky.

Webhook Security: Signature Verification

A webhook URL is public, so anyone can send a fake payload.
Therefore the sender sends an HMAC signature along, and the
receiver must verify it.

Idempotency and Retries

If the network is unstable, the same webhook can arrive multiple
times. It’s safe for the receiving side to keep an idempotency
table keyed by the event ID.

When a request comes in, first attempt INSERT ... ON CONFLICT DO NOTHING, and if there’s a conflict, treat it as an already-processed
event.

Selection Criteria

Situation Recommended
Hard to expose an endpoint to an external system Polling
Simple status lookups, low traffic Polling
Real-time behavior matters, e.g. payments, notifications Webhook
Large-scale events (thousands of TPS) Webhook + a queue
Bidirectional real-time communication WebSocket (see the separate post)

A Hybrid Pattern

In practice, polling is sometimes used as a backup on the
assumption that webhooks can be missed. For example, a payment
system notifies immediately via webhook, but the client also checks
the status once more every 5 seconds on the payment-complete page.
Using both approaches together captures reliability and real-time
behavior at once.

Wrapping Up

Polling is simple and safe but wasteful; webhooks are efficient but
incur operational cost. If you choose along the two axes of “event
frequency” and “real-time requirement,” you’ll rarely go wrong.
