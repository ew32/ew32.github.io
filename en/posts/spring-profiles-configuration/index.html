Spring Profiles and Externalized Configuration - A Per-Environment application.yml Strategy

Posted on 2026-04-11 Edited on 2026-07-20 In Backend , Java
A practical guide to Spring Boot Profiles, per-environment application.yml separation, @Profile, the precedence of environment variables and properties, and secret management.

Why Per-Environment Configuration Separation Is Needed

When you run the same application code on local, dev server,
staging, and production, the DB URL, external API keys, log level,
and feature toggles all differ. Branching on environment in code
soon hits a limit. Spring Boot solves this cleanly with two
mechanisms: Profiles and externalized configuration.

Profiles Basics

A Profile is simply “a named bundle of configuration.” Specify the
active Profile with one of the following.

You can combine multiple with commas:
--spring.profiles.active=prod,monitoring.

Splitting application.yml Files

The traditional way is to split files.

Put common values in application.yml, and the file matching the
active Profile is merged in via override.

Splitting into Multi-Document Within a Single File

You can also write multiple Profile configs in one file with YAML’s
--- separator.

Useful for small projects or library modules.

Activating Beans with @Profile

If you want to register a bean only in a specific Profile, use
@Profile.

Expressions like !prod, prod & monitoring, and prod | staging
are also possible.

Injecting Config Values: @Value and @ConfigurationProperties

@Value

Use it to inject one or two simple values.

@ConfigurationProperties (Recommended)

Bundles many properties into one object. Provides type safety,
autocomplete, and validation.

You can also apply JSR-303 validation.

Property Precedence

When the same key is defined in multiple places, Spring Boot
resolves it by a specified precedence (higher wins).

  1. Command-line arguments (--key=value)
  2. SPRING_APPLICATION_JSON
  3. JNDI / servlet context
  4. OS environment variables
  5. Java system properties (-Dkey=value)
  6. External application-{profile}.yml
  7. External application.yml
  8. Packaged application-{profile}.yml
  9. Packaged application.yml
  10. @PropertySource
  11. Default values

In production, the principle is to inject secrets via environment
variables or command-line arguments, and never put secrets in
the code repository.

Environment Variable Mapping Rules

A property like mail.smtp.host is automatically mapped from the
environment variable MAIL_SMTP_HOST.

Spring Boot maps _ to . and uppercase to lowercase. It’s the
most common approach for Docker container and Kubernetes operations.

${} Substitution and Defaults

In YAML you can reference other properties or specify a default for
an environment variable.

Secret Management

You must not store secrets in plaintext. The recommended methods:

  • Environment variable injection (the simplest)
  • Kubernetes Secret + environment variables
  • HashiCorp Vault + Spring Cloud Vault
  • AWS Secrets Manager / Parameter Store + Spring Cloud AWS

Profiles in Tests

@ActiveProfiles lets you specify the active Profile per test.

Placing an application-test.yml automatically loads test-only
configuration.

Common Mistakes and Tips

  • Profile conflicts: remember that if multiple profiles have the same key, the last activated profile wins.
  • Don’t put production config inside the package: secrets baked into the build artifact risk a leak.
  • The default profile: if no profile is active, the default profile applies. Be careful not to accidentally put production config in the default profile.
  • A change in Spring Boot 2.4+: use spring.config.activate.on-profile instead of spring.profiles (deprecated).
  • Actively use @ConfigurationProperties to keep type-safe configuration.

Wrapping Up

Profiles and externalized configuration are one of Spring Boot’s
biggest strengths. Being able to run the same binary in every
environment without changing a line of code is a big operational
weapon. There are three core principles — common config in
application.yml, environment differences in profile files or
environment variables, and never put secrets in the
repository.
