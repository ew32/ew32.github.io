Getting Started with gRPC and Protocol Buffers

Posted on 2026-05-22 Edited on 2026-07-20 In Backend
How the high-performance RPC framework gRPC works, how to write Protocol Buffers schemas, and pros and cons vs. REST.

What Is gRPC?

gRPC is an open-source RPC (Remote Procedure Call) framework that
Google built and donated to the CNCF. It lets a client call a remote
method as if it were local, with these traits:

  • HTTP/2-based: multiplexing, header compression, bidirectional streaming
  • Protocol Buffers: a compact, fast binary serialization format
  • Polyglot: auto-generates code for Java, Go, Python, Node.js, C++, Kotlin, and more
  • Contract-first: the .proto schema is the API spec

Protocol Buffers Basics

Define messages and services in a .proto file.

Use the protoc compiler to generate client/server stubs for each
language.

The Four RPC Types

  1. Unary: standard request/response
  2. Server Streaming: client sends 1 → server sends N
  3. Client Streaming: client sends N → server sends 1
  4. Bidirectional Streaming: N:N in both directions

Performance Characteristics

  • With binary Protobuf + HTTP/2 multiplexing, response size typically drops 30–50% vs JSON/REST and throughput more than doubles
  • A single TCP connection handles many concurrent requests (eases head-of-line blocking)
  • Static types + code generation cut the cost of writing clients

Pros and Cons vs REST

Pros

  • High performance, low latency
  • Strong contract (schema)
  • Bidirectional / streaming support
  • Auto-generated polyglot clients

Cons

  • Binary format is hard for humans to read (you need debugging tools)
  • Can’t be called directly from a browser (needs gRPC-Web or a gateway)
  • Compatibility issues with some HTTP/1.1 infrastructure (proxies)
  • Learning curve and toolchain dependency (protoc)

gRPC-Web and Gateways

Browsers can’t call regular gRPC. There are two common workarounds.

  1. gRPC-Web: a proxy (Envoy) translates gRPC-Web to regular gRPC
  2. grpc-gateway: adding HTTP mappings in your .proto auto-generates a REST/JSON gateway

Where to Use It

  • Inter-microservice internal communication
  • Mobile-to-backend (efficient in low-bandwidth settings)
  • Real-time bidirectional communication (chat, IoT telemetry, video signaling)
  • Polyglot multi-language environments

Tips on Schema Evolution

  • Never reuse a field number — ever
  • Use reserved instead of deleting fields
  • Avoid required (proto3 makes everything optional by default)
  • Make clients ignore unknown fields

Wrapping Up

gRPC has become almost the default choice for internal microservices
communication. If you need performance and a strong contract, it’s
worth a serious look. The most practical setup is hybrid: REST or
GraphQL for public APIs, gRPC for internal communication.
