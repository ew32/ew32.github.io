The CAP Theorem and PACELC - Trade-offs in Distributed Databases

Posted on 2026-04-17 Edited on 2026-07-20 In Backend , Database
A look at the CAP theorem - a fundamental principle of distributed system design - and its extension PACELC, plus a summary of what real-world distributed databases choose.

What Is the CAP Theorem?

The CAP Theorem was proposed by Eric Brewer in 2000 and formally
proven by Seth Gilbert and Nancy Lynch in 2002. It is a theorem about the
fundamental limits of distributed systems. It states that a distributed
data system can simultaneously satisfy at most two of the following
three properties.

  • C (Consistency): All nodes show the same data at the same point in time
  • A (Availability): Every request gets a response, even in the face of failures
  • P (Partition tolerance): The system keeps operating even when a network partition occurs

Why Can’t You Satisfy All Three at Once?

In a distributed system network failures can happen at any time, so P
is effectively mandatory. The real choice is therefore between CP
and AP, both assuming P.

Consider the following scenario:

If a client now sends a write to Node A, the system must pick one of two
options.

  1. Choose A (availability): Node A accepts the write and responds. But Node B doesn’t know about the latest data → consistency is violated
  2. Choose C (consistency): Node A rejects the write or waits. It doesn’t respond → availability is violated

CP Systems

Systems that prioritize consistency and partition tolerance. During a
network partition, some nodes may refuse to respond.

  • HBase, MongoDB (default settings), etcd, ZooKeeper, Spanner

Example: ZooKeeper refuses writes if it can’t reach a quorum, keeping all
nodes in the same state.

AP Systems

Systems that prioritize availability and partition tolerance. They
respond under any condition, but only offer eventual consistency.

  • Cassandra, DynamoDB, Couchbase, Riak

Example: Cassandra still accepts writes even when some nodes are
disconnected, and reconciles data later via read repair or hinted handoff.

CA Systems?

Theoretically possible, but only if you assume network partitions never
happen. A single-node RDBMS would fall into this category. In
practice, no distributed system that satisfies only CA actually exists.

Limitations of CAP

CAP is a heavily simplified model. Real systems are more complex in the
following ways.

  1. CAP doesn’t distinguish between normal operation and failure scenarios
  2. It doesn’t specify what “consistency” means exactly (linearizability, sequential consistency, etc.)
  3. It doesn’t deal with the latency trade-off

To compensate for these, PACELC was introduced.

The PACELC Model

A model proposed by Daniel Abadi in 2010 that extends CAP.

If Partition, then A vs C; Else, then Latency vs Consistency.

In other words,

  • During a partition: choose between A and C (same as CAP)
  • When there is no partition: choose between latency (L) and consistency (C)

This lets you express clearly what trade-off the system makes even in
normal operation.

PACELC Classification Examples

System During Partition Otherwise Class
Cassandra AP EL PA/EL
DynamoDB (default) AP EL PA/EL
MongoDB CP EC PC/EC
Spanner CP EC PC/EC
MySQL Group Replication CP EC PC/EC
Single-node RDBMS - EC -/EC

PA/EL systems prioritize availability and low latency; PC/EC systems
prioritize consistency.

The Variety of Consistency Models

The word “consistency” has several meanings.

  • Strong Consistency (Linearizability): All operations are observed in real-time order
  • Sequential Consistency: They appear in some consistent order
  • Causal Consistency: Only causally related operations are guaranteed an order
  • Read Your Writes: You can see your own writes immediately
  • Eventual Consistency: Eventually they become the same

Distributed system design starts by clearly defining which of these
levels you need.

Practical Adoption Guide

When to Choose CP

  • When strong consistency is required (financial transactions, inventory)
  • Distributed locks, leader election, configuration stores
  • Domains where refusing a response is better than returning wrong data

When to Choose AP

  • Social media feeds, recommendation systems
  • Log collection, metric storage
  • Cases where allowing some stale data has little business impact

Tunable Consistency

Systems like Cassandra and DynamoDB let you adjust the consistency level
per request.

Choices like ONE, QUORUM, and ALL change the response time and the
strength of the consistency guarantee.

Misconceptions About CAP

  1. The slogan “pick two out of three” is an oversimplification — in practice you only have to choose during a partition
  2. A CP system isn’t 100% consistent (it depends on implementation)
  3. An AP system doesn’t fail to provide any consistency at all (it provides eventual consistency)

Wrapping Up

CAP and PACELC are frameworks that reveal the essential trade-offs in
distributed system design. There is no magic system that meets every
business requirement; you must consciously decide the balance between
consistency, availability, and latency in line with what your service
needs.
