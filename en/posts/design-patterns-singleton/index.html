Design Patterns - The Singleton Pattern

Posted on 2026-05-29 Edited on 2026-05-22 In Backend , Java
The Singleton concept, a comparison of Java implementations, and caveats when using it.

What Is the Singleton Pattern?

The Singleton pattern is in the creational category of the GoF
design patterns. It guarantees that a class has only one instance
and provides global access to it.

Typical use cases include:

  • Configuration manager: keeps app-wide configuration
  • Logging system: a single logger instance unifies output
  • Connection pool: manages the database connection pool
  • Cache: an in-memory cache object

Basic Implementation - The Simplest Form

The most intuitive implementation uses a private constructor and a
public static method.

This works in a single-threaded environment, but in a multi-threaded
one two threads can pass instance == null at the same time and create
two instances.

Thread-Safe Implementation - synchronized

The simplest multi-threaded fix is synchronized on the whole method.

But locking on every call hurts performance. It’s wasteful to keep
locking after the instance has been created.

DCL (Double-Checked Locking)

The pattern that combines performance with thread safety is
double-checked locking.

volatile is the key. Object construction is three steps —
(1) memory allocation, (2) initialization, (3) reference assignment —
and the JVM may reorder them. Without volatile, you can return a
partially initialized object.

Holder Pattern - The Most Recommended

An elegant approach that leverages the class-loading moment of an
inner static class.

Holder is first loaded when getInstance() is called, so lazy
initialization is automatic. At the same time, the class loader’s
JVM-level guarantee on class-init thread safety removes the need for
explicit synchronization.

Enum Singleton - The Safest

The approach Joshua Bloch recommends in Effective Java is to use an
enum.

Usage is as simple as Singleton.INSTANCE.doSomething(). The benefits:

  • Safe under serialization: ordinary singletons can be re-created on deserialization
  • Resistant to reflection attacks: enums can’t be instantiated via reflection
  • Thread safe: guaranteed by the JVM

Drawbacks: it can’t extend a class and there’s no lazy
initialization.

Comparison of Implementations

Approach Thread Safe Lazy Safe vs. Serialization/Reflection Recommendation
Basic X O X Avoid
synchronized method O O X Only for simple cases
DCL + volatile O O X Decent
Holder O O X Recommended
Enum O X O Most recommended

Things to Watch For

A misused singleton easily becomes an anti-pattern.

  1. Global state: accessible from anywhere → hard to trace dependencies
  2. Hard to test: you can’t replace the instance, making mocking difficult
  3. Increased coupling: classes that use the singleton are tightly bound to it
  4. Multiple classloaders: each classloader may create its own instance

Alternative - DI Containers

In modern Java development, DI containers like Spring handle
object lifecycle. Spring beans are singleton-scoped by default — you
don’t need to implement the pattern yourself.

This gives you:

  • Easy mock injection in tests
  • Natural interface-based design
  • Unified lifecycle management

Wrapping Up

Singleton is one of the most familiar and most misused patterns. If
you must implement it yourself, use enum or Holder, and where
possible, delegate to a DI container. The starting point is asking
again whether you really need global state.
