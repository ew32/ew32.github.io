Service Worker and PWA Basics

Posted on 2026-06-22 Edited on 2026-06-05 In Frontend
The basics of the Service Worker lifecycle, caching strategies, the PWA manifest, and offline behavior.

PWA and Service Worker

A PWA (Progressive Web App) is an approach that combines the
universality of the web with the experience of a native app. Its
core engine is the Service Worker. A Service Worker is a
separate worker thread sitting between the browser and the network,
able to intercept all of the page’s requests to return cached
responses or perform background synchronization.

There are prerequisites a Service Worker must satisfy.

  • HTTPS is required (localhost is an exception).
  • No direct DOM access, no window object.
  • Only asynchronous APIs (no LocalStorage).
  • Registered to a specific scope (usually the site root).

Registration and Lifecycle

A Service Worker goes through the following states.

  1. install: runs when first registered or the file changes. Usually prepares the cache.
  2. activate: activates, replacing the previous SW version. Cleans up old caches.
  3. fetch / message / sync / push: handles events while active.

Caching Strategies

Combine these four strategies based on the resource’s nature.

Cache First (for static assets)

Network First (for HTML, APIs)

Stale-While-Revalidate (for resources that change often but where freshness matters less)

Network Only / Cache Only

Sensitive APIs like payments can be Network Only; fonts bundled at
build time can be Cache Only.

Debugging and Pitfalls

  • Chrome DevTools → Application → Service Workers lets you force updates, simulate offline, and inspect caches.
  • Serve the SW file from a stable, changeable URL (usually /sw.js) and set its cache policy to no-cache. Otherwise a new version may never deploy.
  • Include a version string in the cache key to make cleanup timing clear.
  • Using Workbox abstracts the strategies above into a concise API.

Wrapping Up

Service Workers are powerful but correspondingly tricky to debug.
Start small. Even a minimal setup — “Cache First for static assets,
Network First for HTML, leave APIs alone” — gives you both offline
behavior and a perceived speed boost.
