Database Query Optimization Techniques - From EXPLAIN to the N+1 Problem

Posted on 2026-04-18 Edited on 2026-06-05 In Backend , Database
A comprehensive look at SQL query optimization techniques you can apply right now - execution plan analysis, index usage, JOIN optimization, fixing N+1.

Why Query Optimization Matters

The database is the performance bottleneck in most backend services.
The same SQL can have a 10x or 100x difference in response time
depending on how it’s written. Query optimization isn’t merely
polishing SQL syntax — it’s a holistic task that requires
understanding the optimizer’s behavior, index structure, and execution
plans.

Execution Plan Analysis: EXPLAIN

The first step of optimization is always checking the execution plan.

MySQL

Key columns:

  • type: const, eq_ref, ref, range, index, ALL (slower as you go right)
  • key: the index used
  • rows: estimated rows examined
  • Extra: extra info like Using filesort, Using temporary

PostgreSQL

Node info like Seq Scan, Index Scan, Bitmap Heap Scan, Hash Join, Nested Loop is shown.

Principles of Index Usage

1. Leftmost Prefix

A composite index (a, b, c) can be used in these conditions only.

  • WHERE a = ?
  • WHERE a = ? AND b = ?
  • WHERE a = ? AND b = ? AND c = ?

WHERE b = ? alone can’t use the index.

2. Avoid Functions and Computation

3. Leverage High-Cardinality Columns

Low-cardinality columns like gender give little benefit as a
standalone index. Combine like (status, created_at) to raise
selectivity.

4. Covering Index

If every column the query needs lives in the index, the result can
return from the index alone, without touching the table.

JOIN Optimization

JOIN Types and Behavior

  • Nested Loop Join: for each row of the outer, look up the inner via an index. Good when row counts are small
  • Hash Join: build a hash table from the smaller side, scan the larger. Strong for equi-joins
  • Merge Join: merge two sorted tables. Efficient when both have indexes

Writing Tips

  1. Always index the join keys
  2. Reduce the smaller result set first when possible
  3. Exclude unneeded columns from SELECT
  4. If LEFT JOIN isn’t semantically required, switch to INNER JOIN to give the optimizer more freedom
  5. JOINs are generally faster than subqueries, but it varies by DBMS version — compare with EXPLAIN

The N+1 Problem

The most common performance issue with ORMs.

How It Happens

Fixes

  1. Fetch Join (JPQL)

  2. EntityGraph

  3. Batch size setting

  4. Write the single query yourself (native SQL or QueryDSL)

Pagination Optimization

Common Anti-Patterns

1. SELECT *

Select only the columns you need to reduce network and memory use. It
also unlocks covering-index optimizations.

2. Implicit Type Conversion

3. Overusing OR

4. Needless DISTINCT and ORDER BY

If the aggregate is already unique, drop DISTINCT. If you don’t need
sorted output, drop ORDER BY.

5. NOT IN and NULL

If the subquery contains NULL, the result comes back empty. Consider
NOT EXISTS or LEFT JOIN ... IS NULL.

Statistics and the Optimizer

The optimizer chooses plans based on statistics. After data
distributions change significantly, refresh stats.

Caching and Reusing Results

  • Result caching: store frequently queried results in Redis, etc.
  • Materialized views: precompute aggregates
  • Read replicas: route analytical queries to the slave

Optimization Workflow

  1. Identify slow queries: slow query log, pg_stat_statements
  2. Analyze with EXPLAIN: look for full scans and needless sorts
  3. Modify indexes or queries
  4. Compare before/after: response time, rows examined
  5. Monitor in production: watch how performance changes as data grows

Wrapping Up

Query optimization isn’t a one-and-done activity. As data changes and
traffic patterns shift, the same query behaves differently. Make
EXPLAIN a habit and keep index design principles and anti-patterns in
mind to operate a lightweight, scalable database.
