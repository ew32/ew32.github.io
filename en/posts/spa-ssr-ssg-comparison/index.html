SPA vs SSR vs SSG - Comparing Rendering Approaches

Posted on 2026-05-20 Edited on 2026-07-20 In Frontend , JavaScript
A comparison of the operating principles and pros and cons of client-side rendering, server-side rendering, and static site generation.

An Overview of Web Rendering Approaches

The way a web page is shown to users has evolved in many directions
over time. Even with the same React/Vue code, when and where the
HTML is generated greatly changes performance, SEO, and user
experience.

The three main approaches are:

  • SPA (Single Page Application): rendered on the client (browser)
  • SSR (Server Side Rendering): rendered on the server on each request
  • SSG (Static Site Generation): HTML pre-generated at build time

SPA - Client-Side Rendering

The browser receives a nearly empty HTML and a JavaScript bundle,
then JS runs and draws the screen. Page transitions are handled
by JS, so navigation is fast without a refresh.

  • Pros: fast page transitions after the initial load, rich interactivity, low server burden
  • Cons: slow initial load (bundle download + execution), poor SEO due to empty HTML, perceived slowness on low-spec devices

Representative tools: Create React App, Vite + React/Vue (without
SSR)

SSR - Server-Side Rendering

When a request comes in, the server runs React/Vue to respond with
complete HTML. The client sees the content immediately, and JS
then loads to make it interactive (hydration).

  • Pros: fast first paint (FCP), good SEO, dynamic data reflected immediately
  • Cons: server computation on every request, increased infrastructure cost/complexity, TTFB can be slow

Representative tools: Next.js, Nuxt, Remix, SvelteKit

SSG - Static Site Generation

An approach that pre-generates the HTML of all pages at build
time and puts them on a CDN. When a request comes in, the static
file is sent as-is, so it’s very fast.

  • Pros: top-tier response speed, minimal server cost, good security, cache-friendly
  • Cons: increased build time, unsuitable for frequently changing data, limits on per-user content

Representative tools: Next.js (static mode), Hexo, Astro, Gatsby,
Hugo

ISR and Hybrid

Recent frameworks use the approaches above mixed. A
representative pattern is ISR (Incremental Static Regeneration),
which periodically regenerates SSG-built pages in the background.

Also, React Server Components can decide server rendering at the
component level, not the page level, letting you freely mix
static and dynamic areas within a single page.

Comparison Table

Item SPA SSR SSG
When HTML is generated Browser Server, on request At build time
First-paint speed Slow Moderate Very fast
SEO Poor Good Very good
Server cost Low High Low
Dynamic data Good Very good Limited
Build time Short Short Proportional to page count

How to Choose

  • Blogs, docs, marketing sites: SSG (Hexo, Astro, etc.)
  • News, commerce, parts of dashboards: SSR + ISR
  • Post-login admin pages, SaaS internals: SPA or partial SSR
  • Mixed requirements: choose the optimal mode per page with Next.js, Nuxt, SvelteKit

Best Practices

  • Decide the mode per page based on SEO and the frequency of data changes.
  • Make static whatever can be made static.
  • Cache dynamic content as much as possible with cache headers and ISR.
  • Continuously measure Core Web Vitals (LCP, INP, CLS).

Wrapping Up

SPA, SSR, and SSG aren’t substitutes — they’re tools with their
own domains. Modern frameworks mix them, letting you choose the
optimal rendering at the page/component level. Analyze your
project’s content characteristics well and combine the right
approaches.
