Mastering PostgreSQL VACUUM - Everything About MVCC and autovacuum

Posted on 2026-04-13 Edited on 2026-06-05 In Backend , Database
PostgreSQL's MVCC structure, how VACUUM works, autovacuum tuning, and how to resolve dead tuple and bloat problems, from a practical standpoint.

PostgreSQL and MVCC

PostgreSQL controls concurrency through MVCC (Multi-Version
Concurrency Control). Under MVCC, modifying or deleting a row
doesn’t remove it immediately — a new version of the tuple is
created to ensure isolation between transactions. This has the
advantage that reads and writes don’t block each other, but it
produces a byproduct called the dead tuple.

If you don’t clean up these dead tuples, the table bloats,
performance degrades, and eventually you can hit the fatal problem
of transaction ID exhaustion.

What Is VACUUM?

VACUUM is PostgreSQL’s core maintenance operation that reclaims dead
tuples and makes space reusable. Its main roles:

  1. Reclaim dead tuples: remove tuples no transaction can see anymore
  2. Update the Visibility Map: enable Index Only Scans
  3. Update the Free Space Map: used to decide where to insert new data
  4. Prevent Transaction ID Wraparound: perform freezing to prevent XID exhaustion
  5. Gather statistics (VACUUM ANALYZE): refresh statistics for the optimizer

Types of VACUUM

1. VACUUM (lazy vacuum)

Plain VACUUM reclaims dead tuples but doesn’t return disk space to
the OS. That is, the table size doesn’t shrink — only internal
space is made reusable.

2. VACUUM FULL

Rewrites the whole table, fully removing dead tuples and reclaiming
disk space. But it takes an ACCESS EXCLUSIVE lock, so be careful
using it on a live system.

In production, an extension like pg_repack lets you reorganize a
table without a lock.

3. VACUUM ANALYZE

Performs VACUUM and also collects statistics.

How Autovacuum Works

PostgreSQL has autovacuum enabled by default to run VACUUM and
ANALYZE automatically. The autovacuum launcher spawns worker
processes that check, per table, whether thresholds have been
reached.

Monitoring Dead Tuples and Bloat

Large tables with a dead_ratio above 0.1 (10%) become tuning
targets.

The Transaction ID Wraparound Problem

PostgreSQL’s transaction ID is limited to 32 bits (~4 billion) and
has a circular (wraparound) structure. To prevent this, VACUUM marks
old tuples’ XIDs as frozen.

When autovacuum_freeze_max_age (default 200 million) is reached,
an anti-wraparound VACUUM runs forcibly, and at that point the user
can’t even stop it with a VACUUM command.

Common Problems and Fixes

  • Long-running transactions: a long-open transaction prevents VACUUM from removing dead tuples. Track them in pg_stat_activity
  • Unused replication slots: a forgotten slot holding WAL causes a similar problem
  • Accumulated bloat: measure exact bloat with the pgstattuple extension

Debugging When VACUUM Doesn’t Work

Check these items in order.

  1. Whether there are old transactions (pg_stat_activity)
  2. Unused replication slots (pg_replication_slots)
  3. Prepared transactions (pg_prepared_xacts)
  4. A standby with the hot_standby_feedback option on

Wrapping Up

VACUUM is the essential tool that underpins PostgreSQL’s MVCC.
Understanding how autovacuum behaves, steadily watching monitoring
metrics, and applying tuning suited to a table’s characteristics
lets you prevent problems like bloat and XID wraparound in advance.
Remember the saying: “PostgreSQL is happy when it gets to VACUUM
often.”
