A Summary of HTTP/2 vs HTTP/3 Differences

Posted on 2026-05-23 Edited on 2026-05-22 In Network , HTTP
Following HTTP's evolution, the core differences between HTTP/2 and HTTP/3.

The Evolution of HTTP

HTTP began with HTTP/0.9 in 1991 and has evolved for over 30 years.
Each version was designed to overcome the performance limits of the
previous one, and HTTP/2 (2015) and HTTP/3 (2022) are the
standards of the modern web.

  • HTTP/1.1 (1997): text-based, Keep-Alive, pipelining
  • HTTP/2 (2015): binary, multiplexing, header compression
  • HTTP/3 (2022): built on QUIC, uses UDP

The Limits of HTTP/1.1

HTTP/1.1 handles only one request at a time over a single TCP
connection. To work around this, browsers usually open six concurrent
connections per origin, but problems remain.

  • Head-of-Line Blocking: a slow earlier request stalls everything behind it
  • Header duplication: the same headers are sent on every request
  • TCP handshake cost: every new connection burns an RTT

Core Features of HTTP/2

1. Binary Framing

HTTP/2 communicates in binary frames, not text. Messages are
broken into small frames and interleaved.

Frame Type Role
HEADERS Header information
DATA Body data
SETTINGS Connection parameters
PRIORITY Priority hints

2. Multiplexing

Many requests and responses flow over the same TCP connection
concurrently, interleaved. HTTP/1.1’s Head-of-Line Blocking is
solved at the application layer.

3. Header Compression (HPACK)

Previously sent headers are referenced by index to eliminate
duplication. For follow-up requests to the same site, header size
approaches zero.

4. Server Push

The server can push resources to the client before the client
requests them.

(That said, it was hard to use well — Chrome deprecated Server Push
in 2022.)

5. Stream Priorities

Each stream gets a priority so important resources arrive first.

What HTTP/2 Doesn’t Solve

HTTP/2 still runs over TCP, so TCP’s own limits remain.

  • TCP-level Head-of-Line Blocking: one lost packet stalls every stream on the connection
  • Connection setup cost: TCP 3-way + TLS 1.2 handshake needs 2–3 RTTs
  • No connection migration: switching networks requires a new connection

Enter HTTP/3

HTTP/3 runs on a new transport protocol called QUIC. QUIC is
UDP-based and implements reliability, security, and multiplexing at
the application layer.

QUIC’s Traits

  1. UDP-based: evolves quickly, independent of OS upgrades
  2. TLS 1.3 integrated: handshake and encryption in 1 RTT (0 RTT on reconnect)
  3. Stream-level reliability: packet loss on one stream doesn’t affect others
  4. Connection IDs: keeps the connection alive across IP changes (Wi-Fi <-> LTE)

HTTP/2 vs HTTP/3 Comparison

Item HTTP/2 HTTP/3
Transport TCP QUIC (UDP)
Encryption TLS 1.2/1.3 (optional) TLS 1.3 (required)
Head-of-Line Blocking At TCP layer None
Connection setup 2–3 RTT 1 RTT (0 RTT possible)
Header compression HPACK QPACK
Connection migration No Yes
Adoption Very high Rapidly growing

Adoption Considerations

  • HTTP/3 uses UDP 443 — check firewall policy
  • In most environments, Alt-Svc triggers automatic upgrade
  • Verify caching and CDN compatibility
  • Falls back to HTTP/2 if the client doesn’t support HTTP/3

Real-World Performance Differences

Scenario HTTP/2 HTTP/3
Stable wired environment Similar Slightly ahead
Mobile / packet-loss environment Large degradation Holds steady
Network switch Reconnect required Connection survives

Wrapping Up

HTTP/2 lifted web performance dramatically with multiplexing and
header compression, and HTTP/3 builds on the new QUIC foundation
to overcome TCP’s structural limits. The benefits stand out
especially in mobile and unstable networks — HTTP/3 will quickly
become the default protocol of the web.
