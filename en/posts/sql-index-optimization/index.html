A Guide to SQL Index Optimization

Posted on 2026-05-05 Edited on 2026-06-05 In Backend , Database
Practical optimization know-how from how indexes work to composite index design and execution plan analysis.

What Is an Index?

An index is like the index of a book. Just as you find a specific
word in a book by looking at the index rather than reading the whole
thing, a DB can find the data it wants quickly through an index.

The essence of an index is a trade-off that boosts search speed by
using extra storage space. Making more of them isn’t
unconditionally good.

How Indexes Work

Most RDBMSes use a B-Tree (more precisely, a B+Tree) structure.

Characteristics of a B-Tree

  • Keeps data in sorted order
  • A balanced tree → finding any data takes nearly the same time (O(log N))
  • Favorable for range searches
  • Leaf nodes are linked together as a linked list

Why an Index Is Fast

Suppose you search for a value in a 10,000-row table:

  • Full scan: about 5,000 comparisons on average
  • A B-Tree index (assuming a fanout of 100): about 3 node lookups

Composite Indexes

An index that bundles multiple columns; the column order matters
greatly.

This index is effective only under the following conditions:

  • WHERE user_id = ?
  • WHERE user_id = ? AND status = ?

It isn’t used for:

  • WHERE status = ? (the leading column is missing)

Principles for Column Order

  1. Put higher-selectivity columns first (the more distinct values, the better)
  2. Put equality conditions before range conditions
  3. Put frequently used conditions first

Covering Indexes

An index where all columns the query needs are included in the
index, so the result can be produced from the index alone,
without touching the table.

Execution Plan Analysis

In MySQL/PostgreSQL you can check a query’s execution plan with
EXPLAIN.

Key Things to Check

  • type: ALL (full scan) is dangerous; ref/range/const are good
  • key: the index actually used
  • rows: estimated rows examined (fewer is better)
  • Extra: Using index (covering) is good; Using filesort/Using temporary warrant caution

The Cost of Indexes

1. Degraded Write Performance

On INSERT/UPDATE/DELETE, indexes must be updated too. The more
indexes, the worse write performance.

2. Storage Space

Indexes themselves take disk space. With many indexes on a large
table, the indexes can be larger than the data.

3. Optimizer Confusion

With many similar indexes, the optimizer may pick the wrong one.

An Index Design Checklist

  • Identify the frequently used WHERE/JOIN/ORDER BY columns
  • Is selectivity high enough? (recommend under 10%)
  • Is the column order of composite indexes appropriate?
  • Are there duplicate or unused indexes?
  • For write-heavy tables, did you minimize the number of indexes?
  • Did you check actual usage with an execution plan?

Updating Statistics

The optimizer builds execution plans based on statistics. If the
data has changed significantly, refresh the statistics.

Wrapping Up

Indexes are powerful but not a cure-all. Optimization without
measurement is guesswork, so get into the habit of adding or
removing indexes based on execution plans and actual query response
times.
