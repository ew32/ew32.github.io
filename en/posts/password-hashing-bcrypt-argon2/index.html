Password Hashing - bcrypt vs Argon2

Posted on 2026-05-13 Edited on 2026-07-20 In Security , Cryptography
The principles of hashing for safe password storage, and the difference between bcrypt and Argon2 with selection criteria.

Why You Can’t Store Passwords as Plaintext or a Simple Hash

If you store passwords as plaintext, the moment the DB leaks every
account is instantly compromised. But ordinary hashes like MD5 and
SHA-256 aren’t a solution either. Such hashes are designed to be
fast, so a modern GPU can attempt billions of tries per second.
Combined with a rainbow table, passwords of eight characters or
fewer crack in minutes.

A password hash must be slow and different per user. The
approach devised for this is Salt + Slow Hash, and its
representatives are bcrypt and Argon2.

The Three Principles of Password Hashing

  1. Salt: add a random value so the same password produces a different hash per user
  2. Slow: deliberately slow the computation to raise brute-force cost
  3. Parameterized: cost parameters can be adjusted as hardware advances

bcrypt

A Blowfish-based hash algorithm released in 1999 — the
longest-proven password-hashing function.

Characteristics:

  • Adjust the computation amount with a single cost factor
  • Generates and encodes its own salt
  • Input is truncated to 72 bytes (the tail of a long password is ignored)
  • Low memory usage, so relatively weak against GPU/ASIC attacks

Argon2

The winner of the 2015 Password Hashing Competition — currently
the algorithm OWASP most recommends for new systems. It has three
variants.

Variant Characteristics Use
Argon2d Strong GPU resistance, weak side-channel resistance Cryptocurrency mining
Argon2i Strong side-channel resistance Environments with side-channel concerns
Argon2id Combines the strengths of i + d Recommended for password hashing

Characteristics:

  • Memory-hard — requires large amounts of RAM, neutralizing GPU/ASIC attacks
  • Memory, iterations, and parallelism can be tuned independently
  • No input-length limit

bcrypt vs Argon2

Item bcrypt Argon2id
Released 1999 2015
Tuning parameters cost (time) memory + time + parallelism
Memory-hard No Yes
GPU resistance Average Strong
Input length 72-byte limit Unlimited
Standardization De facto standard RFC 9106
Library maturity Very high Increasingly mature

Which Should You Choose?

  • New systems: Argon2id recommended (OWASP guideline)
  • Legacy systems: keep bcrypt but with cost 12 or higher
  • If your environment lacks an Argon2 library, bcrypt (cost ≥ 12) or scrypt is also reasonable.

Parameter Guide (as of 2026)

General recommendations from the OWASP Password Storage Cheat Sheet.

In production, the common balance point is tuning so the login
response time is about 250–500ms.

What You Must Never Do

  • Storing passwords with only plaintext/MD5/SHA-256/SHA-512
  • Using the same salt for all users
  • Storing the pepper (server secret) in the same DB as the password (the pepper goes in a separate KMS)
  • Using your own homemade hashing algorithm

Additional Defenses Beyond Passwords

Hashing alone isn’t enough. Operate the following policies together.

  • Login rate limiting + account lockout
  • Adopt MFA (multi-factor authentication)
  • Block leaked passwords (the haveibeenpwned API)
  • Periodic rehashing of the algorithm (rehash on login)

Wrapping Up

Password hashing is a game of “how much time can you buy assuming
the DB has leaked.” bcrypt is still a safe choice, and Argon2id is a
better default. Remember that what matters isn’t the algorithm name
but sufficient cost parameters, regular reassessment, and
layered defense.
