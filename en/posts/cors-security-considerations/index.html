CORS Security Considerations - A Safe Configuration Guide

Posted on 2026-04-30 Edited on 2026-06-05 In Security , Cryptography
The security risks created by misconfigured CORS, what to watch for when combining Access-Control-Allow-Origin wildcards with credentials, and safe allowlist patterns.

CORS Is Not a Security Feature

A common misconception is that CORS (Cross-Origin Resource Sharing) is
a “security feature.” More precisely, CORS is a relaxation mechanism
for the Same-Origin Policy. That is, it’s a protocol for the server to
explicitly allow cross-origin requests that would otherwise be blocked.

So misconfiguring CORS weakens security — it doesn’t strengthen it.

The Most Dangerous Combination

The following combination is almost always a vulnerability.

If the server puts the request’s Origin header back into the response
verbatim and also allows credentials, any origin can make authenticated
requests. An attacker’s evil.com can call the API with the victim’s
cookies.

The Wildcard Trap

Access-Control-Allow-Origin: * cannot be used together with
credentials. The browser refuses.

Because of this, developers sometimes arrive at the wrong conclusion of
“then I’ll just reflect the Origin.” As we saw above, that workaround
creates a bigger risk.

Beware of null Origin

A sandboxed iframe, a local file (file://), and some redirect
scenarios send Origin: null. The following configuration can be
exploited.

An attacker can spoof a null Origin by making the request from inside
<iframe sandbox>. Never put null on your allowlist.

A Safe Allowlist Pattern

A correct implementation checks an explicit allowlist.

The key points:

  • Put only the exact origin from the allowlist into the response
  • Add Vary: Origin so the CDN doesn’t cache wrongly
  • Match wildcard subdomains strictly with a regex

Pitfalls in Regex Matching

When allowing subdomains, regex mistakes are common.

Always anchor and ensure a full match.

Preflight Requests

Complex cross-origin requests first trigger a preflight (OPTIONS).
The server responds with the following headers.

You can extend the cache time with Access-Control-Max-Age to improve
performance, but consider the delay in propagating policy changes.

Relationship to CSRF

CORS does not block CSRF. CSRF arises from the browser automatically
attaching cookies; CORS protection only blocks reading the response —
the request itself still reaches the server. State-changing APIs must
be protected by one of the following.

  • A CSRF token
  • A SameSite=Strict|Lax cookie
  • A custom header + forced preflight

Audit Checklist

  • Is Allow-Origin an explicit allowlist, not a wildcard or Origin reflection?
  • Is the combination Allow-Credentials: true + * absent?
  • Is the null Origin disallowed?
  • Does the response include a Vary: Origin header?
  • Is the subdomain regex a fullmatch with proper escaping?
  • Do state-changing APIs have separate CSRF protection?

Wrapping Up

CORS is one of the areas where misconfiguration most frequently leads
to security incidents. Resist the temptation of wildcards and use an
explicit allowlist. CORS is a relaxation mechanism for the SOP — not a
substitute for CSRF protection or authentication.
