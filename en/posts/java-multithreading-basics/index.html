Java Multithreading Basics

Posted on 2026-05-08 Edited on 2026-05-26 In Backend , Java
The basic mechanisms for creating and synchronizing threads in Java, plus the concurrency toolbox.

Why Multithreading

Modern CPUs are multi-core, and a single-threaded program uses only
a sliver of them. Multithreading is a key technique to improve
responsiveness and throughput by doing many things at once. But if
you can’t manage the shared-resource contention and unpredictability
that come with concurrent execution, you’ll end up with corrupted
data or deadlocks.

Creating Threads

The most basic ways are the Thread class or the Runnable
interface.

Implementing Runnable is the usual choice over extending Thread.
From Java 21 you can also use lightweight virtual threads.

ExecutorService

In practice you don’t spawn threads directly — manage them with
ExecutorService.

Executor flavors:

Factory Description
newFixedThreadPool(n) Fixed-size pool
newCachedThreadPool() Creates on demand, reclaims after 60s
newSingleThreadExecutor() Single thread
newScheduledThreadPool(n) Delayed/periodic execution
newVirtualThreadPerTaskExecutor() Java 21+ virtual threads

Shared State and Synchronization

When multiple threads touch the same variable, race conditions
appear.

count++ is read-modify-write, so concurrent calls lose updates.
Several fixes:

synchronized

Uses the object’s implicit lock; can be applied at method or block
scope. Simple, but performance degrades when the locked section
grows.

java.util.concurrent.atomic

Atomic classes provide atomic operations without locks.

ReentrantLock

More flexible than synchronized — tryLock, fairness, interruptible
acquire.

volatile and Visibility

The Java Memory Model allows per-thread caching, so one thread’s
write may not be visible to others immediately. The volatile
keyword forces reads and writes to go through main memory.

volatile doesn’t provide atomicity, however, so it can’t replace
compound operations like count++.

Concurrent Collections

Collections in java.util.concurrent are thread-safe.

  • ConcurrentHashMap: hashmap with segment locking
  • CopyOnWriteArrayList: copies the array on write, favorable for read-heavy workloads
  • BlockingQueue family: producer-consumer pattern

CompletableFuture

A modern API for composing asynchronous results.

You can express a dependency graph without callback hell, and error
handling stays consistent.

Avoiding Deadlocks

Deadlocks happen when two threads each hold a lock and wait for the
other.

  • Standardize lock-acquisition order across the codebase
  • Use tryLock(timeout) so threads can give up
  • Keep lock scopes small and hold only one lock at a time when possible

Best Practices

  • Use an ExecutorService or virtual threads instead of creating threads directly
  • Reduce shared mutable state; prefer immutable objects and message passing
  • Use only the concurrency tools whose semantics you fully understand
  • Aggressively expose race conditions with load testing

Wrapping Up

Multithreading is powerful but full of traps. Understand the precise
differences between synchronized, volatile, atomic, and the
concurrent collections, and reach for higher-level abstractions
(ExecutorService, CompletableFuture, virtual threads) first —
that’s the safe and productive path.
