A Complete Guide to Redis Data Structures - String, List, Hash, Set, ZSet, Stream

Posted on 2026-04-20 Edited on 2026-06-05 In Backend , Database
The characteristics, commands, and practical use cases of Redis's core data structures - String, List, Hash, Set, Sorted Set, Stream - with code examples.

Redis and Data Structures

Redis isn’t just a simple key-value cache — it’s called a “data
structures server” for the variety of structures it provides.
Because it operates in memory, microsecond responses are possible,
and each structure has optimized commands, letting you express
complex business logic concisely.

1. String

Redis’s most basic data structure; every key’s value can store up to
512MB. It can handle not just strings but also numbers (with atomic
increment/decrement) and serialized objects (JSON, etc.).

Uses: caching, counters, distributed locks, session storage

2. List

A doubly linked list that can add/remove elements at either end in
O(1). Usable as a queue and a stack.

Uses: job queues, the N most recent logs, message buffers

3. Hash

A map of fields and values. Suited to representing objects. Small
hashes are stored as a ziplist, making them very memory-efficient.

Uses: user profiles, field-level object updates,
memory-efficient data storage

4. Set

An unordered collection with no duplicates. Supports union,
intersection, and difference operations.

Uses: tags, friend relationships, like lists, recommendation
systems, deduplication

5. Sorted Set (ZSet)

A collection sorted by score. The heart of ranking systems.
Add/remove is O(log N) and range queries are O(log N + M) — very
efficient.

Uses: leaderboards, priority queues, chronologically sorted
feeds, delayed-job queues

The Delayed-Job Queue Pattern

6. Stream

A log data structure added in Redis 5.0. It provides an append-only
log similar to Kafka and supports consumer groups.

Uses: event sourcing, message broker, audit logs, async job
dispatch

7. Additional Data Structures

Bitmap

Performs bit-level operations on top of a String. Useful for
memory-efficient attendance tracking and on/off flags.

HyperLogLog

Approximate cardinality estimation. With 12KB it can measure billions
of unique counts with about 0.81% error.

Geo

Stores geographic coordinates and computes distances.

Bloom Filter (Redis Stack)

Probabilistically judges existence. Useful for preventing cache
misses.

A Data Structure Selection Guide

Requirement Recommended Structure
Simple cache String
Storing an object Hash
Job queue List, Stream
Deduplication, set operations Set
Ranking, chronological sorting Sorted Set
Event streaming Stream
Attendance tracking, on/off Bitmap
Unique counts HyperLogLog
Location-based search Geo

Key Naming Convention

Separating namespaces with a colon (:) makes key search and
management easier.

Operations Caveats

  • Set TTLs: assign appropriate expirations to prevent memory leaks
  • Forbid KEYS *: use SCAN in production
  • Beware of large keys: handling large data in a single command can block other commands
  • Set maxmemory-policy: explicitly specify a policy like allkeys-lru
  • Choose persistence: a combination of RDB (snapshot) + AOF (append-only) is recommended

Wrapping Up

Knowing Redis’s varied data structures lets you solve the same
problem far more elegantly and quickly. Rather than using it only as
a simple cache, understand its potential as a data structures server
and choose the structure that fits your business requirements —
that’s how to use Redis most effectively.
