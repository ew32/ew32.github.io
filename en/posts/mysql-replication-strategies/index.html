A Complete Guide to MySQL Replication - From Master-Slave to Group Replication

Posted on 2026-04-12 Edited on 2026-05-26 In Backend , Database
A comparison of MySQL's core replication strategies - Master-Slave, Group Replication, GTID - with details on setup, failure handling, and monitoring in practice.

What Is MySQL Replication?

MySQL replication is the technology that delivers data from one
MySQL server to another, asynchronously or semi-synchronously. It
provides operational benefits such as read-load distribution, high
availability (HA), backups, and separating analytical workloads.
Replication isn’t merely copying data — it’s the core
infrastructure responsible for production stability.

How Replication Works

MySQL replication operates on the binary log (binlog). The
master (source) server records every data change (INSERT, UPDATE,
DELETE, DDL) into the binlog, and the slave (replica) reads and
re-executes them against itself.

The processing flow:

  1. Master: records an event in the binlog on transaction commit
  2. The slave’s IO Thread: reads binlog events from the master into the local relay log
  3. The slave’s SQL Thread: reads the relay log and applies it to the data

Master-Slave (Traditional Asynchronous Replication)

The most basic setup — a single master takes writes and N slaves
replicate the data asynchronously.

Pros

  • Simple to set up with low operational burden
  • Read load can be distributed to slaves
  • On master failure, a slave can be promoted for quick recovery

Cons

  • Being asynchronous, replication lag can occur
  • Stale-read risk when reading from a slave
  • On master failure, data not yet replicated may be lost

Semi-Synchronous Replication

A method introduced to reduce the data-loss risk of asynchronous
replication. The master delays its commit response until at least
one slave acknowledges that it has recorded the binlog event into
its relay log.

With its good balance of performance and stability, it’s frequently
used in finance and similar sectors.

GTID-Based Replication

A GTID (Global Transaction Identifier) assigns a globally unique
ID to each transaction. Traditional binlog filename/position-based
replication made it hard to find the exact position during
failover; GTID solves this.

The GTID format is server_uuid:transaction_id. e.g.,
3E11FA47-71CA-11E1-9E33-C80AA9429562:23.

Benefits of GTID

  • Failover becomes simple (SOURCE_AUTO_POSITION=1)
  • Easy to track which transaction was applied where
  • Easy to change the replication topology

MySQL Group Replication

A multi-master synchronous replication solution introduced in
MySQL 5.7. It uses a Paxos-based consensus algorithm (XCom) and
provides both consistency and availability.

Key Characteristics

  • All nodes in the group are peers (single-primary or multi-primary mode)
  • A transaction commits after majority consensus in the group
  • Automatic detection and group reconfiguration on node failure
  • Includes conflict detection and a certification stage

Setup Example

When Group Replication Is a Good Fit

  • Mission-critical services that need automatic failover
  • Environments where data consistency across multiple data centers matters
  • When you want unified operation through InnoDB Cluster

Monitoring Replication Lag

Tools like pt-heartbeat or PMM enable more accurate lag
measurement.

Practical Operations Tips

  • Standardize on binlog_format = ROW: avoids non-deterministic SQL issues
  • Set slaves to read_only=ON to prevent mistakes
  • Per-channel monitoring: track lag per channel when using multi-source replication
  • Consider failover automation with a VIP or ProxySQL
  • Combine with a backup strategy: run mysqldump or Xtrabackup on a slave

Wrapping Up

MySQL replication isn’t simple data copying — it’s a tool for
balancing availability, consistency, and performance. Choose the
appropriate method among asynchronous, semi-synchronous, and Group
Replication based on the service’s needs, and combine GTID with
automatic-failover tools for more stable operation.
