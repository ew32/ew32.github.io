An Introduction to Spring Batch - Job/Step/ItemReader/Processor/Writer

Posted on 2026-04-08 Edited on 2026-05-22 In Backend , Java
The core concepts of Spring Batch for large-scale batch processing - chunk-oriented processing, restart/skip/retry, and frequently used practical patterns.

The Problem Spring Batch Solves

In scenarios that need to reliably process large amounts of data in
bulk — nightly payment settlement, recalculating member tiers,
synchronizing with external systems, generating reports — the key
challenges are restartability, transaction management, failure
handling, and monitoring. People start with a simple scheduler +
business logic combination, and as the data grows and failure
handling gets complex, everyone ends up rebuilding similar
infrastructure.

Spring Batch is a framework that standardizes these common
requirements. It stores Job metadata in a DB so you can restart
from where you stopped, and lets you declaratively configure
chunk-level transactions, retry, and skip.

The Core Domain Model

  • Job — one unit of batch work. Composed of multiple Steps
  • Step — one stage of a Job. Operates in chunk or tasklet mode
  • ItemReader — reads one item at a time from a data source
  • ItemProcessor — processes, transforms, or filters one item
  • ItemWriter — outputs processed data in bulk
  • JobRepository — stores execution metadata (start time, end status, parameters)
  • JobLauncher — the Job runner

Chunk-Oriented Processing

Spring Batch’s representative processing model.

If you set the chunk size to 100, it reads and processes 100 items,
writes them all at once, and commits the transaction. If it
fails midway, only that chunk is rolled back, and on restart it
continues from after the last successful chunk.

Failure Handling: Skip, Retry, Restart

  • Skip: skip an item that threw a specific exception and continue
  • Retry: retry on a transient failure
  • Restart: resume from after the failed chunk on the next run

You must use JobParameters well for restart to be possible.
Running with the same parameters means an already-completed Job
won’t run again, and a failed Job will continue on re-running
with the same parameters. So for work that must run anew each time,
distinguish it with a parameter like --job.runDate=2026-04-08.

How to Run a Job

  • CommandLineJobRunner / Spring Boot auto-run: runs automatically at boot
  • Calling JobLauncher directly: triggered by a scheduler or API
  • Integration with external schedulers like Quartz, Airflow

Performance Optimization Tips

  • Chunk size: too small means transaction overhead; too large means memory/rollback burden. Tune typically between 100 and 1000.
  • For a paging Reader, matching page size = chunk size is efficient.
  • Multi-threaded Step: split a single Step’s processing across multiple threads.
  • Partitioning: split the data and run multiple Steps in parallel. Most effective at large scale.
  • Bulk Insert: JdbcBatchItemWriter uses PreparedStatement batching for fast writes.

Monitoring and Operations

  • Query the Spring Batch metadata tables (BATCH_JOB_INSTANCE, BATCH_JOB_EXECUTION, etc.) directly
  • Expose metrics with Spring Boot Actuator + Micrometer
  • Use Spring Batch Admin (separate) or build your own dashboard
  • Send failure alerts from a JobExecutionListener via Slack/email

Wrapping Up

Spring Batch is a framework that provides reliability and
operability rather than flashy new features. If a batch built with
a simple scheduler has started to wobble as the data grows, that’s
when it’s time to adopt Spring Batch. Chunk-oriented processing,
metadata-based restart, and standardized failure handling are assets
hard to implement yourself.
