Session vs Cookie vs JWT

Posted on 2024-07-18 Edited on 2026-06-15 In Backend , Security , Authentication
A comparison of the authentication structure, security characteristics, and use scenarios of sessions, cookies, and JWT, with selection criteria.

Explaining Sessions and Cookies, and Comparing with JWT

Session and Cookie

Session

  • Concept: a method that manages authentication by storing a server-generated session ID in the client’s cookie.
  • How it works:
    • When the user logs in, the server generates a unique session ID.
    • This session ID is stored along with user info in the server’s session store.
    • The server stores this session ID in the client’s cookie.
    • On every subsequent request, the client sends this cookie.
    • The server verifies the user by checking the session ID in the session store.
  • Security:
    • Session cookies are mainly managed server-side, so HTTPS is used to encrypt communication and prevent session hijacking.
    • Since the server manages the session, things like session expiration can be set on the server.
    • It can be vulnerable to XSS attacks, so security is strengthened with the HttpOnly, Secure, and SameSite attributes.

Cookie

  • Concept: a small file used to store data on the client side, used to maintain state between server and client.
  • How it works:
    • The server sends a cookie to the client via the Set-Cookie header.
    • The client stores this cookie and sends it on every subsequent request.
    • The server checks the cookie and processes the needed information.
  • Security:
    • Cookies are stored on the client side, so they can be vulnerable to XSS attacks.
    • The Secure attribute can make them sent only over HTTPS connections.
    • The HttpOnly attribute can prevent access from JavaScript.
    • The SameSite attribute can prevent CSRF attacks.

What Is JWT (JSON Web Token)?

  • Concept: JSON Web Token (JWT) is an open standard (RFC 7519) for securely transmitting information between parties as a JSON object.
  • Components:
    • Header: contains the token type (JWT) and the hashing algorithm.
    • Payload: contains the information (claims) to carry in the token.
    • Signature: a signature to verify the token’s integrity.
  • How it works:
    • When the client requests authentication, the server generates a JWT based on the client’s info and responds with it.
    • The client stores the JWT received from the server and includes it in the HTTP header on every subsequent request.
    • The server verifies the JWT’s signature and authenticates the client based on the payload info.

JWT’s Advantages and Security Concerns

Advantages of JWT

1. Stateless

  • Reduced server load: JWT stores no state on the server, so the server doesn’t need to maintain user state to process each request. This reduces server load and improves scalability.
  • Scalability: no session synchronization between servers is needed, making horizontal scaling easy.

2. Flexibility

  • Various use cases: JWT can be used by various clients — web, mobile, IoT — and is widely used in auth protocols like OAuth 2.0.
  • Self-contained: JWT includes the needed information in the payload, so the server can handle authentication with the token alone, without an extra lookup.

3. Security

  • Signature and encryption: a JWT can prevent tampering through its signature and can be encrypted if needed for secure transmission.
  • Short-lived use: setting the token’s validity period to be short strengthens security.

4. Standardization

  • Wide support: JWT follows the RFC 7519 standard and is supported by many languages and frameworks, enabling consistent authentication across various environments.

5. Transmission Efficiency

  • Lightweight: JWT is JSON-formatted text, kept small via Base64 URL encoding, and easily sent in an HTTP header.

6. Easy to Debug and Test

  • Human-readable format: a JWT is Base64 URL-encoded JSON, so it can be decoded and read by people, making debugging and testing easy.
  • Online tools: various online tools can decode and verify JWTs.

7. Client-Side Storage

  • Flexible storage options: a JWT can be stored in client-side storage (localStorage, sessionStorage) or a cookie, fitting various situations.
  • Session persistence: even if the browser is closed and reopened, the session can be maintained via a JWT in local storage.

Summary

The advantages of JWT, summarized as a table:

Advantage Description
Stateless No server state needed, reduced server load, improved scalability
Flexibility Usable across various clients and protocols, self-contained info
Security Prevents tampering via signature and encryption, can be short-lived
Standardization RFC 7519 standard, supported by many languages and frameworks
Transmission efficiency Lightweight, easily sent via HTTP headers
Easy to debug and test Human-readable format, supported by various online tools
Client-side storage Can be stored in local storage or cookies, session persists across browser restarts

JWT Security Concerns and Countermeasures

1. Interception in Transit (MITM Attack)

  • Concern: the token can be intercepted in transit.
  • Countermeasure: always use HTTPS to encrypt transmitted data, protecting the token from interception in transit.

2. Token Storage Attacks

  • Concern: a token stored on the client can be accessed by malicious scripts.
  • Countermeasures:
    • Use HttpOnly cookies: set the HttpOnly attribute to prevent client-side JavaScript from accessing the cookie, mitigating XSS.
    • Use Secure cookies: set the Secure attribute so cookies are sent only over HTTPS.
    • Use SameSite cookies: set the SameSite attribute to prevent CSRF attacks.
    • Be careful with local storage: local storage is vulnerable to XSS, so use cookies where possible and apply XSS defenses.

3. Token Reuse After Theft

  • Concern: if a token is stolen, an attacker can act as the authenticated user.
  • Countermeasures:
    • Set a short validity period: keep the token’s validity short and reissue new tokens via automatic renewal.
    • Token invalidation: implement a way to invalidate stolen tokens by maintaining a server-side blacklist.

4. Token Tampering

  • Concern: an attacker can tamper with the token to escalate privileges.
  • Countermeasures:
    • Signature verification: the server that receives the token must always verify the signature to check for tampering.
    • Use a strong secret key: manage the signing secret key safely and use a sufficiently complex, unpredictable value.

5. Payload Information Leakage

  • Concern: the payload is only Base64 URL-encoded, so anyone can decode it.
  • Countermeasures:
    • Don’t include sensitive info: don’t put sensitive information in the JWT payload.
    • Encryption: if needed, use JWE (JSON Web Encryption) to encrypt the token itself.

Summary

JWT’s security vulnerabilities and countermeasures, summarized:

Security Concern Countermeasure
Interception in transit Use HTTPS
Token storage attacks Use HttpOnly, Secure, SameSite cookies; apply XSS defenses
Token reuse after theft Set a short validity period, implement a token-invalidation system
Token tampering Verify the signature, use a strong secret key
Payload information leakage Don’t include sensitive info; encrypt the token with JWE if needed

Comparing Sessions and Cookies with JWT

Session vs JWT

Item Session JWT
State Server stores the session state Stateless; the server doesn’t store state
Scalability Needs session synchronization between servers No synchronization needed, easy to scale
Storage location Server-side session store Client side (local storage, cookie)
Security Needs HTTPS to prevent session hijacking Prevents tampering via signature and encryption
Validity period Server can set session expiration A short validity period can be set

Cookie vs JWT

Item Cookie JWT
Storage location Client side (cookie) Client side (local storage, cookie)
Data contained Mainly simple data like a session ID Can contain various info in the payload
Security Can be vulnerable to XSS, CSRF Prevents tampering via signature and encryption
Validity period Cookie expiration can be set A short validity period can be set
Transmission efficiency Has a size limit Lightweight, easily sent via HTTP headers

Through these comparisons, you can understand the pros and cons of
sessions, cookies, and JWT, and choose the appropriate
authentication method for your needs.
