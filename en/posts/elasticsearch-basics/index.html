Elasticsearch Basics - From the Inverted Index to the Query DSL

Posted on 2026-04-15 Edited on 2026-07-20 In Backend , Database
The core concepts of Elasticsearch - inverted indexes, mappings, analyzers, and Query DSL - with indexing and search examples to understand how a search engine works.

What Is Elasticsearch?

Elasticsearch is a distributed search and analytics engine built on
Apache Lucene. It stores JSON documents and performs full-text search
and real-time aggregations very fast — used widely for log analytics,
search services, monitoring, and more. It’s also the core component of
the ELK stack (Elasticsearch + Logstash + Kibana).

Core Concepts

Cluster, Node, Shard

  • Cluster: a collection of nodes
  • Node: a single Elasticsearch instance
  • Index: a collection of documents with similar characteristics (think of it as the RDB’s database or table)
  • Document: a unit of data in JSON (RDB row)
  • Shard: a physical split of an index — enables distribution and parallelism
  • Replica: a copy of a shard — improves availability and search throughput

The Inverted Index

Elasticsearch’s fast search comes from the inverted index. A
typical database stores data row by row, but the inverted index
records “which word appears in which documents” up front.

Example: assume these two documents.

  • Doc1: “Elasticsearch is fast”
  • Doc2: “Elasticsearch is distributed”

The inverted index would look like:

Term Documents
elasticsearch Doc1, Doc2
is Doc1, Doc2
fast Doc1
distributed Doc2

Thanks to this structure, searching for “fast” returns Doc1 instantly.

Mapping

A mapping defines the structure and field types of a document.
Automatic mapping is possible, but explicit mapping is strongly
recommended in production.

text vs keyword

  • text: tokenized through an analyzer. For full-text search
  • keyword: stored verbatim, no analysis. For sorting, aggregation, exact match

To use the same field for both purposes, configure a multi-field.

Analyzers

text fields are processed by an analyzer at index time. An analyzer
has three stages.

  1. Character Filter: preprocesses the raw text (strip HTML, etc.)
  2. Tokenizer: splits into tokens
  3. Token Filter: transforms tokens (lowercase, stopwords, stemming, etc.)

For Korean, use analyzers like Nori or eunjeon for morphological
analysis.

Query DSL

Elasticsearch’s power comes from the Query DSL. Using JSON, you
can write very expressive queries.

match query (full-text)

term query (exact match)

range query

bool query (combining)

  • must: must match, affects score
  • filter: must match, no effect on score (cacheable)
  • must_not: exclude if it matches
  • should: bonus if it matches

Aggregations

Aggregations support per-group statistics, time series, histograms,
and more — and they’re the foundation of Kibana dashboards.

Operational Considerations

  • Designing shard count: too many is overhead, too few limits scaling (10–50GB per shard is a common guideline)
  • Index Lifecycle (ILM): automate hot-warm-cold-delete for time-series data
  • Refresh interval: default 1s; raising it during bulk indexing improves throughput
  • Bulk API: prefer _bulk over per-document indexing

Wrapping Up

Elasticsearch isn’t just a data store — it’s a distributed engine
designed for search and analytics. Once you understand the inverted
index, mapping, and analyzers, and can wield the Query DSL freely, you
can deliver powerful search experiences and real-time analytics with
relative ease.
