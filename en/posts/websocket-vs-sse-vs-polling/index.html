WebSocket vs SSE vs Polling: Comparing Real-Time Transport Techniques

Posted on 2026-06-24 Edited on 2026-07-20 In Network , HTTP
A practical comparison of the operating principles and selection criteria of polling, long polling, SSE, and WebSocket.

Real-Time Communication — What to Build It With

There are really only four ways to create “real-time” on the web:
Polling, Long Polling, Server-Sent Events (SSE), and WebSocket. Each
emerged at a different time and works differently, and which one you
pick changes both your infrastructure cost and the user experience.

How the Four Approaches Work

1. Polling

The client sends a request at a fixed interval.

It’s the simplest, but requests keep going out even with no events.
The average latency is half the polling interval.

2. Long Polling

The server holds the response until a change occurs.

Latency is near zero, but the server must hold many open
connections in memory.

3. SSE

Over HTTP, the server unilaterally streams messages. The client
only listens, with EventSource.

Features like automatic reconnection and recovering missed events
via Last-Event-ID are included in the standard.

4. WebSocket

After an HTTP handshake, it upgrades the same TCP connection into a
full-duplex channel.

It’s the most powerful — bidirectional, low-latency, binary
support — but there are also many operational items you must handle
yourself.

A Side-by-Side Comparison

Item Polling Long Polling SSE WebSocket
Direction C → S C ↔ S (repeated) S → C C ↔ S
Protocol HTTP HTTP HTTP WS over TCP
Latency Interval-dependent Very low Very low The lowest
Binary Possible Possible Not possible Possible
Auto-reconnect Natural Natural Built in Implement yourself
Proxy compatibility Very good Very good Good (watch buffering) Needs an upgrade
Server load Low (inefficient) Medium Low Low
Authentication Header per request Header per request Cookie/header Once at handshake
Implementation difficulty The easiest Easy Easy Medium

Recommendations by Case

  • Notification badges, progress display, stock quotes, log live view → SSE.
  • Chat, collaboration tools, games, whiteboards → WebSocket.
  • State changes infrequently and external system integration must be simple → Polling.
  • AI token streaming → SSE is effectively the standard.
  • Mobile push → none of the above, but the OS push channel (FCM/APNs).

Operational Considerations

Horizontal Scaling and Message Routing

With both WebSocket and SSE, a connection is bound to a specific
instance. To broadcast a message to all instances, you need a
message bus like Redis Pub/Sub, NATS, or Kafka.

Heartbeat and Idle Termination

Most load balancers have an idle timeout of around 60 seconds. Keep
the connection alive with ping/pong for WebSocket, and with periodic
comment lines (: ping\n\n) for SSE.

Authentication

WebSocket checks headers only at handshake time. It’s common to put
a token in the query (?token=...) or to complete authentication
in the first message. For SSE, cookie-based authentication is the
simplest.

CORS and Proxies

SSE/Polling follow normal HTTP CORS rules. WebSocket is handled by
Origin header verification rather than CORS, and when placed behind
an Nginx proxy you must add settings to pass through the Upgrade
and Connection headers.

A Decision Tree

  1. Do you definitely need bidirectional communication? → Yes: WebSocket, No: next step.
  2. Do you need server-to-client push? → Yes: SSE, No: next step.
  3. Do events occur often and is latency important? → Yes: Long Polling, No: Polling.

Wrapping Up

The myth that “if it’s real-time, it must be WebSocket” no longer
holds. SSE fits one-way notifications better, and Polling fits
simple lookups better in many cases. Organize your requirements
first, and review starting from the simplest approach.
