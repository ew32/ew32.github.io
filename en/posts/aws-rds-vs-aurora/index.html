AWS RDS vs Aurora - What to Choose and When

Posted on 2026-07-02 Edited on 2026-06-15 In Cloud , Infrastructure
A comparison of AWS's managed relational database services - RDS and Aurora - across architecture, performance, cost, and operations, plus selection criteria.

Where the Two Services Sit

Amazon RDS is a managed service that hosts open-source and commercial
engines — MySQL, PostgreSQL, MariaDB, Oracle, SQL Server — in the cloud as
they are. Amazon Aurora is offered within RDS, but it is a separate engine
that uses a cloud-native storage engine AWS designed itself. It is
compatible with MySQL and PostgreSQL.

Put simply, RDS is a “managed traditional database” and Aurora is a
“cloud-native database compatible with MySQL/PostgreSQL.”

Architectural Differences

RDS

  • A DB instance runs on a single EBS volume.
  • Multi-AZ holds a standby instance based on synchronous replication. On failure, DNS fails over to the standby.
  • Read replicas use asynchronous replication (the engine’s built-in mechanism).

Aurora

  • The storage layer is separated from compute, storing data on distributed storage replicated six ways (3 AZs × 2 copies).
  • One cluster has a single writer and up to 15 reader instances. All of them share the same storage.
  • Failover is usually within 30 seconds — very fast, via reader promotion.
  • Storage auto-scales based on usage (up to 128 TB).

Performance

  • Aurora advertises up to 5x the throughput of MySQL and roughly 3x that of PostgreSQL at the same instance class. The real-world effect varies by workload, but it often has the edge in OLTP, replication, and failover performance.
  • Aurora readers see the same storage as the writer, so replication lag is usually within tens of milliseconds.
  • Aurora Serverless v2 supports per-second auto-scaling in ACU units.

Cost

  • The per-hour instance cost is roughly 20–30% higher for Aurora than for RDS.
  • In exchange, Aurora bills only for the storage you use rather than requiring you to pre-provision capacity. RDS bills based on allocated capacity.
  • Turning on the I/O Optimized option removes Aurora’s I/O charges but raises instance and storage unit prices, so it can be cheaper for I/O-heavy workloads.
  • For small scale with steady traffic, RDS can be simpler and cheaper.

Feature Comparison

Item RDS (MySQL/PG) Aurora
Engine compatibility Identical MySQL/PostgreSQL compatible
Maximum storage 64 TB 128 TB
Number of read replicas Up to 15 (depends on engine) Up to 15
Replication lag Asynchronous, seconds Shared storage, tens of ms
Failover 60–120 seconds Usually within 30 seconds
Backup Automatic snapshots Automatic + continuous restore, strong PITR
Global replication Cross-Region Read Replica Aurora Global Database (sub-second RPO)
Serverless RDS Serverless not supported Aurora Serverless v2 supported

Selection Guide

When RDS Is a Good Fit

  • When you must use an engine Aurora doesn’t support, such as Oracle or SQL Server.
  • An environment with small or predictable traffic where cost optimization matters.
  • Workloads where a single instance is enough and there is no complex HA requirement.

When Aurora Is a Good Fit

  • Production services that need high availability and fast failover.
  • When read traffic is heavy and you need many readers.
  • When you need global multi-region or serverless auto-scaling.
  • New services for which MySQL/PostgreSQL compatibility is sufficient.

Operational Tips

  • Use Performance Insights to visualize load at the query level.
  • Use Enhanced Monitoring to collect OS metrics at one-second granularity.
  • Manage parameter groups and option groups as IaC (Terraform, etc.) to keep a history.
  • Run regular restore drills for backups, snapshots, and PITR.

Wrapping Up

RDS and Aurora both share the broad framing of a “managed RDBMS,” but they
differ significantly in storage, replication, and the serverless area.
Choose by weighing the workload’s availability requirements alongside the
cost model, and if needed, review the migration path from RDS to Aurora
(snapshot conversion or DMS) in advance.
