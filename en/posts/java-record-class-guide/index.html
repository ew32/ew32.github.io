A Complete Guide to Java Record Classes

Posted on 2026-04-01 Edited on 2026-07-20 In Backend , Java
The syntax of records (preview in Java 14, standard in 16), auto-generated members, DTO/VO patterns, and limitations.

What Are Record Classes?

Previewed in Java 14 and standardized in Java 16, record is a new
type for declaring immutable data-carrying classes very
concisely. Previously, building a DTO or VO meant writing fields,
constructor, getters, equals, hashCode, and toString by hand
(or relying on Lombok), but records have the language generate
these automatically.

A common scene in classic Java:

With a record, it’s a single line.

What’s Auto-Generated

Declaring a record makes the compiler generate:

  • private final fields for each component declared in the header
  • A canonical constructor taking all components
  • Accessor methods with the same name as each component (x(), y() — note: not getX())
  • Component-based equals, hashCode, toString

Every record implicitly extends java.lang.Record, is implicitly
final, and can’t extend other classes (implementing interfaces
is fine).

Compact Constructors and Validation

Validation or normalization happens in the compact constructor.
Omit the parameter list — the body is inserted at the start of the
canonical constructor.

Reassigning a parameter like value = ... stores the new value into
the final field. This is the idiomatic way to normalize in records.

Static Factories, Methods, Nested Types

Records can also have static factories, instance methods, and nested
types.

Use in DTOs, VOs, and API Responses

Records shine for passing data between layers. They dramatically
cut boilerplate for Spring request/response objects, JPA
projections, MapStruct results, and so on.

They suit JPA DTO projections perfectly.

Synergy with Pattern Matching

From Java 21, records combine with record deconstruction patterns
for powerful branching code.

Limits and Caveats

  • No inheritance: records are implicitly final and can’t extend other records or classes. They can only implement interfaces.
  • No extra fields: instance fields are limited to the header components. Static fields are fine.
  • Not suitable as JPA entities: entities need a no-arg constructor and mutable fields — at odds with records. Keep entities as regular classes; use records for DTOs/projections.
  • Not deeply immutable: if a component is a mutable collection, it can be mutated externally. If needed, defensive-copy with List.copyOf in the compact constructor.

Wrapping Up

Records make Java more expressive. Beyond reducing code, they make
“this object is a value” obvious in the code itself. Apply them
aggressively wherever you need immutable value objects — DTOs,
VOs, event payloads, pattern-matching targets.
