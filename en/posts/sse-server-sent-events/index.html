Fully Understanding Server-Sent Events (SSE)

Posted on 2026-06-19 Edited on 2026-07-20 In Network , HTTP
How Server-Sent Events work, how to implement them, and the differences from WebSocket, with practical examples.

What Is SSE?

Server-Sent Events (SSE) is an HTML5 standard for streaming a
one-way flow from server to client. Unlike WebSocket, it operates
over plain HTTP without a separate protocol upgrade, and the browser
can easily consume it with the EventSource API.

The core idea is simple. The server keeps the response open and
keeps flowing text/event-stream-formatted messages, and the
browser parses them line by line and converts them into events.

The Message Format

The rules:

  • Each event is separated by a blank line.
  • event: is the event type; omitting it defaults to message.
  • data: is the body (splitting it across lines joins them with newlines).
  • id: is the event identifier. If the connection drops and reconnects, the browser automatically sends it via the Last-Event-ID header.
  • retry: is the reconnection wait time (ms).

Client Implementation

EventSource automatically retries when the connection drops and
sends the last received id via the Last-Event-ID header. The
server can use this value to resend missed events.

Server Implementation (Node.js / Express)

The key is (1) flush the headers first, (2) res.write whenever an
event occurs, (3) clean up the listener when the client disconnects.

Caution: Proxies and Buffering

Putting SSE behind a reverse proxy like Nginx can buffer the
response, so messages aren’t delivered immediately.

If you go through a CDN, likewise exclude the SSE path from caching
and buffering.

SSE vs WebSocket vs Long Polling

Item SSE WebSocket Long Polling
Direction Server → client, one-way Bidirectional Bidirectional (repeated)
Protocol HTTP A separate protocol over TCP HTTP
Auto-reconnect Built in Implement yourself Natural
Binary Not supported Supported Text-centric
Proxy compatibility Very good Needs an upgrade Good
Implementation difficulty The lowest Medium Low

Suitable Use Cases

  • One-way real-time feeds: stock quotes, sports scores, notifications, progress bars.
  • Streaming AI responses: SSE is effectively the standard for token-by-token output.
  • A live view of server logs.
  • Progress of long-running work: build logs, data-processing progress.

Conversely, if you need bidirectional communication like chat,
games, or collaboration tools, WebSocket is the fit.

Operations Tips

  • Memory usage per connection is low, but there’s a limit to the number of concurrent connections on one instance, so consider horizontal scaling and sticky sessions.
  • Send an empty comment line (: ping\n\n) every 30–60 seconds to prevent idle connections from being cut.
  • Handle authentication with cookies or headers, just like regular HTTP.

Wrapping Up

SSE is “the simplest way to implement real-time over HTTP.” If you
don’t need bidirectional communication, think of SSE before the more
complex WebSocket.
