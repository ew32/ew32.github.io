AWS IAM Policy and Role Best Practices

Posted on 2026-07-01 Edited on 2026-06-05 In Cloud , Infrastructure
Best practices for IAM policies, roles, and credential management to run an AWS account securely while honoring the principle of least privilege.

The Basic IAM Model

IAM defines, as a JSON policy, who (Principal) can do what (Action)
on which resource (Resource) and under what conditions (Condition).
During evaluation, an explicit Deny always overrides an Allow, and if there
is no Allow at all, the request is denied by default.

Core Principles

1. Don’t Use the Root Account

Register MFA on the root account and lock it away. Do day-to-day work as
an IAM user or an SSO user.

2. Principle of Least Privilege

Don’t sprinkle * wildcards everywhere — grant only the actions and
resources actually needed. The “policy generation” feature of IAM Access
Analyzer can pick out only the permissions that were actually used, based
on CloudTrail logs.

3. Separate People from Workloads

  • People: federate SAML/OIDC through AWS IAM Identity Center (formerly SSO).
  • Workloads (EC2/ECS/Lambda): attach an IAM Role directly to the instance/task/function.
  • GitHub Actions and external systems: use OIDC federation to get temporary credentials without issuing access keys.

4. Avoid Static Access Keys

Long-lived access keys carry a high leak risk and a heavy rotation
management cost. Where possible, use STS temporary credentials, IAM Roles,
and IAM Identity Center.

Types of Policies

Type Purpose
Managed policy A reusable, standard bundle of permissions
Inline policy Attached to a single identity; hard to manage separately
Resource-based policy Attached to the resource itself — S3 buckets, KMS keys, SQS queues, etc.
Permissions Boundary The upper limit on permissions a delegated admin can create
SCP (Service Control Policy) An organization-wide guardrail at the Organizations level

Common Safety Patterns

Enforce MFA

Block Outside a Specific Region

Tag-Based Authorization

Visibility and Auditing of Permissions

  • IAM Access Analyzer: automatically identifies externally exposed resources and unused permissions.
  • CloudTrail: an audit log of every API call.
  • Credential Report: a CSV download showing the last-used date of users and access keys.
  • AWS Config: tracks IAM change history and evaluates compliance rules.

Operations Checklist

  • Root credentials secured and MFA enabled.
  • A user password policy, with 90-day access key rotation (when unavoidable).
  • All human users sign in once through IAM Identity Center.
  • Workloads use IAM Roles + OIDC instead of access keys.
  • Apply multi-account guardrails with permissions boundaries and SCPs.

Wrapping Up

IAM is more than just granting permissions — it is the core that
determines your organization’s security posture. Building on the three
pillars of least privilege, temporary credentials, and visibility tools,
and reviewing permissions regularly, can sharply reduce the chance of an
incident.
