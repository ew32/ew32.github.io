A Practical Guide to Tree Shaking and Bundle Optimization

Posted on 2026-06-21 Edited on 2026-06-15 In Frontend
How to slim down a frontend bundle — the principle of tree shaking, the sideEffects flag, bundle analysis, and chunk splitting.

What Is Tree Shaking?

Tree shaking is a static-analysis technique that “cuts unused code
out of the bundle.” The name comes from the image of dead leaves
falling when you shake a tree. webpack 2 introduced it, and now
almost every bundler — Vite, esbuild, Rollup, Parcel — supports it.

The core premise is static imports of ES Modules. With
CommonJS’s require(), what gets imported is decided at runtime, so
static analysis is impossible.

1. Prefer the ESM Build

When picking a library, check whether its package.json has a
"module" or "exports" field. Notably, ESM-friendly packages are
good — e.g. lodash-es instead of lodash, plus date-fns,
nanoid, and zod.

2. Set the sideEffects Flag Accurately

In package.json you can tell the bundler which modules have no
side effects.

If there are files with side effects, like CSS imports, specify
them as an array.

Without this flag, the bundler includes every module “just in
case.”

3. Use Only Named Imports

Also, a pattern where a library bundles all features under a single
default export defeats tree shaking. Write code centered on named
exports where possible.

4. Check Your Babel/TypeScript Settings

If you set module to CommonJS, TypeScript converts ESM to CJS
and tree shaking effectively disappears.

5. Analyze the Bundle

If you don’t know what’s inside, you can’t shrink it.

The most frequently encountered problems are:

  • moment.js + all locales (~250KB)
  • all of lodash (~70KB)
  • large SVGs/PNGs inlined as base64
  • both the ESM and CJS versions of the same library included at once

6. Code Splitting

A technique paired with lazy loading. When it meets a dynamic
import, the bundler automatically splits off a chunk.

In Vite you can finely control vendor chunk splitting with
build.rollupOptions.output.manualChunks.

7. Compression and Polyfills

  • Turning on Brotli compression on the server saves an extra 15–25% over gzip.
  • Don’t apply core-js/regenerator-runtime indiscriminately; specify only your actual target browsers in browserslist.
  • Using the module/nomodule pattern to build modern ESM and legacy chunks together delivers only lightweight code to modern browsers.

Metrics

Metric Target
Initial JS transfer size (gzip) < 150KB
Total JS parse/execute time < 300ms (on a mid-range mobile device)
Average route chunk < 50KB
LCP < 2.5s

A Practical Checklist

  • Declare "sideEffects" in package.json
  • Consider alternatives for CommonJS-only libraries
  • Remove import * as and default-only export patterns
  • Split off routes and heavy modules with dynamic import
  • Periodically check with a bundle analyzer
  • Apply Brotli/HTTP2

Wrapping Up

Bundle optimization isn’t a one-time task. The habit of asking “does
this package tree-shake well?” every time you add a dependency is
what ultimately determines page speed.
