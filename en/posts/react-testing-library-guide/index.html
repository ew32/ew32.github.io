Testing Components with React Testing Library

Posted on 2026-06-07 Edited on 2026-05-22 In Frontend , JavaScript
The philosophy of React Testing Library (RTL), commonly used queries, and patterns for testing user interactions.

The Philosophy of React Testing Library

React Testing Library (RTL) follows the principle that “tests should
resemble the way a user uses the component.” It checks behavior
based on visible text, role, and labels — not internal
implementation (state, props shape, etc.).

The advantage of this approach is resilience to refactoring. Even if
you change the component’s internal structure, the test still passes
as long as the user’s perspective is the same.

Basic Usage

Query Priority

RTL recommends queries in this order.

  1. getByRole — most recommended. Based on the accessibility tree
  2. getByLabelText — form elements
  3. getByPlaceholderText
  4. getByText — text in content areas
  5. getByDisplayValue — a form’s current value
  6. getByAltText, getByTitle
  7. getByTestId — the last resort

getBy* errors if not found, queryBy* returns null, and findBy*
waits asynchronously.

Handling Async

findBy* waits up to 1 second by default and uses waitFor
internally.

User Interaction

@testing-library/user-event simulates events close to real
browser behavior.

API Mocking

Using msw (Mock Service Worker) intercepts the network layer for
consistent tests.

Anti-Patterns

  • Using class-based selectors via container.querySelector('.my-class') — coupled to implementation
  • Directly inspecting a component’s internal state — RTL deliberately doesn’t provide this
  • Calling act() manually too often — userEvent and findBy* handle it automatically

Wrapping Up

React Testing Library steers you toward verifying components from
the user’s point of view, producing highly maintainable tests.
Learning three patterns — getByRole-centric queries, simulating
interaction with userEvent, and mocking the network with MSW —
lets you write robust frontend tests for most cases.
