A Complete Guide to OAuth 2.0 Authorization Flows

Posted on 2026-05-12 Edited on 2026-07-20 In Backend , Authentication
The four standard OAuth 2.0 authorization flows and the OIDC and PKCE extensions, with criteria for choosing the right one.

What Is OAuth 2.0?

OAuth 2.0 is a standard protocol that lets a user grant
delegated access to specific resources (email, profile, files,
etc.) without exposing their password directly to a third-party
application. Specified in RFC 6749, nearly every SaaS platform today
adopts it.

OAuth 2.0’s core idea is simple: “Don’t share the password — issue a
permission-limited Access Token.”

Key Roles

Role Description
Resource Owner The owner of the protected resource (usually the end user)
Client The application trying to access the resource
Authorization Server The server that issues tokens after authentication
Resource Server The API server hosting the protected resources

The order in which these four parties exchange messages defines an
OAuth 2.0 Flow.

Authorization Code Flow

The most standard and secure flow, used by server-side web
applications.

Because there’s an extra step of exchanging the code for a token,
the token isn’t exposed in the browser URL.

The PKCE Extension

PKCE (Proof Key for Code Exchange) is an extension defined in
RFC 7636 for environments — like mobile/SPA — that can’t securely
store a client_secret.

By sending code_challenge in the authorize step and verifying
code_verifier during the token exchange, it blocks the
Authorization Code Interception attack. The current OAuth 2.1
draft recommends PKCE as mandatory for all public clients.

Client Credentials Flow

Used for server-to-server (M2M) communication without user
involvement.

Suited to cron jobs, backend-to-backend calls, and issuing CI/CD
tokens.

Device Authorization Flow

A flow for environments where keyboard input is hard, like TVs and
IoT devices. The device shows a user_code on screen, and the
user enters that code on another device (a smartphone) to complete
authentication. Commonly seen with Google TV and AWS CLI login.

The Implicit Flow Is No Longer Recommended

The Implicit Flow once used by SPAs is insecure because the Access
Token is exposed directly in the URL fragment. The IETF now
recommends replacing it with Authorization Code + PKCE.

OpenID Connect (OIDC)

If OAuth 2.0 is an authorization protocol, OIDC is an
authentication layer placed on top of it. The key difference is
the addition of a JWT called id_token, which conveys the user’s
identity information as standard claims (sub, iss, aud, exp).

Best Practices

  • Choose Authorization Code + PKCE by default. It applies to SPA, Mobile, and Server alike.
  • Always perform CSRF defense with the state parameter.
  • Strictly validate redirect_uri against an allowlist and avoid wildcards.
  • Keep Access Tokens short (5–30 minutes), and manage Refresh Tokens with a rotation policy.
  • Storing tokens in localStorage exposes them to XSS. In an SPA, consider HttpOnly cookies + the BFF pattern.
  • Request scope narrowly, following the principle of least privilege.

Wrapping Up

OAuth 2.0 isn’t just a login tool — it’s the standard language of
a protocol for permission delegation. Rather than memorizing flow
names, understand the threat model each flow addresses to make the
right choice. If you design a new service today, in nearly every
case the combination of Authorization Code + PKCE + OIDC will be
the starting point.
