Web Font Optimization: font-display, preload, and subsetting

Posted on 2026-06-14 Edited on 2026-07-20 In Frontend
Core optimization techniques — font-display, preload, subsetting, and more — for reducing the rendering delay and layout shift caused by web fonts.

How Web Fonts Affect Performance

Web fonts define a site’s identity, but used wrong they cause the
following problems.

  • FOIT (Flash of Invisible Text) — text is invisible while the font loads
  • FOUT (Flash of Unstyled Text) — briefly shown in a system font, then swapped
  • CLS (Cumulative Layout Shift) — the layout wobbles as line height/width change on font swap
  • LCP (Largest Contentful Paint) delay — if the large text depends on a font, LCP is delayed

The font-display Strategy

Specify font-display inside @font-face to control behavior
during font loading.

Value Behavior
auto Browser default (usually similar to block)
block Up to 3s invisible → font appears
swap Show fallback immediately → swap when the font loads
fallback 100ms invisible → fallback → swap if it arrives within 3s
optional 100ms invisible → if not received, just keep the fallback

For body text, swap is generally recommended. However, if FOUT
causes a large visual change, consider optional or fallback
matching (next section).

Raising Priority with preload

Preload the first-displayed font with <link rel="preload">.

If you omit crossorigin, the font downloads twice, so always
include it. Fonts are normally discovered only after the CSS is
parsed, so preload is very effective.

WOFF2 and Format Specification

All modern browsers support WOFF2. WOFF2 is about 30% smaller than
WOFF, so it’s the format to prioritize.

Add woff and ttf only when you genuinely need to support old
browsers.

Subsetting

Korean fonts have many glyphs, so the files are large. Keeping only
the glyphs you use shrinks them greatly.

  • Split by Unicode range
  • Include only the 2,350 commonly used KS X 1001 Korean syllables
  • Make a separate subset of just Latin/digits/symbols

The browser fetches a file only when a glyph in that range appears
on the page.

Variable Fonts

A variable font that merges multiple weights/slants into one file
is far more efficient than fetching 6–9 weights separately.

The font-weight: 100 900 range specification is the mark of a
variable font.

Reducing Layout Shift

With size-adjust, ascent-override, and descent-override, you
can match the fallback font’s metrics close to the web font,
nearly eliminating wobble on swap.

Next.js’s next/font automates this work, bringing CLS close to 0.

Checklist

  • Use WOFF2
  • Register only the weights/styles you need
  • Apply font-display: swap (or optional)
  • Use <link rel="preload" ... crossorigin> for above-the-fold fonts
  • For Korean, subset or split by unicode-range
  • Mitigate CLS by adjusting fallback metrics
  • Use a variable font when possible

Wrapping Up

Web font optimization directly affects core metrics like LCP, CLS,
and FOIT. Just applying preload, subsetting, font-display, and
fallback metric matching well noticeably improves perceived load
speed and visual stability. It’s one of the most effective
investments for making a fast site without harming the design.
