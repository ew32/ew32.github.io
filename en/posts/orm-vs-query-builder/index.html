ORM vs Query Builder vs Raw SQL

Posted on 2026-05-11 Edited on 2026-07-20 In Backend , Database
The pros and cons and selection criteria of the three ways to build a data-access layer - ORM, Query Builder, and Raw SQL.

Three Branches of the Data-Access Layer

There are broadly three ways an application talks to the database.

  • Raw SQL: run hand-written SQL as-is
  • Query Builder: assemble SQL in a programming language
  • ORM (Object-Relational Mapping): work with the DB through an object graph

Each has a different level of abstraction, which shifts the balance
of productivity, performance, and control.

Raw SQL

The lowest abstraction. JDBC, MyBatis, and jdbcTemplate fall here.

MyBatis keeps SQL in XML/annotations and automates result mapping.

Pros

  • You can see and tune the SQL directly
  • The fastest execution path
  • Freedom to use DB-specific features

Cons

  • Lots of boilerplate
  • Lacks type safety
  • Hard to refactor (renaming columns, etc.)

Query Builder

Assembles SQL in code but doesn’t map to an object graph. Examples
include jOOQ (Java), QueryDSL (Java), Knex (Node.js),
and SQLAlchemy Core (Python).

Pros

  • Compile-time column/type checking
  • A structure close to 1:1 with SQL, so it’s predictable
  • Convenient for dynamic queries (conditional where)
  • Refactor-friendly

Cons

  • The learning curve can be steeper than an ORM
  • Some mapping logic is still needed

ORM (Object-Relational Mapping)

Maps the object model to tables and automatically generates SQL
through the object graph. Examples: Hibernate/JPA, TypeORM,
Django ORM, ActiveRecord.

Pros

  • Top productivity: simple CRUD is almost free
  • Object-graph traversal feels natural
  • Rich features — first-level cache, dirty checking, cascade
  • Relatively independent of the DB type

Cons

  • Auto-generated SQL can be inefficient
  • Many pitfalls, like the N+1 problem
  • Hard to debug without understanding the internals
  • Insufficient for complex aggregation or window functions

Comparing the Three

Item Raw SQL Query Builder ORM
Abstraction level Low Medium High
Productivity Low Medium-high High
Performance control Very good Good Average
Type safety Weak Strong Strong
Learning curve Low Medium High
Fit for complex queries Very good Good Average

When to Use Which

  • Simple CRUD-centric domain: ORM
  • Complex search/reporting, dynamic queries: Query Builder
  • Large batches, complex analytics, high performance needs: Raw SQL
  • Mixed: it’s common to use ORM + Query Builder + Native Query together in one project

Best Practices

  • Even when using an ORM, always check the generated SQL (show-sql, p6spy).
  • Consciously choose the fetch strategy (LAZY/EAGER).
  • Understand how the first- and second-level caches behave.
  • Use batch insert/update or raw SQL for bulk processing.
  • Don’t bury business logic deep in SQL — clearly split responsibility between code and SQL.

Wrapping Up

ORM, Query Builder, and Raw SQL aren’t competitors — they’re
tools for different layers. Enjoy the ORM’s productivity but
know its limits, and where needed, drop down to a Query Builder or
Native SQL for direct control. A good backend developer is someone
who can judge “which tool fits this query now” rather than “which
tool is right.”
