RESTful API Design Principles

Posted on 2026-05-03 Edited on 2026-06-05 In Backend
The core constraints of REST and practical, applicable API design best practices.

What Is REST?

REST (Representational State Transfer) is an architectural style
for distributed systems first presented in Roy Fielding’s 2000
doctoral dissertation. Designed to make good use of HTTP’s
characteristics, it is the most widely used API design approach
today.

The Six Constraints of REST

  1. Client-Server: separation of concerns
  2. Stateless: each request is independent; the server doesn’t store client state
  3. Cacheable: responses must declare whether they are cacheable
  4. Layered system: the client doesn’t perceive layers other than the directly connected server
  5. Uniform interface: resource identification, manipulation through representations, self-descriptive messages, HATEOAS
  6. Code on demand (optional): the server can send executable code to the client

HTTP Methods and Their Meanings

Method Purpose Idempotent Safe
GET Retrieve a resource Yes Yes
POST Create a resource No No
PUT Replace a resource entirely Yes No
PATCH Partially update a resource No No
DELETE Delete a resource Yes No

Idempotent: sending the same request multiple times yields the
same result
Safe: doesn’t change the resource’s state

Using HTTP Status Codes

Success (2xx)

  • 200 OK: general success
  • 201 Created: created successfully (POST)
  • 204 No Content: succeeded but no response body (DELETE)

Client Errors (4xx)

  • 400 Bad Request: a malformed request
  • 401 Unauthorized: authentication required
  • 403 Forbidden: no permission
  • 404 Not Found: resource doesn’t exist
  • 409 Conflict: a state conflict
  • 422 Unprocessable Entity: validation failed

Server Errors (5xx)

  • 500 Internal Server Error: a general server error
  • 503 Service Unavailable: the service is temporarily down

Pagination

  • Offset-based: simple to implement, but rows can be duplicated/missed when data changes
  • Cursor-based: suited to large-scale/real-time data

Versioning Strategies

  1. URI versioning: /v1/users (the clearest)
  2. Header versioning: Accept: application/vnd.api.v1+json
  3. Query parameter: /users?version=1

Security Recommendations

  • Use HTTPS without exception
  • Use standard authentication methods (OAuth 2.0, JWT)
  • Apply rate limiting
  • Set a clear CORS policy
  • Don’t expose sensitive information in the URI

Wrapping Up

A RESTful API is not just a set of rules — it’s a design
philosophy that makes good use of HTTP’s characteristics. You
don’t have to follow every principle 100%, but a consistent design
greatly reduces the API users’ learning cost.
