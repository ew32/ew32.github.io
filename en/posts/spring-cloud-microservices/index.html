Microservices with Spring Cloud - Eureka, Config Server, Gateway

Posted on 2026-04-09 Edited on 2026-05-26 In Backend , Java
A practical look at the core Spring Cloud components - Eureka service discovery, Config Server central configuration, and Spring Cloud Gateway routing.

Why Spring Cloud?

Moving to a microservices architecture turns everything that was
natural in a monolith into an infrastructure problem.

  • How does a service learn another service’s address? → Service discovery
  • How do you consistently manage config values (DB passwords, external API keys)? → Central configuration management
  • How do external clients call dozens of services? → API Gateway
  • How do you keep failures from propagating? → Circuit breaker

Spring Cloud standardizes these distributed-system patterns on top
of Spring Boot. Let’s look at the core modules.

Service Discovery: Eureka

Netflix Eureka is one of the most widely used discovery servers.
Each service instance registers itself at startup, and other
services look it up by service name.

The Eureka Server

The Eureka Client

This alone registers it as order-service, and other services can
call http://order-service/.... Adding @LoadBalanced to a
RestTemplate or WebClient automatically picks one of the
instances to call.

Central Configuration: Config Server

When dozens of services have different configs per environment
(dev/staging/prod), managing them all as separate files becomes
operational hell. Spring Cloud Config Server uses a Git repository
as its backend and serves configuration centrally.

The Config Server

In the Git repo you place files like order-service.yml and
order-service-prod.yml.

The Client

When the service starts, it automatically fetches
order-service-prod.yml. With the @RefreshScope +
/actuator/refresh combination you can refresh config without a
restart, or apply it to all instances at once with Spring Cloud Bus.

API Gateway: Spring Cloud Gateway

The single entry point for external traffic. It handles routing,
authentication, rate limiting, and load balancing.

lb://order-service finds instances from Eureka and load-balances.
You can also define routes with the Java DSL.

The Gateway is WebFlux-based, so it’s non-blocking and handles
high concurrency.

Circuit Breaker: Resilience4j

A pattern for preventing failure propagation. Spring Cloud Circuit
Breaker abstracts multiple implementations (Resilience4j, Sentinel,
etc.).

Configuration defines the failure-rate threshold, sliding-window
size, the half-open transition time, and so on.

Distributed Tracing: Micrometer Tracing

Since Spring Boot 3, Sleuth has been replaced by Micrometer
Tracing. Request IDs (traceId, spanId) are automatically included in
logs and can be exported to Zipkin, Tempo, Jaeger, and others. It’s
effectively essential for microservices operations.

An Example Architecture

Considerations When Adopting

  • Increased operational complexity: the infrastructure services themselves — discovery, Config, Gateway — must also be run with HA.
  • In Kubernetes environments, discovery is often replaced by K8s Services and Config by ConfigMaps/Secrets. The Spring Cloud Kubernetes module fills that role.
  • Local development: actively use Spring Boot devtools, profile separation, and Testcontainers.
  • Version matrix: always check that the Spring Cloud release train (e.g., 2024.0.x) is compatible with your Spring Boot version.

Wrapping Up

Spring Cloud provides standard components so you don’t have to build
microservices infrastructure yourself. Rather than adopting
everything from the start, it’s safer to adopt gradually in the
order service discovery → Config Server → Gateway → circuit
breaker → tracing. Considering Kubernetes-friendly configurations
together can also reduce the operational burden.
