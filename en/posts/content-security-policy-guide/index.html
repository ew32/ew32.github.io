A Guide to Writing Content Security Policy (CSP) Headers

Posted on 2026-05-01 Edited on 2026-06-15 In Security , Cryptography
The main directives of CSP - the last line of XSS defense - the strict-dynamic pattern with nonce/hash, and how to adopt incrementally with report-only mode.

Why CSP

XSS is still one of the most frequent web vulnerabilities. If input
validation and output encoding are the first line of defense, CSP
(Content Security Policy) is the last safety net. Even if the code has
an XSS bug, a strong CSP prevents an attacker from executing arbitrary
scripts.

CSP is delivered through an HTTP response header that restricts the
origins from which the browser may load and execute resources on the
page.

Key Directives

Directive Meaning
default-src Fallback for other directives
script-src JavaScript sources
style-src CSS sources
img-src Image sources
connect-src XHR, fetch, WebSocket
font-src Fonts
frame-src iframe embed targets
frame-ancestors Who is allowed to embed this page
object-src <object>, <embed>
base-uri The <base> tag
form-action Form action targets

A Common Weak Configuration

The following settings won’t stop XSS.

The moment you allow unsafe-inline, inline scripts can execute, which
destroys the XSS defense.

Strict CSP with Nonces

The modern recommended pattern is a strict CSP combining a nonce with
strict-dynamic.

  • Generate a new nonce on every request
  • Attach the nonce attribute only to trusted script tags
  • strict-dynamic also allows scripts dynamically loaded by a nonced script

Hash-Based Allowlisting

For a static inline script, allow it by SHA hash.

The hash value appears in the browser console’s violation message, or
you can compute it yourself.

Trusted Types

To shut down DOM XSS at the source, adopt Trusted Types.

Under this setting, element.innerHTML = userInput throws. Developers
can assign only values that have been validated by a policy function.

Adopting Gradually with Report-Only

Applying a strict CSP to an existing site overnight will break it.
First run in Report-Only mode and collect violations.

In this mode, violations are only reported — nothing is blocked — so
you can refine the policy without impacting production.

Common Pitfalls

  • Third-party scripts: Google Analytics, ad SDKs, and the like use inline scripts or eval, which conflict with strict CSP
  • CDN wildcards: script-src https://cdn.example.com looks fine, but if anyone can upload to that CDN it’s dangerous
  • JSONP: effectively a CSP bypass pattern — review alternatives

Audit Checklist

  • Are unsafe-inline and unsafe-eval removed?
  • Are scripts controlled by nonce or hash?
  • Are object-src 'none' and base-uri 'none' set?
  • Is frame-ancestors set so clickjacking is also defended against?
  • Are violations being collected via Report-To or report-uri?

Wrapping Up

It’s hard to craft a perfect CSP all at once, but Report-Only mode and
gradual hardening let you adopt it safely. The combination of nonce +
strict-dynamic + Trusted Types is currently the most recommended shape.
