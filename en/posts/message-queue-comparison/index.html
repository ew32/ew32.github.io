Comparing Message Queues - Kafka, RabbitMQ, SQS

Posted on 2026-05-23 Edited on 2026-06-05 In Backend
A comparison of the models, pros and cons, and suitable scenarios of three representative message queue/streaming platforms - Kafka, RabbitMQ, and AWS SQS.

Why a Message Queue

In a distributed system, a message queue lowers coupling between
services, absorbs traffic spikes, and enables asynchronous
processing and event-driven architecture. The three representative
solutions — Kafka, RabbitMQ, and AWS SQS — all “exchange messages,”
but differ in model and strengths.

Apache Kafka

A distributed event-streaming platform that started at LinkedIn.
Strictly speaking, it’s closer to a “distributed commit log” than a
“message queue.”

Model

  • A topic is split into partitions
  • Each partition is an append-only log
  • Consumers manage an offset and can read from any position

Characteristics

  • Very high throughput (hundreds of thousands to millions of messages per second)
  • Retains messages, so multiple consumer groups can reprocess the same data
  • Strong ordering guarantees (within a partition)
  • Pairs with the stream-processing ecosystem (Kafka Streams, Flink)

Weaknesses

  • Operationally complex (brokers, ZooKeeper/KRaft, monitoring)
  • An offset-commit model rather than per-message ack — different usage patterns
  • Overkill infrastructure for large messages or a handful of queues

RabbitMQ

A traditional message broker implementing AMQP (Advanced Message
Queuing Protocol).

Model

  • Producer → Exchange → Queue → Consumer
  • Exchange types (direct, fanout, topic, headers) for flexible routing
  • Supports per-message ack/nack
  • Queues are generally emptied once consumed

Characteristics

  • Rich routing patterns (header-based routing, RPC, dead-letter)
  • Naturally expresses task queues, publish-subscribe, RPC, and more
  • Relatively simple operation; clustering is possible
  • Rich queue features — priority queues, TTL, delayed messages

Weaknesses

  • Lower throughput than Kafka (typically tens of thousands to hundreds of thousands msg/s)
  • Not suited to long retention/reprocessing scenarios

AWS SQS

A fully managed message queue service.

Model

  • Standard queue: nearly unlimited throughput, at-least-once delivery, no ordering guarantee
  • FIFO queue: ordering guarantee + exactly-once processing, throughput limit (3000 msg/s per partition)

Characteristics

  • Zero infrastructure operation (serverless)
  • Natural integration with the AWS ecosystem (Lambda, SNS, EventBridge)
  • Auto-scaling, multi-AZ redundancy
  • Pay-as-you-go

Weaknesses

  • Message retention up to 14 days
  • No consumer-group concept (use SNS fan-out to deliver the same message to several consumers)
  • Polling-based (long polling recommended)
  • 256KB message-size limit

Comparison Summary

Item Kafka RabbitMQ SQS
Model Distributed log Message broker Managed queue
Throughput Very high Medium-high Very high (Standard)
Ordering Within a partition Within a queue FIFO only
Retention Configurable (can be permanent) Removed on consumption Up to 14 days
Reprocessing Offset rewind Hard Not possible
Routing Topics/partitions Rich Exchanges A single queue
Operational burden High Medium None (managed)
Pricing model Self-hosted Self-hosted Usage-based

What to Use When

  • Kafka: event sourcing, stream processing, data pipelines, large volumes of logs/metrics, independent processing by many consumer groups
  • RabbitMQ: task queues, complex routing, RPC, priority/delayed messages, passing commands between microservices
  • SQS: low-ops asynchronous queues in an AWS environment, Lambda triggers, traffic buffering

Related Concepts to Consider

  • Outbox pattern: atomicity between a transaction and message publication
  • Idempotent consumer: prepare for at-least-once delivery
  • Dead Letter Queue: isolate messages that fail processing
  • Backpressure: flow control matched to consumer speed

Wrapping Up

The three solutions are less competitors than tools for different
purposes. Answer “do I need streaming/event logs, task
distribution, or a quick start in an AWS environment?” first, and
the choice follows naturally. In large systems, using all three
together is common.
