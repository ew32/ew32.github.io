Understanding and Tuning the Database Connection Pool (HikariCP)

Posted on 2026-05-11 Edited on 2026-06-15 In Backend , Database
Why a connection pool is needed, how HikariCP works, and its key settings and tuning points.

Why Connections Are Expensive

Every time you create a new database connection, the following happens.

  1. TCP 3-way handshake
  2. (With TLS) TLS handshake
  3. Authentication, authorization
  4. Session variable initialization
  5. Memory allocation

If you go through this on every request, response time grows by tens
of ms or more and the DB server takes a heavy hit. A connection
pool removes this cost by reusing pre-built connections.

How a Connection Pool Works

close() doesn’t actually drop the connection — it returns it to the
pool. So you must always return it via try-with-resources or
finally.

What Makes HikariCP Special

HikariCP has been the default connection pool since Spring Boot
2.x.

  • Very lightweight codebase
  • ConcurrentBag data structure minimizes lock contention
  • Fast acquire/return paths
  • Accurate metrics (HikariPool.MBean)

Key Settings

Property Meaning Recommended Default
maximumPoolSize Max connections Sized per CPU/workload
minimumIdle Min idle connections Equal to maximumPoolSize (recommended)
connectionTimeout Max wait for a connection from the pool 30,000 ms
idleTimeout How long to keep idle connections 600,000 ms
maxLifetime Max lifetime of a connection 1,800,000 ms
validationTimeout Connection validation time 5,000 ms
leakDetectionThreshold Leak detection time Around 5,000 ms in prod

Sizing the Pool

The formula from the HikariCP wiki is the most commonly cited.

For most OLTP workloads, 10–30 is plenty. The intuition that
“bigger is better” is a common misconception — pools that are too
large hurt performance because of DB-side context switching and lock
contention.

Why Smaller Pools Are Faster

  • The DB can only process so many things at once (CPU, disk)
  • A big pool builds a wait queue at the DB, raising response times
  • A small pool with fast turnover produces higher throughput

Aligning maxLifetime with DB Timeouts

maxLifetime must be shorter than the DB’s wait_timeout (MySQL) or
the firewall’s idle TCP timeout. Otherwise the pool hands out zombie
connections it doesn’t know are dead.

Leak Detection

Setting leakDetectionThreshold logs connections that aren’t returned
within the given time.

Metric Monitoring

HikariCP integrates with Micrometer and exposes metrics like:

  • hikaricp.connections.active
  • hikaricp.connections.idle
  • hikaricp.connections.pending (threads waiting)
  • hikaricp.connections.usage (distribution of use time)
  • hikaricp.connections.acquire (time to acquire)

If pending is often above 0, you’re short on pool size; if acquire
time is high, suspect DB-side latency.

Common Traps

  • Holding the transaction for too long: don’t put external API calls inside a transaction.
  • Synchronous external calls inside @Transactional: the pool runs out fast.
  • N+1 queries: increase connection hold time.
  • Long-running batch and OLTP sharing one pool: split into separate datasources.
  • Not using the PreparedStatement cache: turn cachePrepStmts=true on.

Multiple DataSources

When you use a read replica, split the pool.

Combining @Transactional(readOnly = true) with a routing datasource
enables read load balancing.

Tuning Procedure in Brief

  1. Start small (10–20)
  2. Watch pending and acquire under load
  3. Monitor DB CPU/IO simultaneously
  4. If the DB is the limit, don’t grow the pool — optimize the DB and queries first
  5. Set maxLifetime below DB/network timeouts

Wrapping Up

A connection pool looks like a simple setting but directly affects
system response time and stability. HikariCP’s defaults are reasonable,
but pool size and lifetime should be tuned to the environment. Rather
than relying on “more pool = faster,” send only what the DB can
actually handle and let the rest queue — that’s the best tuning.
