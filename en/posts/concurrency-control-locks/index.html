Database Concurrency Control and Locks

Posted on 2026-05-10 Edited on 2026-06-05 In Backend , Database
The kinds of locks a database uses to handle concurrent transactions safely, plus MVCC and deadlock handling.

Why Concurrency Control Is Needed

When several users read and write the same data at once, problems like
the following can occur.

  • Lost Update: two transactions modify the same row and one change is lost
  • Inconsistent Read: another transaction modifies data partway through, breaking consistency
  • Phantom Read: a new row appears in the middle of a range query

The goal of concurrency control is to prevent these while maximizing
throughput.

Classifying Locks

Granularity

Unit Description
Row Lock Per row, maximum concurrency
Page Lock Per page (SQL Server, etc.)
Table Lock The whole table
Database Lock The whole DB (special cases like backup)

Lock Modes

  • Shared Lock (S): read lock. Multiple transactions can hold simultaneously
  • Exclusive Lock (X): write lock. Held exclusively
  • Intent Lock (IS/IX): signals at the upper level the intent of a lower-level lock

Compatibility table:

S X
S O X
X X X

Acquiring Locks Directly in SQL

FOR UPDATE blocks other writes against the same row.

MVCC: Lock-Free Reads

MVCC (Multi-Version Concurrency Control) keeps multiple versions of
the data so reads and writes are decoupled.

  • Reads see a snapshot from the moment the transaction began
  • Writes create a new version; reads take no locks
  • “Readers don’t block writers, and writers don’t block readers”

PostgreSQL cleans up old versions with VACUUM, while MySQL InnoDB
holds them in the undo log.

Optimistic vs Pessimistic Locks

Pessimistic Lock

Assumes conflicts are frequent and takes a lock up front.

Optimistic Lock

Assumes conflicts are rare; proceeds without taking a lock and checks
for a conflict at commit time. Typically uses a version column.

If the update affects 0 rows, a conflict occurred.

Item Pessimistic Optimistic
Conflict frequency High Low
Waiting Yes No
Retry Usually unnecessary Required
Suitable workload Frequent write conflicts Read-heavy

Gap Locks and Next-Key Locks (MySQL)

To prevent phantom reads, InnoDB uses gap locks and next-key
locks.

Because gap locks affect concurrency, index design matters.

Deadlocks

A state where each side waits for a resource the other holds and they
wait forever.

  • Transaction A: locks row 1 → requests row 2
  • Transaction B: locks row 2 → requests row 1

Most DBMSes detect the deadlock and forcibly roll back one side.

Ways to Reduce Deadlocks

  • Lock resources in the same order
  • Keep transactions short
  • Design indexes well to reduce the lock range
  • Implement retry logic

Monitoring

  • PostgreSQL: pg_locks, pg_stat_activity
  • MySQL: INFORMATION_SCHEMA.INNODB_TRX, INNODB_LOCKS
  • Track lock-wait time alongside the slow query log

Wrapping Up

Concurrency control is a core feature of an RDBMS, and locks and MVCC
are its two pillars. Choose isolation levels and lock strategies that
match your workload, and understand subtle behaviors like deadlocks and
gap locks to build a stable system. Rather than solving every conflict
with a lock, the most effective prescription is keeping transactions
short and designing indexes well.
