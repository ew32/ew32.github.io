Next.js Data Fetching Strategies - SSR, SSG, ISR, CSR

Posted on 2026-06-09 Edited on 2026-07-20 In Frontend , JavaScript
The four rendering strategies for building pages in Next.js and how to choose among them in the App Router.

Four Rendering Strategies

A major strength of Next.js is choosing a different rendering
strategy per page. There are broadly four.

Strategy Timing Cache Use Cases
SSG (Static) Build time Permanent Marketing, blogs
ISR Build + periodic regeneration TTL Product lists, frequently updated lists
SSR At request time None Personalized pages
CSR Client-side Memory Some dashboard widgets

In the App Router (13+), all of these are unified through fetch
options and dynamic settings.

SSG (Static Site Generation)

Renders once at build time. All users receive the same HTML — the
fastest and most cache-friendly.

ISR (Incremental Static Regeneration)

Regenerates in the background after a set time, even post-build.

Or trigger it externally with on-demand revalidation.

SSR (Server-Side Rendering)

Renders anew on every request. Suited to pages that must differ each
time, like authenticated or personalized data.

cache: "no-store" or dynamic = "force-dynamic" triggers SSR.

CSR (Client-Side Rendering)

In a "use client" component, fetch with useEffect or
SWR/React Query.

Suited to areas with frequent interaction — search, real-time
widgets, pagination.

App Router Cache Layers

The App Router has a four-tier cache.

  1. Request Memoization — deduplicates fetches within the same render cycle
  2. Data Cache — the next.revalidate option of fetch
  3. Full Route Cache — the HTML/RSC of statically rendered pages
  4. Router Cache — the client router’s prefetch cache

Thanks to this structure, calling just revalidateTag clears both
the data and page caches together.

Strategy Selection Criteria

  • Data barely changes → SSG
  • It’s OK to update every few minutes → ISR (revalidate)
  • Must update on external events → ISR (revalidateTag)
  • Must differ per user every time → SSR (no-store)
  • Areas with frequent interaction → client component + SWR

Wrapping Up

Data fetching in Next.js isn’t “which library to use” — it’s “when
to render this page.” Make it as static as possible with SSG/ISR
and split out only the areas that need personalization into SSR/CSR
— that’s favorable for both performance and cost. Understanding the
App Router’s cache layers makes these choices much more natural.
