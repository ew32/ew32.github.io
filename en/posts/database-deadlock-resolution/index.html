Causes of Database Deadlocks and How to Resolve Them

Posted on 2026-04-16 Edited on 2026-05-26 In Backend , Database
How deadlocks arise in a database and the prevention and resolution methods used in practice - lock ordering, timeouts, retries.

What Is a Deadlock?

A deadlock is a state where two or more transactions wait for each
other to release the resources they hold and end up waiting forever.
In databases it arises from a tangle of row locks, gap locks, table
locks, and the like.

Conditions for a Deadlock (Coffman Conditions)

A deadlock can occur only when all four of the following hold at once.

  1. Mutual exclusion: a resource is held by only one transaction at a time
  2. Hold and wait: a transaction holds resources while waiting for others
  3. No preemption: resources can’t be forcibly taken
  4. Circular wait: a cycle exists in the resource-wait graph

Break any one of these and the deadlock can’t happen. In practice the
common strategy is to prevent the circular wait.

Typical Deadlock Scenarios

Different Lock Orders

T1 tried to lock id=1 then id=2, T2 tried id=2 then id=1 — a circular
wait is formed.

Index Gap Lock Deadlock (MySQL)

In InnoDB’s REPEATABLE READ, gap locks add to the picture and cause
unexpected deadlocks.

Detection and Handling

Most DBMSes detect a deadlock automatically and pick the least costly
transaction as the victim to roll back.

Checking MySQL Deadlock Logs

PostgreSQL runs deadlock detection after deadlock_timeout (default
1s) of lock waiting and returns an error when one is found.

Prevention Strategies

1. Unify the Lock Order

The most effective method. Establish a rule that when updating
multiple rows, you always take locks in the same order (e.g., id
ascending).

2. Keep Transactions Short

The longer the transaction, the longer locks are held and the higher
the chance of conflict. Don’t put external API calls or heavy
computation inside a transaction.

3. Proper Index Design

Without indexes, full scans take locks on more rows. Make sure the
critical update queries have indexes.

4. Adjust the Isolation Level

If REPEATABLE READ’s gap locks frequently cause problems, consider
lowering to READ COMMITTED. Just review the business requirements
first.

5. Set Lock Timeouts

A timeout that prevents indefinite waiting is important.

Implementing Retry Logic

Since deadlocks can’t be eliminated, automatic retry in the
application is the standard pattern.

Spring’s @Retryable or tools like Resilience4j work too.

Resolution Checklist

  1. Collect deadlock logs and analyze patterns
  2. Check the execution plan and lock behavior of the involved queries
  3. Standardize lock order
  4. Shrink transaction scope
  5. Reinforce indexes
  6. Introduce retry logic
  7. Adjust isolation level if needed

Wrapping Up

Deadlocks are unavoidable in a multi-transaction environment, but
consistent lock ordering, short transactions, the right indexes, and
retry logic let you manage them effectively. When one happens, the
habit of analyzing logs to find the cause and applying structural
improvements is what produces a stable service.
