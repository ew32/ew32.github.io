An Overview of Secrets Management and HashiCorp Vault

Posted on 2026-05-03 Edited on 2026-06-05 In Backend , Authentication
Why you should separate secrets from code and environment variables, and the core concepts of HashiCorp Vault (secret engine, auth method, lease, dynamic secret).

Why Secrets Management Is Needed

Leaving secrets like DB passwords, API keys, and certificates
directly in .env files or environment variables causes the following
problems.

  • Accidentally committed to a Git repository
  • Baked into a container image and preserved permanently
  • Operators learn the plaintext, making accountability untraceable
  • Rotation requires redeploying every service

A secrets management system centralizes secrets, audits access, and
enables automatic rotation and dynamic issuance. HashiCorp Vault is
the de facto standard in this field.

Core Vault Concepts

Secret Engine

A backend that stores or dynamically generates secrets.

  • kv: key-value storage (static secrets)
  • database: generates/revokes DB credentials on demand
  • pki: issues private certificates
  • transit: send data to Vault for encryption/decryption (the key is never exposed)
  • aws: issues STS temporary credentials

Auth Method

How a client authenticates to Vault.

  • token: the most basic form
  • approle: machine-to-machine authentication (role_id + secret_id)
  • kubernetes: authenticates with a pod’s ServiceAccount token
  • aws, gcp, azure: based on cloud instance metadata
  • oidc/jwt: integration with an external IdP

Policy

ACL policies define which actions are allowed on which paths.

Lease and Revoke

Secrets Vault issues have a lease (rental period). When it expires
they’re auto-revoked, and in an incident you can revoke them all
with a single command.

Static Secret Usage Flow

From the application’s perspective, you call the Vault API instead
of environment variables.

The Power of Dynamic Secrets

Vault’s real advantage is dynamic secrets. For example, it
generates DB credentials at request time and auto-revokes them after
use.

Now when the application reads database/creds/readonly, a new DB
user valid for only one hour is created.

The benefits:

  • Credentials are short-lived, so a leak causes little damage
  • No one uses the same password, so audit tracing is clear
  • Revocation is immediate

Integration with Kubernetes

The standard way to use Vault in Kubernetes is the Vault Agent
Injector.

The sidecar mounts the secret as a file and auto-renews it before
the lease expires. The application code just reads the file.

Seal/Unseal

Vault starts in a sealed state. The master key is split into
multiple shares via Shamir’s Secret Sharing, and a threshold number
must be gathered to unseal. Auto-unseal using AWS KMS, GCP KMS, and
the like is also possible.

Thanks to the sealing mechanism, even if a single server is seized,
the secrets on disk aren’t immediately exposed.

Alternatives and Selection Criteria

  • AWS Secrets Manager / Parameter Store: AWS-only, simple integration
  • GCP Secret Manager: GCP-only
  • Azure Key Vault: Azure-only
  • HashiCorp Vault: multi-cloud, dynamic secrets, transit, and other rich features
  • Doppler, 1Password Secrets Automation: simple SaaS-style solutions

If you’re tied to a single cloud, that cloud’s managed service is
simpler. For multi-cloud, mixed on-premises, or advanced features,
Vault is powerful.

Audit Checklist

  • Are there no plaintext secrets in code and images?
  • Is every secret access recorded in the audit log?
  • Have you switched to dynamic secrets where possible?
  • Do policies follow the principle of least privilege?
  • Are Vault’s own unseal keys and root token stored securely?

Wrapping Up

Secrets can’t travel with code. Separating them out with a
centralized management system like Vault and switching to dynamic
secrets where possible raises both your security level and your
operational efficiency at the same time.
