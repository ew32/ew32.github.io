Mastering the Spring Bean Lifecycle

Posted on 2026-05-08 Edited on 2026-05-22 In Backend , Java
A step-by-step look at the full flow by which the Spring container creates, injects dependencies into, initializes, and destroys beans.

The Spring Container and Beans

The core of Spring is that the IoC container manages the
lifecycle of beans. The moment of object creation, dependency
injection, initialization callbacks, and destruction callbacks are
all under the container’s control. Understanding this accurately
helps prevent subtle bugs and lets you handle resources safely.

The Lifecycle Stages at a Glance

Stage Description
1. Instantiation Create an instance of the bean class
2. Dependency injection DI via constructor/setter/field
3. Aware callbacks BeanNameAware, ApplicationContextAware, etc.
4. BeanPostProcessor pre-processing postProcessBeforeInitialization
5. Initialization callbacks @PostConstruct, InitializingBean, init-method
6. BeanPostProcessor post-processing postProcessAfterInitialization (when AOP proxies are created)
7. Use The bean is used
8. Destruction callbacks @PreDestroy, DisposableBean, destroy-method

Using Initialization Callbacks

The most common way to put code that runs once after a bean has all
its dependencies is @PostConstruct.

The differences between the three approaches:

Approach Characteristics
@PostConstruct A JSR-250 standard, most recommended
InitializingBean.afterPropertiesSet() Depends on the Spring API
@Bean(initMethod = "...") Useful for initializing third-party library beans

Destruction Callbacks

If you must clean up resources at application shutdown, use
@PreDestroy.

@Bean(destroyMethod = "...") specifies an external class’s
shutdown method in the bean definition. If a bean implements
AutoCloseable, destroyMethod inference automatically calls
close().

Scope and Lifecycle

A bean’s scope greatly affects its lifecycle.

  • singleton (default): created once at container start, destroyed at container shutdown
  • prototype: created anew on each request; the container doesn’t call the destruction callback
  • request / session: created/destroyed per web request or session

For prototype beans, the user is responsible for resource cleanup
themselves.

When Dependency Injection Happens

Constructor injection fills dependencies at the moment of instance
creation, but setter and field injection fill them after instance
creation. So you can’t use other beans in the constructor, but you
can safely use them at the @PostConstruct stage.

The Role of BeanPostProcessor

A BeanPostProcessor is a powerful extension point that can
intercept and modify every bean before and after the initialization
callbacks. The proxy that Spring AOP creates is actually generated
in the post-processing stage by a BeanPostProcessor called
AnnotationAwareAspectJAutoProxyCreator.

This is why, inside @PostConstruct, this is sometimes the
original rather than the proxy, so advice isn’t applied on
self-invocation.

Handling Circular Dependencies

Constructor injection fails immediately at container startup if
there’s a circular dependency. Since Spring Boot 2.6 it isn’t
allowed by default; resolve it with the following.

  • Rethink the design to break the dependency
  • Lazy loading via @Lazy
  • Convert to an indirect dependency, e.g., event publishing

Debugging Tips

To see when a bean is created and what post-processing it goes
through, these tools are useful.

  • Print the auto-configuration report with the --debug option
  • Check registered beans with ApplicationContext.getBeanDefinitionNames()
  • Analyze the initialization order with BeanPostProcessor trace logging

Wrapping Up

The Spring bean lifecycle looks simple, but combined with topics
like AOP proxy creation, scope, and circular references, it produces
subtle behavioral differences. Clearly understanding the role of
each stage lets you easily diagnose problems like resource leaks,
missed initialization, and proxies not being applied.
