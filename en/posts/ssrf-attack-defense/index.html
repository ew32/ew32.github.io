SSRF Attacks and Defense: Request Forgery Targeting the Internal Network

Posted on 2026-04-27 Edited on 2026-06-15 In Security , Cryptography
How SSRF works, real-world damage cases like cloud metadata theft, and a robust defense strategy that even blocks DNS Rebinding.

What Is SSRF?

SSRF (Server-Side Request Forgery) is a vulnerability where an
attacker tricks a server-side application into sending an HTTP
request to an arbitrary URL. The common pattern arises when the
server fetches a user-supplied URL as-is.

An attacker can point at an internal metadata endpoint like
?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/
to steal AWS credentials.

A Real-World Damage Case

The 2019 Capital One incident is the canonical example: an SSRF
stole an IAM token from EC2 IMDSv1 and leaked over 100 million
records from S3 buckets. It was shocking proof that a single SSRF
can expose an entire cloud infrastructure.

SSRF attacks aim at varied targets.

  • Cloud metadata (169.254.169.254, the GCP/Azure equivalent endpoints)
  • Internal management APIs (Consul, etcd — e.g. http://localhost:8500/v1/kv/...)
  • Intranet services (http://10.0.0.5/admin)
  • The file protocol (file:///etc/passwd)

Basic Defense — An Insufficient Approach

A commonly attempted defense is to parse the URL and check whether
the host is a private IP.

There are two problems.

  1. TOCTOU: the DNS result at check time may differ from the
    result at request time (DNS Rebinding).
  2. Bypass techniques: decimal IPs (http://2852039166/), IPv6
    mapping, redirect chaining, and more.

A Robust Defense Strategy

You need layered defense.

1. An Allowlist

If possible, explicitly limit the domains you can call. An allowlist
is always safer than a blocklist.

2. Pinning the DNS Result

To stop DNS Rebinding, resolve the IP yourself, then make the
connection to that IP and set only the Host header to the original
hostname.

For Python’s requests, you hook urllib3‘s connection pool.

3. Handling Redirects

If the server responds with a 30x redirect, the client re-requests
the new URL. So you must repeat the same validation on every
redirect.

4. Protecting Metadata

On AWS you should use IMDSv2 (session-token based). IMDSv2 requires
obtaining a token via a PUT request and including the token header
on GET requests, so a simple SSRF can’t easily steal credentials.

5. Network Isolation

The strongest defense is at the network level. Place a dedicated
proxy for outbound calls, and block egress from the application
subnet to internal metadata/management IPs. Enforcing this with
security groups and VPC routing prevents damage even when there’s a
code bug.

A Review Checklist

  • Do you avoid directly fetching user-input URLs?
  • Do you block private/link-local/reserved IPs?
  • Is there a DNS Rebinding countermeasure?
  • Do you re-validate on every redirect?
  • Are you enforcing IMDSv2?
  • Is there an egress policy for outbound calls?

Wrapping Up

SSRF is a powerful vulnerability that can start from a simple URL
parameter and threaten an entire cloud. Only when you apply
code-level validation, network segmentation, and cloud
configuration (IMDSv2) together are you truly safe.
