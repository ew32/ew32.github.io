Transactions and the ACID Principles

Posted on 2026-05-10 Edited on 2026-07-20 In Backend , Database
A summary of what a database transaction means, the four ACID properties, and how behavior differs by isolation level.

What Is a Transaction?

A transaction is a single logical unit of work that changes the
state of a database. Work that’s only meaningful if both parts
succeed — like a transfer’s “withdrawal” and “deposit” — must be
bundled into a transaction.

If it fails partway, you call ROLLBACK to undo all changes.

The Four ACID Properties

Property Meaning
Atomicity All operations in a transaction must all succeed or all fail
Consistency The database must satisfy its defined rules before and after a transaction
Isolation Concurrently running transactions must not interfere with each other
Durability A committed result must be preserved even through a system failure

Atomicity

Atomicity is usually implemented with an undo log and WAL
(Write-Ahead Logging). If a failure occurs during a transaction,
the undo log is used to recover the pre-change state.

Consistency

It guarantees that the database’s constraints (PK, FK, CHECK,
UNIQUE) and the business rules defined by the application are not
violated. Consistency is a shared responsibility of the DBMS and
the application.

Isolation

The most complex property; the strength of the guarantee varies by
isolation level.

Durability

A committed transaction must be written not only to memory but also
to disk. Most RDBMSes complete a commit only after forcing the redo
log to disk with fsync.

Isolation Levels and Possible Phenomena

Isolation level Dirty Read Non-repeatable Read Phantom Read
READ UNCOMMITTED O O O
READ COMMITTED X O O
REPEATABLE READ X X O (X in MySQL)
SERIALIZABLE X X X

Dirty Read

Reading a change that another transaction has not yet committed.

Non-repeatable Read

Running the same query twice and getting different results. You see
another transaction’s UPDATE result.

Phantom Read

Running the same range query twice and getting a different number
of rows.

An Isolation Level Configuration Example

In Spring you can specify it via an attribute of @Transactional.

MVCC and Isolation

PostgreSQL and MySQL InnoDB use MVCC (Multi-Version Concurrency
Control) to separate reads from writes. Each transaction sees a
snapshot matching its point in time, so consistent reads are
possible without locks.

  • PostgreSQL: all isolation levels are MVCC-based
  • MySQL: REPEATABLE READ is the default, and gap locks prevent phantom reads

Transactions in a Distributed Environment

In a microservices environment, there are many cases that can’t be
bundled into a single DB transaction.

  • 2PC (Two-Phase Commit): strong consistency, performance degradation
  • The Saga pattern: eventual consistency via compensating transactions
  • The Outbox pattern: safely publish the message and DB state together

ACID vs BASE

Distributed NoSQL sometimes adopts the BASE (Basically Available,
Soft state, Eventual consistency) model. ACID prioritizes strong
consistency; BASE prioritizes availability and scalability.

Model Consistency Availability Where used
ACID Strong Moderate Finance, payments
BASE Eventual Strong Social, recommendations

Practical Tips

  • Keep the transaction scope short to reduce lock contention.
  • It’s best not to put external API calls inside a transaction.
  • Setting the isolation level to SERIALIZABLE unconditionally causes a large performance hit.
  • Always handle lock timeouts and deadlock retries.

Wrapping Up

ACID isn’t mere theory — it’s the foundation that upholds data
accuracy. Once you understand the meaning of isolation levels, you
can handle concurrency problems and performance trade-offs more
rationally. In a distributed environment, it’s good to also learn
the balance between ACID and BASE, along with alternative patterns
like Saga.
