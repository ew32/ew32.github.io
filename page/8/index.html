Azure App Service: Differences Between Zip Deploy and WEBSITE_RUN_FROM_PACKAGE

Azure App Service를 사용하다 보면 Zip Deploy와 WEBSITE_RUN_FROM_PACKAGE를 서로 대체되는 배포 방식으로 오해하는 경우가 많다. 그러나 두 개념은 서로 다른 역할을 수행하며, 실제로는 함께 사용될 수도 있다.

본 글에서는 Zip Deploy와 WEBSITE_RUN_FROM_PACKAGE의 개념과 차이점, 그리고 권장되는 사용 방식에 대하여 정리하였다.

Zip Deploy란?

Zip Deploy는 ZIP 파일을 App Service에 업로드한 뒤, 서버에서 압축을 해제하여 wwwroot 디렉터리에 배포하는 방식이다.

배포 과정은 다음과 같다.

주요 특징

  • 서버에 실제 파일이 생성된다.
  • 런타임 중 파일 수정이 가능하다.
  • 배포 과정에서 파일 잠금(Lock) 문제가 발생할 수 있다.
  • 삭제된 파일이 서버에 남아 있을 수 있다.
  • 이전 버전의 파일이 잔존하는 문제가 발생할 수 있다.

WEBSITE_RUN_FROM_PACKAGE란?

WEBSITE_RUN_FROM_PACKAGE=1을 설정하면 ZIP 파일을 압축 해제하지 않고, 패키지 자체를 읽기 전용(Read-Only)으로 마운트하여 실행한다.

실행 과정은 다음과 같다.

주요 특징

  • 압축 해제가 수행되지 않는다.
  • 읽기 전용 환경으로 동작한다.
  • 배포 안정성이 높다.
  • 이전 파일이 남지 않는다.
  • 배포 속도가 빠르다.
  • 롤백이 용이하다.

반면, 애플리케이션 실행 중 파일 생성이나 수정이 제한된다는 단점이 존재한다.

두 방식은 서로 대체 관계가 아니다

Zip Deploy와 WEBSITE_RUN_FROM_PACKAGE를 서로 다른 배포 방식으로 이해하는 경우가 많지만, 실제로 두 개념은 담당하는 영역이 다르다.

마무리

Zip Deploy는 ZIP 파일을 업로드하는 배포 방법이며, WEBSITE_RUN_FROM_PACKAGE는 업로드된 패키지를 실행하는 방식을 결정하는 설정이다.

따라서 두 개념은 서로 대체되는 관계가 아니라 상호 보완적인 관계라고 볼 수 있다.

또한 WEBSITE_RUN_FROM_PACKAGE는 필수 설정이 아니며, 필요에 따라 선택적으로 적용할 수 있다. 다만 운영 환경에서는 Zip Deploy + WEBSITE_RUN_FROM_PACKAGE=1 조합이 널리 사용되며, 안정적인 배포를 위한 권장 구성으로 평가받고 있다.

Android 16KB Page Size 대응이란? (Google Play Console 경고 해결)

최근 Google Play Console에서 다음과 같은 경고를 볼 수 있다.

Your app must support 16 KB memory page sizes

처음 보면 생소하지만, 이는 Android 운영체제의 메모리 관리 방식과 관련된 내용이다.

16KB Page Size란?

운영체제는 메모리를 한 번에 관리하지 않고 일정한 크기의 페이지(Page) 단위로 나누어 관리한다.

기존 Android 기기 대부분은 다음과 같이 4KB 페이지를 사용했다.

하지만 최신 ARM64 기반 Android 기기에서는 다음과 같이 16KB 페이지를 사용하는 경우가 등장하고 있다.

즉, 이번 이슈는 운영체제의 메모리 페이징(Page Size) 자체와 관련된 이야기이다.

왜 Google이 16KB 지원을 요구하는가?

16KB 페이지를 사용하면 다음과 같은 장점이 있다.

  • Page Table 크기 감소
  • TLB(Translation Lookaside Buffer) 효율 향상
  • 메모리 관리 비용 감소
  • 성능 및 전력 효율 향상

특히 대용량 메모리를 사용하는 최신 ARM64 기기에서 효과가 크다.

어떤 앱이 영향을 받을까?

주로 네이티브 코드(.so) 를 사용하는 앱이 영향을 받는다.

영향이 큰 경우

  • NDK 사용
  • C/C++ 라이브러리
  • OpenCV
  • FFmpeg
  • SQLCipher
  • TensorFlow Lite
  • Unity
  • Flutter 플러그인
  • React Native Native Module

대부분 영향이 없는 경우

  • 순수 Java 앱
  • 순수 Kotlin 앱
  • Retrofit
  • Room
  • Firebase

Java/Kotlin 코드는 ART(Android Runtime) 위에서 실행되므로 직접적인 영향을 거의 받지 않는다.

.so 파일이 왜 문제가 되는가?

네이티브 라이브러리(.so)는 실행 시 메모리에 직접 매핑(mmap)된다.

예전에는 대부분 4KB 기준으로 만들어졌다.

하지만 16KB 페이지를 사용하는 환경에서는 다음과 같이 16KB 정렬이 필요하다.
