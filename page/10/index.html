2. 논리 백업(Logical Backup)

의미

DB의 “구조와 데이터의 의미”를 저장하는 방식이다.

예시:

복원 시 해당 SQL을 다시 실행하여 DB를 생성한다.

특징

  • 구조(DDL) 저장
  • 데이터 저장
  • 함수(Function) 저장
  • View 저장
  • Index 저장
  • Constraint 저장

장점

  • 다른 서버 이관 가능
  • OS 변경 가능
  • 버전 업그레이드 가능
  • 특정 테이블만 복원 가능
  • 사람이 확인 가능

단점

  • 물리 백업보다 느림
  • 대용량 DB에서는 시간이 오래 걸림

3. 물리 백업(Physical Backup)

의미

PostgreSQL이 사용하는 실제 데이터 파일을 그대로 복사하는 방식이다.

예시 디렉터리:

특징

DB 엔진이 사용하는 내부 바이너리 파일을 그대로 백업한다.

장점

  • 매우 빠름
  • 장애 복구에 적합
  • 대용량 DB에 유리

단점

  • 버전 의존적
  • OS 의존적
  • 부분 복원 어려움
  • 다른 환경으로 이관 어려움

4. 논리 백업 vs 물리 백업

항목 논리 백업 물리 백업
저장 형태 SQL 또는 논리 포맷 DB 파일
사람이 읽기 가능 O X
부분 복원 O X
버전 업그레이드 O 제한적
속도 느림 빠름
장애 복구 보통 매우 우수
서버 이전 우수 제한적

5. 논리 백업과 물리 백업의 비유

논리 백업

집의 설계도 + 자재 목록

집을 다시 지을 수 있다.

물리 백업

집 자체를 통째로 옮기는 것

복구는 빠르지만 다른 환경으로 이동하기 어렵다.

6. 논리 백업 = Raw SQL 인가?

부분적으로 맞다.

예:

이 경우는 실제 SQL이다.

하지만 PostgreSQL은 SQL 외에도 논리 백업 전용 포맷을 제공한다.

예:

파일은 바이너리지만 논리 백업이다.

이유는 내부 내용이 DB 구조와 데이터의 논리적 표현이기 때문이다.

따라서

“논리 백업 = SQL”

은 완전히 맞는 말은 아니다.

정확한 표현은

“논리 백업 = DB를 재구성할 수 있는 논리 표현”

이다.

8. 물리 백업이 빠른데 왜 논리 백업을 사용하는가?

속도보다 유연성이 중요하기 때문이다.

물리 백업의 문제점

1. 버전 종속

예:

  • PostgreSQL 14 → PostgreSQL 15
  • PostgreSQL 15 → PostgreSQL 18

복원 시 문제가 발생할 수 있다.

2. 부분 복원 어려움

특정 테이블만 복원하기 어렵다.

3. 개발 환경 복제 어려움

운영 DB 전체를 복사해야 한다.

4. 데이터 정제 어려움

불필요한 데이터 제거가 어렵다.

논리 백업의 장점

1. 다른 서버 이관

  • Windows → Linux
  • On-Premise → Cloud

가능

2. 버전 업그레이드

가장 많이 사용하는 방식

3. 부분 복원

  • 특정 스키마
  • 특정 테이블

복원 가능

4. 데이터 가공

마스킹 및 샘플링 가능

9. PostgreSQL이 논리 백업 최강이라고 평가받는 이유

1. pg_catalog 기반

PostgreSQL은 시스템 카탈로그에 모든 메타데이터를 저장한다.

예:

  • Table

  • Function
  • Sequence
  • Index

예:

  • Table → Sequence
  • View → Table
  • Function → Type

관계를 분석하여 올바른 순서로 복원한다.

3. 복원 성공률이 높음
